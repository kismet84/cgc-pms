### 模块 B：运营稳定与流程闭环增强（近期 1~2 周）

#### 任务 B1：分包任务与计量显式关联
- **任务名称与简要说明**  
  把当前 `SubMeasure` 对 `SubTask` 的间接关联升级为显式主从关系，减少业务歧义。
- **输入/输出**  
  - 输入：当前 `projectId + contractId + partnerId` 间接关联  
  - 输出：`sub_measure.sub_task_id` + 前后端关联能力
- **涉及的文件建议**  
  - ✨新增 Flyway migration（MySQL/H2）  
  - 🔨修改 `backend/src/main/java/com/cgcpms/subcontract/**`  
  - 🔨修改 `frontend-admin/src/pages/subcontract/measure.vue`
- **复杂度**：P1
- **依赖**：无
- **验收标准**
  - **Given** 已存在分包任务和计量单  
    **When** 用户创建/编辑计量  
    **Then** 可以明确选择关联 `SubTask`
  - **异常场景**
    - 不允许引用其他项目或不匹配的分包任务

#### 任务 B2：出库业务单据化（领料申请）
- **任务名称与简要说明**  
  将当前手工出库升级为“申请 → 审批 → 自动出库”的单据化流程。
- **输入/输出**  
  - 输入：当前 `POST /inventory/stock/out` 手工出库接口  
  - 输出：领料申请单、审批流、库存出库追溯
- **涉及的文件建议**  
  - ✨新增 `material-requisition` 模块（前后端）  
  - 🔨修改 `MatStockService`
- **复杂度**：P1
- **依赖**：B1 无强依赖，可并行
- **验收标准**
  - **Given** 领料申请通过审批  
    **When** 回调执行  
    **Then** 自动调用 `stockOut(sourceType="MAT_REQUISITION")`
  - **异常场景**
    - 库存不足时拒绝出库并留痕

#### 任务 B3：项目自动归档通知
- **任务名称与简要说明**  
  让项目归档从“人工触发”升级到“自动判定 + 通知提醒”。
- **输入/输出**  
  - 输入：结算、付款、合同状态数据  
  - 输出：归档条件检测 + 消息通知
- **涉及的文件建议**  
  - 🔨修改 `settlement` / `project` / `notification` 相关服务
- **复杂度**：P2
- **依赖**：B2
- **验收标准**
  - **Given** 项目满足全部归档条件  
    **When** 结算审批通过后触发检查  
    **Then** 系统通知项目经理可归档

#### 任务 B4：发票强制关联付款记录
- **任务名称与简要说明**  
  把发票管理与实际支付链路完全绑定，提升财务数据一致性。
- **输入/输出**  
  - 输入：当前 `payRecordId` 可选  
  - 输出：发票创建必须关联付款记录
- **涉及的文件建议**  
  - 🔨修改 `invoice` 前后端
- **复杂度**：P2
- **依赖**：无
- **验收标准**
  - **Given** 用户创建发票  
    **When** 未提供 `payRecordId`

## 三、数据流变化

### 1. 当前系统阶段
用户在前端发起业务操作  
→ 调用 `/api` 后端模块  
→ 审批/成本/库存/付款/发票/结算各链路在现有闭环内流转  
→ 驾驶舱、预警、通知消费这些结构化数据

### 2. 后期路线数据演进

#### A. 流程深化
分包任务 / 领料申请 / 归档通知 / 发票强关联  
→ 把当前“业务可跑通”升级为“业务强约束、强追溯”

#### B. 决策增强
预算、付款、库存、进度、资源数据  
→ 驾驶舱 / 报表 / 预测分析  
→ 管理层看到“计划 vs 实际 vs 预测”

#### C. 现场协同
移动端待办 / 扫码 / 拍照附件  
→ 现场数据更实时进入系统  
→ 反过来提升经营分析与审计质量

#### D. 平台化
文档 / 图纸 / 分包协同 / 设备资源 / AI 助手  
→ 从单体流程系统逐步升级为协同型项目管理平台

## 四、影响范围与回归测试建议

### 1. 分包/计量链路
- **受影响模块**：`subcontract`, `cost`, `workflow`
- **回归建议**：创建分包任务 → 创建计量 → 审批 → 成本生成，确认显式关联后旧链路不回退

### 2. 库存/领料链路
- **受影响模块**：`inventory`, `purchase`, `receipt`
- **回归建议**：领料申请通过后自动出库，验证库存扣减、流水追溯、异常库存拦截

### 3. 发票/付款链路
- **受影响模块**：`payment`, `invoice`
- **回归建议**：创建付款记录后再创建发票，验证不允许无 `payRecordId` 发票入账

### 4. 驾驶舱/报表
- **受影响模块**：`dashboard`, `cost`, `payment`, `contract`
- **回归建议**：报表导出数据应与页面列表/图表口径一致，避免“页面和导出不一致”

### 5. 移动端/现场数据采集
- **受影响模块**：`approval`, `inventory`, `receipt`, `file`
- **回归建议**：手机视口下完成待办审批、库存查询、拍照上传，确认布局与接口都正常

### 6. 文档协同/平台化扩展
- **受影响模块**：`file`, `notification`, `workflow`, `project`
- **回归建议**：文档版本、权限、审阅状态必须可追溯，且不能破坏现有附件能力

## 五、冒烟测试方案

1. 从项目创建到合同审批，再到成本联动，确认当前主链路仍完整可跑。  
2. 创建采购 → 验收 → 入库 → 出库，确认库存追溯不受后续路线规划项影响。  
3. 创建分包任务并计量审批，确认未来 `sub_task_id` 设计不会破坏现有计量成本链路。  
4. 创建付款并回写，再创建发票，验证付款/发票主链路仍稳定。  
5. 创建结算并审批，验证归档守卫与通知设计不影响现有结算流程。  
6. 打开驾驶舱，确认 KPI、预警、角色视图继续可用。  
7. 导出一份成本或合同报表（后续实现），确认与页面数据一致。  
8. 在手机视口打开审批待办和库存查询（后续实现），确认最关键操作可完成。

## 附：路线图结论摘要

### 当前阶段定位
CGC-PMS 已完成“工程总包项目管理系统”的**核心业务闭环版**：

- 项目、合同、采购、验收、库存、分包、成本、付款、发票、结算、审批、预警、驾驶舱、通知、组织、系统管理均已具备基础能力
- 近期已经把 **CI、JaCoCo、SQL Safety、Prometheus、备份** 等平台基础风险大幅收敛

### 后期演进原则
后期不要直接跳去“AI 大而全”，应遵循：

1. **先补运营稳定与流程强约束**
2. **再补经营分析与预测能力**
3. **再补移动端与现场协同**
4. **最后补生态协同与平台化扩展**

### 最优先功能包
1. 分包任务-计量显式关联  
2. 领料申请单据化  
3. 项目自动归档通知  
4. 发票强制关联付款记录  
5. 成本/合同/付款报表导出
