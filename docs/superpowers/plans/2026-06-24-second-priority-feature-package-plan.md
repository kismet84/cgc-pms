# 第二优先级任务包实施计划书

> 编写日期：2026-06-24  
> 目标：在“第一优先级：分包任务与计量显式关联”完成后，继续推进一组高价值、低风险、能直接增强业务闭环与经营输出能力的任务包，为 v1.0-rc 与后续版本排期提供执行基线。

---

## 实施边界（执行前先冻结）

1. **Sprint A 推荐实施顺序固定为：A3 → A1 → A2。**  
   先做“发票强制关联付款记录”这种低耦合约束增强，再做“领料申请模块”这种新增业务单据，最后再做依赖多链路状态的“项目自动归档通知”。

2. **领料申请第一期不做库存预占。**  
   本阶段库存仅在审批通过时校验并实际扣减，不引入预占/释放库存机制，避免范围膨胀为完整库存预留系统。

3. **项目自动归档第一期只做“条件判定 + 通知提醒”，不做自动归档动作。**  
   自动归档属于高风险状态切换，本阶段目标是减少人工判断成本，而不是替代人工最终确认。

4. **报表中心第一期默认采用同步导出。**  
   先满足中小规模数据导出需求；若后续出现数据量/性能瓶颈，再演进为异步导出任务机制。

5. **驾驶舱增强与风险前瞻第一期仅做“规则驱动 + 快照对比 + 趋势展示”，不做黑盒预测模型。**  
   优先建立可信、可解释的经营分析能力，避免在数据治理尚未稳定前引入“看起来高级但不可验证”的伪预测功能。

## 一、任务拆解

### 模块 A：流程约束闭环（Sprint A）

#### 任务 A1：出库业务单据化（领料申请模块）
- **任务名称与简要说明**  
  将当前手工出库升级为“申请 → 审批 → 自动出库”的单据化流程，使库存扣减具备业务单据、审批链和来源追溯。
- **输入/输出**  
  - 输入：现有 `POST /inventory/stock/out` 手工出库流程、`MatStockService`、审批引擎  
  - 输出：领料申请主表/明细表、审批流、自动出库逻辑、库存追溯链路
- **涉及的文件建议**  
  - ✨新增 `material_requisition` / `material_requisition_item` migration（MySQL/H2）  
  - ✨新增 `backend/src/main/java/com/cgcpms/requisition/**`  
  - ✨新增 `frontend-admin/src/pages/requisition/**`  
  - 🔨修改 `MatStockService`
- **复杂度**：P0
- **验收标准**
  - **设计边界**
    - 本阶段不做库存预占
    - 仅在审批通过时校验库存并执行实际扣减
  - **正常场景**
    - Given 用户提交领料申请并审批通过  
      When workflow 回调执行  
      Then 自动调用 `stockOut(sourceType="MAT_REQUISITION")` 并生成库存流水
  - **边界条件**
    - Given 领料明细为空或数量为 0  
      When 提交申请  
      Then 系统拒绝并返回明确错误
  - **异常场景**
    - Given 库存不足  
      When 审批通过准备出库  
      Then 出库失败并保留审批/业务追踪信息，不发生静默扣减

#### 任务 A2：项目自动归档通知
- **任务名称与简要说明**  
  让项目归档从当前人工触发升级为“自动判定 + 提醒通知”的半自动闭环。
- **输入/输出**  
  - 输入：合同、付款、结算、归档条件数据  
  - 输出：异步归档条件检测服务 + 站内通知
- **涉及的文件建议**  
  - 🔨修改 `backend/src/main/java/com/cgcpms/project/**`  
  - 🔨修改 `backend/src/main/java/com/cgcpms/settlement/**`  
  - 🔨修改 `backend/src/main/java/com/cgcpms/notification/**`
- **复杂度**：P1
- **依赖**：任务 A1 可并行，但推荐领料申请完成后推进
- **验收标准**
  - **设计边界**
    - 本阶段仅做“可归档通知”
    - 不在后台自动执行 `archive`
  - **正常场景**
    - Given 项目满足归档条件（合同已结算、付款已付清、结算已定案）  
      When 结算审批通过后触发检查  
      Then 通知项目经理“项目可归档”
  - **边界条件**
    - Given 仅满足部分条件  
      When 检查执行  
      Then 不发送误通知
  - **异常场景**
    - Given 检查过程异常  
      When 异步任务失败  
      Then 主业务审批链路不回滚，并有错误日志或重试策略

#### 任务 A3：发票强制关联付款记录
- **任务名称与简要说明**  
  把发票与付款记录从“可选关联”提升为“新数据强制关联”，强化财务一致性与追溯能力。
- **输入/输出**  
  - 输入：当前 `POST /invoices` 可选 `payRecordId` 逻辑  
  - 输出：前后端强约束校验与表单联动
- **涉及的文件建议**  
  - 🔨修改 `backend/src/main/java/com/cgcpms/invoice/**`  
  - 🔨修改 `frontend-admin/src/pages/invoice/**`
- **复杂度**：P1
- **依赖**：无
- **验收标准**
  - **正常场景**
    - Given 用户选择了合法付款记录  
      When 创建发票  
      Then 发票创建成功并保持付款链路追溯
  - **边界条件**
    - Given 历史发票数据未绑定 `payRecordId`  
      When 系统升级后查询历史数据  
      Then 历史数据不回归、不强制补录
  - **异常场景**
    - Given 未提供 `payRecordId` 或提供跨租户/不存在记录  
      When 提交  
      Then 后端明确拒绝

#### Sprint A 顺序说明
- **推荐顺序**：A3 → A1 → A2
- **原因**：
  - A3 改动最小、耦合最低，可最快落地并提升财务链路一致性；
  - A1 是 Sprint A 主体任务，涉及库存、审批、前后端与 migration；
  - A2 依赖合同/付款/结算状态完整性，适合在前两项稳定后再引入。

- **任务名称与简要说明**  
  把当前“页面可看”升级为“数据可导出、可汇报、可归档”的报表能力。
- **输入/输出**

- **复杂度**：P1
- **依赖**：任务 A1~A3 完成后更有业务价值，但技术上可并行
- **验收标准**
  - **设计边界**
    - 本阶段默认采用同步导出
    - 若数据量超阈值，再单独规划异步导出任务体系

      When 导出

- **依赖**：任务 B1
- **验收标准**

## 三、数据流变化

### 1. 领料申请单据化
用户创建领料申请  
→ 提交审批  
→ workflow 审批通过  
→ `MatStockService.stockOut(sourceType="MAT_REQUISITION")` 自动扣减库存  
→ 库存流水可按申请单追溯

### 2. 项目自动归档通知
结算审批通过  
→ 异步检查合同/付款/结算状态  
→ 满足归档条件  
→ 通知项目经理  
→ 用户决定是否执行最终归档

### 3. 发票强绑付款
用户创建发票  
→ 先选择付款记录  
→ 自动回填合同/供应商/金额上下文  
→ 提交发票  
→ 发票与付款形成稳定链路

### 4. 报表与驾驶舱增强
业务数据（成本/付款/库存/合同/项目计划）  
→ 聚合服务  
→ 报表导出 / 图表展示 / 趋势分析  
→ 管理层消费“计划 vs 实际 vs 预测”

### 5. 移动端与现场数据采集
用户在移动端查看待办 / 扫码 / 拍照  
→ 前端上传结构化数据 / 附件  
→ 后端关联业务对象  
→ 数据进入审批、库存、验收、报表链路

## 四、影响范围与回归测试建议

### 1. 库存与采购/验收链路
- **影响范围**：`inventory`, `purchase`, `receipt`
- **回归建议**：从采购到验收到出库完整跑一遍，确认领料申请引入后不会破坏现有库存追溯

### 2. 项目归档闭环
- **影响范围**：`project`, `settlement`, `payment`, `contract`, `notification`
- **回归建议**：构造“满足条件”和“不满足条件”两种项目，验证通知只在正确场景触发

### 3. 发票与付款链路
- **影响范围**：`invoice`, `payment`
- **回归建议**：创建付款后创建发票，验证不允许脱离付款记录独立建票

### 4. 驾驶舱与报表
- **影响范围**：`dashboard`, `cost`, `contract`, `payment`, `inventory`
- **回归建议**：导出报表与页面展示数字逐项核对，防止统计口径不一致

### 5. 移动端与附件链路
- **影响范围**：`approval`, `inventory`, `receipt`, `file`
- **回归建议**：在手机视口下完成审批、库存查询、拍照上传，确认不丢数据且附件关联正确

## 五、冒烟测试方案

1. 新建领料申请并审批通过，预期自动扣减库存且生成来源为 `MAT_REQUISITION` 的流水。  
2. 构造库存不足的领料申请，预期审批后不会扣库且提示明确失败原因。  
3. 结算审批通过后触发项目归档检查，满足条件时预期收到站内通知。  
4. 创建发票时不选择付款记录，预期后端明确拒绝。  
5. 从付款记录创建发票，预期发票成功入账并可反查付款来源。  
6. 导出成本汇总报表，预期下载成功且金额与页面 KPI 一致。  
7. 驾驶舱查看项目进度甘特和资金对比，预期图表正常渲染且无空指针/白屏。  
8. 在手机视口完成审批待办处理，预期同意/驳回可正常提交。  
9. 在移动端拍照上传验收附件，预期文件自动关联到当前验收单。
