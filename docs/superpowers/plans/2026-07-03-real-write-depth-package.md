# 真实写操作深度增强执行包

## 1. 结论先行

- 当前判定：需要增强，但不是从零开始。
- 当前结论：非阻塞，且应保持“小范围补深”，不扩成全量业务重测。
- 依据：
  - 当前仓库已经覆盖部分真实写操作正向闭环。
  - 现有覆盖更偏“成功路径可走通”，对失败场景和回读校验覆盖不均衡。
  - 当前剩余项要求的是“深度增强”，不是“重做主链路”。

## 2. 当前证据

### 2.1 已存在的真实写操作覆盖

- `frontend-admin/e2e/approval.spec.ts`
  - 已通过 API 创建项目、合作方、合同，并提交审批。
  - 已在 UI 中完成“同意 / 驳回”。
  - 已回读 `/api/workflow/instances/{id}` 验证状态与审批记录。
- `frontend-admin/e2e/contract.spec.ts`
  - 已在 UI 中完成合同草稿创建。
  - 已验证成功跳回台账页，且台账基础结构可见。
- `frontend-admin/e2e/invoice.spec.ts`
  - 已覆盖发票创建、登记、核验主链路。
- `frontend-admin/e2e/c3-failure-path.spec.ts`
  - 已覆盖库存出库超额的失败路径。
  - 当前验证目标不是“写成功”，而是“出现业务级失败反馈，而不是系统异常空白”。

### 2.1A 当前已满足的子目标

- 当前已可确认至少存在以下两类基础证据：
  - 真实写链路正向基础：
    - 审批创建 / 提交 / UI 操作 / 后端回读
    - 发票创建 / 列表回查 / 核验状态变化
  - 至少一个显式失败路径：
    - 库存出库超额 -> 出现业务级反馈，且无 `系统异常，请稍后重试`
- 当前未满足的部分因此收敛为：
  - 库存真实写入后的台账回读
  - 合同失败场景或保存后回读增强
  - 付款链路若要补深，应只补缺口，不重做已有发票基础链路

### 2.2 当前覆盖仍偏浅的区域

- `frontend-admin/e2e/payment-invoice.spec.ts`
  - 当前主要验证付款申请列表、发票列表、筛选和弹窗可见。
  - 对付款真实创建 / 提交 / 回读未形成独立深度覆盖。
- `frontend-admin/e2e/inventory.spec.ts`
  - 当前验证库存交易页、出入库切换、台账渲染。
  - 尚未对“确认入库 / 确认出库”形成真实写入及回读校验。
- `frontend-admin/e2e/contract.spec.ts`
  - 当前验证成功草稿保存，但对失败场景与回读细节仍偏少。

### 2.3 当前最像“已达标基线”的区域

- 审批链路当前已经具备：
  - 真实创建
  - 提交审批
  - UI 操作
  - 后端状态回读
- 该链路不应作为下一轮优先补深对象。

## 3. 决策建议

- 本包定位：执行包，不是执行结果。
- 当前最小可行增强方向：
  - 优先补“库存真实写操作”与“合同失败场景 / 回读校验”
  - 次优先补“付款真实创建 / 提交”
- 不建议本轮同时扩所有模块：
  - 当前目标是收口非阻塞剩余项，不是开启一轮全仓 E2E 扩面工程。

## 4. 执行任务

### 任务 1：补库存真实写操作最小闭环

- 目标：让库存不再只有页面展示验证，而有最小真实写入证据。
- 建议文件：
  - `frontend-admin/e2e/inventory.spec.ts`
  - 或新增 `frontend-admin/e2e/inventory-write.spec.ts`
- 最小覆盖：
  - 准备可用仓库 / 物料样本
  - 执行一次真实入库
  - 回到 `/inventory/stock` 或相关台账确认流水出现
  - 如当前业务允许，再补一次真实出库并回读库存变化
- 当前说明：
  - 失败路径已由 `c3-failure-path.spec.ts` 提供基础证据。
  - 这一任务剩余重点是“成功写入 + 台账回读”，不是再补另一个失败提示。

### 任务 2：补合同失败场景与回读

- 目标：避免合同只验证“能保存成功”，不验证失败保护和数据落库。
- 建议文件：
  - `frontend-admin/e2e/contract.spec.ts`
- 最小覆盖：
  - 缺少必填字段时，保存 / 下一步被阻止且提示明确
  - 成功保存草稿后，在台账中能回读到新建合同
  - 如可低成本实现，再补一次详情页核对合同核心字段

### 任务 3：付款链路择一补深

- 目标：让付款不只停留在列表页结构验证。
- 建议方向二选一：
  - 付款申请真实创建后回读状态
  - 发票登记后回读核验状态变化
- 约束：
  - 只选一个最稳路径，不同时扩两条深水链路
- 当前说明：
  - `invoice.spec.ts` 已覆盖发票创建与核验状态变化基础。
  - 若继续补付款侧，优先补“付款申请真实创建后回读状态”，而不是重复扩写发票链路。

## 5. 禁止事项

- 不把本轮任务扩成所有模块的新增、编辑、删除、审批、回退全覆盖。
- 不为了让测试通过而在一个 spec 中堆过多数据准备逻辑。
- 不重复补已经在 `approval.spec.ts` 中具备真实回读闭环的同类正向路径。

## 6. 建议执行角色

- 主执行角色：前端测试工程执行子智能体
- 协同角色：后端验证执行子智能体
- 主负责人职责：
  - 只负责确认优先级、边界和通过 / 不通过结论

## 7. 验收标准

- 必须至少完成以下两类增强中的两项：
  - 库存真实写入 + 台账回读
  - 合同失败场景或保存后回读增强
  - 付款 / 发票链路新增一个真实写后回读验证
- 且必须满足：
  - 新增覆盖属于“深度增强”，不是重复列表页冒烟
  - 至少有一个场景包含失败保护或异常路径验证
  - 至少有一个场景包含写后回读校验
- 当前验收解释：
  - “至少有一个场景包含失败保护或异常路径验证”这一项，现有 `c3-failure-path.spec.ts` 已提供基础满足。
  - 当前剩余验收压力主要集中在“写后回读校验”和“第二条高风险链路补深”。

## 8. 风险点

- 数据风险：库存、付款等链路对基础主数据依赖较强，若样本不足，执行成本会上升。
- 边界风险：若同时扩太多模块，容易把非阻塞收尾做成新一轮大改造。
- 维护风险：若把数据准备和 UI 断言全堆在单一 spec，中后期会变脆。

## 9. 通过 / 不通过判定口径

### 通过

- 当前写操作覆盖已从“主成功路径”提升到“成功 + 失败 / 回读至少各一部分”。
- 至少两个高风险链路的深度被明确增强。
- 未突破当前非阻塞收尾项的最小边界。

### 不通过

- 仍只有列表结构验证，没有新的真实写入回读。
- 只重复已有审批正向路径，未补新的深度空白。
- 把范围扩大成全仓重做，导致收尾项失控。

## 10. 最小产物建议

- 代码产物：
  - 一个或两个 E2E spec 的定向增强
- 验证产物：
  - 执行结果摘要
  - 关键写后回读证据
- 文档回写建议：
  - 通过后可回写 `docs/未来开发计划.md`
  - 若仅完成一部分，需明确写成“已增强，但未完全关闭”
