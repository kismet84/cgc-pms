# 质量加固设计说明

## 1. 目标

将《未来开发计划》第四条“质量加固”从指标清单转为可持续执行的工程体系，覆盖测试覆盖率、代码可维护性和安全加固。实施完成后，质量要求必须由自动化门禁持续验证，而不是依赖一次性人工检查。

## 2. 审查结论与基线修正

截至 2026-06-22，原计划与仓库现状存在以下偏差：

| 领域 | 原计划描述 | 仓库证据 | 设计处理 |
|---|---|---|---|
| 后端覆盖率 | 当前 67.8%，目标 80% | JaCoCo 已接入，但 `backend/pom.xml` 门禁仍为 60% | 重新生成可信基线，按 68%→72%→80%递增门禁 |
| 前端覆盖率 | 154 case，目标 200+ | 现有 37 个单测文件；Vitest 声明 80% 阈值，但未配置 coverage provider 和 coverage 脚本 | 用覆盖率和关键行为双重约束，不以用例数量作为唯一目标 |
| E2E | 36 case，目标 50+ | 现有 14 个 spec、约 45 个 `test()`；CI 未执行 Playwright | 补齐五条主链路并纳入 CI |
| 结算服务拆分 | 待拆分 | 已拆为 QueryService、WriteService、Assembler 和门面 | 转为架构验收与防回退检查，不重复开发 |
| 状态常量 | 常量类已建、引用待迁移 | Handler、QueryService、WriteService 仍有 7 处字面量 | 完成迁移并增加状态流转测试 |
| 前端大文件 | 列出两个超限文件 | 目前有 6 个 Vue 文件超过 800 行，最高 1333 行 | 按风险和变更频率分批拆分，不做全仓重写 |
| SQL 注入审计 | 审查 Mapper XML | 当前 Mapper XML 为 0 | 改为审查注解 SQL、Wrapper、动态表/列名和字符串拼接入口 |
| API 限流 | 审查敏感端点 | `@RateLimit` 仅覆盖登录和刷新令牌 | 建立端点风险清单并覆盖认证、上传、导出及高成本查询 |
| 文件白名单 | 待实现 | 已有扩展名和 50MB 限制，缺少 MIME、文件签名一致性校验 | 在现有 FileService 上增量加固 |
| 操作审计 | 待实现 | 有请求参数脱敏切面，无持久化业务操作审计 | 新增异步、失败隔离、可查询的审计日志 |
| CI | 第五条另列自动化测试 | CI 已存在，但只监听 `main`；当前主分支为 `master`，MySQL 作业允许失败 | 作为质量门禁底座一并修正 |

## 3. 总体方案

采用“基线校准 → 三轨实施 → 门禁递增 → 发布验收”的渐进方案。

三个工作包依次交付、共享统一 CI 门禁；同一时间只推进一个工作包：

1. 测试与质量门禁：建立可信覆盖率基线，补齐关键业务行为测试和五条主链路 E2E，逐步提高阈值。
2. 代码可维护性：迁移结算状态常量，验证结算服务边界，分批拆分超限 Vue 页面。
3. 安全加固：完成 SQL 注入、限流、文件上传和操作审计四项闭环。

每个工作包必须先增加失败测试或审计规则，再实施最小改动，最后由 CI 固化。禁止通过排除核心包、空断言或降低阈值制造“达标”。

## 4. 工作包 A：测试与质量门禁

### 4.1 可信基线

- 后端统一使用 `mvn verify` 生成 JaCoCo XML/HTML，并记录 instruction、branch、line 三类指标。
- 前端安装 Vitest V8 coverage provider，新增 `test:coverage` 脚本，覆盖率范围限定为可执行的 `src/**/*.{ts,vue}`，明确排除类型声明、生成文件和入口装配文件。
- E2E 用 Playwright 实际发现的测试数作为数量基线，并记录成功率、重试次数和总耗时。
- 基线报告作为 CI artifact 保存，避免手工统计用例数造成漂移。

### 4.2 后端覆盖率提升

优先补测高风险而非低价值 getter：审批状态流转、金额计算、库存幂等、付款防超付、租户隔离、权限拒绝、文件安全和审计失败隔离。门禁按 68%、72%、80%三阶段提升；每一阶段只能在实际覆盖率稳定高于阈值后调整 `pom.xml`。

最终门禁至少包含 instruction 80% 和 branch 70%。核心包可设置更高的独立阈值，配置类、纯 DTO 和生成代码允许明确排除，但排除项必须逐项列出理由。

### 4.3 前端单元测试

测试目标从“200+ case”修正为：关键模块行为完整、全局 lines/functions/statements 80%、branches 70%。重点覆盖表单校验、请求失败、权限按钮、空状态、分页筛选、状态映射以及拆分后的 composable。

现有 200+ case 数量目标保留为观测指标，不作为可通过复制低价值断言达成的硬门禁。

### 4.4 E2E 主链路

以独立测试数据和 API fixture 覆盖五条链路：

1. 项目→合同→提交审批→审批通过。
2. 采购→验收→审批通过→自动入库→来源流水可追溯。
3. 分包→计量→审批通过→成本生成且幂等。
4. 付款申请→支付回写→发票关联，并验证超付拦截。
5. 结算→定案→归档守卫，覆盖不满足条件和满足条件两种结果。

测试总数达到 50+，但验收以五条链路均具备正向、关键失败和幂等断言为准。CI 失败保留 trace、截图和 HTML report。

### 4.5 CI 分层

- 快速层：前端类型检查、lint、单测；后端 H2 单测和 JaCoCo check。
- 数据库层：MySQL + Redis 集成测试，移除 `continue-on-error`。
- E2E 层：构建并启动依赖服务，执行五条链路；PR 必跑 Chromium，主分支可增加完整回归。
- 触发分支同时覆盖 `master` 和 `main`，待仓库主分支统一后再删除兼容项。
- 使用 `pnpm install --frozen-lockfile`，上传覆盖率和失败诊断 artifact。

## 5. 工作包 B：代码可维护性

### 5.1 结算模块

- 将 `SettlementWorkflowHandler`、`StlSettlementQueryService`、`StlSettlementWriteService` 中的 `DRAFT`、`FINALIZED` 等状态字面量全部替换为 `SettlementStatusConstants`。
- 为创建默认状态、审批定案、撤回恢复和查询状态映射增加回归测试。
- 增加架构约束：Controller 只能依赖 QueryService/WriteService 或既有门面，禁止业务逻辑重新堆回门面类。

### 5.2 前端大文件拆分

按优先级处理当前超过 800 行的页面：

1. `org/index.vue`、`dashboard/index.vue`：行数最大，先抽取数据加载、图表配置和弹窗状态。
2. `inventory/stock.vue`、`receipt/index.vue`：属于核心库存链路，抽取查询状态、库存交易表单和仓库/物料选项加载。
3. `invoice/index.vue`、`contract/ContractLedgerPage.vue`：抽取筛选器、表格列、对话框和导出逻辑。

拆分单位包括同目录 `components/`、`composables/` 和纯函数模块。页面文件目标不超过 500 行，新组件原则上不超过 300 行；若模板本身合理且拆分会破坏内聚性，可在审查记录中说明例外。每次只拆一个页面，保持 DOM 关键结构、API 调用顺序和用户行为不变。

## 6. 工作包 C：安全加固

### 6.1 SQL 注入审计

审计范围覆盖所有 Mapper 接口、`@Select/@Update/@Delete/@Insert`、MyBatis-Plus `apply/last/having`、动态表名或列名、原生 JDBC 和字符串拼接 SQL。建立可重复执行的静态扫描脚本，禁止未经白名单映射的用户输入进入 SQL 结构位置。

验收要求：值参数全部绑定；排序字段、表名、列名只能由服务端枚举映射；发现的每个危险入口都有负向测试；CI 扫描无未解释命中。

### 6.2 API 限流

先生成端点风险清单，再按用途设置策略：认证端点按 IP/账号限流，上传按用户限流，导出和高成本统计按租户/用户限流。`RateLimitAspect` 的 key 必须支持 IP、用户、租户和端点组合，并在 Redis 不可用时采用明确的降级策略。

验收覆盖正常请求、超过阈值返回 429、不同用户互不影响、窗口恢复、Redis 异常以及代理头可信边界。

### 6.3 文件上传

保留现有扩展名和大小限制，新增：

- 原始文件名规范化及控制字符拦截。
- 服务端检测 MIME，不信任客户端 `Content-Type`。
- PDF、JPEG、PNG、GIF、WebP、Office/ZIP 类文件进行文件签名或容器格式校验。
- 扩展名、声明 MIME、检测 MIME 三者必须匹配允许矩阵。
- 存储时使用检测后的 MIME；拒绝双扩展名伪装、可执行文件和畸形压缩容器。

当前允许的文档、图片、压缩包范围需由业务白名单显式定义；不在白名单中的格式默认拒绝。

### 6.4 持久化操作审计

新增审计注解、审计事件、异步持久化服务、数据库表和只读查询接口。记录租户、用户、操作类型、业务类型、业务 ID、HTTP 方法、路径、结果、错误码、来源 IP、耗时和时间；不记录 Token、Cookie、密码、文件内容或完整请求体。

审计写入与业务事务隔离，审计失败不得回滚业务，但必须产生受控告警。登录/退出、增删改、审批提交/处理、文件上传/下载/删除、权限失败和认证失败必须覆盖。查询接口只对审计权限开放，并按租户隔离。

## 7. 数据流与失败处理

业务请求先完成认证和限流，再进行输入/文件校验和业务授权；业务方法产生结构化审计事件，事件在请求结束后异步落库。测试和静态扫描结果进入 CI，各门禁失败直接阻止合并。

外部依赖故障按以下原则处理：Redis 限流故障采用配置化降级并告警；MinIO 上传失败不写文件记录；审计落库失败不影响业务事务但计入监控；E2E 环境启动失败与业务测试失败分开报告。

## 8. 分期与交付顺序

### 8.1 共享文件变更控制

为避免共享配置和测试基础设施冲突，以下文件实行单一阶段归属：

- 阶段 0 负责建立指标采集方式，但不提高门禁阈值。
- 阶段 1 集中修改 `backend/pom.xml`、`frontend-admin/package.json`、`frontend-admin/vitest.config.ts` 和 `.github/workflows/ci.yml`，建立门禁骨架。
- 阶段 2 负责 `frontend-admin/playwright.config.ts`、E2E fixture 和主链路 spec；如需调整 CI，仅在既有 E2E job 内增量修改。
- 阶段 3 的页面拆分不得修改全局测试配置和 CI，只能增加页面、组件、composable 及其就近测试。
- 阶段 4 的安全加固不得重构测试底座；安全扫描脚本通过独立 CI step 接入。
- 阶段 5 只提高已存在的覆盖率阈值和生成最终报告，不再改变测试发现规则、排除范围或作业拓扑。

每阶段开始前工作区必须无未提交的计划内改动；结束时单独提交并运行该阶段全量验收。后续阶段需要修改前一阶段归属的共享文件时，必须先记录原因并单独提交，禁止与业务实现混在同一提交中。

| 阶段 | 建议周期 | 交付内容 | 退出条件 |
|---|---:|---|---|
| 0 基线校准 | 1 天 | 覆盖率报告、端点清单、SQL 扫描清单、大文件清单 | 指标可在本地和 CI 重复生成 |
| 1 P1 门禁 | 3 天 | 后端 68% 门禁、关键链路补测、CI 分支修正、MySQL 作业转硬门禁 | 快速层和数据库层稳定通过 |
| 2 主链路与常量 | 3 天 | 后端 72%、50+ E2E、结算常量迁移 | 五条链路均在 CI 通过 |
| 3 可维护性 | 5 天 | 六个大页面分批拆分 | 行数目标、单测和视觉回归通过 |
| 4 安全 P2 | 5 天 | SQL、限流、文件上传加固及测试 | 安全负向测试和扫描通过 |
| 5 审计与最终门禁 | 5 天 | 操作审计、后端 80%、前端覆盖率门禁 | 全量质量门禁通过并形成发布报告 |

阶段周期为单人有效开发日估算。所有阶段严格串行：前一阶段未达到退出条件，不启动后一阶段；不得拆分给多个开发流同时推进。

## 9. 最终验收标准

1. 后端 `mvn verify` 全量通过，instruction ≥80%、branch ≥70%，无核心业务包被无理由排除。
2. 前端类型检查、lint、单测通过，lines/functions/statements ≥80%、branches ≥70%。
3. Playwright 50+ case 通过，五条主链路均有正向、失败和关键幂等验证。
4. CI 在 `master/main` 的 push 和 PR 上执行，MySQL 与 E2E 均为硬门禁。
5. 结算状态业务代码无重复字面量，服务拆分边界有测试保护。
6. 六个超限 Vue 页面完成拆分或具备批准的例外记录，用户行为无回归。
7. SQL 审计无未解释高风险命中；动态结构参数均使用服务端白名单。
8. 敏感端点限流测试通过，返回码和降级策略明确。
9. 上传格式通过扩展名、MIME 和文件签名联合校验，伪装文件负向测试通过。
10. 关键操作可按租户、用户、业务对象和时间检索，审计记录不含敏感内容。

## 10. 明确不纳入范围

- 不在本轮引入商业 SAST/DAST 平台；先使用仓库内脚本、测试和现有 CI 实现可重复门禁。
- 不因覆盖率目标重构无关业务模块。
- 不一次性重写全部前端页面，只处理当前超过 800 行的六个文件。
- 不在审计日志中保存完整请求体或响应体。
- 不将第五条运维自动化中的监控、备份和镜像发布整体并入本方案，仅处理质量门禁直接依赖的 CI 配置。
