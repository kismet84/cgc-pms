## 1. 项目背景

CGC-PMS 是面向建筑工程总包项目全过程管理的前后端分离系统，当前已经覆盖项目、合同、成本、采购库存、分包、付款、发票、审批、预警和驾驶舱等核心业务。

当前系统已经形成以下业务闭环：

```text
项目立项
  -> 合同 / 签证 / 采购 / 分包
  -> 成本归集
  -> 付款 / 发票 / 结算
  -> 审批 / 通知 / 预警
  -> 驾驶舱分析
```

本计划书的目标不是推翻现有系统重做，而是在现有架构和业务基础上，按生产级交付标准进行增强，使系统具备真实试点、生产部署、持续迭代和后续行业深化能力。

## 3. 产品定位

CGC-PMS 的产品定位建议定义为：

> 面向建筑工程总包企业的项目经营、合同成本、采购库存、分包结算、付款发票、审批预警、角色驾驶舱一体化管理平台。

当前系统已经接近 **PMS + 轻 ERP** 形态。后续不要把它仅仅做成项目台账，也不要直接扩成通用 ERP。正确方向是：

1. **短期**：把现有业务闭环做成生产级系统。
2. **中期**：补齐报表、预警治理、进度计划、现场管理。
3. **长期**：向建筑行业深水区扩展，包括供应链、财务、移动端、图纸、RFI、Submittal、BIM、AI 辅助。

1. 建立 CI/CD、测试、扫描、发布、回滚门禁。

3. 完成权限、租户、项目数据边界复核。

5. 完成前端列表页、导出、上传、异常态、空态生产化。
6. 完成监控、日志、备份、恢复、部署手册。
7. 完成试点演示数据、角色手册和验收材料。

### 6.2 中期目标：2–6 个月

中期目标是让系统从“业务闭环可用”升级为“经营分析与执行协同可用”：

1. 建设报表中心。
2. 建设规则治理中心。
3. 建设通知订阅和多渠道触达。
4. 建设 WBS、任务、里程碑、甘特图。
5. 建设项目计划 vs 实际、延期预警。
6. 建设供应商评分、采购比价、库存补货建议。

长期目标是形成建筑行业项目管理平台：

1. 建设施工日报、质量安全、照片证据、整改闭环。

6. 建设 AI 合同审查、发票识别、预警解释、项目知识问答。

2. 不借用经营、付款、风险、设计变更或泛审批数据。
3. 技术问题有责任人、状态、截止日期、关闭记录。
4. 图纸版本可追溯。
5. RFI / Submittal 有完整流程。

---

### 9.4 BIM / IFC 与模型问题管理

#### 功能说明

建设长期数字建造能力。

#### 功能范围

1. IFC 文件上传。
2. Web 模型查看。
3. 构件选择。
4. 构件属性查看。
5. BIM issue 标记。
6. issue 分派。
7. issue 截图。
8. issue 关闭。
9. BIM issue 与技术问题 / 质量问题关联。
10. BIM issue 驾驶舱。

#### 实现方法

1. 前端集成 xeokit 或同类 Web IFC 查看器。
2. MinIO 存储 IFC 文件。
3. 后端记录模型元数据。
4. `bim_issue` 绑定项目、模型、构件、截图、责任人、状态。
5. BIM issue 可转技术问题、质量问题或施工问题。

#### 验收标准

1. 可上传并查看 IFC。
2. 可选择构件并查看属性。
3. 可在模型上创建问题。
4. 问题可分派、处理、关闭。
5. BIM issue 能进入预警和驾驶舱。

---

### 9.5 AI 辅助能力

#### 功能说明

在数据闭环稳定后，建设 AI 辅助能力。

#### 功能范围

1. 合同风险摘要。
2. 付款条件识别。
3. 发票识别增强。
4. 预警原因解释。
5. 项目知识问答。
6. 会议纪要提取待办。
7. 技术问题相似案例推荐。
8. 施工日报自动摘要。
9. 供应商履约风险预测。

#### 实现方法

1. 不直接让 AI 改业务数据。
2. AI 输出必须进入“草稿 / 建议 / 待确认”状态。
3. 所有 AI 建议保留来源引用。
4. 关键业务动作仍由人确认。
5. 建立 AI 调用日志、费用日志和错误日志。
6. 对合同、发票、图纸等敏感文件做权限过滤。

#### 验收标准

1. AI 不能绕过权限访问数据。
2. AI 结果有来源引用。
3. AI 建议不能直接改变审批、付款、结算状态。
4. 用户可采纳、修改或驳回 AI 建议。
5. AI 调用可审计。

## 12. 开发原则

1. 不重做已有稳定业务模块。
2. 不通过改前端文案冒充新角色能力。
3. 不在缺少真实数据契约时上线总工程师驾驶舱。
4. 不把角色默认筛选当成权限控制。
5. 不一次性承诺所有预警规则。
6. 不修改已应用 Flyway migration。
7. 不让 AI 直接修改关键业务数据。
8. 不用历史测试数量替代当前验证结果。
9. 所有核心业务必须可追溯、可审计、可回滚。
10. 每个阶段必须有明确验收标准和质量归档。

## 13. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 业务范围继续膨胀 | 进度失控 | 按 P0 / P1 / P2 / P3 分级，冻结阶段边界 |
| 总工程师角色被强行包装 | 数据失真、验收失败 | 技术域未立项前继续冻结 |
| 预警规则过多 | 用户疲劳、误报增加 | 先做 M1，再做 M2 规则治理 |
| 权限只做前端控制 | 严重安全风险 | 后端 `@PreAuthorize` + Service 数据边界校验 |
| 成本口径不一致 | 驾驶舱失真 | 建来源单据追溯和月度快照 |
| 文件上传风险 | 安全漏洞 | 白名单、大小限制、MIME 校验、下载鉴权 |
| 大数据导出拖垮服务 | 性能问题 | 大导出异步任务化 |
| 没有恢复演练 | 备份不可用 | 每月恢复演练 |
| 移动端过早复杂化 | 研发成本过高 | 先 H5 / PWA，后原生 App |
| BIM / AI 过早进入 | 主线被干扰 | 放入长期 P3，待核心业务稳定后推进 |

## 15. 最终建议

CGC-PMS 当前最适合采用“三步走”：

### 第一步：完成生产准入

1. CI/CD。
2. 权限边界。
3. 预警中心 M1。
4. 主链路回归。
5. 运维监控。
6. 备份恢复。
7. 试点手册。

### 第二步：完成经营与执行增强

1. 报表中心。
2. 规则治理。
3. 通知平台。
4. WBS。
5. 甘特图。
6. 项目计划。
7. 供应商评分。
8. 库存补货。

### 第三步：进入建筑行业深水区

1. 现场日报。
2. 质量安全。
3. 移动端。
4. 财务集成。
5. 图纸。
6. RFI。
7. Submittal。
8. 总工程师技术域。
9. BIM。
10. AI。

当前最优先的执行顺序是：

```text
CI/CD 和上线门禁
  -> 预警中心 M1
  -> 权限、租户、项目数据边界复核
  -> 成本、采购、库存、付款、发票、审批主链路回归
  -> 前端列表页和文件上传安全生产化
  -> 监控、备份、运维、试点材料
  -> 报表、规则、通知、进度、现场、财务、技术域、BIM 和 AI
```

本计划不建议短期盲目重构所有页面，也不建议继续包装总工程师驾驶舱。短期的核心不是“看起来功能更多”，而是“每个已上线功能都可用、可信、可追溯、可验收、可生产部署”。
