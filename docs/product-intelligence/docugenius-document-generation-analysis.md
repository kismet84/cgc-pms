# DocuGenius 业务文档生成能力研究

> 状态：Active Research / 非实施授权
> 核验时间：2026-07-16
> 来源类型：DocuGenius 官方飞书知识库及其官方关联说明
> 研究目的：提炼可用于 cgc-pms 业务单据输出的产品机制，不复制官方全文，不把竞品能力直接转换为 Ready

## 一、结论摘要

DocuGenius“排版打印”的核心不是浏览器打印，而是一个轻量业务文档生成平台：它把审批或多维表格数据映射为稳定字段，通过可发布的模板生成 PDF/Office 文档，再进入打印、归档或邮件交付。

对 cgc-pms 最有价值的不是拖拽编辑器外观，而是以下结构：

1. 业务数据与版式分离。
2. 普通字段、循环明细、条件显隐和格式化组成最小模板语义。
3. 模板草稿、发布、停用、默认和版本复用形成治理闭环。
4. 管理员负责模板，业务用户只使用已发布版本。
5. 预览、生成、归档和交付围绕同一数据与模板版本执行。

cgc-pms 不应照搬完整拖拽低代码、全 Office 格式、任意签章或自动邮件。当前更合理的最小动作是先建立可审计 PDF 生成层，用付款申请单和结算单证明跨域复用，再依据真实需求决定是否扩展。

对应正式计划：[`第48条主线-业务单据模板与可审计PDF生成任务计划书.md`](../plans/第48条主线-业务单据模板与可审计PDF生成任务计划书.md)。

## 四、官方能力事实

### 4.1 产品定位

官方资料展示的稳定链路为：

```text
审批或多维表格数据
  → 字段与关联明细
  → 模板设计和数据绑定
  → PDF/Office 文档
  → 打印、下载、归档或交付
```

因此“打印”只是输出动作，主要产品价值在数据到文档的自动转换。

### 4.2 数据源与字段模型

已核验事实：

1. 审批数据区分表单字段和系统字段。
2. 普通字段可以插入文字等组件，并在实际打印时替换为真实数据。
3. 明细字段和审批节点字段属于集合数据，可插入表格形成循环区域。
4. 多维表格示例使用主表、明细表、产品表和客户表，并通过关联/查找引用形成输出数据。
5. 数据字段是否允许进入打印还受上游审批字段配置影响。

产品含义：模板绑定的对象不是数据库列，而是经过上游业务系统允许暴露的数据字段。

### 4.3 模板表达能力

已核验事实：

- 文本、表格、图片等排版组件。
- 普通字段插值和循环明细。
- 基于字段值的组件显示与隐藏。
- 文本、数字、货币、百分比、日期和日期时间格式化。
- 金额财务大写、前缀后缀和空值显示。
- 纸张、页面、页眉页脚、水印、图片和附件相关能力。
- 在线排版与 Office 文件模板两种路径。

其中循环、条件和格式化是业务模板区别于静态 PDF 的关键能力。

### 4.4 模板生命周期

已核验事实：

1. 一个审批流程可以配置多个打印模板。
2. 模板编辑完成后需要发布，业务用户才能选择。
3. 非默认模板可以停用；默认模板需要先切换后才能停用。
4. 停用模板可再次启用。
5. 模板可以导出并导入其他审批流程。
6. 导入后若字段不一致会出现变量错误，需要重新处理字段。

产品含义：模板复用不能只复制版式，还必须验证目标数据契约。

### 4.5 管理员与业务用户

已核验事实：

- 模板配置依赖审批管理权限，由管理员完成。
- 普通用户不负责模板设计，只使用管理员发布的模板。
- 提交人、审批人、抄送人等相关人员可在其可见审批单上执行打印。
- 首次使用部分自定义能力时存在数据访问授权步骤。

产品含义：模板维护权限和业务数据访问权限是两个不同的授权维度。

### 4.6 生成、归档与交付

已核验事实：

1. 可在预览后打印或保存为 PDF。
2. 对账单和 Invoice 示例把生成文件写回多维表格附件字段。
3. 示例使用按钮或自动化分别触发文档生成和邮件发送。
4. 邮件以生成的 PDF 作为附件。
5. 同一审批支持多个模板，并可按条件选择数据或生成不同文档。

产品含义：文档生成与邮件交付可以拆为两个步骤，生成文件是两者之间的稳定交付物。

### 4.7 附件、签名和图章

已核验事实：

- 审批附件可以进入预览和打印流程。
- 官方说明支持 PDF、DOCX、PPTX 等附件类型。
- 审批节点启用手写签名后，可在附件预览中选择签名和时间并放置。
- 图章以图片形式上传和放置。

证据边界：上述事实只证明产品提供图片化签名/图章和附件输出操作，不证明其在任意司法辖区等同于合规电子签名。

### 4.8 定价和商业边界

审批集成说明包含付费方案、导出次数和增购包。价格及额度属于易变化商业信息，本文不保存具体数值；如采购或成本决策需要，必须重新核验官方最新页面。

## 五、产品机制分析

### 5.1 真正的核心闭环

```mermaid
flowchart LR
    A["业务事实"] --> B["稳定字段模型"]
    B --> C["模板版本"]
    C --> D["预览与校验"]
    D --> E["正式生成"]
    E --> F["归档文件"]
    F --> G["打印或交付"]
```

任何只实现其中一段的方案都不完整：

- 只有浏览器打印，没有稳定模板和历史版本。
- 只有模板编辑，没有业务权限和数据契约。
- 只有 PDF 生成，没有归档与审计。
- 只有邮件发送，没有独立的可校验生成结果。

### 5.2 最小模板语义

对企业单据而言，第一版真正必要的能力是：

1. 标量字段：编号、名称、金额、日期、状态。
2. 集合循环：结算明细、付款来源、发票分配、审批轨迹。
3. 条件显隐：有无发票、不同付款类型、不同合同或结算场景。
4. 格式化：金额、财务大写、币种、百分比、日期和空值。

拖拽、复杂组件和 Office 三格式不是形成首个闭环的必要条件。

### 5.3 双模板路径的价值

在线模板适合结构相对统一、字段变化较多的单据；Office 模板适合已有正式格式、合同或公文。两者共同存在有产品合理性，但会显著增加模板解析、分页、兼容和安全成本。

对 cgc-pms 的推断：MVP 应只冻结一种受控渲染路径；只有真实格式需求证明 HTML/PDF 不足时，再评估 DOCX 模板，不应同时支持 DOCX/XLSX/PPTX。

### 5.4 自动化的正确拆分

官方示例将“生成 PDF”和“发送邮件”拆成两个自动化动作。这一结构值得保留：

- 生成失败不会产生错误附件邮件。
- 生成结果可以先人工校验。
- 邮件重试不需要重新计算业务文档。
- 文件归档可成为审计和交付的统一事实。

## 七、值得借鉴的设计

### 7.1 高优先级借鉴

1. **业务输出模型独立于数据库和页面。** 模板只绑定稳定逻辑字段。
2. **模板版本不可变。** 发布后修改必须产生新版本，历史 PDF 继续引用旧版本。
3. **模板发布前做字段校验。** 字段缺失或类型变化必须阻止发布。
4. **普通字段、明细、条件、格式化优先。** 用最小语义覆盖最多单据。
5. **管理员和业务用户分工。** 模板发布权限不等于业务数据访问权限。
6. **预览和正式生成共用渲染链。** 避免预览与下载结果漂移。
7. **生成与交付解耦。** PDF 归档成功后才能进入邮件或其他交付。
8. **文件作为历史事实。** 生成记录、模板版本和归档文件共同支撑追溯。

### 7.2 中期借鉴

- 多模板和默认模板。
- 模板复制、导入、导出及字段兼容校验。
- 附件清单或受控附件合并。
- 后台批量生成、限流和失败重试。
- 文件交付状态与重复发送防护。

### 7.3 不宜直接照搬

- 第一版建设完整拖拽设计器。
- 同时支持全部 Office 模板格式。
- 允许模板执行任意表达式、脚本或远程资源。
- 将图章图片或手写签名图片直接包装为可信电子签名。
- 把外部邮件发送与业务审批事务同步绑定。
- 使用营销效率数字作为 cgc-pms 验收目标。
- 为覆盖更多单据而绕过付款、结算的金额和状态一致性门禁。

## 十、决策结论

### 10.1 当前结论

DocuGenius 的产品结构与 cgc-pms 的付款、结算、审批、文件和审计基础存在明确互补价值。建议保留第48条主线作为产品 Candidate 输入，但实施前必须先完成当前付款/结算/发票闭环收口、输出契约冻结和渲染器安全尖峰。

### 10.2 当前不应直接形成 Ready 的原因

1. 当前工作区的付款、结算、发票和 migration 仍在变化。
2. 首批正式单据样式、允许状态、脱敏规则、保留期和角色权限尚未冻结。
3. Java 21 PDF 渲染依赖、中文字体、分页、许可证和资源守卫尚未形成技术证据。
4. 当前迭代决策仍以已选产品方向和既有能力证据收口为事实，不能只凭新增竞品研究改变优先级。

### 10.3 进入 Candidate/Ready 的最小条件

- 当前 `project-map.md` 与 `evolution-decision.md` 完成增量评估。
- 付款申请单、结算单的用户价值和首批样式经确认。
- 稳定输出字段、金额来源、允许状态和权限边界可验证。
- 技术尖峰证明后端 PDF 路径安全、可维护、可回滚。
- Ready 明确非目标、migration、测试、独立审查和最小回滚方式。

## 十一、更新与失效策略
