## 决策结论

本轮推荐：**采购低库存补货建议最小闭环**。

选择原因：当前库存余额、流水、低库存提示、采购申请和采购审批已经存在，主要断点是低库存预警不能直接形成采购动作。与 WBS、现场日报相比，它不需要先建立新的大型业务域；与供应商履约档案相比，它的输入口径更直接，能够形成更短的可验证闭环。

本轮不把“统一阈值小于 10”包装成成熟补货规则。第一版只复用现状，完成“低库存识别 → 建议数量确认 → 采购申请预填 → 用户提交现有审批”的链路；物料级安全库存、最大库存、供货周期和需求预测另行评估。

实施回写（2026-07-11）：`ISSUE-037-001` 已完成当前选中低库存项到采购申请预填的最小桥接，并通过真实采购经理角色保存链验收。该结果不改变对安全库存、供货周期、预测和全量补货工作台的缺口判断。

## 候选决策卡

### 1. 采购低库存补货建议最小闭环

| 字段 | 裁决内容 |
| --- | --- |
| 当前缺口 | `MatStockService.getKpi()` 已按可用量小于 10 统计低库存；库存页已显示低库存提示；采购申请页面和审批链已存在，但两者没有直接衔接 |
| 项目证据 | `inventory/service/MatStockService.java`、`pages/inventory/stock.vue`、`pages/inventory/purchase-request.vue`、`purchase/entity/MatPurchaseRequest.java` |
| 外部证据 | 广联达 PMSmart 的物资需用/缺口分析；Odoo 19 的重订货规则、供货周期和补货报告 |
| 用户价值 | 采购经理能从库存风险直接发起采购申请，减少重复查找物料和手工录入 |
| 战略匹配 | 直接加强既有采购—收货—库存—领料闭环，符合总包项目物资管理定位 |
| 最小闭环 | 在库存页列出低库存项；用户确认建议数量；跳转/打开现有采购申请并预填项目、物料和数量；仍由用户提交现有审批 |
| 非目标 | 自动下单、自动提交审批、需求预测、物料级阈值配置、供货周期模型、多仓调拨、供应商自动选择 |
| 依赖 | 库存项必须能确定项目、仓库、物料；采购申请必须支持预填并继续走现有权限和审批 |
| 风险 | 当前阈值统一为 10，不能宣称是智能预测；建议数量必须允许用户修改；不得跨租户/项目带入物料 |
| 工作量 | M |
| 置信度 | 高 |
| 裁决 | 已由 `ISSUE-037-001` 完成最小闭环；扩展能力另行决策 |

#### 建议的最小验收边界

1. 只有当前租户和当前项目可见库存参与建议。
2. 可用量小于 10 且大于等于 0 的库存项可进入建议列表；阈值沿用现状并明确展示口径。
3. 建议数量默认补到 10，但用户可修改，且必须大于 0。
4. 发起采购申请时复用现有页面、保存和审批链，不创建第二套采购对象。
5. 不自动提交，不自动选择供应商，不修改库存。
6. 至少覆盖库存边界、项目隔离、预填、用户修改和保存链测试。

## 本轮未选原因

- WBS 回写：单前置 FS 已证明可在现有分包任务模型内形成最小闭环；完整计划平台仍需独立决策。
- 不先做现场日报：能补生产数据主线，但新对象和现场角色设计范围更大。
- 不先做供应商履约档案：现有评分可复用，但综合评分数据质量尚未核验。
- 不先做接口入口治理：它是必要治理，不是当前产品方向的首选。
- 不先做 AI、BIM、RFI、Submittal：当前数据与业务对象不足，违反最小闭环原则。
