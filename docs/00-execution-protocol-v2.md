# 高韧性通用执行协议 v2 (Enhanced Resilience Protocol)

> **角色**：高级自动化工程师执行规范
> **用途**：全权执行实施方案文档，自愈驱动，非阻塞式容错
> **更新日期**：2026-06-23
> **继承自**：v1 原始协议 + 结构化审查修订

---

## 0. 协议入口与生命周期

### 0.1 方案文档来源

方案文档可以是以下任一形式：
- 用户消息中附带的文件路径（如 `D:\projects-test\cgc-pms\docs\superpowers\specs\xxx-design.md`）
- 用户消息中贴出的 URL
- 会话上下文中的附件

**如方案文档路径模糊、不存在或无法解析，必须先提问澄清，不得猜测启动。**

### 0.2 全局资源限制

| 参数 | 值 | 说明 |
|------|-----|------|
| `maxRetries` | **3** | 同一子任务最大重试次数，超出后强制跳过 |
| `maxParallel` | **3** | 同时并行执行的任务数上限 |
| `globalTimeout` | **3 小时** | 整个执行流程的全局超时。超时后保存现场并输出已完成/未完成清单 |
| `tokenBudget` | **20M** | 输入 token 硬性上限。达到后强制停止，保存现场并输出已完成/未完成清单。每 Phase 结束后输出已消耗 token 估算 |

### 0.3 人工中断机制

用户可在任意时刻发送以下信号中断执行：
- `STOP` — 立即停止，保存现场，输出当前进度报告
- `SKIP <task>` — 跳过指定子任务，继续后续流程

中断后 AI 不得私自恢复执行。

### 0.4 跨会话持久化

每完成一个 Phase，将进度写入：

```
D:\projects-test\cgc-pms\docs\superpowers\specs\.execution-progress.json
```

JSON 结构：

```json
{
  "planFile": "方案文档路径",
  "startedAt": "ISO8601",
  "lastUpdatedAt": "ISO8601",
  "phases": [
    {
      "id": "phase-0",
      "title": "Phase 0: 环境检查",
      "status": "completed | in-progress | skipped | failed",
      "tasksCompleted": 3,
      "tasksTotal": 5,
      "tasksSkipped": ["task-name-1"],
      "notes": ""
    }
  ]
}
```

恢复执行时，先读取此文件，跳过已完成 Phase，从断点继续。

## 1. 绝对完成原则 (The "No-Give-Up" Rule)

### 1.1 核心规则

必须从方案的**第一步串行执行至最后一步**，直至所有阶段标记为 `completed` 或 `skipped`。

### 1.2 硬性中止条件（仅以下两条）

1. 方案文档**白纸黑字写明的中止条件**（如"环境基线低于 X%"）
2. 用户发送 `STOP` 信号

任何其他情况——编译报错、测试失败、依赖缺失、API 返回异常——均视为**待解决的问题**，触发协议 2（智能自愈）处理。

### 1.3 与协议 2 的关系

协议 1 定义**态度**（不放弃），协议 2 定义**机制**（怎么处理失败）。两者不矛盾：

- 遇到失败 → 自动进入自愈循环（协议 2）
- 3 次重试后仍失败 → 记录 + **跳过该子任务**（不是停止整个流程）
- 跳过子任务 → 继续执行下一个子任务或下一个 Phase
- **只有**中止条件触发或用户 `STOP` 才停止整个流程

### 1.4 飞书通知（固定配置）

- **发送消息给用户**：
  ```bash
  lark-cli im +messages-send --user-id ou_20cc1ff2064c5eb97eaf3c6e78309292 --text "<消息内容>" --as bot
  ```
- **拉取用户发送的最新 1 条消息**：
  ```bash
  lark-cli im +chat-messages-list --chat-id oc_d7f0f381ce06dc89f2a3f483be26c825 --page-size 1 --sort desc --as user
  ```

> 飞书 ID 为当前用户固定凭证，硬编码是正确的，无需改为占位符。

### 1.5 飞书确认-回复交互流程

当执行过程中遇到**需要用户决策**的情况（如依赖冲突、业务逻辑修复需人工确认、API Key 缺失等），按以下流程操作：

```
┌─────────────────────────────────────────┐
│  Step 1: 发送确认请求                    │
│  - 飞书消息概述问题 + 列出选项 (A/B/C)    │
│  - 消息末尾标注"请在 1 分钟内回复"        │
│  - 消息格式：                            │
│    【执行确认】Phase X - <简短标题>       │
│    问题：<描述>                          │
│    选项：                                │
│    A) <选项A>                            │
│    B) <选项B>                            │
│    C) <选项C>                            │
│    请回复 A/B/C                          │
├─────────────────────────────────────────┤
│  Step 2: 等待 60 秒                      │
│  - sleep 60s（给用户思考+回复时间）        │
├─────────────────────────────────────────┤
│  Step 3: 拉取最新消息                    │
│  - lark-cli im +chat-messages-list ...   │
├─────────────────────────────────────────┤
│  Step 4: 解析用户回复                    │
│  - 匹配 A/B/C（大小写不敏感）             │
│  - 匹配到 → 按对应选项继续执行            │
│  - 未匹配/无回复 →                       │
│    再次拉取（再等 30s）                   │
│    仍无回复 → 采用默认选项（标注为         │
│    "用户未回复，采用默认 A"）              │
└─────────────────────────────────────────┘
```

**关键规则**：
- 发送确认请求时**暂停执行**，不进入自愈循环
- 等待总时长最多 **90 秒**（60s + 30s 二次拉取），不无限等待
- 无回复时使用**预先指定的默认选项**（消息中标注"如未回复，默认选择 A"）
- 确认消息模板中必须提供**明确的单字母选项**，便于解析

## 2. 智能自愈与故障处理 (Self-Healing Protocol)

### 2.1 自愈循环流程

```
┌─────────────────────────────────────────────────┐
│  子任务执行失败                                    │
│      ↓                                           │
│  [retryCount = 0]                                │
│      ↓                                           │
│  ┌──────────────────────────────────┐            │
│  │  Step 1: 分析 (Analyze)          │            │
│  │  - 读取控制台堆栈和日志            │            │
│  │  - 定位根因                       │            │
│  │  - 分类：环境/依赖/配置/业务逻辑    │            │
│  └──────────────┬───────────────────┘            │
│                 ↓                                │
│  ┌──────────────────────────────────┐            │
│  │  Step 2: 保存现场 (Checkpoint)    │ ← 新增     │
│  │  - git stash 保存当前变更          │            │
│  │  - 记录 stash 消息含任务名+时间     │            │
│  └──────────────┬───────────────────┘            │
│                 ↓                                │
│  ┌──────────────────────────────────┐            │
│  │  Step 3: 修复 (Fix)              │            │
│  │  ← 仅允许以下范围：                │            │
│  │  a) import/path 修正              │            │
│  │  b) 配置文件格式修正 (YAML/JSON)    │            │
│  │  c) 依赖声明修正 (pom/gradle/pkgs) │            │
│  │  d) 类型注解补充                   │            │
│  │  e) 过时 API 调用更新              │            │
│  │                                   │            │
│  │  ← 以下范围禁止自动修改：           │            │
│  │  - 业务逻辑/算法/状态机             │            │
│  │  - 接口契约/API 签名               │            │
│  │  - 数据库 Schema（除 Flyway 格式）  │            │
│  │  - 安全相关代码                    │            │
│  │  → 若根因属于禁止范围，             │            │
│  │    立即跳过（不重试），记录原因      │            │
│  └──────────────┬───────────────────┘            │
│                 ↓                                │
│  ┌──────────────────────────────────┐            │
│  │  Step 4: 重试 (Retry)            │            │
│  │  retryCount++                     │            │
│  │  重新执行子任务                    │            │
│  └──────────────┬───────────────────┘            │
│                 ↓                                │
│         成功？─── 是 ──→ 继续下一子任务            │
│                 ↓                                │
│         否 ──→ retryCount < 3?                   │
│                 ↓ 是              ↓ 否             │
│           返回 Step 2      进入跳过流程 ──→        │
│                            (见 2.2)              │
└─────────────────────────────────────────────────┘
```

### 2.2 跳过流程（不可阻塞）

同一子任务重试 **3 次** 后仍失败，判断为"复杂业务逻辑冲突"或"环境依赖缺失"：

1. **记录**到执行交接清单文件：
   ```
   D:\projects-test\cgc-pms\docs\superpowers\specs\.execution-handoff.md
   ```
   每条记录包含：
   ```markdown
   ## [SKIPPED] <子任务名称>
   - **时间**：<ISO8601>
   - **Phase**：<Phase 编号和名称>
   - **失败命令**：`<完整命令>`
   - **错误摘要**：
     ```
     <完整错误堆栈>
     ```
   - **尝试过的修复**：
     1. <修复 1 描述>
     2. <修复 2 描述>
     3. <修复 3 描述>
   - **疑似根因**：<分析结论>
   - **建议人工介入**：<下一步建议>
   ```

2. **通知用户**（飞书消息）：发送简短摘要，含子任务名称和跳过原因。

3. **继续执行**：立即跳到下一个子任务或 Phase，**不阻塞整体流程**。

### 2.3 自愈范围分类决策表

| 根因类型 | 处理方式 |
|---------|---------|
| 缺失依赖包/版本不匹配 | 自动修复 + 重试 |
| Import/路径错误 | 自动修复 + 重试 |
| 配置格式错误 (YAML/JSON/XML) | 自动修复 + 重试 |
| 类型/编译错误（非业务逻辑） | 自动修复 + 重试 |
| 测试断言过期 | 自动更新断言 + 重试 |
| 业务逻辑缺陷 | **禁止自动修改**，记录 + 跳过 |
| 需要外部 API Key/权限 | 记录 + 跳过 + 飞书通知用户 |
| 基础设施不可用 (DB/Redis down) | 等待 10s 重试 ×3 → 记录 + 跳过 |

## 3. 并行与依赖管理 (Parallelism & Dependencies)

### 3.1 串行约束

严格遵循方案文档中定义的依赖关系：
- Phase 之间默认**串行**（Phase 0 → Phase 1 → Phase 2 → ...）
- 方案内声明为依赖的任务必须串行执行

### 3.2 并行规则

- 方案文档注明"可并行"的任务 → 同时启动
- **并行上限**：`maxParallel = 3`（同一时间最多 3 个任务并发）
- 超出上限的任务排队等待，先进先出

### 3.3 隐式依赖检测

**执行前扫描**：在每个 Phase 开始前，快速检查方案未声明但实际存在的依赖：
- 文件依赖性：任务 B 引用了任务 A 产出的文件，但方案标记为"可并行"
- 数据依赖性：任务 B 需要的数据库表/数据由任务 A 创建
- 状态依赖性：任务 B 依赖任务 A 的副作用（如环境变量设置、服务启动）

**检测到隐式依赖时的处理**：
1. **暂停并行**该组任务
2. **记录**到 `.execution-handoff.md`：
   ```markdown
   ## [IMPLICIT-DEP] 检测到未声明的依赖
   - **任务组**: <任务 A, 任务 B>
   - **依赖关系**: <B 依赖 A 产出 X>
   - **处理**: 已强制串行 A → B
   ```
3. **自动调整为串行**执行，不等待人工确认

### 3.4 依赖冲突升级

如果检测到的隐式依赖与方案声明的依赖关系冲突（方案说 A → B，实际是 B → A），则：
1. 暂停该 Phase
2. 飞书通知用户，附上冲突说明
3. 等待用户决策后继续

## 4. 版本控制策略 (Git Strategy)

### 4.1 提交频率

- **默认**：每完成一个 Phase，执行一次本地提交
- 方案文档指定了其他频率的，按方案执行

### 4.2 暂存方式

**禁止使用 `git add .`**。仅暂存受方案影响的文件：

```bash
# 正确方式：限缩到已知影响的文件
git add <受方案影响的文件路径1> <路径2> ...

# 如不确定影响范围，先用 git status 确认，再精确 add
```

### 4.3 Commit Message 模板

严格遵循 Conventional Commits 格式：

```
<type>(<scope>): [Phase <N>] <简短描述>

- 完成的子任务 1
- 完成的子任务 2
- 跳过的子任务（如有）: <任务名> — <跳过原因>

Ref: <方案文档路径>
```

类型选择：
- `feat` — 新增功能
- `fix` — 修复缺陷
- `refactor` — 重构
- `test` — 测试相关
- `chore` — 配置/构建/基础设施
- `docs` — 文档

示例：
```
chore(db): [Phase 2] complete Flyway migration for soft-delete unique constraints

- V88: fix mat_stock index name reference
- V89: add settlement contract unique constraint
- Skipped: V90 seed data update — 业务逻辑冲突，需人工确认

Ref: D:\projects-test\cgc-pms\docs\superpowers\specs\xxx-design.md
```

### 4.4 关键约束

- **全程仅做本地提交**
- **严禁自动执行 `git push`**
- 全部执行完毕后，生成变更摘要等待人工审查
