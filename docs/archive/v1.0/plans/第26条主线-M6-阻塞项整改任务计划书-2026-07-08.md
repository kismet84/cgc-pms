## 4. 串行阶段计划 M6.1-M6.5

### M6.1 后端 P1 安全修复：B1 + B3

- 目标：先关闭最直接阻塞上线的权限与会话安全问题。
- 包含阻塞项：B1、B3。
- 建议执行顺序：
  1. 修复成本摘要项目级访问控制缺口。
  2. 补成本摘要正反向权限测试。
  3. 收敛 token blacklist / Redis 生产降级语义。
  4. 补 blacklist 降级行为测试与必要文档说明。
- 阶段产出：
  1. 后端实现子任务交付记录。
  2. 定向测试结果。
  3. M6.1 验收结论。
- 阻塞判定：B1 或 B3 任一未关闭，则 M6 不得进入最终通过。

### M6.2 发布包洁净度修复：B2

- 目标：清掉进入版本库和交付面的测试/浏览器运行态产物。
- 包含阻塞项：B2。
- 建议执行顺序：
  1. 确认已跟踪产物是否仍有正式归档价值。
  2. 对无价值运行态产物执行非破坏性移除跟踪。
  3. 对需保留证据迁移为脱敏正式报告附件或文档引用。
  4. 补忽略规则，避免同类产物再次进入版本库。
- 阶段产出：
  1. 洁净度整改记录。
  2. 版本库复核记录。
  3. M6.2 验收结论。
- 阻塞判定：若 `admin-state.json` 或同类运行态浏览器状态文件仍在版本库中，视为未通过。

### M6.3 前端浏览器失败项修复：B4-B7

- 目标：最小范围修复已确认的前端失败项，不扩展到无关页面重构。
- 包含阻塞项：B4、B5、B6、B7。
- 建议执行顺序：
  1. 修复 dashboard 导出。
  2. 修复 alert 导出。
  3. 修复 inventory transaction 空提交/重复点击反馈。
  4. 修复 cost ledger `projectList` 空值告警。
- 阶段产出：
  1. 前端实现子任务交付记录。
  2. 针对性浏览器复测记录。
  3. M6.3 验收结论。
- 阻塞判定：任一页面仍无法证明导出/校验/告警已关闭，则不得进入 M6 收口。

### M6.4 高风险操作可回滚测试数据方案与补测：B8

- 目标：在不污染正式业务数据前提下，补齐上传、最终审批、有效业务提交等高风险操作覆盖。
- 包含阻塞项：B8。
- 建议执行顺序：
  1. 设计可回滚测试数据方案或专用测试实例。
  2. 明确上传、最终审批、有效提交各自的执行前置与回滚方式。
  3. 按方案进行浏览器补测。
  4. 记录通过/不通过与剩余风险。
- 阶段产出：
  1. 高风险操作补测方案。
  2. 补测证据与回滚记录。
  3. M6.4 验收结论。
- 阻塞判定：若仍只能写“未覆盖，不宣称通过”，则 B8 未关闭。

### M6.5 汇总复核与收口报告

- 目标：基于 M6.1-M6.4 的整改与补测结果做最终收口裁决。
- 包含内容：
  1. B1-B8 闭环状态汇总。
  2. 通过/不通过、阻塞/非阻塞裁决。
  3. 剩余风险与后续去向。
  4. 未进入本轮的 P2/P3 后续建议。
- 阶段产出：
  1. `docs/quality` 正式复核报告。
  2. M6 收口验收报告。
- 阻塞判定：只要 B1-B8 存在未关闭项，M6.5 只能输出“不通过 / 阻塞”。

## 6. 每阶段验收标准

### M6.1 验收标准

1. B1 已接入项目级访问控制，且无权限访问为 403。
2. B1 至少覆盖管理员、项目负责人、同租户无权限用户、跨租户用户的正反向测试。
3. B3 已明确生产 blacklist/Redis 异常语义，不能再以“继续放行但无强告警”作为默认收口结论。
4. B3 有对应测试或验证证据，能区分测试环境与生产环境语义。

### M6.2 验收标准

1. 版本库不再跟踪浏览器状态文件、测试截图、覆盖率中间产物等运行态文件。
2. `admin-state.json` 不再以运行态状态文件形式保留在版本库。
3. 需保留的正式证据已脱敏并迁移到合规归档位置。
4. 忽略规则已覆盖本轮确认的问题目录或文件类型。

### M6.3 验收标准

1. `/dashboard` 导出可被浏览器验收明确证明。
2. `/alert` 导出可被浏览器验收明确证明。
3. `/inventory/transaction` 空提交存在可见反馈，重复点击不会形成无感失败。
4. `/cost/ledger` 控制台不再出现 `projectList` 为空的 Vue warn。

### M6.4 验收标准

1. 上传、最终审批、有效业务提交均有明确测试数据方案。
2. 每类高风险操作都有回滚方式或隔离说明。
3. 补测报告对每项操作给出通过/不通过结论，而不是“未覆盖”。
4. 补测不会把正式业务数据污染风险留给 M6.5 兜底。

### M6.5 验收标准

1. B1-B8 均有关闭结论或明确剩余阻塞说明。
2. 最终报告明确“通过/不通过、阻塞/非阻塞、依据、剩余风险”。
3. P2/P3 仅作为后续建议列出，不反向污染本轮阻塞闭环。

## 7. 测试与浏览器复核要求

1. M6.1 后端整改完成后，必须做与 B1、B3 对应的最小定向测试，不以“代码已改”代替验收。
2. M6.2 洁净度整改完成后，必须做只读版本库复核，确认运行态产物已退出交付面。
3. M6.3 复核必须使用内置浏览器，优先入口仍为 `http://localhost:5173/api/auth/dev-login?redirect=/dashboard`。
4. M6.3 浏览器复测至少覆盖 `/dashboard`、`/alert`、`/inventory/transaction`、`/cost/ledger`。
5. M6.4 补测必须覆盖上传、最终审批、有效业务提交；若需要专用测试数据，先给出数据方案再执行。
6. 每次浏览器复核都要保留访问路径、操作步骤、下载/API/控制台证据，以及每项操作的通过/不通过结论。

## 8. 风险与回滚

1. B1、B3 属于安全与权限高风险整改，若修复方式过大，容易引入权限误杀或认证链路回归；应坚持共享入口最小改动。
2. B2 涉及版本库跟踪治理，若未先确认证据归档去向，可能误删仍需保留的正式审计证据。
3. B4-B7 属于前端局部修复，若顺手扩散到页面重构，会拖慢首轮阻塞关闭。
4. B8 涉及真实业务动作，必须先设计可回滚测试数据方案；未经方案确认不得直接执行最终审批或有效提交。
5. 若任一阶段首次复核不通过，主负责人应执行重新分档检查，判断是模型档位不足、任务说明不清、验收标准缺失还是执行边界冲突。
