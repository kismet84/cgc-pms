# 第 10A 主线：权限字典治理与历史权限码治理首批最小修复包任务计划书

## 计划状态

不通过。

本实现包由 `docs/plans/第10条主线-权限字典治理与历史权限码治理裁决任务计划书.md` 派生。主负责人已裁决：本首包样本 `alert:edit` 在小包 1 安全复核阶段不通过，因此本计划不再进入实现。

## 当前终止原因

1. `alert:edit` 不是低风险重复码噪音，而是两个安全级别不同的写动作共用同一后端授权码。
2. `标记已读` 是单条状态修改。
3. `批量评估` 是租户级批处理触发，并伴随 `alert_log` 写入和通知生成。
4. 安全侧已明确判定：该样本仍属于高危需要确认，不宜直接作为首批最小修复样本。

## 当前关闭口径

本计划当前只完成了：

1. 首包样本定义。
2. 后端 / 数据库 / 安全 / 代码质量小包 1 只读复核。
3. 主负责人最终否决该样本进入小包 2。

本计划未进入且不得进入：

1. migration 实施
2. 后端权限注解修改
3. 前端权限感知同步
4. 运行态刷新
5. 测试验收

后续若继续推进 10A，只允许另选更低风险样本后重新开包；本包不再复用。

## 一、主负责人裁决

1. 10A 首批最小修复包**只纳入**：`alert:edit` 同业务域重复 `perms` 清理。
2. 10A 首批最小修复包**明确不纳入**：
   - `contract:submit` 重复
   - 所有 `submit` 未绑定项 6 条
   - 所有 `DB-only` 历史残留码
   - `84` 个 authority 缺口
   - `meta.permission` / `v-permission` / dashboard 观察项
3. 选择 `alert:edit` 的原因：
   - 它是重复码问题里风险最低的一项。
   - 两个菜单都在预警中心业务域内，不跨系统管理、不跨审批流转。
   - 不涉及 `submit`、`approve`、`delete`、`system:*` 等高危权限。
4. 本包目标不是“修完 10A”，而是验证：
   - `sys_menu` 点状修复迁移
   - MySQL/H2 对等
   - 最小定向测试
   - 第 9 条权限清单与审批入口基线不回退

## 二、前置事实

依据 `第10条主线-权限字典治理与历史权限码治理裁决任务计划书` 已完成裁决：

1. `alert:edit` 当前存在重复 `perms`。
2. 重复菜单为：
   - `767 标记已读`
   - `768 批量评估`
3. 两者同属预警中心操作按钮，不跨业务域。
4. 该项虽然仍属“高危需要确认”范畴，但相较 `contract:submit`、`submit` 未绑定项、`system:*` / `project:delete`，风险最低，适合作为首批最小修复样本。

## 三、修复目标

将预警中心两条重复 `perms` 从“一码多菜单”收敛为“语义一一对应”，避免后续权限字典继续带着重复码运行。

最小预期结果：

1. `标记已读` 与 `批量评估` 不再共享同一个 `alert:edit`。
2. 修复后：
   - 不新增新权限事实源
   - 不新增权限字典表
   - 不补全其他历史权限问题
3. 仅允许修改：
   - `sys_menu` 对应 seed / migration
   - 必要的后端 `@PreAuthorize` 或前端权限引用（仅在确有实际使用点时）

## 四、纳入范围

### 仅允许范围

1. 只读复核 `alert:edit` 两条菜单的真实语义、真实调用点、真实角色绑定。
2. 若确认两条菜单语义不同，则把其中一条改为新且语义自洽的权限码。
3. 仅允许 `sys_menu` 点状修复迁移。
4. 若代码中存在对应旧权限码字面量使用点，可做最小同步修正。
5. 补最小测试：
   - migration 完整性
   - 权限码唯一性
   - 若代码改动则补对应定向测试

### 明确不纳入

1. 不碰 `sys_role_menu` 全量重绑。
2. 不补 `submit` 未绑定 6 条。
3. 不修 `contract:submit` 重复。
4. 不删任何 `DB-only` 历史残留码。
5. 不补 authority 缺口。
6. 不碰 `system:*`、`project:*` 高危系统权限。
7. 不做第二租户、复杂 `data_scope`、项目级强隔离。
8. 不碰第 10B / 10C / 10D 任何链路。

## 五、阻塞项

当前无主负责人层面的范围阻塞。执行中若命中以下任一情况，必须停止并回主线程裁决：

1. `alert:edit` 两条菜单其实是同一语义的历史兼容别名，不能简单拆分。
2. 任一非管理角色当前真实依赖其中某条菜单的旧权限码，且改码会影响运行态可见性，但无法提供最小回滚方案。
3. 修复需要同步改多个业务域、多个接口或多个前端页面，超出“点状修复”边界。
4. 修复被证明需要同时改 `sys_role_menu` 大量绑定、补新表、补接口或补前端平台逻辑。
5. live MySQL 只读复验发现迁移链静态事实与运行态严重不一致。

### 通过条件

1. `alert:edit` 重复问题完成只读确认并有明确语义拆分。
2. 修复仅限一个最小样本项。

4. 为修一个重复码而触发大面积角色绑定调整。
