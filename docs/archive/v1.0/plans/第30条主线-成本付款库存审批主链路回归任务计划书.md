# 第30条主线-成本付款库存审批主链路回归任务计划书

**Goal:** 以最小可行方案完成成本、付款、库存、审批主链路回归，验证核心业务链路在后端状态流转、金额口径、前端关键页面、驾驶舱来源下钻四个层面保持一致，为后续 P0 收口提供可执行基线和正式验收依据。

**Architecture:** 严格复用现有后端测试基座、现有业务测试数据、Playwright、`/api/auth/dev-login?redirect=/dashboard` 浏览器验收入口和 `docs/quality` 正式归档规则；不引入新平台、新框架、新矩阵系统，不重构审批引擎，不扩大权限模型，不并行推进前端列表页生产化、文件上传/发票识别安全增强、报表中心建设。执行顺序采用“先冻结矩阵、再做后端一致性、再做前端冒烟、再核驾驶舱、最后归档”的单线收口方式。

---

## 1. 结论与执行边界

### 1.1 本主线结论优先

第30主线不是新功能扩张任务，而是对既有主链路做生产级回归和一致性核验。核心目标是回答三类问题：

1. 来源单据、状态流转、台账汇总是否一致。
2. 前端关键页面和审批主链路是否可在真实浏览器中走通。
3. 驾驶舱指标是否能下钻到来源单据并解释口径。

### 1.2 主线范围

本主线仅覆盖以下九条主链路：

1. 合同 -> 合同清单 -> 付款条件
2. 签证 / 变更 -> 成本 / 收入调整
3. 采购订单 -> 收货 -> 入库 -> 库存
4. 领料 -> 出库 -> 项目成本
5. 分包任务 -> 计量 -> 结算
6. 付款申请 -> 审批 -> 财务回写
7. 发票登记 -> 付款关联 -> 发票台账
8. 审批中心 -> 状态流转 -> 通知 / 预警
9. 驾驶舱 -> 汇总指标 -> 来源单据下钻

### 1.3 非目标

以下内容明确不纳入第30主线：

1. 不做 `7.5 P1-2` 前端列表页生产化。
2. 不做 `7.6 P1-3` 文件上传与发票识别安全增强。
3. 不做报表中心建设。
4. 不重构审批引擎。
5. 不扩大权限模型。
6. 不新增大平台、大框架、大矩阵系统。
7. 不把一次回归扩展成全仓重测。

### 1.4 子智能体执行边界

后续实施默认由主线程派发子智能体执行，边界如下：

1. 实现型子智能体：允许补测试、修复后端一致性缺陷、必要时修正少量前端链路问题。
2. 验收型子智能体：允许运行目标测试、浏览器验收、生成正式结论。
3. 审计/归档型子智能体：允许写入 `docs/quality` 正式报告，不改业务代码。
4. 运维型子智能体：仅在浏览器验收需要真实运行态时，执行最小刷新和只读核验。

---

## 2. 总体策略

### 2.1 最小可行方案

采用四层回归口径：

1. 后端链路一致性为主，优先用现有集成测试和最小补测证明状态、金额、日期一致。
2. 前端只做关键页面冒烟，不扩大到全量列表页体验治理。
3. 驾驶舱只验证关键 KPI 的来源解释能力，不扩展为报表中心。
4. 验收证据以测试结果、浏览器证据、下钻截图/日志摘要、正式报告为准，不保留过程性噪音。

### 2.2 证据口径

每阶段至少产出以下一种正式证据：

1. 后端测试结果摘要。
2. 浏览器验收结果摘要。
3. 来源单据下钻核验记录。
4. `docs/quality` 正式验收或归档报告。

---

## 3. 阶段拆分

## M1 现状盘点与回归矩阵冻结

### 目标

冻结第30主线的真实回归范围、页面入口、接口入口、测试入口和验收口径，避免后续阶段边做边扩。

### 范围

1. 盘点九条主链路对应的后端 Service / 测试类 / 页面入口 / 驾驶舱指标。
2. 识别已有自动化覆盖和明显缺口。
3. 形成“主链路 -> 来源对象 -> 核验字段 -> 验收方式”的回归矩阵。
4. 冻结本主线允许修改的模块边界和不允许扩展的内容。

### 产出物

1. 第30主线回归矩阵。
2. 分阶段派工清单。
3. 每条主链路的验收证据要求。

### 验收标准

1. 九条主链路全部落到具体对象和入口，不停留在模块名。
2. 每条链路至少明确一个后端核验点和一个验收方法。
3. 非目标写清并冻结。

### 风险点

1. 盘点不落到真实类和页面，后续执行会扩大范围。
2. 把驾驶舱、审批、库存问题混成一个大杂烩，导致阶段失焦。

### 建议模型分档

`gpt-5.5 + medium`

原因：需要跨多个业务域盘点真实调用链和测试口径，但仍以只读分析和矩阵冻结为主。

---

## M2 后端链路一致性回归

### 目标

对九条主链路做后端一致性回归，优先证明状态流转、金额口径、日期口径、关联关系和关键台账结果一致。

### 范围

1. 优先复用现有后端测试类、H2 / 集成测试基座和已存在测试数据。
2. 对缺口最明显的链路补最小测试，不引入新的大测试框架。
3. 覆盖重点：
   - 合同、变更、签证对成本/收入的影响
   - 采购、收货、入库、领料对库存和项目成本的影响
   - 付款申请、审批、财务回写、发票关联的一致性
   - 审批状态流转、通知/预警联动

### 产出物

1. 定向后端测试清单。
2. 每条链路的关键断言说明。
3. 需要修复的问题列表与修复提交。

### 验收标准

1. 主链路相关后端测试退出码为 0。
2. 金额字段、日期字段、状态字段断言可解释。
3. 发票与付款记录关联关系可被自动化证明。
4. 审批流转后的业务状态与台账结果一致。

### 风险点

1. 历史测试数据耦合过深，导致单条链路难以独立证明。
2. 一致性问题可能跨两个以上模块，容易演化成临时大改。
3. 库存、成本、付款口径若定义不清，会反复返工。

### 建议模型分档

`gpt-5.5 + high`

原因：涉及跨模块后端行为、状态机、金额口径和数据一致性，属于高风险实现与回归任务。

---

## M3 前端/浏览器冒烟验收

### 目标

用真实浏览器对主链路关键页面进行最小冒烟，验证回归结果不只停留在后端测试通过。

### 范围

1. 复用现有 Playwright 能力和 `dev-login` 入口。
2. 仅覆盖主链路关键页面和关键动作，不展开到列表页生产化治理。
3. 验证重点：
   - 页面能进入
   - 关键单据能查看
   - 审批待办/已办主路径可见
   - 关键跳转和状态展示与后端结果一致

### 产出物

1. 浏览器冒烟脚本或脚本补丁。
2. 冒烟结果摘要。
3. 失败页面与复现路径清单。

### 验收标准

1. 关键页面可打开且不因前端异常阻断主链路。
2. 审批、付款、库存、发票相关关键路径至少各有一条真实浏览器证据。
3. 若发现运行态问题，需区分环境前置和真实质量问题。

### 风险点

1. 本地运行态漂移可能造成假失败。
2. 前端代理缓存旧后端地址时，浏览器证据会失真。
3. 把冒烟验收误扩展成完整 UI 回归，会拖慢主线收口。

### 建议模型分档

`gpt-5.5 + medium`

原因：需要汇总浏览器证据并排除环境噪音，但实施范围集中在关键页面冒烟。

---

## M4 驾驶舱与来源下钻核验

### 目标

验证驾驶舱关键指标可以解释来源，并且下钻链路与来源单据口径一致。

### 范围

1. 仅选择与第30主线直接相关的 KPI，不扩展为报表中心普查。
2. 重点核验：
   - 成本/付款/库存/审批相关驾驶舱指标
   - 指标到来源单据的下钻路径
   - 来源金额、状态、时间口径的一致性

### 产出物

1. KPI 对照表。
2. 指标到来源单据的下钻核验记录。
3. 需修复的口径问题清单。

### 验收标准

1. 关键 KPI 能说明来源对象和计算口径。
2. 下钻结果与来源单据一致，不出现无法解释的汇总偏差。
3. 驾驶舱问题若属于历史已知非本主线范围，必须在报告中明确隔离。

### 风险点

1. 指标定义分散在多个服务，排查成本高。
2. 驾驶舱问题可能暴露更早期的数据沉淀问题，不一定能在本主线一次修净。

### 建议模型分档

`gpt-5.5 + medium`

原因：需要汇总多证据并形成口径判断，风险高于普通文档工作，但低于后端高风险实现。

---

## M5 正式验收与质量归档

### 目标

对第30主线给出正式通过/不通过结论，完成质量归档和剩余风险收口。

### 范围

1. 汇总 M1-M4 证据。
2. 输出正式验收报告。
3. 明确阻塞项、非阻塞项、未覆盖范围和后续建议。
4. 不在本阶段继续扩展实现范围，只做裁决和归档。

### 产出物

1. `docs/quality` 正式验收报告。
2. 第30主线通过/不通过结论。
3. 剩余风险清单。

### 验收标准

1. 结论明确区分通过/不通过、阻塞/非阻塞。
2. 报告可追溯到测试结果、浏览器证据、下钻核验记录。
3. 未运行、未覆盖、待后续处理内容必须明确列出。

### 风险点

1. 若前四阶段证据不完整，最终报告会失去裁决基础。
2. 若把过程日志当正式交付物，归档会变得冗杂且不可复用。

### 建议模型分档

`gpt-5.4 + medium`

原因：以证据汇总、质量裁决和正式归档为主，判断空间较小，但需要结论严谨。

---

## 4. 派工建议

### 4.1 建议任务分类

1. M1：审计/盘点型任务
2. M2：实现型任务
3. M3：验收型任务
4. M4：验收/审计混合任务
5. M5：审计/归档型任务

### 4.2 证据要求

每个阶段回报至少包含：

1. 变更文件或核验对象清单
2. 执行命令摘要
3. 测试或浏览器结果摘要
4. 阻塞与非阻塞风险
5. 是否可进入下一阶段

---

## 5. 最终验收口径

第30主线最终通过至少同时满足以下条件：

1. 九条主链路已有冻结矩阵并全部被覆盖到明确证据类型。
2. 相关后端定向测试通过，关键金额、状态、日期、关联关系一致。
3. 前端关键路径真实浏览器冒烟通过，不存在阻断主链路的页面问题。
4. 驾驶舱关键 KPI 可解释来源，下钻链路成立。
5. 正式验收报告完成，阻塞项为零。

若不满足上述任一条件，则不得宣告第30主线收口。
