## 主负责人结论

第 15 条主线可以正式进入 `M3`。

`M3` 的唯一定位是：**把当前已经可治理的预警能力，从“页面内可见、可处理”提升到“按人精准触达、按权限隔离、按订阅收敛、可继续扩展渠道”**。

本阶段只处理以下 4 类事项：

1. 通知平台最小落地
2. 多渠道触达最小骨架
3. 强权限隔离
4. 订阅策略

本阶段不承担以下交付：

1. 不建设复杂通知编排中心
2. 不建设全量规则设计器后台
3. 不一次性接入全部外部消息渠道正式生产发送
4. 不重构全站权限体系，只处理预警中心强隔离闭环

裁决口径如下：

1. `M3` 必须覆盖通知平台、多渠道触达、强权限隔离、订阅策略四项，不再把它们写成“暂不纳入范围”。
2. `M3` 仍然坚持最小生产增强，不得借机扩成“消息平台重构”或“全系统 RBAC 重构”。
3. 若实现过程中发现必须先解决全局权限底座缺口，需按阻塞项回报主线程裁决，不得默认扩包。

## 一、立项目标

`M3` 完成后，预警中心至少应达到以下状态：

1. 新生成或状态变更的预警可落到站内通知，不再只停留在列表被动查看。
2. 预警触达对象可按用户、角色、业务域和严重度做最小订阅控制。
3. 用户只能看到自己权限范围内的预警，不再只是“默认筛选”层面的弱约束。
4. 通知发送链路具备多渠道扩展位，至少完成站内信实装，并为邮件/企微/短信保留统一派发口径。
5. 整体能力具备真实验收证据，包括权限穿透验证、订阅策略验证和消息触达验证。

## 二、纳入范围

### 2.1 本阶段纳入范围

1. 站内通知最小实现：
   - 预警生成通知
   - 预警状态变更通知
   - 通知已读 / 未读最小闭环
2. 通知目标人计算最小实现：
   - 基于用户角色
   - 基于项目参与关系或业务归属
   - 基于预警域与严重度
3. 订阅策略最小实现：
   - 是否接收
   - 接收渠道
   - 接收严重度阈值
   - 接收业务域范围
4. 强权限隔离最小实现：
   - 列表查询隔离
   - 详情查看隔离
   - 状态操作隔离
   - 批量操作隔离
5. 多渠道触达最小骨架：
   - 统一通知派发接口
   - 渠道路由枚举或配置
   - 失败回退与发送记录基础字段
6. 与上述能力对应的后端测试、前端静态验证、真实页面复验和质量归档。

### 2.2 本阶段不作为交付项

1. 不建设可视化通知编排平台。
2. 不建设全量规则设计器后台。
3. 不一次性接入短信、邮件、企微、钉钉、电话等全部正式外部发送能力。
4. 不重构整个系统的用户、组织、岗位、数据权限底层模型。
5. 不把预警中心扩成独立 IM 或消息中心产品。

说明：

1. 上述内容仅表示“不作为 `M3` 当前交付项”。
2. 上述内容不等于“永久不在计划范围内”。
3. `M3` 必须至少把站内通知、多渠道骨架、订阅策略和强权限隔离做成真实可验收能力。

## 三、设计原则

### 3.1 通知平台原则

1. 优先复用现有系统消息、站内提醒、用户身份与权限体系，不单独造新的用户中心。
2. 先落地站内通知闭环，再保留统一渠道派发接口。
3. 通知模板只覆盖当前已落地预警类型，不为未来所有消息场景过度抽象。

### 3.2 多渠道触达原则

1. 多渠道先做“统一派发口径 + 渠道占位”，至少一条渠道真实可用。
2. 若外部渠道配置、凭据或合规条件缺失，允许先输出可运行占位链路，但必须明确标注“需要确认”。
3. 不允许用“只写文档不进代码”的方式把多渠道触达假装完成。

### 3.3 强权限隔离原则

1. 权限隔离作用于数据查询与操作，不等于页面默认筛选。
2. 后端接口必须成为最终权限边界，前端仅做协同展示，不得承担真实隔离职责。
3. 隔离优先围绕“项目归属、业务域权限、角色职责”实现最小闭环，不默认拉入全系统所有数据权限模型。

### 3.4 订阅策略原则

1. 订阅策略先服务当前已落地预警规则，不为未来未知规则预埋复杂 DSL。
2. 订阅策略必须与强权限隔离并存，订阅范围不得越过可见范围。
3. 允许角色默认订阅 + 用户个性化覆盖并存，但必须有明确优先级。

## 四、分阶段交付包

### 交付包 A：通知平台最小落地

- 执行类型：实现型任务
- 建议执行角色：后端工程执行子智能体
- 目标：
  - 为预警生成与状态流转建立站内通知闭环
- 执行要求：
  1. 识别当前仓库是否已有可复用通知/消息模块。
  2. 若已有模块，优先复用；若没有，只做预警域最小通知表、服务与接口。
  3. 至少覆盖：
     - 新预警生成通知
     - 预警被处理/归档时对相关责任人或创建人通知
  4. 不得直接扩成全站消息中心重构。
- 验收输出：
  - 通知实体 / 表结构 / 服务清单
  - 触发时机清单
  - 接收对象计算口径

### 交付包 B：多渠道触达最小骨架

- 执行类型：实现型任务
- 建议执行角色：后端工程执行子智能体
- 目标：
  - 为站内通知之外的渠道预留统一派发链路
- 执行要求：
  1. 明确渠道枚举、派发接口和失败回执口径。
  2. 真实落地站内信渠道。
  3. 对邮件/企微/短信等外部渠道，至少补齐：
     - 渠道配置位
     - 发送记录字段
     - 调用适配接口
  4. 若无现成外部渠道能力，本阶段不强行打通真实外发，但不得缺少后续接入骨架。
- 验收输出：
  - 渠道枚举与派发接口
  - 发送记录或日志口径
  - 外部渠道占位说明

### 交付包 C：强权限隔离

- 执行类型：实现型任务
- 建议执行角色：后端工程执行子智能体
- 目标：
  - 将预警中心从“默认筛选”升级为“真实可见范围控制”
- 执行要求：
  1. 明确谁可以看全部预警，谁只能看本项目、本业务域或本人相关预警。
  2. 后端列表接口必须按当前登录用户权限过滤。
  3. 详情、状态更新、批量处理也必须做同口径权限校验。
  4. 如发现现有权限模型不足以支撑，必须输出缺口与最小降级方案。
  5. 不得只在前端隐藏按钮而不做后端校验。
- 验收输出：
  - 权限矩阵
  - 接口隔离点清单
  - 正反例测试样本

### 交付包 D：订阅策略最小实现

- 执行类型：实现型任务
- 建议执行角色：前后端协同执行子智能体
- 目标：
  - 建立角色默认订阅 + 用户覆盖的最小可用机制
- 执行要求：
  1. 明确默认订阅维度：
     - 业务域
     - 严重度
     - 渠道
     - 是否接收状态变更通知
  2. 明确用户覆盖策略：
     - 用户可缩小范围
     - 用户是否允许扩大范围，如涉及越权则禁止
  3. 前端页面只做最小可用配置界面，不做复杂运营化工作台。
  4. 订阅配置必须受权限边界约束。
- 验收输出：
  - 订阅字段清单
  - 优先级规则
  - 页面改动说明

### 交付包 E：真实验收与质量归档

- 执行类型：验收型任务 / 运维型任务 / 审计归档型任务
- 建议执行角色：测试验收员 + 运维负责人 + 代码质量审计员
- 目标：
  - 对 `M3` 的通知、权限、订阅和渠道骨架做最终裁决
- 执行要求：
  1. 明确通过 / 不通过。
  2. 明确阻塞 / 非阻塞。
  3. 明确哪些外部渠道因环境或配置原因仅完成占位。
  4. 明确强权限隔离是否真的在后端生效。
  5. 不允许用截图代替后端权限证据。
- 验收输出：
  - `docs/quality/mainline-15-m3-alert-notification-and-access-acceptance-YYYY-MM-DD.md`

## 五、执行顺序

1. 先做通知平台最小落地。
2. 再做多渠道触达最小骨架。
3. 再做强权限隔离。
4. 再做订阅策略最小实现。
5. 最后做运行态刷新、真实验收与质量归档。

## 六、子智能体模型分配规则

本阶段必须按“任务类型 + 风险等级 + 产出要求”逐任务分档。

禁止事项如下：

1. 禁止沿用主线程当前模型。
2. 禁止沿用上一个子智能体配置。
3. 禁止整条主线统一使用同一 `model + thinking`。
4. 禁止省略 `model`、`thinking`、`reason`。

### 6.1 子智能体模型分配表

| 任务/角色 | 任务分类 | model | thinking | reason |
| --- | --- | --- | --- | --- |
| 通知平台最小落地 | 实现型任务 | `gpt-5.5` | `high` | 涉及事件触发链路、状态流转、可能的表结构与现有系统消息复用，属于跨模块后端高风险实现 |
| 多渠道触达最小骨架 | 实现型任务 | `gpt-5.5` | `high` | 涉及统一派发接口、失败回执、渠道扩展位和后续兼容性，判断空间大 |
| 强权限隔离 | 实现型任务 | `gpt-5.5` | `high` | 涉及后端真实权限边界、查询与操作隔离、潜在越权风险，必须高档位 |
| 订阅策略最小实现 | 实现型任务 | `gpt-5.4` | `medium` | 以前后端字段、页面表单和优先级规则为主，复杂度中等，但仍需严守权限边界 |
| 运行态刷新与环境核查 | 运维型任务 | `gpt-5.4` | `low` | 主要是固定步骤执行、端口检查、容器状态和日志回传 |
| 真实验收与质量归档 | 验收型任务 / 审计归档型任务 | `gpt-5.5` | `medium` | 需要跨代码、运行态、浏览器与权限证据做正式裁决，不能低档位草率收口 |

### 6.2 分档依据

#### 实现型任务

1. 后端通知、权限、渠道派发链路，默认使用 `gpt-5.5 + high`。
2. 订阅配置页面与字段联动，默认使用 `gpt-5.4 + medium`。
3. 若订阅策略升级为跨模块复杂归因或出现权限穿透争议，应升档到 `gpt-5.5 + high`。

#### 验收型任务

1. 涉及正式“通过 / 不通过”“阻塞 / 非阻塞”裁决，默认使用 `gpt-5.5 + medium`。

#### 运维型任务

1. 运行态刷新、容器检查、日志摘录、端口与健康检查，默认使用 `gpt-5.4 + low`。

#### 审计/归档型任务

1. 质量报告、权限裁决、触达闭环审计，默认使用 `gpt-5.5 + medium`。

### 6.3 升档条件

满足任一条即可考虑升档：

1. 需要同时理解 2 个及以上模块的真实调用链。
2. 涉及数据库 schema、Flyway migration、权限模型、安全边界、租户隔离。
3. 需要在多个候选根因之间做排除式判断。
4. 需要输出可作为上线、阻塞、通过/不通过裁决依据的正式结论。

### 6.4 降档条件

满足多数条即可考虑降档：

1. 任务步骤固定、判断空间小。
2. 只需核对单一文件、单一页面、单一接口。
3. 输出只是状态回传、截图证据、端口检查、构建结果、日志摘录。

### 6.5 本主线模型分配经验记录

1. 通知派发链路、权限隔离和渠道适配类任务，不应低于 `gpt-5.5 + high`。
2. 订阅页表单与普通页面联动，不应错误升到与权限审计同档位，`gpt-5.4 + medium` 通常足够。
3. 运维刷新与验收裁决必须分档，不能统一成同一高推理档位。
4. “因为主线程当前是 5.4/5.5，所以子智能体全部跟随”的做法在本主线明确判定为不合格。

### 派工单 4：订阅策略最小实现

- `任务名称=预警订阅策略最小实现`
- `目标=建立角色默认订阅与用户覆盖机制，并与通知渠道和权限边界打通`
- `范围=订阅配置字段、订阅查询与保存接口、前端最小配置界面`
- `禁止事项=禁止让用户通过订阅扩大越权范围；禁止建设复杂运营后台`
- `model=gpt-5.4`
- `thinking=medium`
- `reason=以字段设计、页面联动和优先级规则为主，复杂度中等`
- `验收输出=订阅字段清单、优先级规则、页面变更说明、前端验证结果`

### 派工单 5：真实验收与质量归档

- `任务名称=M3 真实验收与归档裁决`
- `目标=验证通知、权限、订阅和渠道骨架是否真实闭环，并出具正式验收报告`
- `范围=运行态刷新、真实账号验证、浏览器验收、接口证据、质量报告`
- `禁止事项=禁止只依据代码 diff 下结论；禁止缺少权限穿透验证就判定通过`
- `model=gpt-5.5`
- `thinking=medium`
- `reason=需要综合代码、运行态、浏览器和权限证据形成正式裁决`
- `验收输出=验收报告路径、通过/不通过、阻塞/非阻塞、剩余风险`

## 九、验收标准

`M3` 判定通过需同时满足：

1. 预警生成或状态流转已能触发站内通知。
2. 通知接收对象具备最小可解释计算口径。
3. 系统已形成统一派发接口和渠道占位，不再只有页面内列表。
4. 用户只能看到自身权限范围内的预警，且后端接口已做真实校验。
5. 详情查看、状态更新、批量处理均不能绕过权限边界。
6. 已形成角色默认订阅 + 用户覆盖的最小订阅策略。
7. 订阅范围不能突破权限范围。
8. 后端相关测试、前端静态验证、运行态复验通过。
9. 真实浏览器与接口证据已覆盖通知、订阅和权限场景。
10. 质量归档报告完成，并明确哪些外部渠道仍为占位接入。

## 十、阻塞项

执行中若命中以下任一情况，必须停止当前实现并回报主线程裁决：

1. 需要重构全系统权限底座后才能继续。
2. 需要先建设完整消息中台后才能落地预警通知。
3. 需要一次性接入多个外部正式发送渠道且依赖缺失。
4. 需要把订阅策略扩成通用规则编排系统。
5. 需要突破现有用户、角色、项目归属数据模型才能判断接收人。

## 十一、风险点

1. 越权风险：
   - 若权限隔离只做在前端，真实接口仍可能泄露数据。
2. 误触达风险：
   - 若接收人计算口径错误，会把预警发给不相关角色或漏发给责任人。
3. 过度抽象风险：
   - 若多渠道骨架被做成通用消息平台，`M3` 会失控扩包。
4. 订阅与权限冲突风险：
   - 若允许用户通过订阅扩大范围，会直接破坏强权限隔离。
5. 环境依赖风险：
   - 外部渠道凭据、租户配置、回调地址等若缺失，只能先完成站内闭环和占位接入。

## 十二、通过 / 不通过判断

### 通过

1. 站内通知已真实可用。
2. 强权限隔离已在后端接口生效。
3. 订阅策略可用且未越权。
4. 多渠道已具备统一派发骨架。
5. 验收报告能清楚区分：
   - 已完成能力
   - 占位能力
   - 后续待扩展渠道

### 不通过

1. 仍停留在“页面里能看，没有主动触达”。
2. 权限仍只是默认筛选，没有后端真实隔离。
3. 订阅策略只写在文档里，没有真正进入代码与运行态。
4. 多渠道只有口头预留，没有统一派发接口。
5. 把 `M3` 做成全站消息平台或全局权限体系重构，导致无法收口。

## 十三、阶段关闭口径

`M3` 关闭时必须同步输出以下内容：

1. 最终结论：通过 / 不通过。
2. 阻塞判断：阻塞 / 非阻塞。
3. 本阶段已完成能力。
4. 本阶段仍为占位接入的能力。
5. 后续专项建议清单。
6. 质量归档文档路径。

## 十四、主负责人建议

1. 按“通知平台 -> 多渠道骨架 -> 强权限隔离 -> 订阅策略 -> 真实验收”的顺序串行推进。
2. 第一优先级不是追求渠道数量，而是先把站内通知和后端权限边界做实。
3. 若现有仓库已有消息/通知模块，优先复用，避免为预警中心单独再造一套平行体系。
4. `M3` 关闭前，必须清楚记录哪些外部渠道只是占位，哪些已经真实打通，避免验收口径失真。
