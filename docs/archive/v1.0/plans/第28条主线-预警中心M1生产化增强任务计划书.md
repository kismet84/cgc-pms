# 第28条主线-预警中心 M1 生产化增强任务计划书

**Goal:** 将生产级增强总计划 `7.2 P0-2：预警中心 M1 生产化增强` 落为第28条主线执行计划，把预警中心从“能看基础列表”补齐为“后端分页、可筛选、可按角色默认聚焦、可处理、可批量已读、可业务跳转、可验收留痕”的生产化工作台。  
**Architecture:** 以现有预警模块为基线做最小增强，优先复用 `AlertController`、`AlertEvaluationService.page(...)`、既有前端页面和 store；先核对 `AlertLog` 与 Flyway 字段，再补缺失字段和兼容映射；不重建规则治理中心，不引入通知平台或多渠道触达，不扩大到 M2 规则治理中心。

## 结论优先

第28条主线不是从零建设预警中心。当前后端已经具备预警分页、筛选、已读、处理状态更新和批量评估等基础能力，本次应围绕总计划 7.2 的生产化缺口推进：

1. 保留并复用现有后端接口能力，前端可封装为 `page`、`batch-read`、`evaluate` 语义；只有在兼容成本低且能减少前端特殊分支时，才增加轻量兼容别名。
2. 先核对 `AlertLog` 与 Flyway 字段，缺什么补什么，不盲目重复创建 `alert_domain`、`biz_type`、`biz_id`、`triggered_at`、`handled_status`、`handled_by`、`handled_at`。
3. 重点补齐分类字段展示、业务跳转、采购交期逾期规则、角色默认筛选、前端体验和验收证据闭环。
4. 总工程师继续冻结，不参与默认筛选；生产经理按已有现场近似数据降级处理。

## 背景与真实现状

生产级增强总计划要求将预警中心增强为“可筛、可分页、可按角色默认聚焦、可处理、可跳转”的工作台。结合当前仓库已核对事实，本主线基线如下：

1. `AlertController` 当前已有 `GET /alerts`、`PUT /alerts/{id}/read`、`PUT /alerts/batch/read`、`PUT /alerts/{id}/status`、`PUT /alerts/batch/status`、`POST /alerts/batch-evaluate`。
2. `AlertEvaluationService.page(...)` 已支持 `tenant`、`projectId`、`ruleType`、`alertDomain`、`severity`、`isRead`、`triggeredStart`、`triggeredEnd`、`processStatus`，并做角色、项目、域访问范围过滤。
3. 总计划建议字段包括 `alert_domain`、`biz_type`、`biz_id`、`triggered_at`、`handled_status`、`handled_by`、`handled_at`；实施前必须核对现有 `AlertLog` 实体和 Flyway migration，避免重复字段、重复语义和迁移冲突。
4. 前端重点文件为 `frontend-admin/src/pages/alert/index.vue`、`frontend-admin/src/api/modules/alert.ts`、`frontend-admin/src/types/alert.ts`、`frontend-admin/src/stores/alert.ts`。
5. 现有分页与权限范围不是主要缺口，主要缺口是总计划要求中的分类字段可用性、采购交期逾期规则、业务跳转体验、角色默认筛选落地和完整验收留痕。

## 目标与非目标

### 目标

1. 让预警列表使用后端分页结果，展示 `total`、`pageNo`、`pageSize`、`records` 并保证翻页总数准确。
2. 在前端补齐项目、严重度、已读状态、规则类型、预警分类、触发时间范围等最小筛选，并支持用户覆盖默认筛选。
3. 按角色默认聚焦预警范围，多角色按并集处理。
4. 落地采购交期逾期预警，基于采购订单、计划交期、收货状态、入库状态生成记录，并具备去重策略。
5. 支持批量已读和处理状态更新，保留操作审计。
6. 支持可定位预警跳转到业务单据或目标页，不可定位预警不得伪造跳转。
7. 补齐后端测试、前端静态验证、真实浏览器验收和质量归档报告。

### 非目标

1. 不建设规则治理中心 M2，不提供规则可视化配置、规则编排或规则版本管理。
2. 不建设通知平台，不引入短信、邮件、企业微信、飞书、多渠道触达。
3. 不重构预警中心整体架构，不拆分大模块服务边界。
4. 不重写权限体系、租户体系或项目访问范围解析。
5. 不扩展总工程师默认筛选；总工程师继续冻结。
6. 不把生产经理现场规则强行做成完备模型，本阶段按已有现场近似数据降级处理。

## 影响范围

### 后端

1. `AlertController`：原则上复用现有接口，必要时增加兼容别名而不移除现有端点。
2. `AlertEvaluationService`：补齐采购交期逾期规则、字段写入、去重和处理状态语义。
3. `AlertLog` 与 Mapper：先核对现有字段，再补缺失字段或映射。
4. Flyway migration：仅在字段确实缺失时新增最小迁移。
5. 后端测试：覆盖分页筛选、角色范围、采购逾期规则、批量已读、处理状态和业务跳转字段。

### 前端

1. `frontend-admin/src/pages/alert/index.vue`：列表、筛选、分页、批量操作、处理状态、跳转入口和验收体验。
2. `frontend-admin/src/api/modules/alert.ts`：基于现有接口封装 page、batch read、status update、evaluate 语义。
3. `frontend-admin/src/types/alert.ts`：补齐分类、业务定位、处理状态、分页参数和响应类型。
4. `frontend-admin/src/stores/alert.ts`：默认筛选、用户覆盖、批量操作后的状态刷新。

### 文档与验收

1. 质量归档报告记录后端、前端、浏览器验收结果。
2. 验收证据记录真实浏览器截图、接口请求结果、测试命令输出和失败修复过程。

## 阶段拆解

### M1.1 基线复核与字段差异清单

目标：确认现有预警能力和字段现状，形成最小改动清单。

执行要点：

1. 复核 `AlertController` 已有端点，不删除、不改坏既有调用方。
2. 复核 `AlertEvaluationService.page(...)` 已支持的筛选参数和权限范围过滤。
3. 复核 `AlertLog`、Mapper、DTO、Flyway migration 中是否已有 `alert_domain`、`biz_type`、`biz_id`、`triggered_at`、`handled_status`、`handled_by`、`handled_at` 或等价字段。
4. 输出字段差异清单，明确“已存在直接复用”“已存在但需兼容映射”“缺失需迁移补齐”三类。

退出条件：

1. 后端接口和字段现状有代码证据。
2. 不存在盲目重复建字段的方案。
3. 前后端字段命名和接口语义已对齐。

### M1.2 后端生产化补齐

目标：在不重建预警模块的前提下补齐采购交期逾期、业务定位、处理状态和审计闭环。

执行要点：

1. 复用 `GET /alerts` 作为分页查询能力，必要时在前端 API 层命名为 `pageAlerts`；如确需兼容总计划端点，可增加 `GET /alerts/page` 别名并指向同一服务逻辑。
2. 复用 `PUT /alerts/batch/read` 作为批量已读能力；如确需兼容总计划端点，可增加 `POST /alerts/batch-read` 别名并保持同一审计链路。
3. 复用 `POST /alerts/batch-evaluate` 作为批量评估能力；如确需兼容总计划端点，可增加 `POST /alerts/evaluate` 别名并保持同一服务逻辑。
4. 新增或补齐采购交期逾期规则：数据源为采购订单、计划交期、收货状态、入库状态；触发条件为超过计划交期且未完成收货或入库。
5. 去重策略采用同一采购订单、项目、规则类型、日期周期内不重复生成。
6. 为可定位预警写入业务定位字段，无法定位时保持空值并由前端禁用跳转。
7. 处理状态更新必须记录处理人、处理时间和处理备注，保留现有审计能力。

退出条件：

1. 分页、筛选、处理状态和批量已读保持可用。
2. 采购交期逾期规则可基于真实数据源生成记录。
3. 业务跳转字段具备可追踪来源，不伪造目标。

### M1.3 前端工作台增强

目标：将预警中心页面补齐为可用工作台，支持后端分页、筛选、默认聚焦、批量处理和跳转。

执行要点：

1. `alert.ts` API 层统一封装分页查询、批量已读、状态更新和批量评估，不把端点差异扩散到页面。
2. `types/alert.ts` 补齐分页参数、筛选参数、预警分类、业务定位、处理状态、处理人和处理时间等类型。
3. `stores/alert.ts` 管理默认筛选和用户覆盖状态，避免页面内散落角色判断。
4. `index.vue` 使用后端分页，不做前端切片；分页切换、pageSize 切换和筛选提交都重新请求后端。
5. 页面筛选至少覆盖项目、严重度、已读状态、规则类型、预警分类、触发时间范围。
6. 角色默认筛选：
   - 超级管理员：全部。
   - 项目经理：全部。
   - 采购经理：采购类，优先采购交期逾期。
   - 商务经理：成本、合同、付款、变更。
   - 生产经理：按已有现场近似数据降级处理。
   - 总工程师：继续冻结，不参与默认筛选。
   - 多角色：按并集生成默认范围。
7. 用户手工修改筛选后，页面必须以用户选择为准，刷新时保留本次工作台上下文。
8. 可定位预警提供明确跳转按钮，不可定位预警展示原因或禁用态。

退出条件：

1. 页面无前端切片分页。
2. 默认筛选可见、可覆盖、可恢复。
3. 批量已读、状态更新和业务跳转在真实浏览器中可验收。

### M1.4 测试、验收与质量归档

目标：用后端测试、前端静态验证和真实浏览器验收形成可收口证据。

执行要点：

1. 后端补充或调整测试，覆盖分页总数、筛选组合、角色范围、采购逾期生成、去重、批量已读、处理状态和业务定位字段。
2. 前端执行类型检查、lint 或项目当前可用的轻量静态验证。
3. 真实浏览器验收预警中心页面：默认筛选、手工筛选、分页、批量已读、处理状态、可跳转和不可跳转两类预警。
4. 输出质量归档报告，包含命令、结果、截图或文字证据、残余风险和上线裁决建议。

退出条件：

1. 总计划 7.2 的 12 条验收标准全部有证据。
2. 失败项已修复或明确列为阻塞，不以口头说明替代证据。

## 任务分类

### 实现型任务

1. 后端字段差异补齐、采购交期逾期规则、业务定位和处理状态闭环。
2. 前端 API、类型、store、页面交互和跳转体验。

### 验收型任务

1. 后端测试与前端静态验证。
2. 真实浏览器验收和证据采集。

### 审计/归档型任务

1. 字段差异清单复核。
2. 质量归档报告和上线裁决证据整理。

### 运维型任务

1. 本主线默认不涉及环境刷新、发布配置、CI 门禁或运行态变更。
2. 如验收需要本地重建前后端，应单独派发运维刷新任务，不与实现任务混在同一子智能体中。

## 验收标准映射

1. 预警列表已改为后端分页，翻页和总数准确：用接口响应和页面翻页证据证明 `total`、`pageNo`、`pageSize`、`records` 一致。
2. 现有规则不回退：通过后端测试和批量评估记录证明既有规则仍可生成或查询。
3. 采购交期逾期预警已按真实数据源落地，并可生成记录：用采购订单、计划交期、收货/入库状态的测试数据证明。
4. 页面支持规则类型、分类、严重度、已读状态、时间范围等最小筛选：用浏览器筛选结果和请求参数证明。
5. 角色默认筛选正确生效，多角色按并集处理：用不同角色账号或模拟权限上下文证明默认筛选范围。
6. 用户可手工覆盖默认筛选：用浏览器操作证明用户筛选优先于默认筛选。
7. 批量已读可用：用勾选多条预警、调用批量已读、刷新后状态变化证明。
8. 可定位预警可跳转到对应业务单据或目标页：用可定位预警跳转截图或路由记录证明。
9. 后端测试通过：记录 Maven 测试命令、结果和失败修复结论。
10. 前端静态验证通过：记录 `pnpm` 类型检查、lint 或项目当前可用静态验证命令结果。
11. 真实浏览器验收通过并保留证据：记录浏览器路径、账号角色、操作步骤、截图或文字证据。
12. 质量归档报告完成：记录正式归档文件路径和主负责人验收结论。

## 风险与回滚

### 主要风险

1. 字段重复风险：总计划字段可能已由既有 migration 或实体字段覆盖，实施前必须先做字段差异清单。
2. 权限范围风险：默认筛选不能替代后端租户、项目、域访问范围过滤，任何新增查询必须继续使用现有范围控制。
3. 规则误报风险：采购交期逾期必须以真实采购、计划交期、收货和入库状态为依据，不能用前端展示状态推断。
4. 跳转误导风险：不可定位预警不得伪造业务链接，避免用户进入错误单据。
5. 多角色默认筛选风险：多角色按并集处理，不能因角色优先级导致用户看不到应看的预警。
6. 生产经理降级风险：现场近似数据口径需要在页面和报告中明确，不应伪装为完整生产现场规则。

### 回滚策略

1. 后端字段迁移如引发兼容问题，优先回滚新增字段读写入口和接口别名，保留既有端点。
2. 采购逾期规则如误报，先停用该规则生成入口或从批量评估中临时排除该规则，再保留已生成记录供审计。
3. 前端工作台如出现分页或筛选阻断，回滚到原列表展示，同时保留后端接口不变。
4. 业务跳转如定位错误，立即禁用对应跳转按钮，保留预警文本和处理能力。

## 收口证据

第28条主线收口时必须具备以下证据：

1. 字段差异清单：说明 `AlertLog` 和 Flyway 中每个目标字段的现状与处理方式。
2. 后端测试结果：覆盖分页、筛选、角色范围、采购逾期、去重、批量已读、处理状态和业务定位。
3. 前端静态验证结果：类型、lint 或项目当前认可的静态验证命令。
4. 真实浏览器验收记录：默认筛选、手工筛选、分页、批量已读、处理状态、可跳转和不可跳转预警。
5. 质量归档报告：包含通过/不通过、阻塞/非阻塞、上线风险、回滚策略和剩余问题。
6. 主负责人最终裁决：明确是否达到生产化增强 M1 收口标准。
