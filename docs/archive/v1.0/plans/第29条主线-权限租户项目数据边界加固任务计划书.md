# 第29条主线-权限租户项目数据边界加固任务计划书

**Goal:** 将总计划 `7.3 P0-3：权限、租户、项目数据边界加固` 落为可执行主线，按“现状审计 -> 高风险最小加固 -> 统一断言与审计复用 -> 隔离验收 -> 归档裁决”五阶段推进，确保核心接口不依赖前端隐藏按钮做权限控制，并以最小实现方式补齐后端权限、租户、项目、业务状态校验闭环。  
**Architecture:** 以仓库现有能力为基线推进最小加固，优先复用 `ProjectAccessChecker`、`AlertAccessScopeResolver.assertProjectAccess/assertAlertAccess`、`OperationAuditEvent`、`OperationAuditService`、`OperationAuditController` 等既有断言与审计链路，先盘点高风险 Controller/Service 的真实缺口，再对最核心接口做后端 fail-close 加固；不重做完整 RBAC，不把角色默认筛选当权限，不一次性重构所有服务，不修改已应用 Flyway，不扩 P1 主链路回归。

## 结论优先

第29条主线不是建设新的权限平台，也不是把全仓所有接口一次性重构。当前仓库已经存在局部可复用的数据边界能力：

1. `AlertAccessScopeResolver` 已具备 `assertProjectAccess`、`assertAlertAccess`、角色域限制和可访问项目集合解析，说明项目级访问控制已有可复用实现样式。
2. `ProjectAccessChecker` 已被合同、工作流、成本、文件等调用，说明项目访问校验已有共享入口，不应再在每个模块重复发明一套。
3. `OperationAuditEvent` 与 `OperationAuditService` 已具备异步审计落库能力，`OperationAuditController` 已按 `tenantId` 过滤查询，说明关键操作留痕已有链路，不需要新建审计平台。

因此本主线应采用最小可行方案：

1. 先做高风险接口现状审计，冻结“首批必须加固”的 Controller / Service / 操作清单。
2. 只对高风险接口补后端硬校验，不把整仓改造扩成全量权限治理工程。
3. 在 Service 层收敛 `assertTenantAccess`、`assertProjectAccess`、`assertBusinessStatus`、`assertApprovalEditable` 等统一断言复用方式，避免 Controller 零散补丁。
4. 复用现有操作审计能力补齐关键变更留痕，不重做日志体系。
5. 最终以横向越权、项目隔离、非法状态流转和审计可追踪四类证据做通过/不通过裁决。

## 背景与当前基线

总计划 `7.3 P0-3` 已明确要求：生产环境不能依赖前端隐藏按钮做权限控制，所有核心接口必须在后端完成权限、租户、项目、业务状态校验，验收标准为：

1. 核心接口无裸权限。
2. 横向越权测试不通过。
3. 项目级数据隔离有效。
4. 状态非法流转被拒绝。
5. 审计日志可追踪关键操作。

结合当前代码基线，本主线以以下事实为出发点：

1. 预警域已有较完整的角色域与项目访问控制样式，可作为后续模块复用样板，而不是另起一套权限模型。
2. `ProjectAccessChecker` 已被多业务域调用，但调用覆盖和测试覆盖并不均匀，首要任务是盘点“哪些模块已接入、哪些关键接口仍裸奔”。
3. 审计链路已存在且查询侧已带租户过滤，因此本主线只需补齐关键写操作事件，不需要引入新的审计存储。
4. 当前最大风险不是“有没有权限概念”，而是核心详情、编辑、提交流程、审批流转、文件访问等高风险链路是否真的在后端 fail-close。

## 目标与非目标

### 目标

1. 形成核心 Controller / Service 权限、租户、项目、业务状态校验盘点清单。
2. 冻结首批高风险接口加固范围，并完成最小后端补强。
3. 复用现有共享校验与审计能力，减少散落在 Controller 的重复判断。
4. 形成横向越权、项目隔离、非法状态流转、审计追踪的正式验收证据。
5. 输出可供主负责人直接裁决的收口材料。

### 非目标

1. 不重做完整 RBAC。
2. 不把角色默认筛选当权限能力。
3. 不一次性重构所有服务。
4. 不修改已应用 Flyway；如确有 schema 缺口，只允许新增版本化 migration，并且仅在主线程后续明确派发实现任务时执行。
5. 不扩展 `7.4 P1-1` 的主链路回归范围。
6. 不建设新的权限中心、审计中心、租户平台或项目授权平台。
7. 不以补前端隐藏、禁用按钮或路由守卫代替后端鉴权。

3. 文件与审批相关的高风险跨对象访问链路。

## 阶段拆解

### M1 现状审计与风险清单

目标：冻结首批高风险接口和真实缺口，避免后续实现盲扫全仓。

执行要点：

1. 逐域盘点核心 Controller / Service，检查以下五项：
   - 是否有 `@PreAuthorize`
   - 是否校验 `tenantId`
   - 是否校验 `projectId`
   - 是否校验业务状态
   - 是否存在越权查询、越权更新、IDOR 风险
2. 优先梳理既有共享能力：
   - `ProjectAccessChecker`
   - `AlertAccessScopeResolver`
   - 审批流中的项目角色/租户校验链路
   - `OperationAuditEvent` / `OperationAuditService`
3. 输出三类清单：
   - 已具备后端硬校验，可直接复用
   - 有局部校验但不完整，需要最小补强
   - 裸权限或明显越权风险，必须进入 M2
4. 冻结首批高风险接口包，控制在“少量核心详情/变更/审批/文件链路”，不扩成全量改造。

退出条件：

1. 有明确的高风险接口清单。
2. 每个接口缺口有代码证据。
3. 已区分工具配置/环境前置/真实质量安全问题，不把所有异常都归因为代码问题。

### M2 高风险接口最小加固

目标：对 M1 冻结的高风险接口做后端最小加固，消除裸权限与横向越权主风险。

执行要点：

1. 优先在共享 Service 或共享校验入口修复根因，避免逐个 Controller 打补丁。
2. 详情查询类接口至少补齐：
   - `tenantId` 归属断言
   - `projectId` 访问断言
   - 对象主键归属校验
3. 状态变更类接口至少补齐：
   - 业务状态合法流转断言
   - 审批中/已完成/已作废等不可编辑态拦截
4. 文件、审批、业务单据深链访问优先做 fail-close，不允许“查不到关系时默认放行”。
5. 仅在确有必要时新增最小迁移；不得修改已应用 Flyway。

退出条件：

1. 首批高风险接口无裸权限。
2. 横向越权与明显 IDOR 主路径已被后端拒绝。
3. 业务状态非法流转主路径已被拒绝。

### M3 Service 层统一断言与审计复用

目标：把 M2 中证明有效的校验收敛到共享断言和审计复用方式，降低重复实现和遗漏概率。

执行要点：

1. 按实际代码现状收敛以下断言能力的复用入口：
   - `assertTenantAccess`
   - `assertProjectAccess`
   - `assertBusinessStatus`
   - `assertApprovalEditable`
2. 能复用 `ProjectAccessChecker` 或现有模块断言的地方不再重写新 helper。
3. 对关键写操作复用 `OperationAuditEvent` 和 `OperationAuditService` 记审计，不新建独立日志方案。
4. 明确哪些模块继续沿用局部断言，哪些模块已经可上升为共享模式，形成后续扩展基线。

退出条件：

1. 首批实现范围内的共享断言入口清晰。
2. 关键操作审计可追踪操作人、时间、对象、结果。
3. 没有把主线扩成大规模架构重构。

### M4 权限/租户/项目隔离测试

目标：以真实测试和只读证据证明 M2/M3 的加固有效。

执行要点：

1. 建立最小负向测试矩阵：
   - 管理员正向
   - 同租户有项目权限账号正向
   - 同租户无项目权限账号负向
   - 非参与人直连详情负向
   - 非法状态流转负向
2. 横向越权测试至少覆盖：
   - 查询他人对象
   - 更新他人对象
   - 借主键直接访问跨项目对象
   - 审批/文件/业务单据深链越权
3. 审计验证至少覆盖一条成功操作和一条拒绝操作的可追踪证据。
4. 明确失败分类：工具配置类、环境前置类、真实质量/安全类。

退出条件：

1. 验收标准五项均有证据。
2. 失败项已明确阻塞/非阻塞。
3. 不以口头说明替代测试或日志证据。

### M5 验收归档

目标：形成主负责人可直接使用的正式收口材料。

执行要点：

1. 归档现状审计报告、高风险接口整改清单、测试与浏览器/接口证据摘要。
2. 对每个遗留问题标注：
   - 是否阻塞
   - 影响模块
   - 是否可回滚
   - 是否应转入后续主线
3. 回看本轮子智能体分档是否合理，记录是否出现临时升档、补派或换模。
4. 输出通过/不通过、阻塞/非阻塞、剩余风险结论。

退出条件：

1. 正式归档文件齐全。
2. 主负责人可直接做收口裁决。

分档说明：

1. 本主线同时包含审计、实现、验收、归档四类任务，不能统一使用同一模型和同一推理强度。
2. 后端高风险加固必须升档到 `gpt-5.5 + high`，因为它直接触及权限、安全、租户隔离和数据一致性。
3. 测试验收使用 `gpt-5.5 + medium`，因为其输出会直接作为通过/不通过裁决依据。
4. 盘点与归档保持 `gpt-5.4 + medium`，符合“多证据整理但不直接实施高风险改动”的任务特征。

## 验收标准

### 通过标准

以下条件全部满足，方可判定第29条主线通过：

1. 核心接口盘点清单完整，且已标出是否有 `@PreAuthorize`、`tenantId`、`projectId`、业务状态校验。
2. 首批高风险接口无裸权限。
3. 横向越权主路径测试不通过，不能通过主键直连访问或修改他人/他项目对象。
4. 项目级数据隔离有效，同租户无项目权限账号不能访问目标项目核心数据。
5. 非法状态流转被拒绝，审批中、已完成、已作废等不可编辑态具备后端硬拦截。
6. 关键写操作审计可追踪操作人、操作时间、操作对象和结果。
7. 已输出正式验收归档材料，能支撑主负责人直接裁决。

### 不通过标准

存在任一情况即判定不通过：

1. 仍存在核心接口仅靠前端隐藏按钮控制权限。
2. 仍可通过主键直连跨项目读取或更新对象。
3. 业务状态非法流转仍可绕过后端校验。
4. 关键写操作无法回溯审计。
5. 首批高风险接口范围失控，扩成全仓重构而未形成可收口结果。
6. 没有正式证据，仅有口头说明或零散日志摘录。

## 风险点

1. 范围膨胀风险：总计划覆盖业务域很多，若 M1 不冻结首批高风险接口，M2 会迅速膨胀为全量权限治理。
2. 断言分散风险：若在每个 Controller 各自补判断，短期看似完成，长期仍会遗漏兄弟路径。
3. 审计缺口风险：只补权限不补关键写操作审计，最终无法满足“可追踪关键操作”的验收要求。
4. 历史数据兼容风险：个别老接口可能依赖宽松状态判断或缺少完整项目关系数据，实施时需先确认是代码缺口还是数据前置问题。
5. 测试矩阵失真风险：若只测管理员正向，不测同租户无权限账号与非法状态流转，无法证明真正收口。

## 回滚/止损策略

1. 若共享断言改动导致多模块连带异常，优先回滚共享入口的新校验，并保留首批高风险接口的局部保护，避免全站不可用。
2. 若某条状态断言引发历史数据兼容问题，先将其收缩为最小阻断范围，并把问题单列为阻塞项，不得整体放开。
3. 若新增审计事件影响主链路稳定性，先保留关键业务操作审计，暂停低优先级事件补录，避免因审计写入拖垮主流程。
4. 若验证发现问题主要来自环境前置或测试数据缺口，应先恢复环境或补齐数据前置，再复验，不得错误回滚正确的安全加固。

## 正式交付物

第29条主线收口时至少应交付以下正式产物：

1. 第29条主线任务计划书
2. M1 现状审计与高风险接口清单
3. M2/M3 高风险接口最小加固说明
4. M4 权限/租户/项目隔离测试验收结果
5. M5 正式质量归档或收口报告
6. 阻塞/非阻塞遗留问题清单
7. 分档复盘结论

## 重新分档检查节点

以下节点必须由主线程重新评估子智能体分档，而不是自动沿用上一阶段配置：

1. M1 完成进入 M2 前
2. 任一子智能体首次回报“不通过”或“阻塞”后
3. 任务边界从实现切换为验收/裁决时
4. 用户新增更高验收标准或要求扩大覆盖范围时

每次重分档至少重答四项：

1. 当前阶段属于实现、验收、运维还是审计
2. 是否已升级为跨模块问题
3. 是否涉及权限、安全、数据一致性
4. 输出是否将直接用于通过/不通过或上线裁决

## 主负责人裁决建议

1. 可进入执行：是。本主线已有总计划锚点和局部可复用能力，适合按“先盘点、再最小加固、再统一验收”的顺序推进。
2. 当前阻塞项：无预设阻塞项，但 M1 若无法冻结首批高风险接口范围，主线将失去收口边界。
3. 当前非阻塞风险：
   - 某些历史接口可能存在测试覆盖缺口；
   - 共享断言入口在不同模块的语义可能不完全一致；
   - 审计事件字段完整性可能需要补证据而不是立刻扩表。
4. 执行原则：优先修复共享根因，优先复用既有能力，优先收敛首批高风险链路，不做全量平台化改造。
