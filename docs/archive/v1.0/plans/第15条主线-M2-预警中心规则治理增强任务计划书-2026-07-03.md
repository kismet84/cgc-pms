## 主负责人结论

第 15 条主线可以正式进入 `M2`。

`M2` 的唯一定位是：**把当前已经落地的预警能力，从“能运行”提升到“口径稳定、可治理、可解释、可验收”**。

本阶段只处理以下 5 类事项：

1. 规则分类治理
2. 规则标签补齐
3. 抑制策略
4. 去重策略
5. 归档口径

本阶段不承担以下交付，但以下事项明确保留为后续阶段任务：

1. 通知平台
2. 多渠道触达
3. 强权限隔离
4. 通用规则引擎设计器
5. 更大范围的规则扩包与订阅机制

裁决口径如下：

1. 上述后续项不是本阶段交付物，也不作为 `M2` 当前阻塞项。
2. 上述后续项不是永久排除项，应在 `M3` 或后续专项中继续推进。
3. `M2` 必须坚持最小治理增强，不得借机扩成“预警平台重构”。

## 一、立项目标

`M2` 完成后，预警中心至少应达到以下状态：

1. 当前已落地规则具备稳定分类口径，不再依赖临时解释或页面侧兜底。
2. 当前核心规则具备最小标签治理能力，可支撑默认视图、统计和后续订阅策略。
3. 同类预警具备最小抑制 / 去重闭环，避免短时间重复轰炸。
4. 当前规则的关键阈值、时间窗、开关具备最小配置化能力，不再全部硬编码散落。
5. 已读、已处理、已归档、失效等状态口径被明确区分，不再混用。

## 二、纳入范围

### 2.1 本阶段纳入范围

1. 现有已落地规则的分类治理与标签清单收敛。
2. `PURCHASE / CONTRACT / COST / PAYMENT / VARIATION` 等业务域的最小规则分类矩阵。
3. 基于现有规则的抑制策略与去重策略。
4. 基于现有规则的最小配置化方案。
5. 预警状态流转、归档口径、报表口径的最小统一。
6. 与上述能力对应的后端测试、前端验证、真实页面复验和质量归档。

### 2.2 本阶段不作为交付项

1. 不建设通用规则设计器后台。
2. 不建设通知平台、订阅中心、消息中心。
3. 不接入短信、邮件、企微、站内信等多渠道触达。
4. 不把默认视图升级成强权限隔离模型。
5. 不一次性纳入所有未确认数据源的逾期类规则。
6. 不重做驾驶舱全部告警卡片。

说明：

1. 上述内容仅表示“不作为 `M2` 当前交付项”。
2. 上述内容不等于“永久不在计划范围内”。
3. 其中规则引擎、通知平台、多渠道触达、强权限隔离，统一归入后续最小增强任务。

## 三、治理原则

### 3.1 分类治理原则

1. 分类先服务当前已落地规则，不为未来假设规则预埋复杂层级。
2. 若某条规则缺乏稳定业务域或细分类标签，不得强行标成“已治理完成”，必须明确列入缺口清单。
3. 角色默认视图依赖规则标签，但规则标签本身不承担权限边界职责。

### 3.2 抑制与去重原则

1. 去重优先围绕“同项目 + 同规则 + 同业务对象 + 同时间窗”建立。
2. 抑制优先解决“短时间重复触发”的噪音问题，不先做复杂通知级节流系统。
3. 去重和抑制不得吞掉业务上应保留的不同预警事实。

### 3.3 配置化原则

1. 只配置当前已落地规则的关键阈值、开关、时间窗，不做通用编排平台。
2. 配置项必须有默认值、回退策略和缺省行为。
3. 配置化不能破坏现有真实样本和已验收规则的触发口径。

### 3.4 归档原则

1. 已读不等于已处理。
2. 已处理不等于已归档。
3. 归档不等于删除。
4. 若当前业务没有稳定“处理完成”来源，必须先定义最小归档口径，不能假装闭环完整。

## 四、分阶段交付包

### 交付包 A：规则分类与标签治理

- 执行类型：实现型任务
- 建议执行角色：后端工程执行子智能体
- 目标：
  - 收敛当前规则的分类、标签、角色映射依据
- 执行要求：
  1. 输出当前规则标签矩阵。
  2. 补齐可正式落地的业务域 / 细分类标签。
  3. 对暂时不能稳定落地的项输出“缺口清单 + 降级口径”。
  4. 不得为少数规则引入重型元数据平台。
- 验收输出：
  - 规则标签矩阵
  - 差异清单
  - 涉及实体、VO、枚举或 migration 的变更清单

### 交付包 B：抑制策略与去重策略最小闭环

- 执行类型：实现型任务
- 建议执行角色：后端工程执行子智能体
- 目标：
  - 为当前核心规则补最小可解释的去重与抑制策略
- 执行要求：
  1. 定义去重键。
  2. 定义时间窗口。
  3. 区分“同一事实重复触发”和“不同事实同时触发”。
  4. 至少覆盖采购交期逾期及一个非采购类已落地规则。
  5. 不得直接上升为通用规则引擎。
- 验收输出：
  - 去重键定义
  - 抑制窗口定义
  - 正反例样本

### 交付包 C：规则配置化最小实现

- 执行类型：实现型任务
- 建议执行角色：后端工程执行子智能体 / 数据库与迁移负责人
- 目标：
  - 将当前核心规则的关键参数从散落硬编码收敛为最小配置
- 执行要求：
  1. 只覆盖当前已落地规则。
  2. 配置字段最小集优先：
     - 是否启用
     - 时间窗口
     - 阈值或边界值
     - 默认回退值
  3. 不做复杂 UI 配置后台。
  4. 如涉及 migration，必须新增版本化脚本。
- 验收输出：
  - 配置字段清单
  - 默认值与回退策略
  - migration 与样本数据说明

### 交付包 D：归档口径与页面联动

- 执行类型：实现型任务
- 建议执行角色：前端工程执行子智能体
- 目标：
  - 将治理后的分类、状态和归档口径收敛到预警页面最小可用展示
- 执行要求：
  1. 页面只补 `M2` 必要展示，不重做布局体系。
  2. 分类、状态、归档口径展示必须与后端一致。
  3. 不得伪造“已处理 / 已归档”能力。
  4. 若后端暂只提供最小状态，前端按最小真实状态落地。
- 验收输出：
  - 页面变更说明
  - 定向测试结果
  - 真实页面复验关注点

### 交付包 E：口径复核与质量归档

- 执行类型：验收型任务 / 审计归档型任务
- 建议执行角色：产品/权限边界复核子智能体 + 测试验收员 + 代码质量审计员
- 目标：
  - 对 `M2` 的分类、去重、抑制、配置化、归档口径做最终裁决
- 执行要求：
  1. 明确通过 / 不通过。
  2. 明确阻塞 / 非阻塞。
  3. 明确哪些项进入 `M3`，哪些项仍留在 `M2` 未闭环。
  4. 不允许用“后续再看”掩盖当前口径缺陷。
- 验收输出：
  - `docs/quality/mainline-15-m2-alert-rule-governance-acceptance-YYYY-MM-DD.md`

## 五、执行顺序

1. 先做规则分类与标签治理。
2. 再做抑制策略与去重策略最小闭环。
3. 再做规则配置化最小实现。
4. 再做归档口径与页面联动。
5. 最后做口径复核、真实页面验收与质量归档。

## 六、子智能体模型分配规则

本阶段必须按“任务类型 + 风险等级 + 产出要求”逐任务分档。

禁止事项如下：

1. 禁止沿用主线程当前模型。
2. 禁止沿用上一个子智能体配置。
3. 禁止整条主线统一使用同一 `model + thinking`。
4. 禁止省略 `model`、`thinking`、`reason`。

### 6.1 子智能体模型分配表

| 任务/角色 | 任务分类 | model | thinking | reason |
| --- | --- | --- | --- | --- |
| 规则分类与标签治理 | 实现型任务 | `gpt-5.5` | `high` | 涉及后端规则、角色映射、已有数据口径和后续默认视图依据，判断复杂度高 |
| 抑制与去重最小闭环 | 实现型任务 | `gpt-5.5` | `high` | 涉及时间窗、业务对象键、误吞风险和跨规则边界 |
| 规则配置化最小实现 | 实现型任务 | `gpt-5.5` | `high` | 涉及 migration、默认值、回退逻辑和运行态兼容 |
| 归档口径与页面联动 | 实现型任务 | `gpt-5.4` | `medium` | 以前端联动为主，但需严格服从后端口径 |
| 口径复核与质量归档 | 验收型任务 / 审计归档型任务 | `gpt-5.5` | `medium` | 需要跨代码、运行态、报告证据做综合判断并形成正式裁决 |
| 运行态刷新与环境核查 | 运维型任务 | `gpt-5.4` | `low` | 判断空间小，以固定步骤执行和证据回传为主 |

### 6.2 分档依据

#### 实现型任务

1. 后端复杂规则治理、migration、数据口径、状态语义定义，默认使用 `gpt-5.5 + high`。
2. 前端页面展示与接口联动，默认使用 `gpt-5.4 + medium`；若升级为跨模块复杂归因，再考虑升档。

#### 验收型任务

1. 涉及真实页面、真实样本、最终通过/不通过裁决，默认使用 `gpt-5.5 + medium`。

#### 运维型任务

1. 运行态刷新、端口检查、容器状态核查、日志回传，默认使用 `gpt-5.4 + low`。

#### 审计/归档型任务

1. 质量报告、口径复核、阶段关闭结论，默认使用 `gpt-5.5 + medium` 或 `gpt-5.4 + medium`。
2. 只要输出需作为正式裁决依据，优先提升到 `gpt-5.5 + medium`。

### 6.3 升档条件

满足任一条即可考虑升档：

1. 需要同时理解 2 个及以上模块的真实调用链。
2. 涉及数据库 schema、Flyway migration、权限模型、安全边界、租户隔离、金额口径。
3. 需要在多个候选根因之间做排除式判断。
4. 需要输出可作为上线、阻塞、通过/不通过裁决依据的正式结论。

### 6.4 降档条件

满足多数条即可考虑降档：

1. 任务步骤固定、判断空间小。
2. 只需核对单一文件、单一页面、单一接口。
3. 输出只是状态回传、截图证据、端口检查、构建结果、日志摘录。

### 6.5 本主线模型分配经验记录

1. 规则治理、migration、状态机定义类任务，实际需要升档到 `gpt-5.5 + high`。
2. 运行态刷新、环境检查、命令执行型动作，可稳定降档到 `gpt-5.4 + low`。
3. 前端联动和普通文档归档不应与后端高风险治理使用同一档位。
4. “整条主线统一分档”被判定为不合理做法，后续阶段必须继续逐任务重评。

## 八、验收标准

`M2` 判定通过需同时满足：

1. 当前已落地规则具备稳定分类口径，角色默认视图不再依赖临时模拟或猜测。
2. 当前核心规则至少形成一套正式可解释的去重 / 抑制策略。
3. 当前核心规则的关键参数具备最小配置化能力。
4. 已读、已处理、已归档、失效等口径已明确区分，且不互相冒充。
5. 前端页面对分类、状态、归档口径的展示与后端一致。
6. 相关后端测试、前端静态验证、运行态复验通过。
7. 质量归档报告完成，并明确 `M2` 与 `M3` 的边界。

## 九、阻塞项

执行中若命中以下任一情况，必须停止当前实现并回报主线程裁决：

1. 需要把 `M2` 扩成通用规则引擎或设计后台。
2. 需要把默认筛选升级为强权限隔离。
3. 需要引入通知平台、多渠道消息发送或订阅中心。
4. 需要一次性补齐所有未确认数据源的逾期规则。
5. 需要跨多个业务域做大范围模型重构才能继续。

## 十、风险点

1. 标签误标风险：
   - 若业务域或细分类标错，会直接污染默认视图、统计和后续订阅策略。
2. 去重误吞风险：
   - 若去重键设计错误，会把不同业务事实合并成同一预警。
3. 配置化过度风险：
   - 若把少量参数抽成复杂配置体系，`M2` 会失控扩包。
4. 状态混用风险：
   - 若已读、已处理、已归档口径混用，后续报表和闭环统计会失真。
5. 边界漂移风险：
   - 若把 `M3` 的通知、权限、订阅建设提前带入，`M2` 将无法按最小治理增强收口。

## 十一、通过 / 不通过判断

### 通过

1. 已有规则标签体系稳定可解释。
2. 至少一类真实重复触发问题被正式治理。
3. 当前关键规则参数已完成最小配置化。
4. 归档口径明确，前后端一致。
5. 归档报告能清楚区分：
   - 已完成治理项
   - 未纳入 `M2` 的项
   - 应转入 `M3` 的项

### 不通过

1. 仍停留在“只有页面能看，没有治理口径”。
2. 去重 / 抑制没有真实闭环，只停留在口头原则。
3. 配置化只写在文档里，没有真正落到代码与运行态。
4. 状态仍混用，无法区分已读、处理、归档。
5. 把 `M2` 做成通知平台、权限平台或规则引擎预演。

## 十二、阶段关闭口径

`M2` 关闭时必须同步输出以下内容：

1. 最终结论：通过 / 不通过。
2. 阻塞判断：阻塞 / 非阻塞。
3. 本阶段已完成治理项。
4. 本阶段未纳入交付项。
5. 应转入 `M3` 或后续专项的任务清单。
6. 质量归档文档路径。

## 十三、主负责人建议

1. 按“规则标签治理 -> 去重抑制 -> 配置化 -> 归档口径 -> 质量归档”的顺序串行推进。
2. 第一优先级不是继续补更多规则数量，而是先把当前已落地规则治理稳定。
3. `M2` 关闭前，必须明确哪些缺口继续保留到 `M3`，不得模糊带过。
