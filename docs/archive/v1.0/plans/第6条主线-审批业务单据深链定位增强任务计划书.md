# 第 6 条主线：审批业务单据深链定位增强任务计划书

## 计划状态

正式版。

本计划先由主会话形成草案，现已完成产品/需求负责人、前端设计师、后端设计师、安全审计员、数据库与迁移负责人、测试验收员六方只读研判，并已按回传结论收敛为正式版。

## 一、正式结论

第 5 条主线已经完成审批详情“查看业务单据”入口 MVP：

1. 合同审批直达合同详情。
2. 采购申请审批跳采购申请列表。
3. 分包计量审批跳分包计量列表。

第 6 条主线在此基础上继续推进，但只做一件事：让采购申请审批、分包计量审批也能从审批详情按 `businessId` 精确定位到目标业务单据，并自动打开现有查看弹窗。

本主线不建设新的采购申请/分包计量独立详情体系，不新增 workflow 聚合接口，不改 workflow 后端契约，不改数据库结构，不扩展为审批中心全量导航重构。

## 二、范围与边界

本主线包含：

1. `approval/todo.vue` 与 `approval/detail.vue` 中采购申请、分包计量入口改为携带 `businessId` 查询参数。
2. 采购申请列表页读取 `businessId` 后，调用目标模块详情接口与明细接口，自动打开现有查看采购申请弹窗。
3. 分包计量列表页读取 `businessId` 后，调用目标模块详情接口，自动打开现有查看分包计量弹窗。
4. 成功或失败尝试后清理查询参数，避免刷新、分页、筛选后重复自动弹窗。
5. 合同审批现有 `/contract/${businessId}` 深链能力不回退。
6. 保持内嵌审批详情与独立审批详情页策略一致。
7. 补齐前端单测、构建验证、运行态刷新、真实浏览器复验和质量归档。

本主线不包含：

1. 不新增采购申请独立详情路由。
2. 不新增分包计量独立详情路由。
3. 不新增 workflow 业务详情聚合接口。
4. 不修改 workflow 的 `businessType/businessId` 返回契约。
5. 不新增字段、索引或 Flyway migration。
6. 不把项目级数据范围强化混入本主线。
7. 不处理付款申请、新审批类型、搜索/高级筛选、移动端、返回审批详情入口、面包屑增强或审批中心全页签回归。

## 三、冻结口径

### 1. 入口文案

统一为：`查看业务单据`

### 2. 入口策略

| 业务类型 | 第 6 条正式目标 | 说明 |
|---|---|---|
| 合同审批 | `/contract/${businessId}` | 保持第 5 条已通过链路不变 |
| 采购申请审批 | `/inventory/purchase-request?businessId=${businessId}` | 进入列表页后自动打开现有查看弹窗 |
| 分包计量审批 | `/subcontract/measure?businessId=${businessId}` | 进入列表页后自动打开现有查看弹窗 |

### 3. 一致性规则

1. 内嵌审批详情和独立审批详情页必须采用同一深链目标与同一失败口径。
2. 未识别 `businessType` 或缺少 `businessId` 时不展示入口。
3. 查询参数名统一使用 `businessId`，不改成 `id`、`requestCode` 或 `measureCode`。
4. 自动打开尝试后统一清理 `businessId` 查询参数。

### 4. 失败口径

1. 无权限：使用目标业务模块现有 403/404/空态或无权限表现，不泄露业务详情。
2. 单据不存在：使用目标模块现有“记录不存在 / 数据不存在”口径，或最小统一提示“业务单据不存在或已不可访问”。
3. 接口失败：使用目标模块现有加载失败口径，最小可接受提示为“业务单据加载失败，请稍后重试”。
4. 不要求审批中心侧预判权限，不做权限感知隐藏入口；那属于后续 UX 增强，不在本主线实现。

## 四、技术路线

正式版采用最小实现路径：

1. 审批详情入口只负责把 `businessId` 传到目标模块页面。
2. 采购申请页面读取 `businessId` 后，调用 `getPurchaseRequestDetail(id)` 与现有明细接口，构造现有查看弹窗所需数据并自动打开。
3. 分包计量页面读取 `businessId` 后，调用 `getMeasureDetail(id)` 或现有详情加载逻辑，自动打开现有查看弹窗。
4. 自动打开前必须经过目标模块详情接口，不允许只用 workflow 传来的 `businessId` 或未校验的列表快照直接渲染详情。
5. 合同审批继续沿用既有深链，不改合同详情链路。

不采用的方案：

1. 不新增采购申请/分包计量独立详情页。
2. 不新增 workflow 详情聚合业务详情返回。
3. 不新增数据库字段、索引、迁移或 workflow 额外 business 编号字段。

## 五、专业研判结论汇总

### 产品/需求负责人

1. 同意第 6 条聚焦“采购申请/分包计量按 ID 定位业务单据”。
2. 同意“列表页查询参数 + 自动打开现有查看弹窗”作为 MVP。
3. “返回审批详情”不列为硬性要求。
4. 本线只做更准确落到目标业务单据，不扩成新详情页体系建设。

### 前端设计师 / 前端架构助手

1. 采购申请页和分包计量页均已有 `handleView(record)` 查看弹窗能力，可复用。
2. 当前缺口在于审批详情没有携带 `businessId`，目标页也没有读取 query 自动打开逻辑。
3. 推荐查询参数名统一为 `businessId`。
4. 推荐自动打开尝试后用 `router.replace` 清理 query 参数。

### 后端设计师 / 后端架构助手

1. 采购申请已有 `GET /purchase-requests/{id}` 与明细接口。
2. 分包计量已有 `GET /sub-measures/{id}` 与明细接口。
3. workflow 已稳定返回 `businessType/businessId`，且采购申请与分包计量提交时写入的就是业务主键 ID。
4. 不需要改 workflow 后端，不需要新增 workflow 聚合接口。

### 安全审计员

1. workflow 详情权限不等于目标业务单据查看权。
2. 查询参数 `businessId` 只是可篡改输入，不能作为授权依据。
3. 打开查看弹窗前必须走目标模块详情接口。
4. 无权限、跨租户、单据不存在时必须不泄露标题、编号、明细、项目、合同、分包商等字段。

### 数据库与迁移负责人

1. `wf_instance.business_id` 已足以定位采购申请和分包计量。
2. 现阶段不需要新增字段、索引或 Flyway migration。
3. 当前数据库口径是数值型主键 ID，不是业务编号。
4. demo 数据已具备采购申请和分包计量深链样本，孤儿关联为 0。

### 测试验收员

1. 浏览器矩阵收敛为“2 条新链路 + 1 条旧链路回归 + 1 条只读抽查”。
2. 负向权限样本不是硬门禁，按“有则验、无则记风险”处理。
3. 本线的验收单位不是“到了正确模块”，而是“目标单据已被自动定位并呈现”。
