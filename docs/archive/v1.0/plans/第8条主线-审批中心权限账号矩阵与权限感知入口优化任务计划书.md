# 第 8 条主线：审批中心最小权限账号矩阵与权限感知入口优化任务计划书

## 计划状态

正式版。

本计划先由主会话形成草案，现已完成产品/需求负责人、前端设计师、后端设计师、数据库与迁移负责人、安全审计员、测试验收员、部署运维负责人、代码质量审计员八方只读研判，并已按回传结论收敛为正式版。

## 一、正式结论

第 8 条主线定位为“审批中心最小权限账号矩阵与权限感知入口优化”。

目标不是建立完整权限平台，也不是覆盖全角色、全租户、全 `data_scope`，而是在第 7 条审批中心生产化闭环增强之后，补齐同租户内最小可登录账号矩阵，并验证三类已冻结业务在审批详情入口上的权限表现：

1. 有目标业务权限时，入口可用且业务单据可达。
2. 缺目标业务权限时，入口禁用并提示，不误导为可访问。
3. 目标业务后端接口继续承担真实授权边界。
4. 只读页签和非参与人边界不回退。

本主线拆成 4 个串行小包：

1. 账号矩阵设计与 seed 必要性裁决
2. 最小 seed / 后端边界确认
3. 前端入口展示最小实现
4. 浏览器矩阵验收、安全复核与质量归档

## 二、正式范围

### 纳入本主线

1. 仅覆盖审批中心已冻结三类业务：
   - 合同审批
   - 采购申请
   - 分包计量
2. 建立同租户 `tenant=0` 下最小可登录账号矩阵：
   - `admin`：全权限正向账号，沿用 `admin/admin123`。
   - `workflow-only`：可见 workflow 详情，但缺合同、采购申请、分包计量目标业务查询权限。
   - `cc-readonly`：可见 `/approval/cc` 抄送详情，但不具备审批动作。
   - `non-participant`：同租户非参与人，不应看到无关审批详情。
3. 覆盖最小权限边界：
   - 同租户缺目标业务权限
   - workflow 非参与人不可见
   - `/approval/cc` 只读页签不误显动作
   - 目标业务入口无权限时不泄露合同、采购申请、分包计量详情字段
4. 优化审批详情入口展示：
   - 有目标业务访问权限时展示可点击 `查看业务单据`。
   - 识别业务类型且存在 `businessId`，但缺目标权限时展示禁用态 `查看业务单据`，提示 `无权访问该业务单据`。
   - 未识别业务类型或缺少 `businessId` 时不展示入口。
   - 如果运行时无法可靠判断目标权限，则保留入口并由目标模块拒绝，但不得在拒绝前渲染目标业务详情字段。
5. 形成真实浏览器证据与质量归档。

### 不纳入本主线

1. 不扩展付款申请或更多 workflow 业务类型。
2. 不重构全站 RBAC、菜单权限、数据权限体系。
3. 不引入第二租户样本。
4. 不做复杂 `data_scope` 全矩阵覆盖。
5. 不新增 workflow 业务详情聚合接口。
6. 不让 workflow 详情接口返回目标业务权限结果。
7. 不新增权限配置中心、入口注册表或后端业务授权字段。
8. 不做审批中心 UI 大改版。

## 三、冻结口径

### 1. 产品口径

1. 主线名称和范围按“最小账号矩阵”收口，不按“完整账号矩阵”理解。
2. 无目标业务权限时首选 `禁用并提示`，不采用直接隐藏作为默认口径。
3. 本主线只覆盖合同审批、采购申请、分包计量三类业务。
4. `todo.vue` 与 `detail.vue` 两处入口策略必须一致。
5. 最小提示文案为：`无权访问该业务单据`。

### 2. 前端口径

1. 前端复用 `frontend-admin/src/stores/user.ts` 中的 `permissions` / `hasPermission(code)`。
2. 三类业务入口和权限映射继续集中在 `frontend-admin/src/pages/approval/workflowDisplay.ts`。
3. 不复用 route meta 判断三类业务权限，因为当前合同、采购申请、分包计量路由未稳定配置目标详情权限。
4. 权限映射固定为：
   - `CONTRACT` / `CONTRACT_APPROVAL` -> `contract:query`
   - `PURCHASE_REQUEST` -> `purchase:request:list`
   - `SUB_MEASURE` -> `subcontract:measure:query`
5. 小包 3 最小修改范围优先控制在：
   - `frontend-admin/src/pages/approval/workflowDisplay.ts`
   - `frontend-admin/src/pages/approval/todo.vue`
   - `frontend-admin/src/pages/approval/detail.vue`
   - `frontend-admin/src/pages/approval/__tests__/ApprovalWorkList.test.ts`
   - `frontend-admin/src/pages/approval/__tests__/ApprovalConfirm.test.ts`

### 3. 后端口径

1. 不新增权限感知后端接口。
2. 不让 workflow 详情返回目标业务权限结果。
3. workflow 详情继续只返回 `businessType`、`businessId`、流程详情、workflow 动作集合等审批信息。
4. 目标业务详情权限继续由目标模块接口承担：
   - 合同：`contract:query`
   - 采购申请：`purchase:request:list`
   - 分包计量：`subcontract:measure:query`
5. 本主线只验目标业务权限码拒绝，不纳入项目级 `data_scope` 专项。

### 4. 数据库与迁移口径

1. 需要补最小 demo-only seed，支撑真实浏览器账号矩阵验收。
2. 不新增表、字段、索引。
3. 不新增 `sys_menu/perms` 定义，沿用现有权限码。
4. 新增范围限定为 demo-only 用户、角色、用户角色绑定、角色菜单绑定、必要 workflow 参与关系样本。
5. MySQL/H2 需要对等 migration；不得只补 MySQL。
6. 不引入第二租户或复杂 `data_scope` 样本。
7. `wf_cc` 无 `deleted_flag`，新增抄送 seed 必须按真实表结构写幂等逻辑。

### 5. 安全口径

1. 权限感知入口只是前端体验优化，不是授权边界。
2. 浏览器端隐藏、禁用或提示均不能替代目标业务接口鉴权。
3. 目标业务无权限时不得泄露目标业务标题、编号、金额、项目、供应商、分包商、明细等字段。
4. 采购申请、分包计量深链必须继续先调用目标模块详情接口，再打开查看弹窗；不得用 workflow 快照渲染目标详情。
5. 第二租户、复杂 `data_scope`、项目级强隔离浏览器硬验另列后续专项，不作为本线阻塞。

### 6. 测试口径

1. 采用 `3+1` 最小账号矩阵，不做账号 × 页签 × 三业务全排列。
2. `admin` 正向硬验三类业务入口可达。
3. `workflow-only` 负向账号对三类业务均形成真实负向证据，证明 workflow 可见但目标业务无权限不泄露详情字段。
4. `cc-readonly` 只验 `/approval/cc` 可见且无 `同意/驳回/撤回/重提`。
5. `non-participant` 只验至少 1 条无关审批不可见或详情接口拒绝。
6. 跨租户和复杂 `data_scope` 继续静态安全复核兜底。
7. 证据目录建议为：`D:/projects-test/cgc-pms/output/playwright/approval-center-permission-matrix-acceptance-20260701/`

### 7. 运维口径

1. 若新增 migration/seed 或后端代码，必须走后端 `clean package`、确认新 JAR、重启 `cgc-pms-backend-dev`、等待 3 分钟、health/Flyway/接口核验。
2. 若只改前端入口展示，重启 `cgc-pms-frontend-dev`、等待 3 分钟、验证 5173 与 HTTP 模块。
3. 若前后端都改，顺序固定为后端先、前端后，不能并行刷新。
4. 只有宿主源码已新但 5173 HTTP 模块仍旧时，才清理 `/app/node_modules/.vite` 和 `/app/node_modules/.vite-temp`。

## 四、串行小包

### 小包 1：账号矩阵设计与 seed 必要性裁决

执行角色：

- 产品/需求负责人
- 数据库与迁移负责人
- 安全审计员
- 测试验收员

目标：

1. 固化 4 类账号：`admin`、`workflow-only`、`cc-readonly`、`non-participant`。
2. 明确每个账号的角色、权限、租户、workflow 参与关系。
3. 明确三类业务负向样本如何支撑 `workflow-only` 验收。
4. 明确哪些样本需要 seed，哪些复用现有数据。

验收：

1. 每个账号有唯一职责，不让一个账号同时承担多个负向角色。
2. 每条验收路径能落到真实页面或接口。
3. 不引入第二租户、复杂 `data_scope` 或多余角色变体。

### 小包 2：最小 seed / 后端边界确认

执行角色：

- 数据库与迁移负责人
- 后端设计师 / 后端架构助手
- 安全审计员

目标：

1. 新增 MySQL/H2 对等 demo-only migration。
2. 新增最小用户、角色、用户角色绑定、角色菜单绑定、workflow 参与关系。
3. 复核目标业务模块权限和租户校验仍由目标模块承担。
4. 确认不新增 workflow 权限判断接口。

验收：

1. `admin`、`workflow-only`、`cc-readonly`、`non-participant` 均可按预期登录或参与验收。
2. `workflow-only` 能看到指定 workflow 详情，但不具备 `contract:query`、`purchase:request:list`、`subcontract:measure:query`。
3. `cc-readonly` 具备抄送可见性，但不具备审批动作。
4. `non-participant` 不具备指定 workflow 实例可见性。
5. 后端不新增 workflow 聚合业务详情接口和业务权限预判接口。

### 小包 3：前端入口展示最小实现

执行角色：

- 前端设计师 / 前端架构助手

目标：

1. 在 `workflowDisplay.ts` 增加三类业务目标权限映射。
2. 在 `todo.vue` 与 `detail.vue` 保持同一入口展示策略。
3. 对缺目标权限的已识别业务显示禁用态 `查看业务单据`，提示 `无权访问该业务单据`。
4. 未识别业务类型或缺 `businessId` 继续隐藏入口。
5. 不新增大组件、不做审批详情重构。

验收：

1. 有权限账号仍能进入合同、采购申请、分包计量业务单据。
2. 无目标业务权限账号看到禁用态和泛化提示。
3. 入口判断复用 `userInfo.permissions` / `hasPermission`。
4. `done/cc` 只读动作口径不回退。
5. 定向测试通过：
   - `pnpm exec vitest run src/pages/approval/__tests__/ApprovalWorkList.test.ts src/pages/approval/__tests__/ApprovalConfirm.test.ts`
   - `pnpm build`

### 小包 4：浏览器矩阵验收、安全复核与质量归档

执行角色：

- 部署运维负责人
- 测试验收员
- 安全审计员
- 代码质量审计员

目标：

1. 刷新运行态。
2. 使用真实浏览器按最小账号矩阵验收。
3. 复核权限边界未扩大。
4. 生成最终质量归档。

核心验收矩阵：

1. `admin` 正向：
   - 合同入口可达
   - 采购申请入口可达
   - 分包计量入口可达
2. `workflow-only` 负向：
   - workflow 详情可见
   - 合同、采购申请、分包计量均显示禁用入口或目标模块拒绝
   - 不泄露目标业务详情字段
3. `cc-readonly`：
   - `/approval/cc` 可见
   - 详情只读
   - 无 `同意/驳回/撤回/重提`
4. `non-participant`：
   - 至少 1 条无关审批不可见或详情接口拒绝

验收产物：

1. 截图。
2. 关键 network。
3. DOM/入口状态证据。
4. `evidence.json` 或等价 summary。
5. `docs/quality/approval-center-permission-matrix-acceptance-2026-07-01.md` 质量归档。

## 五、正式关闭条件

1. 最小账号矩阵可登录、可复验。
2. MySQL/H2 seed 对等，且不新增表、字段、索引。
3. 三类业务 `admin` 正向入口可达。
4. `workflow-only` 对三类业务的无目标权限表现通过，且不泄露目标业务详情字段。
5. `cc-readonly` 只读口径不回退。
6. `non-participant` 不可见或详情拒绝通过。
7. 前端定向测试、构建通过。
8. 必要后端测试或迁移验证通过。
9. 运行态刷新与真实浏览器验收通过。
10. 安全复核与质量归档完成。
