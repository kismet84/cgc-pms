# 第 15 条主线：预警中心生产化增强任务计划书

## 计划状态

正式版。

本计划由主会话基于当前代码现状、既有预警规则盘点和角色默认筛选讨论结果整理形成，可作为后续子智能体执行分派、验收和归档依据。

子智能体分派规则在本主线内单独明确如下：

1. 每个子智能体必须显式指定 `model` 和 `thinking`。
2. 子智能体不得继承主线程当前模型，也不得沿用上一个子智能体配置。
3. 模型和推理强度按任务复杂度、风险和产出类型单独匹配，不允许整条主线统一一个模型档位。
4. 一次并行分派 2 个及以上子智能体前，主线程必须先给出“子智能体模型分配表”，至少写清 `任务/角色`、`model`、`thinking`、`reason`。
5. 每个派工单必须写明 `目标`、`范围`、`禁止事项`、`验收输出`，防止把模型分档规则写成空原则。
6. 当任务从实现转为验收、从单模块转为跨模块、或首次出现“不通过 / 阻塞”时，主线程必须重新评估分档，不得沿用上一轮配置。

## 一、结论

第 15 条主线定位为“预警中心最小生产化增强”。

本主线只做三件事：

1. 把当前预警中心从“能看基础列表”增强为“可筛、可分页、可按角色默认聚焦”的工作台。
2. 补齐与当前业务口径直接相关、且已明确数据来源的逾期/风险规则，优先处理采购交期逾期。
3. 保留最小实现原则，不扩成通知平台、全量规则设计器或跨渠道消息中心。

当前已确认事实：

- 后端当前接口只支持列表、单条已读、手动批量评估。
- 列表筛选当前仅有 `projectId`、`severity`、`isRead`。
- 前端当前分页是前端切片，不是后端分页。
- 当前规则类型只有 8 类。
- 采购经理驾驶舱中的“逾期交货”当前是驾驶舱 KPI，不属于预警中心规则。

因此，第 15 条主线的核心裁决是：

- 先补“规则口径 + 字段标签 + 页面可用性”。
- 不先做复杂规则引擎。
- 不先做邮件、短信、企微、站内信多渠道触达。
- 不把“默认筛选”误做成“强权限隔离”。
- 上述三项不是永久排除，而是本主线最小增强完成后的后续顺延任务。

## 二、边界

本主线包含：

- 为预警记录补充最小可用分类口径，如 `alertDomain` 或等价字段。
- 预警列表改为后端分页，并补齐规则类型、分类、触发时间范围等筛选。
- 增加角色默认筛选策略，支持多角色并集和用户手工覆盖。
- 新增采购交期逾期预警规则；其他逾期类规则只纳入已确认数据源的小包。
- 增加批量已读、业务单据跳转入口、处理状态基础字段的最小闭环。
- 完成后端测试、前端静态验证、真实页面定向验收和质量归档。

本主线不包含：

- 不在本主线 `M1` 最小增强内建设通用规则引擎或规则设计后台。
- 不在本主线 `M1` 最小增强内建设消息中心、通知订阅中心或多渠道触达。
- 不在本主线 `M1` 最小增强内把角色默认筛选提升为强权限隔离模型。
- 不一次性纳入“所有逾期都预警”；未确认数据源的逾期项只能先列入后续候选。
- 不重做驾驶舱全部告警卡片。

上述未纳入 `M1` 的事项不视为排除项，而视为第 15 条主线后续 `M2/M3` 候选范围。

## 三、角色默认策略

本主线采用“角色默认值，不等于权限边界”的策略。

默认矩阵如下：

| 角色 | 默认关注域 |
|---|---|
| 超级管理员 | 全部 |
| 项目经理 | 全部 |
| 采购经理 | 采购类，当前优先落地采购交期逾期 |
| 商务经理 | 成本、合同、付款、变更 |
| 生产经理 | 当前仅保留“待细分类标签补齐后再增强”的降级口径，`M1` 不假装已精确落地人工/机械细域 |
| 总工程师 | 当前角色冻结，`M1` 不作为已正式落地的默认筛选对象；如后续启用，需单独补规则标签与验收 |

硬规则：

1. 多角色用户按并集取默认筛选。
2. 用户手工改筛选后，允许覆盖默认值。
3. 默认筛选只是页面默认视图，不替代业务权限校验。
4. 若底层规则缺少分类标签，不得假装矩阵已完成，必须先补标签或降级处理。
5. 对于“角色已冻结”或“规则细标签尚未落地”的项，计划书中只能写降级策略，不能写成已完成能力。

## 四、任务拆解

### 任务 1：后端最小规则与字段增强

执行角色：后端工程执行子智能体。

建议模型：`model=gpt-5.5`，`thinking=high`。

输入：

- 本计划书
- `backend/src/main/java/com/cgcpms/alert/controller/AlertController.java`
- `backend/src/main/java/com/cgcpms/alert/service/AlertEvaluationService.java`
- 相关 `alert_log` 实体、mapper、测试文件
- 采购交期相关数据来源代码

执行要求：

1. 在不重构为规则引擎的前提下，为预警记录补最小分类/标签字段。
2. 列表接口改为后端分页，并支持规则类型、分类、时间范围筛选。
3. 新增采购交期逾期预警规则，明确去重口径、触发条件和样本边界。
4. 为业务跳转保留最小必要定位信息，不新增跨模块聚合接口。
5. 若其他逾期项数据源未稳定，标记为未纳入，不得强行一并实现。

验证要求：

- 相关后端测试通过。
- 如涉及 migration，必须新增版本化脚本，不修改历史 migration。
- `git diff --check -- <本次修改文件>` 通过。

### 任务 2：前端页面最小生产化增强

执行角色：前端工程执行子智能体。

建议模型：`model=gpt-5.4`，`thinking=medium`。

输入：

- 本计划书
- `frontend-admin/src/pages/alert/index.vue`
- `frontend-admin/src/stores/alert.ts`
- `frontend-admin/src/api/modules/alert.ts`
- `frontend-admin/src/types/alert.ts`

执行要求：

1. 将当前前端切片分页改为后端分页。
2. 页面补齐规则类型、分类、触发时间范围、只看默认域等筛选能力。
3. 新增批量已读入口。
4. 为可定位的预警增加“查看业务单据”入口；不可定位的规则不得伪造跳转。
5. 页面结构继续沿用现有列表页设计语言，不单独发明新布局体系。

验证要求：

- `pnpm type-check`
- `pnpm build`
- 如已有对应页面测试，补齐或更新最小测试
- `git diff --check -- <本次修改文件>`

### 任务 3：角色默认筛选与边界复核

执行角色：产品/权限边界复核子智能体。

建议模型：`model=gpt-5.4`，`thinking=medium`。

复核要求：

1. 确认默认筛选口径和当前角色职责一致。
2. 确认“默认筛选”未越界为“权限控制”。
3. 确认总工程师默认域不错误混入人工、机械成本。
4. 对缺少规则标签的数据项明确标注“待补字段后才能完全落地”。

输出：

- 通过/不通过
- 阻塞/非阻塞
- 需不需要调整角色矩阵

### 任务 4：运行态刷新与真实页面验收

执行角色：部署运维负责人 + 测试验收员。

建议模型：

- 运维：`model=gpt-5.4`，`thinking=low`
- 测试：`model=gpt-5.5`，`thinking=medium`

执行要求：

1. 刷新前后端运行态。
2. 若发生重启，按仓库约定等待 3 分钟后再做验收。
3. 验证预警页可正常加载、分页准确、筛选可用、批量已读可用。
4. 验证新增采购交期逾期规则能在真实样本上出现。
5. 验证角色默认筛选符合预期，但用户仍可手动调整。

输出：

- 通过/不通过
- 阻塞/非阻塞
- 命令级验证结果
- 浏览器证据路径

### 任务 5：质量归档

执行角色：代码质量审计员。

建议模型：`model=gpt-5.4`，`thinking=medium`。

输出文件：

- `docs/quality/mainline-15-alert-center-enhancement-acceptance-YYYY-MM-DD.md`

归档内容：

1. 最终结论：通过/不通过，阻塞/非阻塞。
2. 实际修改文件与 migration。
3. 后端测试、前端静态验证、浏览器复验证据。
4. 角色默认筛选复核结论。
5. 未纳入本主线的后续项。

## 五、执行顺序

1. 后端最小规则与字段增强。
2. 前端页面最小生产化增强。
3. 角色默认筛选与边界复核。
4. 运行态刷新。
5. 真实页面验收。
6. 质量归档。
7. 主会话最终裁决并关闭第 15 条主线。

## 六、最终验收标准

本主线判定通过需同时满足：

1. 预警列表已改为后端分页，翻页和总数准确。
2. 现有 8 条规则不回退。
3. 采购交期逾期预警已按真实数据源落地，并可生成记录。
4. 页面支持规则类型、分类、严重度、已读状态、时间范围等最小筛选。
5. 角色默认筛选正确生效，多角色按并集处理。
6. 用户可手工覆盖默认筛选。
7. 批量已读可用。
8. 可定位的预警可跳转到对应业务单据或目标页。
9. 后端测试通过。
10. 前端静态验证通过。
11. 真实浏览器验收通过并保留证据。
12. 质量归档报告完成。

## 七、剩余风险

1. “所有逾期都预警”当前只能分批落地，未确认数据源的项不能在本主线内一次性完成。
2. 若当前 `alert_log` 或相关 VO 缺少稳定分类字段，角色默认矩阵只能部分落地。
3. 业务跳转能力受各模块现有详情路由和权限链约束，不保证所有规则都能直达详情。
4. 规则引擎、通知平台、多渠道触达、强权限隔离应视为第 15 条主线的后续增强阶段，不应误判为永久排除；但在 `M1` 验收中不作为未完成项阻塞当前最小增强收口。

## 八、后续增强预留

第 15 条主线后续建议按串行阶段继续推进：

1. `M1`：最小生产化增强
   完成后端分页、角色默认筛选、采购交期逾期规则、批量已读、业务跳转和真实验收。
2. `M2`：规则治理增强
   引入规则分类治理、规则配置化、抑制策略、去重策略和归档口径。
3. `M3`：通知与权限深化
   推进通知平台、多渠道触达、强权限隔离和更精细的订阅策略。
