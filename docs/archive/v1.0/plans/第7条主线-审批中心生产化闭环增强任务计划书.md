# 第 7 条主线：审批中心生产化闭环增强任务计划书

## 计划状态

正式版。

本计划先由主会话形成草案，现已完成产品/需求负责人、前端设计师、后端设计师、数据库与迁移负责人、安全审计员、测试验收员、部署运维负责人、代码质量审计员八方只读研判，并已按回传结论收敛为正式版。

## 一、正式结论

第 7 条是大主线，但不作为一次性大包平推，而是作为一条串行主线拆成 4 个小包依次推进：

1. 后端服务端筛选契约小包
2. 前端筛选接入与轻量映射收口小包
3. 审批详情/业务入口/只读口径回归小包
4. 负向场景、运行态验收与质量归档小包

本主线定位为“审批中心生产化工作台增强”，重点是让审批中心四页签在当前三类核心业务类型下具备稳定的高频检索、入口可达、只读/动作口径一致和可交付验收能力。

## 二、正式范围

### 纳入本主线

1. 四页签核心筛选 MVP：
   - `keyword`
   - `instanceStatus`
   - `businessType`
   - `startTime/endTime` 仅在后端低成本支持前提下纳入；否则降为抽样增强项
2. 三类核心业务类型：
   - `合同审批`
   - `采购申请`
   - `分包计量`
3. 审批详情与内嵌详情中的：
   - 业务类型中文化
   - 业务单据入口映射
   - 只读页签无动作误显
   - 待办/我发起动作口径不回退
4. 安全与负向场景：
   - 同租户无目标业务权限
   - `businessId` 篡改
   - 单据不存在/已删除
   - `done`/`cc` 只读页签不误显动作
5. 运行态刷新、真实浏览器验收、质量归档

### 不纳入本主线

1. 不纳入付款申请或下一批未稳定业务类型
2. 不重做审批中心整体 UI
3. 不重写 workflow 引擎
4. 不新增 workflow 业务详情聚合接口
5. 不新建业务类型配置中心、字典表或插件式入口注册表
6. 不做审批中心全页签 × 全筛选项 × 全业务类型的全排列验收
7. 不做移动端专项
8. 不做全站权限体系或项目级数据范围强化重构

## 三、冻结口径

### 1. 产品口径

1. 第 7 条聚焦审批中心生产化工作台增强，不滑向全面产品化重构。
2. 正式业务类型范围只含：合同、采购申请、分包计量。
3. 筛选 MVP 默认锁定：`keyword`、`instanceStatus`、`businessType`。
4. `时间范围` 不是默认硬前置，只有在后端确认低成本支持时才纳入正式实现。
5. 负向提示至少统一为审批中心链路内三类最小口径：
   - 无权限：`无权访问该业务单据` 或目标模块现有等价口径
   - 单据不存在/已删除：`业务单据不存在或已不可访问`
   - 加载失败：`业务单据加载失败，请稍后重试`

### 2. 前端口径

1. `todo/done/cc` 不允许做前端本地假筛选分页数据。
2. 业务类型中文化与业务单据入口映射允许抽审批模块内轻量 helper。
3. 不抽共享详情大组件，不做大型共享筛选框架。
4. 内嵌详情与独立详情页保持：
   - 业务类型中文化一致
   - 业务入口映射一致
   - 失败口径一致

### 3. 后端口径

1. 第 7 条若要做稳定筛选，必须走后端最小参数扩展。
2. 统一筛选参数名采用：
   - `keyword`
   - `businessType`
   - `instanceStatus`
   - `startTime`
   - `endTime`
3. 各页签时间字段语义固定为：
   - `todo = receivedAt`
   - `done = record.createdAt`
   - `cc = cc.createdTime`
   - `mine = instance.createdAt`
4. 必须保留原身份谓词：
   - `todo = tenantId + approverId + PENDING`
   - `done = tenantId + operatorId`
   - `cc = tenantId + ccUserId`
   - `mine = tenantId + initiatorId`
5. 业务类型中文化本轮仍放前端 helper，不加后端 `businessTypeName`。

### 4. 数据库口径

1. 草案阶段不新增 migration 的判断成立。
2. 正式版优先看后端最终查询形态，再决定是否补最小索引。
3. 当前最可能的性能热点是 `done` 对 `wf_record` 的筛选。
4. 若需要数据库优化，优先关注 `wf_record`，不预先对四张表一并加索引。
5. `cc` 时间跨度样本不足，如要做真实多时间范围硬验，后续再补最小 demo-only 样本。

### 5. 安全口径

1. 筛选条件只能作为原身份查询后的附加 `AND` 条件，不能重构成通用大接口绕过页签边界。
2. 业务入口继续由目标模块鉴权；workflow 权限不等于目标业务详情权限。
3. `businessId/status/businessType/startTime/endTime` 都视为不可信输入。
4. 权限感知隐藏入口不是本主线阻塞项，仅是后续体验优化。

## 四、四个串行小包

### 小包 1：后端服务端筛选契约

执行角色：

- 后端设计师 / 后端架构助手
- 数据库与迁移负责人
- 安全审计员（只读复核）

目标：

1. 为 `todo/done/cc/mine` 建立统一筛选参数契约。
2. 确认是否全部支持 `keyword/businessType/instanceStatus/startTime/endTime`。
3. 在保留原身份谓词前提下实现后端服务端筛选。
4. 只做最小查询扩展，不改 workflow 引擎、不做通用大列表接口。

验收：

1. `todo/done/cc/mine` 参数命名统一。
2. 分页 `total`、页码和筛选结果一致。
3. 安全复核确认未扩大身份边界。
4. 如需最小索引优化，只补必要索引。

### 小包 2：前端筛选接入与轻量映射收口

执行角色：

- 前端设计师 / 前端架构助手

目标：

1. 在四页签接入正式支持的服务端筛选项。
2. 抽审批模块内轻量 helper，收口三类业务的：
   - 中文名称
   - 入口映射
   - 未识别兜底显示
3. 不做大型筛选框架，不做共享详情大重构。

验收：

1. 四页签筛选条按页签最小可用落地，不要求完全同构。
2. `todo/done/cc/mine` 查询和结果一致。
3. 三类业务入口与中文化不回退。

### 小包 3：详情一致性与入口/只读回归

执行角色：

- 前端设计师 / 前端架构助手
- 测试验收员
- 安全审计员

目标：

1. 确保第 5/6 条已通过能力不回退。
2. 确保内嵌详情和独立详情页在三类业务上的：
   - 中文化一致
   - 入口策略一致
   - 失败口径一致
3. 确保 `done/cc` 只读页签无 `同意/驳回/撤回/重提` 误显。
4. 确保 `todo/mine` 动作口径不回退。

验收：

1. 合同入口仍直达。
2. 采购申请、分包计量深链仍自动定位并打开目标单据。
3. 只读口径通过。

### 小包 4：负向场景、运行态验收与质量归档

执行角色：

- 测试验收员
- 安全审计员
- 部署运维负责人
- 代码质量审计员

目标：

1. 覆盖核心真实账号负向场景。
2. 运行态刷新与浏览器矩阵验收。
3. 生成最终质量归档。

验收：

1. 同租户无目标业务权限、`businessId` 篡改、只读页签误显动作等核心负向场景通过。
2. 跨租户和复杂 `data_scope` 如无现成样本，可由安全静态复核兜底并显式记录剩余风险。
3. 归档报告完成。

## 五、浏览器验收矩阵

### 核心矩阵（硬验）

1. `/approval/todo`
   - `keyword` 1 组
   - `businessType` 1 组
   - `instanceStatus` 1 组
   - 待办动作仍可进入详情
2. `/approval/done`
   - `keyword + businessType` 或 `instanceStatus + timeRange` 1 组
   - 业务入口可达
3. `/approval/cc`
   - 只读页签筛选 1 组
   - 无动作误显
4. `/approval/mine`
   - 状态筛选 1 组
   - 翻页后筛选条件保持
5. 三类业务类型必验：
   - 合同审批
   - 采购申请
   - 分包计量

### 抽样矩阵（非全量硬验）

1. `timeRange` 只在 `done/mine` 各抽 1 页签
2. 其他业务类型仅在有稳定样本前提下抽 1-2 类
3. 多筛选组合只抽：
   - `keyword + businessType`
   - `instanceStatus + timeRange`

### 负向矩阵

真实硬验：

1. 同租户无目标业务权限
2. `cc` 只读
3. `done` 只读
4. `businessId` 篡改

静态兜底：

1. 跨租户
2. 复杂 `projectId/data_scope`
3. 缺少现成样本的组合越权场景

## 六、正式验收标准

本主线最终通过需同时满足：

1. 四个串行小包各自给出 `通过/不通过` 与 `阻塞/非阻塞`。
2. 后端筛选契约落地，且不扩大页签身份边界。
3. 前端接入后分页、`total` 和筛选结果一致。
4. 三类核心业务类型中文化、入口策略、详情表现一致。
5. 第 5/6 条已通过能力不回退。
6. `done/cc` 只读口径不回退。
7. `todo/mine` 动作口径不回退。
8. 单测通过。
9. 若后端有改动，对应后端测试通过；若后端无改动，由后端助手确认。
10. `pnpm build` 通过。
11. 如涉及后端改动，后端运行态健康检查通过。
12. 真实浏览器核心矩阵通过。
13. 质量归档报告完成。

## 七、运维执行路径

### 仅前端改动

1. 重启 `cgc-pms-frontend-dev`
2. 等待 3 分钟
3. 验证 `http://localhost:5173/` 返回 200
4. 取证运行态 HTTP 模块关键特征
5. 仅在源码已新但 HTTP 模块仍旧时，清 `/app/node_modules/.vite` 与 `/app/node_modules/.vite-temp` 后重启

### 前后端同时改动

1. 后端：
   - `clean package`
   - 重启 `cgc-pms-backend-dev`
   - 等待 3 分钟
   - 验证 `http://localhost:8080/api/actuator/health` 为 `UP`
2. 前端：
   - 重启 `cgc-pms-frontend-dev`
   - 等待 3 分钟
   - 验证 `http://localhost:5173/` 返回 200
   - 取证运行态关键特征

不得并行刷新前后端。
