# 成本管理模块UI优化 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在不更换设计体系、不新增依赖、不重构后端的前提下，按现有灰白企业后台风格统一重设计成本管理模块核心页面，沿用已收敛的列表页骨架，覆盖 `成本列表`、`项目成本明细核对`、`成本目标列表`、`成本目标编辑/详情`，并保持完整交互。

**Architecture:** 以现有 `lg-list-page`、`lg-search-bar`、`lg-list-table-panel`、`lg-analysis-rail`、`ColumnSettingsButton`、`useColumnSettings`、Ant Design Vue、VxeGrid 为唯一骨架来源。优先复用已在项目列表、合同列表、签证列表、预警中心验证过的布局和密度规则；只做页面层统一，不抽新设计系统，不新建通用组件库。

**Tech Stack:** Vue 3、TypeScript、Vite、Ant Design Vue、VxeGrid、Vitest、Playwright、现有成本管理 API。

---

## 规则边界

- 主线程只负责计划、分派、督办和验收，不亲自实现代码。
- 仓库存在 `.codegraph/`，执行者理解页面前必须优先使用 CodeGraph。
- 不新增 UI 框架、图表库、状态库、样式体系、设计 token 体系。
- 不改数据库 schema，不改成本业务口径，不改后端接口契约。
- 不把本轮 UI 优化扩散为“成本管理模块功能重构”。
- 不覆盖非核心外围页；`成本科目` 默认不在本轮主交付内，除非主页面统一后仍存在明显断层且用户追加要求。
- 保持完整交互：筛选、分页、列设置、详情、弹窗、保存、提交审批、离开拦截都必须可用。
- 完成后需给出桌面端与移动端真实验收结论。

## UI统一目标

### 统一骨架

- 顶部统一为：`breadcrumb + 页面标题`，副标题弱化，不再各页自由发挥。
- 主区统一为：左侧业务主区 + 右侧分析栏。
- 左侧统一顺序：`筛选区 -> KPI区 -> 表格/主内容区`。
- 右侧统一顺序：`结论摘要 -> 结构分布 -> 风险提示/操作提示`。

### 统一筛选区

- 统一成一行主筛选项、一行关键词和操作按钮的紧凑结构。
- 控件宽度按可用空间动态撑满，不保留随意空白。
- 查询按钮为主色，重置按钮弱化，按钮顺序统一。

### 统一KPI区

- 卡片高度、圆角、留白、数字字号、标签位置统一。
- 仅保留高价值指标，不做驾驶舱式炫技扩展。
- 状态色统一使用现有企业后台强对比色规则。

### 统一表格区

- 工具栏统一：`标题 + 数量 + 辅助说明 + 列设置 + 刷新 + 主操作按钮`。
- 表头高度、行高、金额列右对齐、操作列居中统一。
- 空态、分页、移动端卡片 fallback 统一。

### 统一分析栏

- 成本列表：来源结构、科目结构、异常提示。
- 成本明细：偏差结论、超标科目、联动说明。
- 成本目标列表：版本状态、生效版本、待审批。
- 成本目标编辑：版本摘要、金额校验、提交流程提示。

### 统一编辑页语言

- 编辑页不再像独立表单系统，改成与列表体系相容的业务页。
- 左侧主表单 + 明细表格，右侧摘要信息栏。
- 保存、提交审批、返回动作区位置统一。

## 重新分档检查点

- M19-UI-1 完成后：若页面差异远超预期，需把实现任务拆得更细，避免单子智能体写集中过大。
- M19-UI-2 完成后：进入 M19-UI-3 前，重新确认编辑页是否需要升档。
- M19-UI-3 若发现表单交互明显复杂于预期：可升到 `gpt-5.5 / medium`。
- M19-UI-4 若发现问题已从 UI 一致性升级为权限/接口/数据口径问题：转交其他主线，不在本主线硬扛。

## 最终上线裁决标准

通过条件：

- 成本管理四页视觉上属于同一后台模块，而非四套不同语言。
- 筛选区、KPI区、工具栏、右侧分析栏顺序和密度统一。
- 编辑页与列表页的设计语言统一，且完整交互不回退。
- `pnpm type-check` 通过。
- `pnpm build` 通过。
- 浏览器桌面端与移动端验收通过。
- 无本轮 UI 改动引入的控制台红错或关键 API 失败。

不通过条件：

- 仍存在明显页面割裂，无法看出属于同一模块。
- 编辑页交互被破坏。
- 移动端出现明显不可用。
- 构建或类型检查失败且无法证明无关。
- 真实浏览器验收出现本轮引入的关键错误。
