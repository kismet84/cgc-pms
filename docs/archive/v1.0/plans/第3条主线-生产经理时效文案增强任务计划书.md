# 第 3 条主线：生产经理时效文案增强任务计划书

## 结论

本主线只处理上一条观察项收口主线中的 GAP-004：生产经理页 `近期验收` 未来日期显示 `0天` 的可读性增强。目标是把生产经理 `近期验收` 的时效列从单纯展示 `pendingDays` 改为基于真实日期的中性时效文案，避免未来日期显示 `0天` 造成理解成本。

本主线不处理 GAP-005 审批详情业务入口。GAP-005 应作为第 4 条主线单独规划，避免把驾驶舱文案增强与审批导航增强混在一起。

本计划书已根据产品/需求负责人、前端设计师、测试验收员三方只读研究结论修订。原始版本仅作为主负责人草案，当前版本作为第 3 条主线正式计划。

## 一、任务拆解

### 任务 1：冻结生产经理时效文案口径

- 简要说明：将生产经理 `近期验收` 的时效列定义为中性时间距离，不直接套用采购经理的逾期语义。
- 输入：`docs/13-驾驶舱摘要字段生成与展示规范.md`、`docs/quality/production-timing-and-approval-business-entry-closure-2026-07-01.md`、运行态 GAP-004 证据。
- 输出：明确口径写入正式基线：生产经理 `近期验收` 使用中性时间距离文案，未来 `剩余X天`，当天 `今日`，过去 `已过X天`，无日期或非法日期 `-`。
- 涉及文件建议：`docs/13-驾驶舱摘要字段生成与展示规范.md`
- 复杂度：P0
- 验收标准：
  - Given 生产经理 `近期验收` 存在未来日期，When 查看时效列，Then 显示 `剩余X天`，不显示 `0天`。
  - Given 生产经理 `近期验收` 日期为当天，When 查看时效列，Then 显示 `今日`。
  - Given 生产经理 `近期验收` 日期为过去，When 查看时效列，Then 显示 `已过X天`，不显示 `逾期X天`，避免伪造逾期结论。
  - Given 日期为空或非法，When 查看时效列，Then 显示 `-`。

### 任务 2：前端生产经理页局部实现

- 简要说明：仅在生产经理页 `近期验收` 表格中改用 `record.date` 计算时效文案，不改变后端 `pendingDays` 语义，也不影响同页 `近期领料`。
- 输入：`ProductionManagerDashboardVO.recentReceipts[]` 中的 `date` 字段。
- 输出：生产经理 `近期验收` 时效列显示中性文案。
- 涉及文件建议：`frontend-admin/src/pages/dashboard/components/DashboardProductionView.vue`
- 复杂度：P1
- 依赖：任务 1
- 验收标准：
  - Given `record.date=2026-07-05` 且当前日期为 2026-07-01，When 渲染生产经理 `近期验收`，Then 时效显示 `剩余4天`。
  - Given `record.date=2026-07-01`，When 渲染生产经理 `近期验收`，Then 时效显示 `今日`。
  - Given `record.date=2026-06-30`，When 渲染生产经理 `近期验收`，Then 时效显示 `已过1天`。
  - Given `record.date` 为空或非法，When 渲染生产经理 `近期验收`，Then 时效显示 `-`。
  - Given 生产经理 `近期领料` 仍使用现有 `pendingDays` 口径，When 渲染页面，Then 不因本任务被改成 `record.date` 口径。

### 任务 3：补充最小前端测试

- 简要说明：补充或扩展现有 dashboard 测试，覆盖生产经理未来、当天、过去、空值四类文案。
- 输入：现有 dashboard 测试结构。
- 输出：最小测试覆盖，能防止 `0天` 回退。
- 涉及文件建议：`frontend-admin/src/pages/dashboard/__tests__/DashboardReferenceFidelity.test.ts`
- 复杂度：P1
- 依赖：任务 2
- 验收标准：
  - Given 测试读取生产经理组件源码，When 检查时效逻辑，Then 能证明不再只依赖 `pendingDays === 0 -> 今日`。
  - Given 测试样本包含未来日期，When 执行相关测试，Then 断言包含 `剩余X天` 规则。
  - Given 测试样本包含过去日期，When 执行相关测试，Then 断言包含 `已过X天` 规则。
  - Given 测试读取生产经理组件源码，When 检查近期领料分支，Then 能证明近期领料未被本次近期验收逻辑误伤。

### 任务 4：运行态定向复验与归档

- 简要说明：刷新前端运行态后，按“1 主验 + 2 抽查”复验：主验生产经理 `近期验收`，抽查采购经理与总工程师时效列不回退。
- 输入：前端构建结果、Docker/Vite 运行态、生产经理真实页面。
- 输出：复验证据目录与主线收口结论。
- 涉及文件建议：`docs/quality/production-manager-timing-copy-acceptance-2026-07-01.md`
- 复杂度：P1
- 依赖：任务 3
- 验收标准：
  - Given 前端运行态已刷新并等待 3 分钟，When 浏览器进入 `/dashboard` 生产经理页，Then 未来日期不再显示 `0天`。
  - Given 浏览器进入 `/dashboard` 采购经理页，When 查看 `待验收入库`，Then 既有 `剩余X天 / 今日 / 逾期X天 / -` 不回退。
  - Given 浏览器进入 `/dashboard` 总工程师页，When 查看时效列，Then 既有 `剩余X天 / 今日到期 / 逾期X天 / -` 不回退。
  - Given 复验证据已保存，When 主负责人验收，Then 能看到截图或 evidence 文件路径。
  - Given GAP-004 完成复验，When 写入归档报告，Then 明确本主线通过 / 不通过、阻塞 / 非阻塞、剩余风险。

## 三、数据流变化

当前数据流保持不变：

`用户进入 /dashboard -> 前端请求生产经理驾驶舱接口 -> 后端返回 recentReceipts[].date 与 pendingDays -> 前端生产经理页渲染近期验收表格`

本主线只改变前端展示解释：

- 现状：`pendingDays=0` 会显示 `今日` 或 `0天`，未来日期容易被误读。
- 目标：生产经理 `近期验收` 使用 `record.date` 计算展示文案；生产经理 `近期领料` 仍沿用现有 `pendingDays` 展示。
- 后端契约：不改变 `pendingDays` 当前“已等待 / 已过天数”语义。
- 状态管理：不新增 store，不新增接口字段。

## 四、影响范围与回归测试建议

- 生产经理页 `近期验收`：重点复验未来日期、当天日期、过去日期、空日期显示。
- 生产经理页 `近期领料`：确认未被 `近期验收` 局部函数改动误伤。
- 生产经理页 `分包计量`：确认无时效列相关回退。
- 采购经理页 `待验收入库`：抽查未来日期仍显示 `剩余X天`，确认没有共享函数误改。
- 总工程师页时效列：抽查未来 / 今日 / 逾期文案不变。
- 前端构建：运行 `pnpm build`，确认类型和构建通过。

## 五、冒烟测试方案

1. 浏览器进入 `/dashboard`，切换 `生产经理`，`近期验收` 未来日期显示 `剩余X天`。
2. 浏览器进入 `/dashboard`，切换 `生产经理`，`近期验收` 不再出现未来日期显示 `0天`。
3. 若运行态已有当天 / 过去 / 空日期样本，顺手记录 `今日 / 已过X天 / -`；若没有样本，明确标注“样本不足，未覆盖”，不为此补数。
4. 切换 `生产经理`，确认 `近期领料` 未被 `近期验收` 时效逻辑误伤。
5. 切换 `采购经理`，确认 `待验收入库` 既有 `剩余X天 / 今日 / 逾期X天 / -` 不回退。
6. 切换 `总工程师`，确认技术事项时效口径不被生产经理改动影响。
7. 运行 `pnpm exec vitest run src/pages/dashboard/__tests__/DashboardReferenceFidelity.test.ts`。
8. 运行 `pnpm build`，确认前端构建通过。
