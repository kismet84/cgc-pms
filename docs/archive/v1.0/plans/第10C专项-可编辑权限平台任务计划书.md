# 第 10C 专项：可编辑权限平台任务计划书

## 计划状态

正式版。

本专项由第 10 条 10A 权限字典治理裁决拆出。经产品、前端、后端、数据库、安全、测试、运维、代码质量 8 个专业角色只读研判，当前可升级为“正式版候选”，但仅限只读盘点与安全设计裁决，不允许直接实现可编辑权限码页面，不允许新增权限字典表，不允许开放生产在线编辑权限码本体。

## 当前执行进度

### 已回传研判

1. 产品/需求负责人：
   - 结论：可升级为“正式版候选”。
   - 明确要求：当前只允许“角色菜单绑定编辑增强”，不开放权限码本体编辑。
   - 风险判断：范围膨胀风险中低，但必须继续保持“高危写能力 MVP 裁决专项”定位。
2. 测试验收员：
   - 结论：可升级为“正式版候选”，但只能作为“安全设计与边界裁决正式版候选”。
   - 明确要求：若后续进入实现，必须先具备审计、防自我提权、高危权限保护和恢复路径。
   - 最低验证：角色菜单绑定正向修改、重新进入或重新登录后权限变化可观察、无权限账号不可提交、自我提权/自我锁死被阻止。
3. 前端设计师 / 前端架构助手：
   - 结论：可升级为“正式版候选”。
   - 明确要求：只复用现有 `PermissionModal` 和角色菜单绑定链路，不新做权限码编辑平台，不新增权限事实源。
4. 部署运维负责人：
   - 结论：可升级为“正式版候选”，但三专项里风险最高。
   - 明确要求：若后续进入实现，必须附带备份、回滚、审计运行态核验；否则不得进入运行态。
5. 代码质量审计员：
   - 结论：可升级为“正式版候选的只读研判/裁决计划”，不建议直接升级为实现计划。
   - 明确要求：正式版应进一步收窄为“角色菜单绑定编辑安全加固评估”，而不是“可编辑权限平台实现计划”。
   - 风险判断：若不先冻结安全门禁，会误变成权限平台重构。
6. 后端设计师 / 后端架构助手：
   - 结论：具备升级为“正式版候选”的后端条件，但未补齐审计、防自我提权/锁死、高危权限保护前阻塞实现。
   - 明确要求：继续复用 `PUT /system/roles/{id}/menus -> SysRoleService.assignMenus -> sys_role_menu` 现有链路，不开放权限码本体编辑。
   - 已识别缺口：角色菜单绑定未见变更前后 diff 审计、未见防自我提权/自我锁死、高危权限二次确认。
7. 安全审计员：
   - 结论：暂不建议直接升级为“实现正式版”，只建议升级为“只读盘点/安全设计正式候选”。
   - 明确要求：只要涉及角色-菜单绑定编辑或权限配置变更，必须先具备后端强鉴权、操作审计、变更前后差异、操作者/目标角色/结果记录、防自我提权、防自我锁死、最后管理员保护、高危权限二次保护、事务一致性、失败回滚/恢复路径。
   - 阻塞规则：缺任一项均阻塞进入实现。
8. 数据库与迁移负责人：
   - 结论：不建议直接升级为可执行正式版候选。
   - 明确要求：现有 `sys_operation_audit_log` 只够粗粒度操作留痕，不足以支撑权限变更 diff 审计与回滚边界。
   - 数据侧判断：若后续允许编辑 `sys_role_menu`，应先冻结“最小差异审计/回滚数据结构”边界，再决定是否进入实现。

## 一、专项目标

评估并定义可编辑权限平台的最小安全 MVP。

默认 MVP 只考虑“角色菜单绑定编辑增强”，不开放权限码本体编辑。权限码本体继续以代码注解、`sys_menu.perms`、迁移 seed 为事实来源。

## 二、纳入范围

1. 盘点现有角色菜单绑定编辑能力：
   - 角色管理页面
   - `PermissionModal`
   - `PUT /system/roles/{id}/menus`
   - `sys_role_menu`
2. 盘点现有审计能力：
   - `sys_operation_audit_log`
   - `@AuditedOperation`
   - 是否记录变更前后值
3. 定义可编辑平台最低安全门禁：
   - 后端强鉴权
   - 防自我提权
   - 防自我锁死
   - 高危权限二次保护
   - 审计日志
   - 事务一致性
   - 回滚 / 恢复路径
4. 裁决是否需要补最小审计 diff 能力。

## 三、明确不纳入范围

1. 不编辑权限码本体。
2. 不新增权限字典事实表。
3. 不做动态权限 DSL。
4. 不做全站 RBAC 重构。
5. 不做跨租户权限管理平台。
6. 不把入口注册体系并入本专项。
7. 不把项目级强隔离并入本专项。

## 四、串行小包

### 小包 1：现有可编辑能力只读盘点

执行角色：

- 前端设计师 / 前端架构助手
- 后端设计师 / 后端架构助手
- 数据库与迁移负责人
- 安全审计员

目标：

1. 梳理现有角色菜单绑定编辑链路。
2. 梳理现有鉴权、审计、事务边界。
3. 标注缺失的安全门禁。

验收：

1. 形成链路清单。
2. 形成安全缺口清单。
3. 严格只读。

### 小包 2：MVP 边界裁决

执行角色：

- 产品/需求负责人
- 安全审计员
- 测试验收员
- 代码质量审计员

目标：

1. 裁决是否允许进入实现。
2. 裁决 MVP 是否仅限角色菜单绑定编辑。
3. 裁决审计与回滚最低要求。

验收：

1. 没有安全门禁缺失时才允许实现。
2. 若缺审计、防自我提权、高危保护，则不进入实现。

### 小包 3：最小实现包定义

执行角色：

- 主负责人裁决

目标：

1. 定义最小前后端改动。
2. 定义 migration / 零 migration 依据。
3. 定义测试与运维路径。

验收：

1. 不触碰权限码本体编辑。
2. 不引入新权限事实源。

## 五、通过 / 不通过标准

### 通过条件

1. 现有编辑链路已盘点。
2. 安全门禁已裁决。
3. MVP 范围足够小。
4. 是否进入实现已有明确结论。

### 不通过条件

1. 未补审计、防自我提权、高危权限保护就进入实现。
2. 直接开放权限码本体编辑。
3. 新增权限事实源。
4. 把可编辑平台与项目强隔离、全站入口注册混在一起。

## 六、当前建议

先做只读安全设计。可编辑权限平台是高危写能力，缺少审计和防自我提权前不得进入实现。

1. 多角色只读研判完成。

3. 安全门禁与 MVP 边界裁决完成。
4. 主负责人已明确“不进入实现”的阶段性决策。
