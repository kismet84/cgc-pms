# 独立 `project.spec.ts` 补齐执行包

## 1. 结论先行

- 当前判定：需要补齐。
- 当前结论：非阻塞，但应优先收口。
- 依据：
  - 当前项目链路只有 `frontend-admin/e2e/ui-refactor-smoke.spec.ts` 中的顺带冒烟覆盖。
  - 现有覆盖只证明 `/project/list` 页面能渲染，不证明“列表进入总览”的独立链路稳定。
  - 仓库已存在项目列表页、项目总览路由和后端接口，不需要为此新增抽象或扩面改造。

## 2. 当前证据

### 2.1 当前没有独立规格化用例

- `frontend-admin/e2e/` 下当前不存在 `project.spec.ts`。
- `docs/未来开发计划.md` 已把“独立 `project.spec.ts` 补齐”列为当前待增强项。
- `docs/quality/mainline-13-closeout-2026-07-03.md` 已把该项列为非阻塞剩余风险。

### 2.2 现有项目覆盖过浅

- `frontend-admin/e2e/ui-refactor-smoke.spec.ts`
  - 当前只验证：
    - 进入 `/project/list`
    - 搜索栏可见
    - 表格可见
  - 当前未验证：
    - 项目编号入口是否可点击
    - 是否能进入 `/project/:id/overview`
    - 项目总览页是否稳定渲染

### 2.3 页面已有明确最小链路

- `frontend-admin/src/pages/project/index.vue`
  - 页面加载后会调用 `getProjectList(...)` 拉取项目列表。
  - 表格中的“项目编号”按钮会跳转 `router.push(\`/project/${row.id}/overview\`)`。
  - 工具栏已有“刷新项目列表”按钮，页面主容器和表格结构稳定。
- 后端已存在项目相关接口：
  - `GET /projects`
  - `GET /projects/{id}`
  - `GET /projects/{projectId}/overview`

## 3. 决策建议

- 本包定位：执行包，不是执行结果。
- 当前最小可行方案：
  - 新增独立 `frontend-admin/e2e/project.spec.ts`
  - 只覆盖“列表可见 -> 点击项目编号 -> 进入总览页 -> 总览核心区域可见”
  - 不在本次范围内扩到创建、编辑、删除、成员管理
- 不建议一次性补齐全 CRUD：
  - 这会把“独立规格化用例”扩成“深度业务写操作增强”，和当前待办边界混在一起。
  - 写路径已在另一条非阻塞剩余项“真实写操作深度增强”中单列，不应混做。

## 4. 执行任务

### 任务 1：新增独立 `project.spec.ts`

- 目标：让项目链路拥有独立 Playwright 入口，不再依赖 UI 冒烟顺带覆盖。
- 文件范围：
  - `frontend-admin/e2e/project.spec.ts`
- 约束：
  - 复用现有 Playwright 登录态方案，不新增新的认证脚手架。
  - 复用仓库现有断言风格，不新增复杂 helper。

### 任务 2：覆盖最小独立链路

- 目标：证明项目链路最小闭环稳定。
- 最小覆盖步骤：
  - 打开 `/project/list`
  - 验证项目页主容器可见
  - 验证搜索栏可见
  - 验证表格区域可见
  - 若表格存在真实项目编号入口，点击首个可点击项目编号
  - 验证 URL 进入 `/project/:id/overview`
  - 验证项目总览页核心容器、标题、卡片或表格区域至少一种稳定元素可见

### 任务 3：处理空数据环境

- 目标：避免把“当前环境没有项目数据”误判成用例失败设计缺陷。
- 执行要求：
  - 若列表为空，测试必须明确给出“当前环境缺少项目样本，无法继续总览跳转验证”的结论。
  - 允许在用例中按现有仓库习惯使用条件分支，但不能把空数据静默当成功。
- 禁止事项：
  - 不允许为了让用例通过而在该用例里临时引入创建项目写操作。
  - 不允许把本用例改造成数据准备脚本。

### 任务 4：与现有冒烟职责去重

- 目标：避免重复堆砌同类断言。
- 要求：
  - `ui-refactor-smoke.spec.ts` 继续保留“页面不空白”职责。
  - 新增 `project.spec.ts` 只承担“项目链路独立规格化”职责。
  - 不需要把同一批截图、同一批基础断言在两个 spec 中重复一遍。

## 5. 建议执行角色

- 主执行角色：前端测试工程执行子智能体
- 协同角色：QA 验证执行人
- 主负责人职责：
  - 只负责边界确认、结果验收、是否通过判断

## 6. 验收标准

- 必须同时满足以下条件：
  - 已新增独立 `frontend-admin/e2e/project.spec.ts`
  - 用例不再仅依赖 `ui-refactor-smoke.spec.ts` 对项目页的顺带覆盖
  - 已覆盖“项目列表 -> 项目总览”最小独立链路
  - 空数据场景被明确处理，而不是静默跳过或伪通过
  - 未把范围膨胀为创建 / 编辑 / 删除 / 成员管理全覆盖

## 7. 风险点

- 数据风险：当前环境如果没有真实项目记录，用例只能验证列表空状态，无法完成总览跳转。
- 选择器风险：若过度依赖脆弱文案或临时 DOM 结构，后续页面微调会导致假失败。
- 边界风险：若把创建、编辑、删除也混入本 spec，会把“独立用例补齐”做成另一轮深写回归。
- 职责风险：若同时修改冒烟 spec 和独立 spec，容易造成覆盖重叠和维护成本上升。

## 8. 通过 / 不通过判定口径

### 通过

- 存在独立 `project.spec.ts`。
- 该 spec 能清楚证明项目链路已拥有最小独立规格化覆盖。
- 空数据场景、跳转场景、总览渲染场景三者至少被明确区分并留有结果。

### 不通过

- 仍旧只有 `ui-refactor-smoke.spec.ts` 的顺带覆盖。
- 新增 spec 只重复“页面可见”而没有进入项目总览。
- 通过临时创建数据或大量 helper 扩写来掩盖当前环境问题。
- 把当前任务扩成 CRUD 全链路，导致边界失控。
