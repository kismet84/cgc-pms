# 第 5 条主线：审批详情业务入口增强任务计划书

## 计划状态

正式版。

本计划先由主会话形成草案，已完成产品/需求负责人、前端设计师、测试验收员、安全审计员四方只读研判，结论均为“修改后采用”。本版已吸收四方调整意见，可作为后续执行分派依据。

## 一、结论

本主线只处理 GAP-005：审批详情缺少显式业务单据入口。

本主线是后续增强，不是阻塞修复。目标是在不重开审批中心闭环主线、不新增后端权限包、不扩成审批中心全量导航重构的前提下，在审批中心内嵌详情和独立审批详情页补一个一致、最小、可验收的“查看业务单据”入口。

MVP 入口策略：

| 业务类型 | 入口文案 | 目标路径 | 说明 |
|---|---|---|---|
| 合同审批 | 查看业务单据 | `/contract/${businessId}` | 当前已有合同详情深链 |
| 采购申请审批 | 查看业务单据 | `/inventory/purchase-request` | 当前先跳列表，不做按 ID 深链 |
| 分包计量审批 | 查看业务单据 | `/subcontract/measure` | 当前先跳列表，不做按 ID 深链 |

硬规则：

1. 内嵌详情与独立审批详情页必须采用同一入口策略。
2. 只有 `businessType` 可识别且 `businessId` 存在时才展示入口。
3. 未识别类型或缺少 `businessId` 时不展示入口，避免误导。
4. workflow 详情只提供导航所需的 `businessType/businessId`，不授予业务单据查看权。
5. 目标页面和目标接口必须继续走业务模块自身鉴权、租户校验和数据范围校验。

## 二、边界

本主线包含：

- 在 `frontend-admin/src/pages/approval/todo.vue` 的内嵌详情区域补“查看业务单据”入口。
- 在 `frontend-admin/src/pages/approval/detail.vue` 的独立审批详情页补同口径入口。
- 补最小前端单测，覆盖入口可见性、跳转映射、未知类型/缺 ID 不展示。
- 运维刷新前端运行态后，做真实浏览器定向复验。
- 归档最终验收报告。

本主线不包含：

- 不新增 workflow 后端接口。
- 不修改 workflow 详情返回契约。
- 不让 workflow 详情绕过业务模块权限。
- 不新增采购申请或分包计量按 ID 详情深链。
- 不处理 GAP-004。
- 不重做审批中心全量导航、搜索、高级筛选、移动端或新审批页面重构。

## 三、任务拆解

### 任务 1：前端最小实现

执行角色：前端设计师 / 前端架构助手。

建议模型：`model=gpt-5.5`，`thinking=high`。

输入：

- 本计划书
- `frontend-admin/src/pages/approval/todo.vue`
- `frontend-admin/src/pages/approval/detail.vue`
- `frontend-admin/src/api/modules/workflow.ts`

执行要求：

1. 在内嵌详情和独立详情页都展示一致的“查看业务单据”入口。
2. 合同审批跳 `/contract/${businessId}`。
3. 采购申请审批跳 `/inventory/purchase-request`。
4. 分包计量审批跳 `/subcontract/measure`。
5. 未识别 `businessType` 或缺少 `businessId` 时不展示入口。
6. 不新增共享组件或跨模块通用映射文件，除非现有代码已有直接可复用模式。
7. 不改审批动作显隐逻辑。

验证要求：

- `pnpm exec vitest run src/pages/approval/__tests__/ApprovalWorkList.test.ts`
- 如独立详情页断言落在现有测试文件中，再补跑对应测试，优先 `src/pages/approval/__tests__/ApprovalConfirm.test.ts`
- `pnpm build`
- `git diff --check -- <本次修改文件>`

### 任务 2：安全边界复核

执行角色：安全审计员。

建议模型：`model=gpt-5.5`，`thinking=medium`。

输入：

- 前端实现回传
- 本计划书
- workflow 详情鉴权链路
- 合同、采购申请、分包计量业务模块权限链路

验收约束：

1. workflow 授权只表示用户可查看审批详情，不表示可查看业务单据详情。
2. 合同深链必须进入合同详情页和 `/contracts/{id}` 权限链。
3. 未授权用户应由合同模块现有 403/404/空态处理，不得由 workflow 返回快照兜底渲染合同详情。
4. 采购申请和分包计量只跳列表，不自动打开或模拟详情。
5. 若后续要做采购申请/分包计量按 ID 深链，必须另拆任务补目标路由、目标 API、tenant/project/data-scope 安全验收。

输出：

- 通过/不通过
- 阻塞/非阻塞
- 是否存在绕过业务模块鉴权的风险
- 是否需要追加修复

### 任务 3：运行态刷新

执行角色：部署运维负责人。

建议模型：`model=gpt-5.4`，`thinking=low`。

触发条件：前端实现和静态验证通过后执行。

执行要求：

1. 刷新 `cgc-pms-frontend-dev` 前端运行态。
2. 如发生重启，等待 3 分钟后再检查。
3. 验证 `http://localhost:5173/` 返回 200。
4. 验证运行态模块已加载新增入口相关关键特征。
5. 若运行态仍旧，再升级为 Vite 缓存清理，不默认先清缓存。

输出：

- 服务名、端口、刷新方式
- 3 分钟等待结果
- 5173 可访问性
- 运行态关键特征命中情况

### 任务 4：真实页面定向复验

执行角色：测试验收员。

建议模型：`model=gpt-5.4`，`thinking=medium`。

浏览器验收范围：3 条主链路 + 1 条只读口径抽查。

主链路：

1. 合同审批详情：点击“查看业务单据”，直达合同详情页。
2. 采购申请审批详情：点击“查看业务单据”，进入采购申请列表页。
3. 分包计量审批详情：点击“查看业务单据”，进入分包计量列表页。

只读抽查：

- `/approval/cc` 或 `/approval/done` 至少抽 1 个页签，确认新增入口后审批详情主体仍可查看，且同意/驳回/撤回/重提等只读口径不回退。

非硬性浏览器项：

- `mine` 页签不作为硬性必验，除非实现明确覆盖该入口形态。
- 未知类型/缺少 `businessId` 以单测为主，不作为真实页面硬性验收样本。

输出：

- 通过/不通过
- 阻塞/非阻塞
- 三条主链路的页面证据
- 只读抽查结论
- 截图/evidence 路径

### 任务 5：质量归档

执行角色：代码质量审计员。

建议模型：`model=gpt-5.5`，`thinking=medium`。

触发条件：实现、安全复核、运行态刷新、测试复验均完成后执行。

输出文件：

- `docs/quality/approval-business-entry-acceptance-2026-07-01.md`

归档内容：

1. 最终结论：通过/不通过，阻塞/非阻塞。
2. 实际修改文件。
3. 命令级验证结果。
4. 浏览器复验证据路径。
5. 安全边界结论。
6. 未纳入本主线的后续增强：采购申请/分包计量按 ID 深链详情。

## 四、执行顺序

1. 前端实现。
2. 安全边界复核。
3. 前端运行态刷新。
4. 测试真实页面复验。
5. 代码质量归档。
6. 主会话最终裁决并关闭第 5 条主线。

## 五、最终验收标准

本主线判定通过需同时满足：

1. 内嵌详情和独立审批详情页入口策略一致。
2. 合同审批可从审批详情直达合同详情。
3. 采购申请审批可从审批详情进入采购申请列表。
4. 分包计量审批可从审批详情进入分包计量列表。
5. 未识别类型或缺少 `businessId` 不展示入口。
6. 不回退 `/approval/cc` 或 `/approval/done` 的只读动作口径。
7. 前端定向单测通过。
8. `pnpm build` 通过。
9. 安全复核未发现绕过业务模块鉴权。
10. 真实浏览器复验通过并保存证据。
11. 质量归档报告完成。

## 六、剩余风险

1. 采购申请和分包计量当前只跳列表，不支持按 ID 自动打开详情；这是本主线明确接受的 MVP 限制。
2. 有 workflow 详情权限但无业务模块权限的用户，点击入口后可能被目标业务模块拒绝；这是正确边界，不作为本主线失败。
3. 若后续产品要求“入口必须只在用户有目标业务模块权限时才显示”，需要另拆权限感知 UX 小包，不在本主线内实现。
