## 主负责人裁决

第 10A 首批最小修复包第二版样本选定为：

- `notification:list`
- 菜单行：`761 消息列表`
- 来源：`V39__init_phase4_menu_perms.sql` / `migration-h2/V39__init_phase4_menu_perms.sql`
- 当前分类：DB-only 历史残留旧读码
- 首选修复方向：将 `sys_menu.id = 761` 的 `perms` 清空为 `NULL`，不改为 `notification:view`

选择理由：

1. `notification:list` 当前只在 migration 与计划/报告文档中出现，未发现后端 `@PreAuthorize`、前端 `hasPermission`、审批 registry 或业务测试把它作为真实权限码消费。
2. 当前后端通知接口的真实读权限是 `notification:view`，写权限是 `notification:edit`；`notification:list` 属于旧 `:list` 风格残留。
3. `notification:list` 不是 `system:*`、`project:*`、`delete`、`submit`、`approve`、`edit` 或批处理触发类权限。
4. 静态 migration 可见 `761 消息列表` 默认只随第 4 阶段菜单授权给 `role_id=1`，未见 `V42` 等后续种子把它授予业务角色。
5. 不把 `notification:list` 改成 `notification:view`，避免把旧残留码转换为真实后端读权限后扩大已绑定角色能力。
6. 清空旧 `perms` 只移除无代码消费的登录权限残留，触达面小于 `alert:edit` 拆码，也小于 `contract:submit`、submit 未绑定、系统管理旧码和项目旧码。

## 冻结范围

## 阻塞项

执行中若命中以下任一情况，必须停止并回主线程裁决：

1. live MySQL 只读复验发现 `sys_role_menu` 中 `menu_id=761` 已被非管理角色绑定。
2. 后端、前端、测试或运行态证据发现 `notification:list` 仍被真实消费。
3. 清空 `perms` 会导致通知页面导航、消息读取、未读数量或系统权限清单出现可见回退。
4. 修复需要同步改 `NotificationController`、前端通知页面、角色绑定或其他权限码。
5. 需要把 `notification:list` 改成 `notification:view` 才能通过验收。
6. 无法访问 live MySQL，导致 `menu_id=761` 运行态绑定范围无法只读确认。
