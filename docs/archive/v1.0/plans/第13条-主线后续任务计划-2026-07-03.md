# 第13条-主线后续任务计划

**创建日期**: 2026-07-03  
**适用阶段**: BCD 模块收口后至上线准备前  
**版本**: v2.0  
**计划属性**: 主线计划，不是功能愿望清单

---

## 一、结论先行

当前阶段的主线不应继续铺开新功能，必须先完成以下三件事：

1. **校准当前真实状态**
   - 重新确认前后端质量门禁、关键链路测试、历史阻断项处理结果。
   - 不沿用旧测试数量、旧审计结论、旧“已完成”表述。

2. **收口上线前阻塞项**
   - 聚焦成本、付款、审批、库存、驾驶舱等高风险链路。
   - 对数据库、认证授权、CI、生产配置按串行方式推进。

3. **输出当前有效上线结论**
   - 明确通过/不通过、阻塞/非阻塞、依据、剩余风险。
   - 历史报告只作为背景，不作为当前放行依据。

**决策建议**: 将后续任务拆为“近期主线”和“中长期储备”两层。近期只做收口与放行准备；报表中心增强、移动端适配、通知运营化、性能深化等进入后续阶段，不占用当前主线资源。

---

## 二、综合研判

### 2.1 当前计划存在的问题

原版本存在以下偏差：

- 把 `266/266`、`89 个 E2E`、`1396/1397`、`发布门禁通过` 等表述直接当作当前状态，但这些数字**需要确认**，不能直接作为现在的项目结论。
- 将“技术债清理”“覆盖率提升到 80%”“E2E 扩展到 150+”“移动端适配”“报表中心增强”等不同性质任务混放在一份主线计划里，导致优先级失真。
- 将“已完成”“已通过”与“需重新验证”的事项混在一起，不符合当前仓库 `docs/未来开发计划.md` 的原则。
- 对上线前真正高风险的事项没有形成收口顺序，尤其是：
  - 历史审计阻断项是否真的关闭；
  - 成本口径在合同、台账、驾驶舱是否一致；
  - 文档索引是否仍把历史报告误当成当前结论。

### 2.2 当前应采用的主线原则

- **原则 1：证据优先**
  - 没有本周证据的“通过”一律不算通过。
- **原则 2：阻塞优先**
  - 优先处理影响上线判断的事项，不优先处理好看但不影响放行的优化项。
- **原则 3：串行推进**
  - 数据库、认证授权、CI、生产配置不可并行大面积推进。
- **原则 4：最小可上线**
  - 先达成“可验证、可交付、可判断能否上线”，再谈扩展能力建设。

## 三、主线目标

### 3.1 近期主线目标（2026-07-03 至 2026-07-11）

在本阶段结束时，项目应达到以下状态：

1. 有一份**当前有效**的质量门禁证据。
2. 有一份**当前有效**的历史阻断项复核结论。
3. 有一份**当前有效**的核心业务链路回归结论。
4. 有一份**明确的上线判断**，而不是停留在“感觉差不多”。

### 3.2 中期目标（上线准备完成后）

- 强化测试覆盖率与自动化深度。
- 完成生产配置、CI/CD、监控、安全加固闭环。
- 再决定是否启动报表中心增强、通知运营化、移动端适配等扩展项。

## 四、主线任务拆解

## 任务 M1：质量门禁复核与证据固化

**定位**: P0，核心阻塞  
**目标**: 重新形成当前真实可复核的前后端质量证据

**执行内容**:
- 复跑后端发布前门禁和关键测试。
- 复跑前端构建、类型检查和必要测试。
- 记录执行时间、分支/提交、环境前提、成功/失败结果。

**输入**:
- 当前仓库代码
- 现有测试集
- `docs/未来开发计划.md`

**输出**:
- 当周门禁报告
- 失败项清单
- 阻塞/非阻塞判断

**验收标准**:
- Given 当前代码基线，When 执行前后端质量门禁，Then 有当周时间戳证据，不依赖历史报告。
- Given 发现失败项，When 汇总结论，Then 每项都有影响范围和责任归属。
- Given 某项只存在旧截图或旧日志，When 判断项目状态，Then 该项标记为“需要确认”。

**风险点**:
- 旧结果和当前代码不一致，导致“历史通过、当前失败”。
- 构建依赖外部环境，若环境不稳定，结论可能失真。

---

## 任务 M2：历史阻断项逐条重判

**定位**: P0，核心阻塞  
**目标**: 对历史审计与修复项重新给出当前有效结论

**执行内容**:
- 整理 2026-06-25 之后的阻断项及修复记录。
- 逐项复核：通过 / 不通过 / 需补证据。
- 对不能稳定复现的项明确标注“需要确认”。

**输入**:
- 历史审计报告
- 修复记录
- 当前运行结果

**输出**:
- 阻断项复核清单
- 剩余风险列表
- 是否允许进入上线准备的判断

**验收标准**:
- Given 一条历史阻断项，When 复核完成，Then 必须能说明当前是否已解决，而不是只说明“曾经修过”。
- Given 修复项无当前证据，When 出具结论，Then 不能标记为“通过”。
- Given 复核发现跨模块回归，When 记录问题，Then 必须同步更新主线风险清单。

**风险点**:
- 修复提交存在，但实际部署链路或数据链路仍未闭环。
- 历史报告描述与当前代码行为可能已经脱节。

---

## 任务 M3：核心业务链路最小回归

**定位**: P0，核心阻塞  
**目标**: 用最小成本验证项目最关键业务链是否可上线

**覆盖范围**:
- 登录与权限
- 项目
- 合同
- 成本
- 付款
- 审批
- 库存
- 驾驶舱

**执行内容**:
- 先跑主成功路径。
- 再补至少一类关键失败路径。
- 重点检查来源单据、台账、汇总、驾驶舱是否一致。

**输入**:
- 可用测试账号
- 稳定测试数据
- 关键页面与接口

**输出**:
- 核心链路回归报告
- 阻塞问题清单
- 建议修复顺序

**验收标准**:
- Given 一条业务链，When 完成主路径操作，Then 页面状态、接口结果、台账汇总三者一致。
- Given 成本链路涉及月份切换，When 在“全部/单月”之间切换，Then 指标和明细变化符合预期。
- Given 涉及审批、付款、库存等失败场景，When 触发异常路径，Then 系统有明确反馈，且不会产生静默错误。

**风险点**:
- 只验证页面，不验证底层台账和汇总，会留下账实不一致风险。
- 只验证 happy path，不验证失败路径，会掩盖上线后高频问题。

---

## 任务 M4：文档索引校准与当前结论统一

**定位**: P0，核心阻塞  
**目标**: 避免历史报告继续冒充当前结论

**执行内容**:
- 校准 `docs/README.md` 中质量报告索引。
- 区分“当前有效结论”和“历史记录”。
- 将本阶段结论挂到统一入口。

**输入**:
- `docs/README.md`
- `docs/quality/`
- 历史报告目录

**输出**:
- 更新后的文档索引
- 当前结论入口

**验收标准**:
- Given 任意读者进入 `docs/README.md`，When 查找质量状态，Then 能直接看到当前有效结论，不会误读历史结论。
- Given 历史报告仍保留，When 在索引中展示，Then 必须明确标注其历史属性和日期。

**风险点**:
- 文档不校准，管理层或执行人会误以为系统已经具备当前上线条件。

---

## 任务 M5：上线准备决策包输出

**定位**: P0，核心阻塞  
**目标**: 形成真正可用于决策的收口报告

**执行内容**:
- 汇总 M1-M4 结果。
- 输出通过/不通过、阻塞/非阻塞、依据、剩余风险。
- 明确下一阶段是否进入生产准备。

**输入**:
- 门禁结果
- 阻断项复核结果
- 核心链路回归结果
- 文档索引校准结果

**输出**:
- 周收口报告
- 上线准备建议
- 剩余风险清单

**验收标准**:
- Given 本阶段工作完成，When 输出收口报告，Then 结论必须明确，不能使用“基本可以”“大体没问题”这类模糊表达。
- Given 仍有阻塞项，When 给出建议，Then 必须标清阻塞原因、影响范围、建议负责人和建议顺序。

**风险点**:
- 没有统一决策包，会导致执行层、管理层和测试层对项目状态理解不一致。

---

## 五、执行顺序

建议严格按以下顺序串行推进：

1. `M1` 质量门禁复核与证据固化
2. `M2` 历史阻断项逐条重判
3. `M3` 核心业务链路最小回归
4. `M4` 文档索引校准与当前结论统一
5. `M5` 上线准备决策包输出

**不建议并行的事项**:
- 数据库结构调整
- 认证授权改动
- CI/CD 流水线重构
- 生产配置改造

原因：这些事项一旦并行推进，会让当前主线的结论失去稳定基线。

## 八、里程碑与验收门

### 里程碑 G1：真相校准完成

**通过条件**:
- 前后端门禁有当前证据
- 历史阻断项已形成最新清单

**不通过条件**:
- 仍依赖历史数字或旧报告作为当前结论

### 里程碑 G2：核心链路收口完成

**通过条件**:
- 成本、付款、审批、库存、驾驶舱核心链路已复核
- 至少完成主成功路径与关键失败路径抽样

**不通过条件**:
- 成本口径在来源单据、台账、汇总、驾驶舱之间不一致且无解释

### 里程碑 G3：上线决策包完成

**通过条件**:
- 有统一的当前有效结论
- 有明确阻塞项和剩余风险
- 可以给出是否进入上线准备的判断

**不通过条件**:
- 仍存在“结论模糊”或“证据缺失”的关键项

## 十、最终判断口径

本计划完成后，只允许输出以下三类结论之一：

1. **通过，可进入上线准备**
   - 依据：门禁、核心链路、阻断项、文档索引均已闭环

2. **不通过，存在阻塞项**
   - 依据：任一 P0 链路或历史阻断项未关闭

3. **需要确认，证据不足**
   - 依据：当前缺乏可复核证据，不能做确定判断

禁止输出：

- “基本可以上线”
- “理论上没问题”
- “大概率通过”

## 十二、按天执行安排

### Day 1：基线校准日

**目标**: 先拿到当前真实状态，不带历史滤镜

**执行任务**:
- 完成 `M1` 质量门禁复核与证据固化
- 启动 `M2` 历史阻断项清单整理

**当日产物**:
- 前后端门禁结果记录
- 当前失败项清单
- 历史阻断项台账初稿

**当日验收口径**:
- 通过：前后端门禁结果可追溯到当天执行证据
- 不通过：仍在引用旧测试数、旧报告、旧截图
- 需要确认：环境不稳定导致结果不可复核

### Day 2：阻断项重判日

**目标**: 明确哪些问题是真的关闭了，哪些只是“修过但未证实”

**执行任务**:
- 完成 `M2` 历史阻断项逐条重判
- 同步更新主线风险清单

**当日产物**:
- 阻断项复核表
- 当前阻塞/非阻塞分类结果

**当日验收口径**:
- 通过：每条阻断项都有“通过 / 不通过 / 需要确认”结论
- 不通过：仍存在“已修复但无当前证据”的模糊项

### Day 3：核心链路主路径回归日

**目标**: 先验证系统是不是“能跑通”

**执行任务**:
- 执行 `M3` 核心业务链路主成功路径回归
- 覆盖登录、项目、合同、成本、付款、审批、库存、驾驶舱

**当日产物**:
- 主路径回归记录
- 首批阻塞问题列表

**当日验收口径**:
- 通过：主成功路径可闭环，页面、接口、台账/汇总一致
- 不通过：任何一条 P0 业务链无法完整走通

### Day 4：关键失败路径与口径核对日

**目标**: 验证系统是不是“经得起出错”

**执行任务**:
- 补跑审批拒绝、库存不足、权限不足、月份筛选异常等失败路径
- 重点完成成本链路口径交叉核对

**当日产物**:
- 失败路径回归记录
- 成本口径差异清单

**当日验收口径**:
- 通过：失败路径有明确反馈且不产生静默错误
- 不通过：成本口径在来源单据、台账、汇总、驾驶舱之间不一致且无解释

### Day 5：索引校准与决策包输出日

**目标**: 把结果固化成真正可用于管理决策的材料

**执行任务**:
- 完成 `M4` 文档索引校准
- 完成 `M5` 上线准备决策包输出

**当日产物**:
- 更新后的 `docs/README.md` 索引方案
- 周收口报告
- 是否进入上线准备的结论

**当日验收口径**:
- 通过：管理层或执行层可直接基于文档得出一致结论
- 不通过：文档中仍混有历史结论冒充当前状态的内容

## 十三、执行分派建议

### 1. 后端执行助手

**职责范围**:
- 负责后端门禁复核
- 负责后端失败项定位
- 负责成本、审批、库存后端口径核对

**建议模型**: 高能力模型（具体型号以当前环境可用模型清单为准）
**建议 thinking**: `high`

**分派任务**:
- `M1` 后端部分
- `M2` 中涉及后端修复闭环的阻断项复核
- `M3` 中涉及后端数据一致性的链路核查

**验收要求**:
- 必须给出命令、结果、失败点、影响范围
- 不能只说“测试通过”，必须说清楚哪类测试、哪条链路、基于哪个提交

### 2. 前端执行助手

**职责范围**:
- 负责前端构建与页面回归
- 负责驾驶舱、合同、成本等重点页面验证
- 负责页面交互异常与 UI 一致性问题归类

**建议模型**: 高能力模型（具体型号以当前环境可用模型清单为准）
**建议 thinking**: `medium`

**分派任务**:
- `M1` 前端部分
- `M3` 中前端主路径与失败路径回归
- `M4` 中涉及前端验收入口的文档核对支持

**验收要求**:
- 必须有页面级证据或操作步骤
- 驾驶舱、成本、合同等重点页面不能只验界面，必须核对关键数值联动

### 3. 测试执行助手

**职责范围**:
- 负责核心链路回归编排
- 负责失败复现与证据记录
- 负责将问题归类为阻塞/非阻塞/需确认

**建议模型**: 轻量模型（具体型号以当前环境可用模型清单为准）
**建议 thinking**: `medium`

**分派任务**:
- `M2` 复核记录整理
- `M3` 全链路回归执行
- `M5` 收口报告证据汇总

**验收要求**:
- 每条问题必须标注复现前提、操作步骤、实际结果、预期结果
- 无法稳定复现的问题不能直接判关闭

### 4. 运维执行助手

**职责范围**:
- 负责环境基线核对
- 负责生产配置、部署前置条件、运行依赖确认
- 负责判断环境问题是否污染测试结论

**建议模型**: 轻量模型（具体型号以当前环境可用模型清单为准）
**建议 thinking**: `low`

**分派任务**:
- `M1` 环境一致性支持
- `M5` 上线准备前置项核对

**验收要求**:
- 必须明确哪些问题属于代码，哪些属于环境
- 环境未稳定前，不能输出“系统通过”的工程结论

---

## 十四、每日站会口径

每日汇报统一使用以下四段：

1. **昨日完成**
   - 只报已落证据的事项，不报口头推进

2. **今日主线**
   - 只报与 `M1-M5` 直接相关的任务

3. **当前阻塞**
   - 明确是代码阻塞、数据阻塞、环境阻塞还是证据阻塞

4. **需要决策**
   - 只上报主线程必须拍板的事项，例如是否降级处理、是否暂停上线准备、是否允许插队修复

---

## 十五、风险升级机制

出现以下任一情况，必须立即升级为主线程关注项：

1. 成本口径在合同、台账、驾驶舱之间出现无法解释的差异
2. 历史阻断项出现“代码修了但当前仍失败”
3. 环境不稳定导致两次以上结果不一致
4. 文档结论与实际运行状态冲突
5. 出现涉及数据库、认证授权、生产配置的新增高风险问题

**升级后的处理原则**:
- 暂停非主线任务
- 先定性是否阻塞上线
- 再决定是否插队修复

---

## 十六、收口标准补充

只有同时满足以下条件，才允许进入“可准备上线”状态：

1. `M1-M5` 全部完成
2. 所有 P0 问题均已关闭或已明确降级且被批准
3. 当前结论基于本周证据，不依赖历史报告
4. 主线程已完成最终通过/不通过判定

若不满足，则默认结论为：

**不通过，继续收口**
