## 1. 报告摘要

### 1.1 审计背景

本次审计针对 CGC-PMS（建筑业企业资质与合同项目管理平台）系统进行全量代码审计，覆盖系统源代码（555个Java文件、93个Vue文件、165个TS/JS文件）、配置文件、第三方依赖、接口调用、权限控制、数据处理逻辑、日志记录、异常处理、安全防护机制等内容。

审计目标包括但不限于：

1. 识别代码中存在的安全漏洞；
2. 发现潜在的业务逻辑风险；
3. 检查敏感信息泄露问题；
4. 评估身份认证、权限控制、数据校验等关键安全机制；
5. 发现代码质量、可维护性、稳定性方面的问题；
6. 提出可落地的整改建议。

### 1.2 审计结论

经审计，本次共发现问题 **41** 项，其中：

| 风险等级 | 数量 |
| ---- | -: |
| 严重   |  2 |
| 高危   |  5 |
| 中危   |  9 |
| 低危   |  19 |
| 信息   |  6 |

总体来看，系统当前安全状态评估为：

**□ 良好　□ 一般　☑ 存在较高风险　□ 存在严重风险**

主要风险集中在以下方面：

1. **生产环境凭据硬编码**：`deploy/.env` 文件中包含真实数据库密码、JWT密钥、Jasypt密钥等，已提交至Git仓库
2. **限流覆盖不全**：仅4个端点配置了速率限制，关键写接口（支付、合同、审批）缺少保护
3. **支付回写幂等性缺陷**：`externalTxnNo`为空时可绕过幂等性检查，导致重复支付回写
4. **日志中泄露敏感数据**：PDF发票识别结果（含完整文本）、外部交易号等被记录在INFO日志
5. **第三方依赖漏洞**：Spring Boot 3.3.5、Guava等可能存在已知CVE

## 4. 系统概览

### 4.1 系统架构说明

系统采用前后端分离架构，前端通过 HTTPS 调用后端 REST API。后端服务使用单体架构（但模块化分包），模块之间通过 Spring Event / ApplicationEventPublisher 解耦通信。核心数据存储在 MySQL，缓存使用 Redis，文件存储使用 MinIO，异步任务通过 @Async 处理。审批工作流使用自定义工作流引擎（基于数据库状态机 + CAS乐观锁）。

### VUL-006：全局限流仅覆盖4个端点

**风险等级：** 严重
**问题类型：** 限流/漏洞防护
**影响模块：** 后端-通用
**影响文件：** 全局Controller（40+个）
**发现方式：** 静态代码扫描 + 人工审计
**状态：** 未修复

#### 6.1 问题描述

项目中实现了@RateLimit注解和Redis限流器，但仅应用于4个端点（登录5次/60s、刷新5次/60s、文件上传20次/60s、仪表盘60次/60s）。所有其他写接口（付款申请、合同创建、审批操作、用户管理）完全没有速率限制。也没有全局IP级别的限流过滤器。恶意攻击者可以对支付、审批等关键接口进行暴力请求或DoS攻击。

#### 6.2 影响范围

1. 所有未配置@RateLimit的Controller端点；
2. 支付、合同、审批、用户管理等关键业务模块。

#### 6.6 修复建议

1. 为所有写操作端点添加@RateLimit注解（建议：20次/60s/用户）；
2. 增加全局IP级别速率限制过滤器（建议：1000次/分钟/IP）；
3. 对支付、审批等关键操作增加更严格的限制。

#### 6.8 整改优先级

**优先级：** P0
**建议完成时间：** 2026-07-04

### VUL-007：PDF发票识别结果在INFO日志泄露

**风险等级：** 高危
**问题类型：** 敏感信息泄露
**影响模块：** 后端-发票识别
**影响文件：** /opt/cgc-pms/backend/src/main/java/com/cgcpms/invoice/service/InvoiceService.java
**影响行号：** 241-247
**发现方式：** 静态代码扫描 + 人工审计
**状态：** 未修复

#### 6.1 问题描述

InvoiceService在提取PDF文本内容后，将完整的PDF文本（最多500字符）以及发票识别结果（发票号码、金额、税率、买卖方名称、卖方税号）全部记录在INFO级别日志中。发票PDF可能包含大量敏感信息（收件人地址、电话、银行账户等），在日志中泄露可能导致合规风险。

#### 6.2 影响范围

1. 所有调用了发票识别功能的操作；
2. 日志文件、日志收集系统。

#### 6.6 修复建议

1. 删除 `log.info("Extracted PDF text ...")` 行；
2. 将识别结果日志降级为DEBUG级别；
3. 对卖方税号等敏感字段进行脱敏。

#### 6.8 整改优先级

**优先级：** P1

#### 6.1 问题描述

PayRecordService.writeback()方法的幂等性检查通过externalTxnNo进行，但当externalTxnNo为null或空时，幂等性检查被完全跳过。支付回写是涉及资金的敏感操作，重复回写可能导致财务账目错误。

#### 6.6 修复建议

1. 当externalTxnNo为null时，生成唯一幂等性令牌（如基于组合字段：orderId+amount+timestamp）；
2. 在数据库层面增加唯一约束防止重复回写；
3. 使用Redis分布式锁确保回写操作的排他性。

**建议完成时间：** 2026-07-09

### VUL-010：@Async缺省ThreadPool配置

**风险等级：** 中危
**问题类型：** 代码质量
**影响模块：** 后端-通用
**影响文件：** /opt/cgc-pms/backend/src/main/java/com/cgcpms/CgcPmsApplication.java
**发现方式：** 静态代码扫描
**状态：** 未修复

#### 6.1 问题描述

@EnableAsync声明在Application类上，但未配置自定义AsyncConfigurer或ThreadPoolTaskExecutor。Spring Boot默认使用SimpleAsyncTaskExecutor，每次@Async调用都创建新线程，无池化、无队列限制。在并发高峰时可能导致OOM。

#### 6.6 修复建议

添加自定义AsyncConfigurer配置类，设置核心线程数、最大线程数、队列容量和拒绝策略。

**优先级：** P2
