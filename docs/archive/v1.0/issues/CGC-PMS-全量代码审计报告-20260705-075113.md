## 4. 数据层正确性

### 4.1 发现的问题

#### [严重] WorkflowQueryService.inSql() SQL 注入
- **文件**: `backend/src/main/java/com/cgcpms/workflow/service/WorkflowQueryService.java`
- **行号**: 84-87
- **描述**: `getMyTodos` 方法中使用 `wrapper.inSql()` 构建子查询，但 `tenantId` 通过字符串拼接嵌入 SQL：
  ```java
  .inSql(WfTask::getInstanceId,
      "SELECT id FROM wf_instance WHERE deleted_flag = 0"
          + " AND tenant_id = " + tenantId
          + " AND instance_status = '" + WorkflowConstants.INSTANCE_RUNNING + "'");
  ```
  虽然 `tenantId` 来自 UserContext（非用户输入），但 `inSql` 完全未转义，若未来变量来源变化则直接暴露 SQL 注入风险。
- **建议**: 替换为 `apply` + `{0}` 参数占位符。

#### [高危] N+1 批量操作 — AlertEvaluationService.batch()
- **文件**: `src/main/java/com/cgcpms/alert/service/AlertEvaluationService.java`
- **行号**: 891-951
- **描述**: batchMarkRead/batchUpdateStatus 循环中每条 SELECT + UPDATE，产生 2N 次数据库调用。
- **建议**: 使用 LambdaUpdateWrapper + in 条件批量更新。

#### [高危] @Transactional 缺少 rollbackFor（3处）
1. **CostSummaryService.refreshSummary** — `@Transactional(isolation = Isolation.REPEATABLE_READ)` 缺 rollbackFor
2. **OrgInitService.backfillTenant** — `@Transactional(propagation = Propagation.REQUIRES_NEW)` 缺 rollbackFor
3. **OperationAuditService.handleAuditEvent** — `@Transactional(propagation = Propagation.REQUIRES_NEW)` 缺 rollbackFor

- **建议**: 统一添加 `rollbackFor = Exception.class`。

#### [中危] Flyway 迁移缺少回滚脚本
- **文件**: `backend/src/main/resources/db/migration/V116-V123`
- **描述**: 最近 8 个迁移文件均无回滚脚本，V119 和 V121 包含不可逆数据变更。
- **建议**: 为包含数据变更的迁移准备回滚 SQL。

#### [中危] WfInstanceMapper.selectAllIncludingDeleted 无租户过滤
- **文件**: `WfInstanceMapper.java:26-28`
- **描述**: @Select 查询不包含 tenant_id 条件。
- **建议**: 添加 `@InterceptorIgnore(tenantLine = "true")` 注解明确声明绕过。

#### [中危] V119 批量 UPDATE 可能锁表
- **文件**: `V119__backfill_alert_domain.sql`
- **描述**: 对 alert_log 表执行大规模 UPDATE 操作，百万级数据可能产生长时间锁表。
- **建议**: 分批执行（LIMIT 循环），低峰期执行。

#### [低危] Flyway V121 列添加缺少幂等保障
- **文件**: `V121__alert_rule_governance.sql`
- **描述**: `ALTER TABLE alert_log ADD COLUMN ...` 不支持 IF NOT EXISTS。
- **建议**: 使用存储过程检查 INFORMATION_SCHEMA.COLUMNS。

#### [低危] Flyway V120 硬编码 ID 可能多环境冲突
- **描述**: 种子数据使用硬编码用户 ID（980000000000000021 等）。
- **建议**: 使用 INSERT IGNORE + 数据库自增。

#### [低危] alert_log 索引选择性低
- **描述**: `idx_alert_process_status` 索引只针对 4 个枚举值的列。
- **建议**: 使用复合索引替代单列索引。

## 8. 前端与交互质量

### 8.1 发现的问题

#### [严重] profile/index.vue 密码正则表达式损坏
- **文件**: `src/pages/profile/index.vue:25`
- **描述**: 密码正则 `PASSWORD_PATTERN` 显示为 `/^(?=....,}$/`，无效正则导致前端密码强度校验完全失效。

#### [高危] alert/index.vue 存在 7 处未保护 console.error
- **文件**: `src/pages/alert/index.vue:235,321,340,388,589,620,674`
- **描述**: console.error(error) 未被 `import.meta.env.DEV` 守卫包裹，生产环境可能泄露内部错误信息。注：esbuild drop:console 在构建时移除，但开发服务器运行时仍有风险。
- **建议**: 使用 `if (import.meta.env.DEV) { console.error(error) }` 包裹。

#### [中危] sessionStorage 存储用户权限和角色信息
- **文件**: `src/stores/user.ts:44-46,57`
- **描述**: 用户信息（roles, permissions）存储在 sessionStorage，同源 XSS 可读取权限结构。
- **建议**: 权限数据仅存储于 Pinia 内存中。

#### [中危] localStorage 引用数据缓存未在登出时清除
- **文件**: `src/stores/reference.ts:48,62`
