# 第32条主线 M6 UI 与质量遗留统一复验入口

日期：2026-07-09
阶段：第32条主线 M6
结论：通过建立正式复验入口；六类遗留已纳入治理清单，但本轮未执行浏览器、覆盖率或 CI 实测，不能宣告问题已验证通过。
阻塞：无主线级硬阻塞；存在两类专项待复验，未复验前不得把旧覆盖率或 E2E 结论当成当前事实。

---

## 1. 本轮裁决

本轮只做统一入口，不做 UI 重构，也不直接硬上覆盖率或 E2E 门禁。

六类遗留按两条复验基线拆分：

1. `UI/体验基线`：UI 一致性、可访问性、登录页品牌、`440px` 移动端布局。
2. `自动化/门禁基线`：覆盖率、E2E CI。

处理顺序统一为：

1. 先复验。
2. 再分类为设计一致性问题、真实功能风险、工具配置问题或环境前置问题。
3. 最后再决定是否拆出整改型 Ready Issue。

本轮不允许直接把六类问题混成一条“大前端重构任务”，也不允许在没有新基线的情况下把覆盖率或 E2E 直接升格为硬门禁。

## 3. 六类问题分组与处理策略

| 类别 | 分组 | 当前口径 | 先复验还是先整改 | 复验重点 |
|---|---|---|---|---|
| UI 一致性 | UI/体验基线 | 有历史桌面审查，但非当前基线 | 先复验 | 列表页结构、工具栏顺序、空态、共享 token 偏离 |
| 可访问性 | UI/体验基线 | 只有历史风险提示，非完整合规结论 | 先复验 | icon-only 按钮语义、键盘可达、读屏文本 |
| 登录页品牌 | UI/体验基线 | 有旧品牌色偏离线索 | 先复验 | token 是否统一、品牌色/圆角/阴影是否仍硬编码 |
| `440px` 移动端布局 | UI/体验基线 | 历史报告明确为高风险但暂缓 | 先复验 | 侧栏挤压、横向溢出、首屏可用性 |
| 覆盖率 | 自动化/门禁基线 | 旧报告数字不可直接采信为当前值 | 先复验 | 当前测量值、阈值声明、CI 是否真阻断 |
| E2E CI | 自动化/门禁基线 | 旧报告与实际状态曾出现冲突 | 先复验 | CI 是否存在 job、是否硬门禁、主链路是否真的纳入 |

结论：

1. 六类里没有一项适合在本轮直接判定“已整改完成”。
2. `440px` 移动端与可访问性要先做真实复验，不能继续只挂观察项。
3. 覆盖率与 E2E CI 要先核对“声明目标 / 当前测量 / CI 执行”三件事，不能只看配置文件或旧报告。

---

## 4. 建议执行顺序

### 4.1 第一优先级：UI/可访问性/移动端复验基线

目标：

1. 用统一页面样本、统一视口、统一检查表重跑一轮基线。
2. 只产出问题分组和阻塞等级，不直接承诺全站整改。

建议样本：

1. 合同台账
2. 成本汇总
3. 结算列表
4. 项目列表
5. 系统字典
6. 登录页

建议视口：

1. `1600 x 1200`
2. `440 x 956`

建议产出：

1. 每页“通过 / 不通过 / 观察项”
2. UI 一致性问题清单
3. 可访问性问题清单
4. 登录页品牌偏离清单
5. `440px` 布局问题清单

### 4.2 第二优先级：覆盖率 / E2E CI 基线复验

目标：

1. 把“当前测量值”“配置阈值”“CI 是否真阻断”三张表对齐。
2. 只建立基线，不直接抬高阈值，不直接硬上门禁。

建议产出：

1. 后端覆盖率当前值、门禁阈值、偏差
2. 前端覆盖率当前值、门禁阈值、偏差
3. E2E 现有 spec / test 数、当前 CI job 状态、是否硬门禁
4. 哪些是工具配置问题，哪些是环境前置问题，哪些才是真实质量债
