# CI 红灯分诊模板

用于 GitHub Actions、本地门禁脚本或验收阶段出现红灯时的最小分诊。先分类，再处理，未分类前不要直接判成代码失败。

## 三类失败

### 1. 工具配置类

典型信号：

- 本机缺少工具或版本不对
- CI job 入口、工作目录、触发分支、凭据注入错误
- 规则文件未加载、脚本路径写错、命令参数失配

处理动作：

1. 复述失败 job、步骤名、原始报错关键词。
2. 核对 workflow、脚本入口、所需工具与环境变量。
3. 先修配置或补前置，再复验。

### 2. 环境前置类

典型信号：

- Docker / WSL / 数据库 / 端口未就绪
- 服务未达到稳定等待时间
- 前端 dev server 仍指向旧 backend
- 测试数据、账号、种子或外部依赖未准备好

处理动作：

1. 核对依赖服务、端口、容器、数据库和等待时间。
2. 若页面回到 `/login` 且日志指向旧 `172.19.x.x:8080`，先刷新前端运行态。
3. 环境恢复后，再复验同一门禁。

### 3. 真实质量/安全类

典型信号：

- 测试失败、构建失败、类型检查失败
- 接口契约不符、权限越权、安全扫描真实命中
- 数据一致性错误、迁移失败、逻辑回归

处理动作：

1. 固定失败用例、接口或模块范围。
2. 记录复现命令与最小证据。
3. 进入实现整改或阻塞裁决。

## 最小处理顺序

1. 记录失败事实：job、步骤、分支、提交、报错关键词。
2. 先做失败分类：工具配置 / 环境前置 / 真实质量安全。
3. 工具配置类先修配置。
4. 环境前置类先恢复环境并复验。
5. 只有真实质量/安全类才进入代码整改、阻塞判定或上线驳回。

## 最小回报模板

```text
失败任务=
失败步骤=
失败分类=工具配置类 / 环境前置类 / 真实质量安全类
关键证据=
当前处理=
下一步=
是否阻塞=阻塞 / 非阻塞 / 需要确认
```
