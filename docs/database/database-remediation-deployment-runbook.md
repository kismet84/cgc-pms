# 数据库模型治理 V195—V210 部署与回滚手册

> 适用范围：CGC-PMS 从已验证的 V194 基线升级到 V210。本文不构成生产执行授权。

## 1. 发布前硬门禁

1. 确认目标环境、schema、当前 Flyway 版本和备份恢复责任人；生产操作必须另获批准。
2. 创建可验证的全量备份，并在隔离环境完成一次恢复校验，记录恢复耗时和校验结果。
3. 在生产等价脱敏副本执行 `scripts/database/database-remediation-preflight-v194.sql`。任何 `BLOCK` 结果非 0 均停止；`REVIEW` 结果必须逐项签字。
4. 核对应用版本包含 V195—V210 的 MySQL/H2 migration、对应实体/服务和测试，不允许只发布 DDL。
5. 估算核心表行数和索引构建窗口；变更窗口内禁止并发运行旧版本写节点。

## 2. 发布顺序

1. 停止旧版本写流量，保留只读健康检查。
2. 再次记录 `flyway_schema_history` 最大成功版本、失败数和 schema 校验和。
3. 启动单个新版本后端节点，让 Flyway 顺序执行 V195—V210；禁止手工跳版本或关闭校验。
4. 确认 Flyway 最大版本为 210、失败记录为 0，再运行 `scripts/database/database-remediation-postflight.sql`。
5. 所有 `BLOCK` 必须为 0；`REVIEW` 必须与发布前签字结果一致或得到新的明确处置。
6. 执行登录、RBAC、审批提交、采购/退料、银行回单转回款、项目列表和物料分类冒烟。
7. 恢复流量并观察错误率、慢查询、锁等待、库存/成本守恒和回款幂等指标。

## 3. 迁移检查重点

|版本|重点检查|
|---|---|
|V195—V196|活动业务键重复为 0；`cost_item` 同源同类型没有重复事实|
|V197|用户—角色、角色—菜单无孤儿和错租户|
|V198—V200|退料来源完整；累计退料不超原出库；质量处置和供应商退货可追溯|
|V201|已匹配入账回单唯一关联回款；重复外部流水不重复入账|
|V202—V204|审批规则唯一确定；项目审批 handler 可回写业务状态|
|V205—V209|冗余状态不再写；材料都有同租户分类；JSON 合法且不超过 1 MiB；类型注册完整；双预警中心同步|
|V210|14 个旧 `deleted_token` 均不存在；15 个 `active_unique_token` 及其唯一索引完整保留|

## 4. 回滚策略

- V195—V210 包含结构约束、新事实和兼容列退役，禁止通过反向 DROP 脚本“快速回滚”。首选回滚应用流量并保留已迁移结构；回退应用必须先确认不读取已退役的 `deleted_token`。
- 若迁移尚未开始，直接恢复旧应用；若 Flyway 中途失败，停止所有写入，保留现场，按失败版本修复后追加新 migration，不篡改既有脚本和 checksum。
- 若迁移完成但业务冒烟失败，先回滚应用节点并进入只读模式；对新事实进行差异审计。只有在确认没有 V195+ 新写入或已完整导出后，才允许从已验证备份整体恢复。
- `mat_material_return_reversal`、供应商退货、回款、成本和会计事实均采用冲销/补偿，不物理删除原事实。
- `overhead_allocation_record` 不属于 V195—V210 物理删除范围；其退役由 `ISSUE-049-001` 独立审批和发布。

## 5. 发布后收口

1. 重新运行 `scripts/database/generate-schema-docs.ps1`，审查结构字典和 ER 差异。
2. 保存关键命令摘要：Flyway 版本、postflight 结果、冒烟状态、回滚点和观察窗口结论；不保存含密钥的原始日志。
3. 若任一数据正确性、租户隔离、金额守恒或幂等检查失败，发布结论必须为不通过，不能降级为“非阻塞”。
