# 11 安全规范

## 角色入口

主负责人：安全审计员。审计类任务以仓库现状和代码证据为准，正式审计报告写入 `docs/quality/`。

## 认证安全

1. JWT + HttpOnly Cookie，前端不存 Token
2. 生产 Cookie 必须 Secure + HTTPS
3. JWT secret 来自环境变量，不用默认值
4. Refresh Token 失败清理用户状态并跳转登录

## 授权安全

1. 所有保护接口必须鉴权
2. 关键接口 `@PreAuthorize` 或等价校验
3. Service 层校验租户和业务数据归属
4. 管理员权限不被普通角色通过菜单配置绕过

## 多租户与项目隔离

1. 主业务表必须有 `tenant_id`
2. 查询按当前租户过滤
3. 项目级业务按项目权限过滤
4. 导出、统计、驾驶舱不绕过数据权限

```java
Long tenantId = UserContext.getCurrentTenantId();
queryWrapper.eq("tenant_id", tenantId);
```

## 输入校验

1. 后端强制校验必填、长度、枚举、金额范围
2. 前端校验仅为体验优化
3. 文件名、备注等自由文本不进入 SQL 拼接
4. SQL 使用 MyBatis 参数绑定
5. 导入/上传限制文件类型和大小

## SQL 注入静态扫描门禁

### CI 集成

SQL 注入扫描已通过 `scripts/check-sql-safety.ps1` 纳入 GitHub Actions CI（`sql-safety-scan` job），每次 push/PR 自动执行。

扫描规则覆盖 6 种常见注入模式：

| # | 规则 | 说明 |
|---|------|------|
| 1 | `${}` MyBatis 字符串替换 | 字符串拼接替代参数绑定 |
| 2 | `@Select/Update/Delete/Insert` + 字符串拼接 | 注解 SQL 与 Java 字符串拼接 |
| 3 | `.apply()` | MyBatis-Plus 动态 SQL 注入点 |
| 4 | `.last()` | 拼接任意 SQL 片段 |
| 5 | `.having()` | 动态 HAVING 子句 |
| 6 | `Statement` | 原始 JDBC Statement |

### 豁免机制

如果扫描命中的代码确实安全（如服务端枚举值插值），在同一行添加注释：

```java
// SQL-SAFETY: server-side-enum
```

豁免标记必须在出现 `apply`/`last`/`having` 调用的代码行上。注释缺省时 CI 会按注入风险报告。

### 误报处理流程

1. 确认命中行属于服务端枚举值直传（不接受外部字符串输入）
2. 添加 `// SQL-SAFETY: server-side-enum` 注释到同一行
3. 重新运行 `powershell -ExecutionPolicy Bypass -File scripts\check-sql-safety.ps1` 确认通过
4. 提交代码，CI 将识别豁免并放行

### 注意事项

- 豁免是**诊断性质**的，不是掩盖问题的手段。如果对某行是否需要豁免有疑问，优先补参绑定而非豁免
- 不得在 `.apply()` 中拼接任何用户输入或不可信值
- 门禁的 pipeline 权限不应被临时关闭；误报走豁免流程，设计缺陷走修复流程

## 文件安全

1. 通过后端上传，前端不持有永久 MinIO 密钥
2. 下载校验当前用户对业务对象的权限
3. 预签名 URL 设置有效期
4. 合同、发票、结算附件不公开桶访问
5. 文件记录保留上传人、时间、业务类型、业务 ID

### MinIO 服务账号（R-47e）

> **安全原则：应用不应使用 MinIO root 凭据进行日常操作。**

#### 推荐做法

1. **创建专用服务账号**：为 CGC-PMS 应用创建一个独立的 MinIO 用户（非 root），仅授予应用所需桶的最小权限。
2. **权限最小化**：为每个桶配置精确的 IAM policy，只开放应用实际需要的操作（`s3:GetObject`, `s3:PutObject`, `s3:DeleteObject`, `s3:ListBucket`），禁止管理类操作（如创建/删除桶、修改用户）。
3. **凭据隔离**：服务账号的 Access Key / Secret Key 通过环境变量 `MINIO_ACCESS_KEY` / `MINIO_SECRET_KEY` 注入，不与 `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD` 共用。

#### 创建命令示例

```bash
# 1. 进入 MinIO 容器
docker exec -it cgc-pms-minio-dev /bin/sh

# 2. 创建 IAM policy（保存为 /tmp/cgc-pms-policy.json）
cat > /tmp/cgc-pms-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::cgc-pms-*"
      ]
    }
  ]
}
EOF

# 3. 创建应用服务账号
mc admin policy create myminio cgc-pms-app /tmp/cgc-pms-policy.json
mc admin user add myminio cgc-pms-app-user <生成的强密码>
mc admin policy attach myminio cgc-pms-app --user cgc-pms-app-user

# 4. 生成 Access Key / Secret Key
mc admin user svcacct add myminio cgc-pms-app-user \
  --access-key "<生成的AK>" \
  --secret-key "<生成的SK>"
```

#### 环境变量配置

```bash
# .env 中增加（不要与 MINIO_ROOT_* 相同）
MINIO_ACCESS_KEY=<服务账号 AK>
MINIO_SECRET_KEY=<服务账号 SK>
```

## 敏感配置

不提交到仓库：生产数据库密码、JWT secret、MinIO secret key、Redis 密码、HTTPS 私钥、第三方系统密钥。

`.env.example` 只提供示例值。`.env` 按环境隔离管理。

## 密钥管理策略

### 目标：向 12-Factor App 演进

当前项目混合使用两种密钥注入方式：

| 方式 | 载体 | 用途 | 状态 |
|------|------|------|------|
| 环境变量 (`JWT_SECRET` 等) | `.env` / Docker | 运行时注入 | ✅ 推荐 |
| Jasypt `ENC()` 加密值 | `application-*.yml` | 静态配置加密 | ⚠️ 过渡方案 |

演进路径：

1. **短期（当前）**：Jasypt 保留，用于 dev/test 环境向后兼容。`JASYPT_ENCRYPTOR_PASSWORD` 通过环境变量注入，不写入 YAML。
2. **中期**：将 `application-*.yml` 中的 `ENC()` 值逐个迁移为环境变量占位符 `${VAR_NAME}`，减少对 Jasypt 的依赖。
3. **长期**：生产环境引入 Spring Cloud Config + HashiCorp Vault（或云原生 Secret Manager），实现密钥集中管理、动态刷新、审计追溯。

### Jasypt 使用规范

- `JASYPT_ENCRYPTOR_PASSWORD` **必须**通过环境变量传入，禁止硬编码在任何配置文件中。
- `JASYPT_ENCRYPTOR_PASSWORD` 与 `JWT_SECRET` **必须使用不同的值**，禁止共用。
- 新增配置项优先使用环境变量占位符，而不是新增 `ENC()` 值。
- 生产环境密码不要用 Jasypt CLI 在线加密后提交；应使用 CI/CD 管道在部署时注入明文或通过 Vault 动态获取。

### 环境变量检查清单

启动前确认以下变量已设置且互不相同：

| 变量 | 用途 | 最低要求 |
|------|------|----------|
| `JWT_SECRET` | JWT 签名 | ≥ 32 字符随机串 |
| `JASYPT_ENCRYPTOR_PASSWORD` | 配置文件解密 | ≥ 32 字符随机串，**与 JWT_SECRET 不同** |
| `MYSQL_ROOT_PASSWORD` | 数据库 root | 强密码 |
| `MYSQL_PASSWORD` | 数据库应用账号 | 强密码，不与 root 相同 |
| `REDIS_PASSWORD` | Redis 认证 | 强密码 |
| `MINIO_ROOT_PASSWORD` | MinIO 管理员 | 强密码 |

> 生成命令：`openssl rand -base64 32`

## JWT 过期配置统一

### 现状

各 profile 的 JWT `expiration` 值不一致：

| Profile | 文件 | `jwt.expiration` | 等效值 |
|---------|------|------------------|--------|
| `dev` | `application-dev.yml` | `900000` | **15 分钟** |
| `prod` | `application-prod.yml` | `900000`（可通过 `${JWT_EXPIRATION}` 覆盖） | **15 分钟** |
| `test` | `application-test.yml` | `86400000` | **24 小时** |
| `local` | `application-local.yml` | `86400000` | **24 小时** |

> `refresh-expiration` 四个环境一致为 `604800000`（7 天），无需调整。

### 风险

- **test 和 local 使用 24h 过期**，与实际生产（15min）偏差过大，导致：
  - 开发/测试环境无法暴露 Token 过期相关的 UI 交互问题（如自动刷新、过期跳转）
  - CI 集成测试在长时间运行中可能使用过期 Token 而不报错，掩盖鉴权缺陷
  - 开发者习惯 24h 不过期后，容易忽视生产环境的短 Token 约束

### 建议

统一为 **15 分钟**，local 环境作为开发便利可放宽至 **1 小时**：

| Profile | 推荐值 | 理由 |
|---------|--------|------|
| `dev` | `900000`（15 min） | ✅ 与生产对齐，已达标 |
| `prod` | `900000`（15 min） | ✅ 安全基线，已达标 |
| `test` | `900000`（15 min） | 🔧 需修改，与生产对齐以暴露鉴权问题 |
| `local` | `3600000`（1 h） | 🔧 需修改，开发便利与安全实践的折中 |

### 执行步骤

1. 修改 `application-test.yml`：`expiration: 900000`
2. 修改 `application-local.yml`：`expiration: 3600000`
3. 前端确认 Token 过期后的自动刷新/跳转逻辑在 15min（及 1h）场景下正常工作
4. CI 流水线中 Token 过期相关测试用例覆盖 15min 场景

## 日志与审计

必须记录：登录/退出、增删改、提交审批、审批处理、文件上传下载、权限不足和认证失败。

不记录：明文密码、Token、Cookie 完整值、密钥、大文件内容。

### 日志脱敏增强（R-47d）

后端提供 `SensitiveDataUtils` 工具类用于在日志中屏蔽敏感字段：

```java
// 密码字段
log.info("Login: {} {}", username, SensitiveDataUtils.maskPassword());
// → Login: admin ****

// Token 值 — 仅显示前 8 字符
log.debug("Token: {}", SensitiveDataUtils.maskToken(rawToken));
// → Token: eyJhbGci...

// 按字段名自动判断
SensitiveDataUtils.maskFieldValue("password", "secret123");  // → ****
SensitiveDataUtils.maskFieldValue("accessToken", "abc123def456..."); // → abc123de...
```

`SensitiveDataMaskingAspect` 自动拦截 Controller 层方法调用，在 DEBUG 日志中输出脱敏后的请求参数。

## Nginx 安全头（R-25）

> 生产 Nginx 配置 `frontend-admin/nginx.conf` 已包含以下安全头，部署时无需额外配置。

| 头部 | 配置值 | 作用 |
|------|--------|------|
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains; preload` | 强制浏览器使用 HTTPS |
| `Content-Security-Policy` | `default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'; frame-ancestors 'none'; form-action 'self'` | 限制资源加载来源，防 XSS |
| `X-Frame-Options` | `DENY` | 禁止页面被嵌入 iframe，防点击劫持 |
| `X-Content-Type-Options` | `nosniff` | 禁止浏览器 MIME 类型嗅探 |
| `X-XSS-Protection` | `1; mode=block` | 启用浏览器 XSS 过滤器（旧版浏览器兼容） |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | 控制 Referer 头传递策略 |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` | 禁用不必要的浏览器 API |

### CSP 定制说明

当前 CSP 中的 `'unsafe-inline'` 和 `'unsafe-eval'` 是为兼容 Ant Design Vue 等 UI 框架的内联样式。如果后续迁移到 strict CSP：

1. 使用 nonce 或 hash 替代 `'unsafe-inline'`
2. 将内联脚本提取为外部 `.js` 文件
3. 对于 `'unsafe-eval'`（Vue 模板编译器需要），生产环境可用 CSP `script-src` 配合 `'wasm-unsafe-eval'` 替代

### 生产环境检查

部署后使用以下命令验证安全头：

```bash
curl -I https://your-domain.com 2>&1 | grep -iE 'strict-transport|content-security|x-frame|x-content|x-xss|referrer-policy|permissions-policy'
```

1. 生产 `JWT_SECRET` 已替换
