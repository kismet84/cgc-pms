# 现场确认结果回填安排单

## 1. 文档定位

本文件用于组织一次真实的“现场确认结果回填”会话，目标是在一次会话内把 seeded 文件按真实现场情况完成回填准备。

使用边界：
- 本文件是回填安排单，不是已完成回填结果，也不是已完成签收结论。
- 本文件不填写真实账号、真实密码、真实审批模板名、真实回填结论。
- 本文件只定义这次回填会要做什么、谁参加、带什么资料、现场产出什么，以及会后如何更新 seeded 文件与正式手册。

## 2. 目标

本次回填会的直接目标是：
- 现场更新 `pilot-confirmation-record-seeded.md`
- 现场更新 `pending-items-signoff-seeded.md`

本次会话要把账号、角色、密码、审批配置与生产经理能力边界放在同一次回填中完成记录与签收准备，不拆成两场会。

## 3. 范围

本次仅覆盖以下事项的真实现场确认与回填安排：
- 试点租户范围
- 账号开通范围
- 角色归属
- 初始密码发放口径
- 密码重置口径
- 审批模板命名规范
- 审批人映射方式
- 生产经理能力边界

本次不覆盖：
- 项目周会机制
- 长期培训运营机制
- 新增培训材料体系
- seeded 文件之外的提前正式回写

## 4. 会前准备

会前必须准备并携带以下资料：
- `external-input-checklist.md`
- `pilot-execution-task-card.md`
- `pilot-confirmation-runbook.md`
- `pilot-confirmation-record-seeded.md`
- `pending-items-signoff-seeded.md`
- `confirmation-writeback-sop.md`
- `../manuals/production-manager-capability-boundary.md`（如本次涉及生产经理能力边界核对）

会前准备要求：
- 实施负责人先确认本次试点范围、会议时间、参会角色和记录人。
- 系统管理员准备租户、账号、角色、密码口径相关现场核对资料。
- 流程管理员准备审批模板、审批人映射相关现场核对资料。
- 培训讲师准备生产经理最小演示路径与当前培训口径。
- 业务代表准备现场确认意见，不得以历史口头认知替代本次会话确认。

## 5. 参会角色

| 角色 | 会议职责 |
|------|------|
| 实施负责人 | 锁定范围、主持会议、处理阻塞项 |
| 系统管理员 | 提供账号、角色、密码相关资料与说明 |
| 流程管理员 | 提供审批模板、审批人映射相关资料与说明 |
| 培训讲师 | 按当前手册边界演示生产经理最小操作路径 |
| 业务代表 | 对试点使用口径和生产经理能力边界给出确认意见 |
| 记录人 | 现场回填 seeded 文件，记录证据、结论和未完成项 |

## 6. 会议议程

| 顺序 | 议程 | 责任角色 | 现场产出 |
|------|------|------|------|
| 1 | 确认本次回填范围与参与角色 | 实施负责人 + 记录人 | 本次确认范围、到场角色、缺席角色说明 |
| 2 | 核对外部输入资料是否齐备 | 实施负责人 + 系统管理员 + 流程管理员 | 可确认项清单、缺失资料清单 |
| 3 | 逐项核对账号、角色、密码、审批事项 | 系统管理员 + 流程管理员 + 记录人 | `pilot-confirmation-record-seeded.md` 对应项现场记录 |
| 4 | 同场核对生产经理能力边界 | 培训讲师 + 业务代表 + 记录人 | 生产经理能力边界现场记录与确认意见 |
| 5 | 汇总是否可签收 | 实施负责人 + 业务代表 + 记录人 | `pending-items-signoff-seeded.md` 初步签收结果 |
| 6 | 明确会后回写与未完成项责任人 | 实施负责人 + 记录人 | 待回写项列表、未完成项补齐计划 |

## 7. 现场要回填的文件

本次会议现场必须直接更新以下文件：
- `pilot-confirmation-record-seeded.md`
- `pending-items-signoff-seeded.md`

现场更新要求：
- 先更新 `pilot-confirmation-record-seeded.md`，逐项写明状态、责任角色、证据、现场结论和后续动作。
- 再更新 `pending-items-signoff-seeded.md`，逐项写明“通过 / 不通过 / 保留待确认”。
- 仅当签收结论为“通过”时，相关事项才允许进入 `confirmation-writeback-sop.md` 的正式回写流程。

## 8. 现场产出

本次会话结束时，至少应形成：
- 已现场补充的 `pilot-confirmation-record-seeded.md`
- 已现场补充的 `pending-items-signoff-seeded.md`
- 一份待回写事项列表
- 一份未完成项清单，注明缺什么、谁补、何时补

## 9. 完成标准

本次回填安排执行完成，必须同时满足：
- seeded 记录单已覆盖本次范围内全部确认项；
- seeded 签收单已对每项给出“通过 / 不通过 / 保留待确认”之一；
- 生产经理能力边界已并入同一次会话完成记录与签收判断；
- 只有签收为“通过”的事项被纳入 `confirmation-writeback-sop.md`；
- 未通过或待确认事项未被写成正式结论。

## 10. 未完成项处理

若现场仍有事项未完成，按以下方式处理：
- 缺资料：在 `pilot-confirmation-record-seeded.md` 和 `pending-items-signoff-seeded.md` 保留“待确认”或“保留待确认”，并注明缺失资料与责任人。
- 口径冲突：签收结论写“不通过”或“保留待确认”，不得直接进入正式回写。
- 生产经理能力边界演示不完整：继续保留待确认，不单独拆第二场会，待补证据后回到同一回填链路处理。
- 会后仅整理已签收“通过”的事项进入 `confirmation-writeback-sop.md`；其余事项继续留在 seeded 文件中跟踪。
