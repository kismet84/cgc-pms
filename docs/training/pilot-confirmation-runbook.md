# 试点外部确认执行说明

## 1. 文档定位

本文件是“如何执行试点外部确认”的说明文档，不是已执行结果，也不是已确认结论。

使用边界：
- 只说明从确认清单出发，到现场记录，再到签收的最小执行顺序。
- 不填写真实账号、真实密码、真实模板名、真实签收结论。
- 未完成现场记录和签收的内容，不得当作已确认事实。

## 2. 前置条件

- 已明确本次确认对应的试点范围、确认场景和组织角色。
- 已准备 [试点账号与审批模板确认清单](pilot-account-and-approval-checklist.md) 作为确认起点。
- 已准备 [试点现场确认记录单模板](pilot-confirmation-record.md) 用于逐项留痕。
- 已准备 [待确认事项签收单模板](pending-items-signoff.md) 用于形成最终签收结论。
- 涉及生产经理培训边界时，已同步准备 [生产经理能力边界确认单](../manuals/production-manager-capability-boundary.md)。

## 3. 参与角色

| 角色 | 主要职责 | 主要输出 |
|------|------|------|
| 实施负责人 | 组织确认范围、召集参与人、协调阻塞项 | 确认场景说明、阻塞升级结论 |
| 系统管理员 | 提供账号、角色、密码重置、环境配置相关证据 | 账号与权限类现场口径 |
| 流程管理员 | 提供审批模板、审批人映射相关证据 | 审批配置类现场口径 |
| 培训讲师 | 按当前有效手册控制培训口径，不放大承诺 | 培训边界说明 |
| 业务代表 | 对试点使用口径做现场确认 | 业务侧确认意见 |
| 记录人 | 统一填写记录单和签收单 | 现场记录、签收留痕 |

## 4. 执行顺序

### 第一步：确定本次确认范围

输入：
- `pilot-account-and-approval-checklist.md`
- `production-manager-capability-boundary.md`（如本次涉及生产经理）

执行动作：
- 逐项标出本次必须确认的事项。
- 将“账号与审批类事项”和“生产经理能力边界事项”放入同一批现场确认范围，不拆成两套流程。

输出：
- 本次确认项清单
- 参与角色名单

通过条件：
- 每个确认项都有责任角色和预期证据方式。

失败处理：
- 若责任角色不清，记录人为“需要确认”，不得进入现场结论阶段。

### 第二步：按清单逐项现场核对

输入：
- 第一步确认项清单
- 对应截图、清单、演示、口头说明等现场证据

执行动作：
- 按 `pilot-confirmation-record.md` 逐项填写“当前状态、责任角色、证据要求、现场结论、后续动作、回写位置”。
- 对账号、密码、审批模板、审批人映射逐项核对。
- 如涉及生产经理，按 `production-manager-capability-boundary.md` 核对“当前可培训范围、当前不能承诺范围、仍需确认范围”。

输出：
- 已填写的现场确认记录

通过条件：
- 每个项至少形成一种可回查证据，且现场结论只能写“已确认 / 待确认 / 不通过 / 不适用”之一。

失败处理：
- 只有口头说明、没有证据时，现场结论固定写“待确认”。
- 发现与现有手册冲突时，现场结论写“不通过”，并记录冲突点。

### 第三步：整理可签收项与不可签收项

输入：
- 已填写的 `pilot-confirmation-record.md`

执行动作：
- 从记录单中筛出具备责任角色、证据、现场结论、回写位置的事项。
- 区分哪些事项可以进入签收判断，哪些事项只能继续保留待确认。

输出：
- 待签收事项列表
- 保留待确认事项列表

通过条件：
- 待签收事项都能指出来源记录位置和拟回写位置。

失败处理：
- 若回写位置不明确，不进入签收，继续保留待确认。

### 第四步：形成签收结论

输入：
- `pilot-confirmation-record.md`
- `pending-items-signoff.md`

执行动作：
- 由责任角色按记录单逐项给出“通过 / 不通过 / 保留待确认”。
- 对生产经理能力边界，只有在现场已核对角色范围、入口、最小操作路径和培训口径一致时，才允许签收为“通过”。

输出：
- 已填写的签收单

通过条件：
- 签收为“通过”的事项，均写明签收依据和是否允许回写正式手册。

失败处理：
- 任一项缺少签收依据，结论不得写“通过”。
- 生产经理能力边界若仍依赖历史认知或口头承诺，结论固定为“保留待确认”或“不通过”。

### 第五步：移交回写

输入：
- 已签收的 `pending-items-signoff.md`

执行动作：
- 只把签收为“通过”的事项移交给文档回写责任人。
- 未通过或保留待确认事项继续留在原清单和记录单，不进入正式手册。

输出：
- 待回写事项列表

通过条件：
- 每个待回写事项都能对应到明确目标文件。

失败处理：
- 若拟回写内容仍包含未确认表述，退回记录人与审核人复核。

## 5. 每一步输入输出速查

| 步骤 | 输入 | 输出 | 完成条件 |
|------|------|------|------|
| 范围确定 | 确认清单、能力边界确认单 | 本次确认项清单 | 确认项、责任人、证据方式齐全 |
| 现场核对 | 确认项清单、现场证据 | 现场确认记录 | 每项有状态和结论 |
| 签收准备 | 现场确认记录 | 待签收事项列表 | 已能区分可签收与不可签收 |
| 签收判断 | 现场确认记录、签收单 | 已填写签收单 | “通过”项有依据和回写位置 |
| 回写移交 | 已填写签收单 | 待回写事项列表 | 只包含允许回写事项 |

## 6. 常见失败口径

| 场景 | 统一口径 |
|------|------|
| 现场只有口头说法，没有截图、清单或演示 | 记为“待确认”，不得口头升级为已确认 |
| 账号、密码、审批模板存在真实敏感信息 | 不写入本执行说明、记录单模板或签收单模板，只记录确认方式和责任角色 |
| 审批模板命名或审批人映射不同人说法不一致 | 记为“不通过”，待责任人统一口径后再复核 |
| 生产经理能力边界与手册冲突 | 以当前有效手册和确认单为准，冲突项不得签收通过 |
| 找不到正式回写位置 | 不进入回写，继续保留待确认 |

## 7. 完成定义

本执行说明对应的流程完成，必须同时满足：
- 确认清单已逐项进入现场记录；
- 现场记录已形成签收判断输入；
- 签收单已区分“允许回写”和“禁止回写”；
- 生产经理能力边界如在本次范围内，已并入同一记录与签收流程；
- 未确认事项仍保留为未确认，不被写成既成事实。
