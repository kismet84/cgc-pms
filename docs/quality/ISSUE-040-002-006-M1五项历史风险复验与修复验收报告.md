## ISSUE-040-006 / V-06

### 首次失败分类与修复

1. `mysql-restore.ps1` 用 `\`` 错误拼接数据库标识符，MySQL 收到非法 `\$` 命令；归类为真实脚本质量问题。
2. `mysql-backup.ps1` 把 `mysqldump` 先捕获为 PowerShell 文本再重编码，中文 SQL 被破坏并在恢复时触发唯一键碰撞；恢复端文本管道存在同类风险。
3. 最小修复：正确生成反引号标识符；备份改为原始 stdout 临时文件再用二进制流 gzip；恢复改为二进制解压文件并通过标准输入导入。

### 真实恢复演练

- 环境：`dev`。
- 数据库绑定：`127.0.0.1:3307`。
- 数据重置 marker：存在。
- 源库：本地 `cgc_pms`。
- 目标：隔离库 `cgc_pms_restore_test`，未连接或覆盖生产。
- 结果：备份 38194 bytes，恢复耗时 7.13 秒，恢复后 74 张表。
- 脱敏校验：SHA-256 后 6 位 `AE95BC`。
- 样本计数一致：`sys_dict_data 92/92`、`sys_menu 131/131`、`flyway_schema_history 143/143`、`pm_project 1/1`、`ct_contract 0/0`。

### 凭据与 CI 只读核对

- `.github/workflows/ci.yml` 引用 `CI_TEST_JWT_SECRET`、`CI_MYSQL_PASSWORD`、`CI_MYSQL_ROOT_PASSWORD`，并提供明确的测试级 fallback。
- 当前 GitHub 仓库 Repo Secrets、Repo Variables 和 Environments 查询结果均为空；未读取或输出任何 Secret 值。
- GitHub Actions 当前启用，允许全部 Actions；默认工作流令牌权限为只读且不能审批 PR。当前没有受保护的生产 Environment，因此不得把生产凭据写入仓库级 Secrets。
- CI 契约静态核对通过：三项 Secret 名称、MySQL service 注入、Spring datasource 覆盖、`application-test.yml` 的 `TEST_JWT_SECRET` 入口，以及生产 Compose 对 JWT/MySQL/Redis/MinIO 必填约束均一致。
- CI fallback 仅用于单次测试运行，MySQL 口令按 `github.run_id` 生成；它不替代生产 Secret 管理。
- `docs/standards/10-部署运维手册.md` 已增加无明文轮换顺序、回滚、验证、旧版本失效和双人复核清单；当前架构明确由目标主机 secret 管理或 Vault/KMS 保管生产值，未来自动部署必须先建立受保护的 `production` Environment。
- 当前没有可识别的生产环境、生产凭据或受保护的 GitHub `production` Environment；未来一旦建立目标环境，必须重新按手册执行，不得复用本机凭据或把本报告当作生产轮换记录。

### 本机真实凭据轮换演练

- 范围：仅 `local-dev`，MySQL 绑定 `127.0.0.1:3307`；未连接生产系统，未清库、删数据或重置测试数据。
