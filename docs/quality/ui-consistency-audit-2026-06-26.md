## 一、总体评价

- 桌面端 UI 一致性评分：7.4 / 10
- 本轮处理范围：暂时忽略移动端问题，桌面端以 1600 x 1200 Playwright 截图为准；移动端问题保留在证据中，但不参与当前优先级排序。
- 当前主要问题概述：
  - 合同台账已经形成较完整的桌面端最终方案：统一的 `lg-list-page` 容器、搜索区、KPI、表格面板、右侧分析栏、金额右对齐和行操作可访问标签。
  - 桌面端最大一致性风险不是单个页面布局，而是样式治理未闭环。当前静态扫描命中 647 处一致性问题，其中 `inline-style` 377 处、裸 `hex-color` 237 处。
  - 成本汇总桌面端已完成第一轮对齐：页头、筛选区、统一内容面板和空态已向合同台账收敛；系统字典、结算、项目等页面在工具栏、空态、表格实现、操作按钮顺序上仍未完全统一。
  - 登录页和部分图表/状态色仍使用旧品牌色或页面级硬编码，品牌 token 没有贯穿。
- 对用户体验的主要影响：
  - 桌面端用户在不同业务模块间切换时，会遇到不同的视觉密度、按钮位置、空态表达和确认文案，学习成本偏高。
  - 部分页面在未选条件或无数据时信息层级仍偏弱，容易被理解为半加载或内容缺失。
  - 样式散落在页面内联代码中，后续继续迭代时很容易出现“合同台账已修好、其他页面又偏离”的回归。

## 三、重点问题详细说明

### 1. 桌面端样式治理未闭环

- 问题现象：`check-ui-style-consistency.mjs --json` 当前命中 647 处，其中 `inline-style` 377 处、裸 `hex-color` 237 处；结算详情、合同变更、项目列表等页面仍有页面级硬编码。
- 为什么影响体验：每个页面对颜色、间距、卡片、空态、图表自定义一套，会导致后续局部修复后继续分叉，合同台账的最终方案难以复制。
- 推荐修改方案：把裸色值迁移到 `designTokens`、`chartPalette`、CSS 变量；把内联布局迁移到页面 class；共享搜索栏、工具栏、空态、KPI 和面板组件。
- 修改前后预期效果：桌面端视觉一致性可被脚本自动守住，新增页面更容易复用合同台账方案。

### 2. 成本汇总桌面端已完成第一轮结构对齐，仍需补充状态复验

- 问题现象：成本汇总未选项目状态已从旧的独立大标题、顶部散乱工具栏和大面积空白，调整为与合同台账一致的紧凑页头、筛选区、`lg-list-table-panel` 内容面板和统一空态。
- 为什么仍需跟进：2026-06-26 Playwright 已验证 1600 x 1200 Docker 前端未选项目状态：无重复“动态成本汇总”标题、无横向溢出、无错误 overlay、控制台无 error/warn；已选项目后的 KPI、图表、右侧分析、加载失败状态还需要带真实项目数据复验。
- 推荐修改方案：用一条真实项目数据验证已选项目状态；若图表、KPI 或右侧分析出现密度/顺序问题，再按合同台账的 KPI + 主表 + 右侧分析模式继续收敛。
- 修改前后预期效果：未选项目状态不再像半加载页面；已选项目状态也能保持和合同台账一致的阅读顺序。

### 3. 列设置、工具栏和表格组件仍有多套表达

- 问题现象：合同台账、结算列表、项目列表、系统字典等页面在“新建/刷新/列设置”的位置、顺序和按钮形态上不一致；表格实现同时存在 `vxe-grid`、`a-table` 和页面自定义列表/面板。
- 为什么影响体验：同类操作在不同页面位置变化，用户需要重新寻找；组件多套并存也会造成 hover、focus、disabled、empty 等状态不一致。
- 推荐修改方案：定义桌面列表页工具栏协议：主操作在左侧首位，辅助操作在右侧，顺序为列设置、刷新、导出；列设置只保留一套共享组件；表格页明确优先组件和例外条件。
- 修改前后预期效果：桌面端用户在任意列表页都能按同一空间记忆找到新建、刷新、列设置和分页。

### 4. 操作文案和确认按钮不统一

- 问题现象：删除按钮文案有 `确定`、`确认删除`、`删除`、`删除节点`；成功反馈有 `删除成功`、`已删除`、`创建成功`、`已保存`。
- 为什么影响体验：用户无法形成稳定预期，尤其在删除、清空、提交审批等高风险操作中会降低确认信任感。
- 推荐修改方案：制定操作文案表，并封装 `confirmDelete`、`confirmSubmit`、`confirmDiscard` 等通用确认方法。
- 修改前后预期效果：同一操作在所有桌面页面中的确认语气、按钮和 toast 结果保持一致。

### 5. 可访问性在合同台账已有进步，但未覆盖全局

- 问题现象：合同台账刷新按钮和行操作菜单已有明确 `aria-label`；结算、项目等截图中仍出现仅名为 `reload`、侧栏图标名如 `home/file-text` 的按钮。
- 为什么影响体验：键盘和读屏用户无法判断按钮的真实业务含义，图标按钮在 collapsed 或工具栏紧凑状态下尤其明显。
- 推荐修改方案：所有 icon-only 按钮必须带业务语义 `aria-label`，侧边栏 collapsed 状态也要暴露菜单中文名。
- 修改前后预期效果：读屏路径可理解，键盘用户能稳定定位“刷新结算列表”“项目管理”等目标。

## 四、统一规范建议

- 色彩规范：主色 `--primary #2563eb`，状态色只使用 `--success/--warning/--error/--info`；图表使用 `chartPalette`；禁止业务页新增裸 hex，白色/黑色也应使用语义变量。
- 字体规范：全局 `14px/1.57`；页面标题 22px/700；模块标题 15px/700；辅助文案 13px；金额和数量统一 `tabular-nums`。
- 间距规范：桌面页面外边距 20-24px；搜索区和 KPI 间距 16px；面板内边距 16/20px；紧凑工具栏按钮间距 8px；列表页横向分栏间距 16px。
- 组件规范：桌面列表页固定结构为页头、`lg-search-bar`、`lg-kpi-strip`、`lg-grid`、`lg-list-table-panel`、`lg-toolbar`、`lg-table-wrap`、`lg-pagination`；主从页允许变体但要复用头部、面板和空态。
- 状态规范：空态分为“暂无数据”“请选择条件”“加载失败”“无权限”；加载态禁止只留大片空白；错误态要保留当前筛选和重试入口。
- 交互规范：主操作按钮靠左并优先；辅助操作靠右，顺序为列设置、刷新、导出；删除/清空等危险操作使用危险按钮和明确对象名。
- 文案规范：删除确认统一为“确认删除”，主按钮“确认删除”，成功“已删除”；提交审批统一“确认提交/已提交审批”；保存统一“保存/已保存”；避免“确定/确认/提交”按页面随意混用。
- 桌面端适配规范：本轮验收以 1280px、1440px、1600px 三档桌面宽度为主；所有列表页首屏应完整展示页头、主要筛选、主操作、表格标题和分页区域。

## 五、落地改进建议

1. P0 先建立桌面端 UI 一致性门禁：
   - 当前基线：`pnpm exec node scripts/check-ui-style-consistency.mjs --json` 为 647 处。
   - 验收：总数先降到 300 以下；新增/修改页面不得新增裸 hex、静态内联 `style=`、非 token 圆角/阴影。
2. P1 继续完成成本汇总桌面端状态复验：
   - 已完成：未选项目状态已有统一页头、筛选区、内容面板和空态；1600px Docker 前端截图无横向溢出，无重复页标题，控制台无 error/warn。
   - 验收：已选项目、加载失败两种状态也符合合同台账结构，不出现大面积无解释空白。
3. P1 统一列设置与工具栏：
   - 验收：所有列表页只使用一套 `ColumnSettingsButton` 或增强后的共享组件；刷新按钮必须有 `aria-label`；工具栏主/辅操作顺序一致。
4. P1 统一桌面端表格、面板和空态组合：
   - 验收：业务页无裸 `<a-empty />`；需先选择条件的页面统一显示标题、说明和主操作；表格无数据和页面无上下文状态区分清楚。
5. P2 统一文案和确认弹窗：
   - 验收：删除/提交/保存/取消/放弃/清空文案有规范表；代码中删除确认不再出现 `okText: '确定'` 或 `okText: '删除'`。
6. P2 登录页品牌改造：
   - 验收：登录页不再使用旧主色裸值；圆角、阴影、logo 渐变走 token；与主应用品牌色一致。
7. 暂缓移动端专项：
   - 验收暂不纳入本轮；后续再以 440px、768px、1280px 作为响应式专项矩阵。
