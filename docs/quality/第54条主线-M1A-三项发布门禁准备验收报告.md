## 1. REL-CREDENTIAL-ROTATION准备

### 轮换对象

| 对象 | 仓库配置键 | 新旧版本验证 |
| --- | --- | --- |
| MySQL root | `MYSQL_ROOT_PASSWORD` | 新root健康；旧root拒绝 |
| MySQL应用账号 | `MYSQL_USER`、`MYSQL_PASSWORD` | 应用最小权限迁移与业务健康；旧密码拒绝 |
| Redis | `REDIS_PASSWORD` | 认证PING与应用缓存健康；旧密码拒绝 |
| MinIO | `MINIO_ROOT_USER`、`MINIO_ROOT_PASSWORD` | 桶读取/写入/签名URL；旧凭据拒绝 |
| JWT | `JWT_SECRET` | 新登录、新token有效；旧token失效 |
| Jasypt | `JASYPT_ENCRYPTOR_PASSWORD` | 加密配置可解密且应用健康；旧口令失效 |
| TLS | 目标环境实际启用的证书、私钥和信任链 | 握手、域名、有效期、旧证书退役 |

### 责任与顺序

- 发布负责人：冻结待发布SHA、维护窗口和回滚窗口。
- Secret保管人：创建新版本并受控注入；不向仓库、日志或报告暴露值。
- 环境执行人：按MySQL应用账号、Redis、MinIO、JWT/Jasypt、TLS的依赖顺序逐服务切换。
- 独立复核人：只复核Secret版本、旧凭据拒绝、健康证据和脱敏清单。
- 回滚：旧Secret版本在受控窗口内保持可恢复；恢复后重跑健康与权限验证，失败原因脱敏记录。

当前缺失：目标环境身份、具体人员、维护时间和Secret版本。必须在M6专项授权时补齐；M1A不读取任何真实值。

## 2. REL-FILE-RESCAN准备

### 对象清单

1. 从`sys_file`取得`id/tenant_id/bucket_name/storage_path/file_size/virus_scan_status`等最小字段。
2. 从MinIO按受控桶形成对象清单。
3. 双向核对：数据库缺对象、对象缺记录、重复引用、大小异常均单独计数，不静默跳过。
4. 服务端生成文档继续纳入对象守恒；其`SERVER_GENERATED`来源单独标识，不直接等同本次真实复扫通过。

### 执行契约

- 每个对象读取真实字节并复用现有`VirusScanner`。
- `CLEAN`：仅在扫描器明确返回CLEAN后写入。
- `INFECTED`：阻断下载并转受控隔离，不物理删除。
- `UNAVAILABLE`、读取失败、对象缺失或未知结果：fail-close，保持不可下载并进入失败重试清单。
- 禁止`UPDATE sys_file SET virus_scan_status='CLEAN'`式批量放行。
- 结果只保存总数、成功、感染、失败、缺失、重试及脱敏对象标识；不保存文件内容或签名URL。

回滚：只回滚本轮状态写入和隔离引用；已确认感染对象不得恢复为可下载。当前缺目标桶、对象总数、隔离桶、重试上限和执行授权，M6前保持`RELEASE_GATE`。
