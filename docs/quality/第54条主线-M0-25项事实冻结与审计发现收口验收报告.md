## 3. 上一轮4项审计发现裁决

### 3.1 定时财务对账事务与可观测性

结论：确认，正式承接为`AUDIT-FINANCE-RECONCILIATION-ATOMICITY`。

证据：`FinanceOperationsService.scheduledReconciliationAndAlerts()`捕获并忽略`RuntimeException`；定时入口直接调用同类私有`runReconciliationForTenant()`，不会经过`runReconciliation()`上的Spring事务代理。对账运行、旧问题删除、新问题插入和最终状态更新存在部分提交风险。

边界：采用现有Spring事务能力建立每租户独立事务并增加脱敏失败记录；不新增调度器或队列。

### 3.2 Prometheus抓取认证缺口

结论：确认配置冲突，正式承接为`AUDIT-PROMETHEUS-SCRAPE-AUTH`，保持`NEEDS_CONFIRMATION`。

证据：`deploy/monitoring/prometheus.yml`只配置目标和路径，未携带认证；`ActuatorMetricsTest`明确断言未登录访问`/api/actuator/prometheus`被拒绝。

解除条件：明确目标环境网络边界、Secret载体和机器认证机制。禁止匿名开放指标端点或提交静态凭据。

### 3.3 现金预测版本号并发冲突

结论：确认，正式承接为`AUDIT-CASH-FORECAST-VERSION-CONCURRENCY`。

证据：`ProjectCashForecastService.createCycle()`读取`MAX(version_no)+1`后插入；数据库唯一键`uk_cash_forecast_cycle_version(tenant_id,project_id,scenario,version_no)`只能拒绝碰撞，不能保证并发请求取得独立版本。

边界：优先复用项目行与现有事务建立固定顺序锁；不引入分布式锁或序列服务。

### 3.4 现金预测约1830次数据库操作

结论：证据不足，关闭。

依据：代码将预测区间限制为最多366天；按每日查询与写入得出的约1830次只是静态上界，没有运行时查询计数、超时复现、硬件基线或业务SLA。当前不得据此制造性能Backlog。若后续在受控数据量下形成稳定超时与量化阈值，再重新立项。
