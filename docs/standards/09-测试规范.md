# 09 测试规范

> 适用版本：v1.5。历史结论和旧路径仅作回溯，当前实现以本分支代码、配置和实时验证为准。

## 角色入口

主负责人：测试验收员。前端 UI 验收补读 [前端 Docker UI 测试规则](../prompt/frontend-docker-ui-test-rules.md)。验收结论必须给出通过/不通过、阻塞/非阻塞、依据、剩余风险。

## 质量目标

核心保护：
1. 登录、权限、菜单
2. 合同、成本、付款、审批、库存状态流转
3. ID、金额、日期、分页前后端契约
4. Flyway 迁移与逻辑删除唯一约束
5. 审批并发、重复提交、状态回写

## 后端测试

技术：JUnit 5、Spring Boot Test、MockMvc

```bash
cd backend
.\mvnw.cmd -Ptest-order-independence test "-Djasypt.encryptor.password=dev-jasypt-key"
.\mvnw.cmd verify "-Djasypt.encryptor.password=dev-jasypt-key"
.\mvnw.cmd "-Dtest=CostTargetControllerTest" test
```

提交前必须按上述顺序执行两个门禁：`test-order-independence` 将资金、结算、发票、用户、告警、驾驶舱和工作流等历史高风险测试按相反顺序单线程复跑；随后执行 `verify`，其生命周期与 CI 一致，包含全量测试、覆盖率检查和打包校验，并保证最终覆盖率报告来自全量套件。新增跨模块集成测试如共享数据库状态，必须纳入该 profile。

### 测试数据隔离

1. 测试类必须拥有独立的 ID、编码和业务单号命名空间，不得写入或修改演示数据及其他测试类的夹具。
2. 除专门验证管理员账号的测试外，禁止修改 `sys_user.id = 1`；需要用户时创建本测试类专属用户。
3. `@BeforeEach` 必须清除本类可能遗留的数据，`@AfterEach` 必须再次清理，保证测试失败后也不污染后续测试。
4. 清理必须按外键依赖的反向顺序执行。MyBatis-Plus 的 `delete` 可能是逻辑删除，隔离清理应使用 `JdbcTemplate` 或等价方式物理删除。
5. 测试不得依赖类执行顺序、历史数据库状态、其他测试先创建的数据或其他测试负责清理的数据。
6. 共享只读基础数据必须保持不可变；需要变更时先复制为本测试类专属数据。

必须覆盖：

| 场景 | 测试点 |
|------|--------|
| Controller | HTTP 状态、响应结构、权限、JSON 字段类型 |
| Service | 事务、状态流转、业务异常 |
| 审批 | 提交/同意/驳回/重复提交/并发 |
| 数据库 | 逻辑删除、唯一索引、Flyway |
| 成本 | 来源幂等、金额计算、汇总口径 |

## 前端测试

技术：Vitest、Vue Test Utils、Playwright

前端 UI 验收默认以 Docker 前端 `http://localhost:5173/` + Codex 内置浏览器为准；默认缩放口径为 `67%`，实测 viewport 约 `1714x964`。严格 `1920x1080` 仅用于外部 Playwright/Chrome 或用户明确要求的严格尺寸验收；若窗口或缩放变化，先记录 `window.innerWidth` / `window.innerHeight`。详细规则见 [前端 Docker UI 测试规则](../prompt/frontend-docker-ui-test-rules.md)。Playwright 用于 E2E 和自动化回归。

```bash
cd frontend-admin
pnpm build              # 类型检查 + 构建
pnpm test:unit          # 单元测试
pnpm test:coverage      # 覆盖率
pnpm exec playwright test   # E2E
```

必须覆盖：
- 页面加载和空状态
- 查询、分页、遴选、刷新
- 新增、编辑、删除、提交审批
- 表单校验
- 权限按钮显示
- 接口错误提示

## E2E 关键路径

| 路径 | 验证点 |
|------|--------|
| 登录 | Cookie、用户信息、菜单跳转 |
| 合同 | 新建、保存、提交、列表刷新 |
| 审批 | 待办、详情、同意、驳回 |
| 库存 | 仓库、入库、出库、余额 |
| 发票 | 登记、列表 |

E2E 不依赖不可控历史数据，通过 API 或测试夹具创建。

## 回归策略

每修复一个缺陷补一个回归验证：
1. 先写失败测试或记录证据
2. 修复最小范围
3. 同一测试确认通过

| 问题类型 | 回归方式 |
|----------|----------|
| Long ID 精度丢失 | MockMvc 断言 JSON ID 是字符串 |
| 逻辑删除唯一冲突 | 插入删除记录后建同编码 → 断言业务错误 |
| 审批重复提交 | 连续提交两次 → 断言第二次被拦截 |
| 付款超额 | 构造超合同付款 → 断言被拦截 |

## 验收标准

功能验收：
1. 用户能完成业务主流程
2. 列表、详情、表单、状态一致
3. 失败时提示明确，无空白页
4. 数据可追溯到项目和来源单据

技术验收：
1. 相关自动化测试通过
2. 前端构建通过
3. 后端启动无迁移错误
4. 日志无明显系统异常

## 禁止事项

1. 只靠手工点页面就说修复完成
2. 忽略失败测试
3. 为通过测试降低业务校验
4. 测试依赖执行顺序或其他测试留下的隐式状态
5. 用生产数据做自动化测试数据源
