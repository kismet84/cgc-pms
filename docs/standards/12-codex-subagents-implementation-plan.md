# Codex Subagents 本地实施方案

> 适用版本：v1.5。历史结论和旧路径仅作回溯，当前实现以本分支代码、配置和实时验证为准。

## 1. 结论

当前采用“本地最小版”方案，不落地 8-agent 全量结构，也不建立复杂台账流程。

本项目当前目标是本地使用：主会话负责总控、决策和验收，不直接改代码；代码实施交给 `implementation_worker`；结果验证交给 `verifier`。审查类 agent 只读，`implementation_worker` 作为单点代码修改入口，`verifier` 只验证不修代码。

`.codex/` 已被 `.gitignore` 忽略，本方案仅作为本地协作配置使用，不要求提交到仓库。本文只记录本项目当前决策，不代表官方验证结果。

## 2. 当前已落地配置

当前最小配置已落地在本地 `.codex/`：

| 文件 | 用途 |
| --- | --- |
| `.codex/config.toml` | 本地 Codex 配置入口 |
| `.codex/agents/code-mapper.toml` | 代码结构梳理与影响范围分析 |
| `.codex/agents/security-reviewer.toml` | 安全审查，只读 |
| `.codex/agents/test-reviewer.toml` | 测试与质量审查，只读 |
| `.codex/agents/implementation-worker.toml` | 唯一代码实施角色 |
| `.codex/agents/verifier.toml` | 验证角色，只验证不修代码 |

该配置已经覆盖当前本地协作的核心需要：分析、实施、验证、审查。

## 3. 推荐日常工作流

1. 主会话确认目标、边界、影响范围和验收标准。
2. 需要理解代码结构时，先交给 `code-mapper` 做只读梳理。
3. 需要改代码时，只分派给 `implementation_worker`。
4. 代码完成后，交给 `verifier` 按指定范围验证。
5. 涉及安全或测试风险时，再分别交给 `security-reviewer` 或 `test-reviewer` 做只读审查。
6. 主会话汇总结果，给出通过/不通过、阻塞/非阻塞、剩余风险和下一步。

## 4. 模型分配规则

按任务复杂度选模型，不默认高配。

| 场景 | 默认模型 |
| --- | --- |
| 单文件文档 / 本地配置 | `gpt-5.4` + `medium` |
| 日常代码实施 | `gpt-5.4` + `medium` |
| 普通代码审查 / 测试审查 | `gpt-5.4` + `medium` |
| 安全审查 / 复杂架构 / 跨模块高风险实现 | `gpt-5.5` + `high` |
| verifier 默认 | `gpt-5.4` + `medium` |

主线程总控不因任务小就亲自改代码，但分派时必须按复杂度选模型。

## 5. Agent 职责矩阵

| Agent | 是否可改代码 | 主要职责 | 禁止事项 |
| --- | --- | --- | --- |
| 主会话 | 否 | 总控、决策、任务分派、验收结论 | 直接修改代码、直接修复问题 |
| `code-mapper` | 否 | 梳理代码结构、调用关系、影响范围 | 改代码、跑破坏性命令 |
| `implementation_worker` | 是 | 按明确边界实施最小必要改动 | 扩大范围、改无关文件、替代验收 |
| `verifier` | 否 | 运行指定验证、复核结果、输出证据 | 改源码、改配置、改文档、变更 Git 状态、扩大验证范围 |
| `security-reviewer` | 否 | 安全风险审查 | 实施修复 |
| `test-reviewer` | 否 | 测试覆盖和质量风险审查 | 实施修复 |

`verifier` 保留 `workspace-write`，但仅用于测试、构建、日志或临时证据产物；不得借此修改源码、配置、文档或 Git 状态。

## 6. 暂缓落地项

以下内容暂缓，不纳入当前本地最小版：

| 暂缓项 | 原因 |
| --- | --- |
| 8-agent 全量拆分 | 当前本地使用偏重，角色过多会增加分派和验收成本 |
| 独立台账全量结构 | 当前任务规模不需要额外流程资产 |
| 自动化多 agent 编排 | 现阶段人工主控更可控，避免误改代码 |
| 统一提交 `.codex/` 配置 | `.codex/` 属于本地使用配置，当前不要求入库 |
| 复杂评分或审计仪表盘 | 当前只需要任务闭环证据，不需要运营化度量 |

## 7. 何时扩展

满足以下条件之一时，再考虑扩展：

| 触发条件 | 可增加内容 |
| --- | --- |
| 多人共同使用同一套 agent 规则 | 评估是否提交部分规范文档，而不是直接提交 `.codex/` |
| 并行任务明显增多 | 增加更细的只读审查 agent 或任务模板 |
| 审计、测试、安全任务频繁重复 | 增加轻量 checklist 或报告模板 |
| agent 分派结果难以追踪 | 增加简单任务记录，不先上复杂台账 |
| worktree 并行开发成为常态 | 补充 worktree 命名、清理和合并规则 |

## 8. Worktree 使用边界

当前不默认使用 worktree。

仅在以下场景使用 worktree：

1. 两个以上代码改动任务需要并行推进。
2. 当前工作区已有未提交改动，且不能混入新任务。
3. 高风险改动需要隔离验证。

不因普通文档、单点修复、只读审查任务创建 worktree。worktree 使用后必须明确分支、目录、合并方式和清理责任。

## 9. 风险与约束

| 风险 | 约束 |
| --- | --- |
| 主会话误变成执行者 | 主会话只做总控、决策和验收，不直接改代码 |
| 多个 agent 同时改代码导致冲突 | 只允许 `implementation_worker` 单点改代码 |
| 审查 agent 越权修复 | `code-mapper`、`security-reviewer`、`test-reviewer` 只读 |
| 验证阶段顺手修复 | `verifier` 只验证，不修代码 |
| 本地配置被误当成团队规范 | `.codex/` 已被 `.gitignore` 忽略，当前仅本地使用 |
| 方案写成官方结论 | 本文只记录本项目当前决策，不声称官方验证 |
