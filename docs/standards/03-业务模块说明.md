# 03 业务模块说明

> 适用版本：v1.5。历史结论和旧路径仅作回溯，当前实现以本分支代码、配置和实时验证为准。

## 总体闭环

系统围绕工程项目全过程管理：

```text
项目立项 -> 合同/签证/采购/分包 -> 成本归集 -> 付款/发票/结算 -> 审批/通知/预警 -> 驾驶舱分析
```

## 业务域

| 业务域 | 前端入口 | 后端模块 | 说明 |
|--------|----------|----------|------|
| 工作台与预警 | `/dashboard`、`/alert` | `dashboard`、`alert` | 多角色经营驾驶舱、预警规则和批处理 |
| 项目与主数据 | `/project`、`/partner`、`/org`、`/material` | `project`、`partner`、`org`、`material` | 项目、合作方、组织、材料字典 |
| 合同与签证 | `/contract`、`/variation` | `contract`、`variation` | 合同台账、清单、付款条件、合同变更、签证 |
| 成本与目标 | `/cost`、`/cost-subject`、`/cost-target` | `cost`、`revenue`、`overhead`、`accounting` | 成本科目、成本台账、动态汇总、目标成本 |
| 采购与库存 | `/purchase`、`/receipt`、`/inventory`、`/requisition` | `purchase`、`receipt`、`inventory`、`requisition` | 采购订单、收货、仓库、库存、领料 |
| 分包与结算 | `/subcontract`、`/settlement` | `subcontract`、`settlement` | 分包任务、计量、结算 |
| 付款与发票 | `/payment`、`/invoice` | `payment`、`invoice` | 付款申请、财务回写、发票登记、PDF 识别 |
| 审批中心 | `/approval` | `workflow`、`notification` | 待办、已办、抄送、流程管理、站内通知 |
| 系统管理 | `/system`、`/settings` | `system`、`auth`、`audit`、`file` | 用户角色、菜单权限、字典、审计、文件 |

## 关键业务规则

- 项目是大多数业务数据的主线维度。
- 租户隔离是后端强制边界，前端隐藏按钮只用于体验。
- 合同、签证、付款、结算等状态流转必须经过业务校验和审批校验。
- 成本口径涉及合同、采购、库存、分包、签证、结算等来源，改动前必须确认汇总来源和月份口径。
- 发票必须关联付款记录；文件上传和 PDF 识别受安全规范约束。

## 驾驶舱最终角色

当前驾驶舱最终角色口径固定为以下 5 类：

| 角色 | 首页关注主线 | 当前可复用数据基础 |
|------|--------------|--------------------|
| 项目经理 | 项目执行协同、待办推进、进度预警、合同履约 | `dashboard.project-manager`、项目总览、预警、审批 |
| 总工程师 | 技术方案、设计协调、技术审核、重大技术问题 | 当前冻结，仓库缺少独立技术驾驶舱聚合 |
| 商务经理 | 合同、变更、索赔、成本控制、结算 | `dashboard.cost-manager`、成本汇总、合同/付款/预警 |
| 采购经理 | 采购执行、供应商、交货、验收、库存联动 | 采购、验收、库存、领料 |
| 生产经理 | 施工组织、现场协调、施工进度、质量、劳务 | 验收、领料、库存、分包计量；完整进度/劳务数据需要确认 |

约束：

- 不得仅通过修改前端 tab 文案冒充新角色驾驶舱。
- 角色名称、接口语义、VO 字段、权限口径必须一致。
- 当前 `商务经理` 直接承接现有 `成本经理驾驶舱` 语义；不得再将其展示为“成本经理”。
- 当前 `采购经理`、`生产经理` 已完成第二阶段放行，默认演示环境已补最小 demo 数据，不再按空白态验收。
- 当前 `总工程师` 仍冻结，后续需独立立项，不能用经营、风险、付款或设计变更硬包装上线。

## 第一阶段上线边界

第一阶段只上线 2 个角色：

1. `商务经理`：沿用当前 `DashboardCostView` 的核心数据语义，定位为经营与成本控制首页。
2. `项目经理`：基于 `DashboardPmView` 重构为执行协同视角首页。

第一阶段明确不包含：

- `采购经理`
- `生产经理`
- `总工程师`
- 任何将财务、管理层、安全等旧角色包装后继续上线的方案

第一阶段页面口径约束：

- `商务经理` 首屏保留经营结果、成本偏差、预警、待付款/逾期事项、商务台账，不强调库存、领料、现场执行。
- `项目经理` 首屏保留待办、待审批、滞后、预警、临期合同、项目协同事项，不展示“成本构成分析”“资金收支概览”等经营/财务主图。
- 若现有数据无法支撑角色主线，应先降级区块或留空，不得用其他角色数据顶替。

## 第二阶段上线边界

第二阶段评估过以下 3 个角色：

1. `采购经理`
2. `生产经理（MVP）`
3. `总工程师（最小可用版评估项，未放行）`

第二阶段页面口径约束：

- `采购经理` 只允许使用采购、验收、库存、领料等真实采购执行语义；不得复用商务经理的利润、成本偏差、结算主视图。
- `生产经理（MVP）` 只允许使用验收、领料、库存、分包计量等现场执行近似数据；不得冒充完整施工进度、劳务、机械、产值管理驾驶舱。
- `总工程师` 只有在存在“技术方案、设计协调、技术审核、重大技术问题”真实数据契约时才能上线；若仅有“设计变更”作为签证类型或泛审批数据，不得包装为总工程师驾驶舱。

第二阶段已知数据边界：

- `采购经理` 当前可直接复用：采购订单状态、审批状态、未入库金额、供应商、材料验收质量状态、库存 KPI、低库存预警。
- `生产经理（MVP）` 当前可直接复用：材料验收质量状态、领料出库状态、分包计量审核状态、库存出入库统计、低库存预警。
- `总工程师` 当前仅发现“设计变更”作为 `variation` 业务类型的一部分，未发现独立“技术审核 / 设计协调 / 技术问题闭环”业务类型和 dashboard 聚合，因此默认阻塞并冻结。

## 总工程师单独立项前置条件

总工程师驾驶舱已从阶段性交付中拆出，按单独立项处理；当前默认结论为“不能立项”，直至以下前置条件满足：

1. 存在独立技术域业务对象，至少覆盖：`技术方案`、`设计协调`、`技术审核`、`重大技术问题闭环`。
2. 上述对象具备明确状态流转、项目维度归属、责任人归属和列表/统计查询接口。
3. 驾驶舱首页指标、事项列表、底表均来自技术域真实数据，不借用经营、付款、风险、设计变更或泛审批页面拼装。
4. 权限、角色名、接口路径、VO 字段命名与“总工程师”语义一致。

## 验收关注点

| 类型 | 必查项 |
|------|--------|
| 业务闭环 | 数据是否能从来源单据流入台账、汇总和驾驶舱 |
| 状态流转 | 草稿、审批中、已审批、驳回、作废等状态是否互斥 |
| 权限边界 | API 是否强制校验租户、项目和权限 |
| 数据口径 | 金额、日期、月份快照、成本来源是否一致 |
| 前端体验 | 列表、表单、弹窗、详情入口是否符合 `lg-*` 基线 |
