# Codex 任务执行策略

状态：Active
适用范围：cgc-pms 普通交互、主线/Backlog、验收、发布收口与 AutoPilot 桌面主线程
上位规则：`AGENTS.override.md`、`AGENTS.md`

本文是 cgc-pms 通用任务执行协议的唯一权威正文。授权、安全、数据、发布和 AutoPilot 专项硬门禁仍以上位规则及专项行为契约为准；专项 Skill 只保存领域步骤和回报骨架，不复制本文全文。

## 1. 核心原则

1. 先识别授权、模式和事实，再实施状态变化。
2. 当前分支源码、配置、日志、运行态和正式治理载体是最终事实源；图谱只提供检索加速和关系导航。
3. 每个问题先走一个确定的首选路径，失败后分类并改变策略，不做无前置变化的原样重试。
4. 验证按风险分层，允许复用仍绑定当前代码、环境和命令的证据，不以降噪替代验证。
5. commentary 只在有信息增益的事件发生时播报，最终答复必须自包含。
6. 中断恢复以任务胶囊、Git 事实和绑定证据为准，不凭会话记忆重做已完成工作。

## 2. 状态机与粘性模式

```text
接收请求 -> 识别授权与模式 -> 只读前置检查 -> 分析或实施
        -> 分层验证 -> 审查/失败分类 -> 归档/Git 收口 -> 最终裁决
                                      \-> 需要补修 -> 实施
```

同一时刻只保留一个当前阶段。任务状态至少记录：目标、授权范围、当前模式、当前阶段、分支与工作区、已完成变更、有效验证、失败分类与阻塞、剩余动作、是否允许提交/推送/合并、正式承接的后续项。

模式具有粘性，直到用户指令或阶段边界明确改变：

- `分析模式`：只读调查、方案和评审，不实施变更。
- `执行模式`：实施已授权变更并验证，不默认提交或发布。
- `验收模式`：复核证据并裁决，不顺带扩大实现。
- `连续迭代模式`：遵守 Ready、checkpoint、fencing、逐 Issue 收口和上限。
- `发布收口模式`：处理提交、推送、PR、合并与分支清理等独立 Git 状态变化。

模式切换只播报一次：原模式、目标模式和切换原因。

## 3. 授权与短指令语义

| 指令 | 默认语义 | 不自动包含 |
| --- | --- | --- |
| `继续` | 从当前任务胶囊的下一未完成步骤继续 | 扩大范围、重做已完成步骤 |
| `执行` | 在已确认方案和授权边界内实施并做必要验证 | 提交、推送、合并、生产变更 |
| `修复` | 修复当前已确认失败并复验 | 无关重构、处理无证据建议 |
| `提交` | 核对分支、状态、验证和提交范围后创建本地提交 | 推送、PR、合并 |
| `推送` | 推送当前已授权功能分支 | 直接推送 `master`、合并 |
| `合并` | 核对目标分支和门禁后执行已授权合并 | 绕过 PR、保护分支或失败门禁 |
| `清理分支` | 先预览，仅清理已合并且无 worktree 占用的授权分支 | 删除未合并成果或远端批量清理 |
| `启动迭代-N` | 按项目关键词协议启动最多 N 个合格实施型 Issue | 放宽 Ready、checkpoint、金丝雀或 no-push 门禁 |

没有可靠任务胶囊时，短指令只授权安全只读核验；涉及数据库删除、生产、权限、安全、金额、租户、提交、推送、合并或分支删除时必须遵守更高优先级的明确授权门。

## 4. 确定性工具路由

| 查询目标 | 首选路径 | 必需核验或备用路径 |
| --- | --- | --- |
| 已知文件、配置键、错误文本、精确字符串 | `rg` + 直接读取 | Git diff、当前配置或日志 |
| Java、TypeScript、Vue 未知符号与局部引用 | CodeGraph，一次有界查询 | 当前源码；召回不足时 `rg` + 直接读取 |
| 跨前后端、跨语言、多跳调用链、架构边界 | `codebase-memory-mcp` | `rg` + 当前源码交叉核验 |
| PowerShell 定义、引用、动态调用和调用链 | `codebase-memory-mcp` | `rg` + 直接读取；禁止使用 CodeGraph |
| CSS、SQL、YAML、JSON、日志、Git、CI | 对应直接工具或文本检索 | 当前配置、日志或运行态 |
| 存量问题、优先级和关联导航 | 项目知识图谱 | `sourceRefs` 对应的当前分支与唯一正式载体 |
| 页面行为与交互结果 | 内置浏览器 | 网络响应、服务日志、当前源码 |

调用预算：

1. 首选路径失败或召回不足时，默认最多切换一个有信息增益的备用路径。
2. 上位规则要求的源码交叉核验、跨层补充查询、PowerShell 核验和高风险复核属于必需证据，不计入备用路径预算。
3. 只有参数已纠正、环境已恢复、出现新证据或明确瞬时故障时才允许复跑。
4. 首选与备用都失败时，停止无变化重试，输出失败分类、已有证据、缺失前置和下一决策。
5. 图谱未命中已知文件、字段或符号统一归类为 `retrieval_gap`，不得解释为代码不存在。
6. `codebase-memory-mcp` 的 project 标识是运行态事实：同一会话复用工具已确认的标识，不得猜测或再次使用已被工具拒绝的值；尚无已验证标识时，只允许一次有界发现并采用工具返回的可用项目。

## 5. 失败分类

| 分类 | 典型情况 | 处理原则 |
| --- | --- | --- |
| `tool_config` | 工具未加载、索引缺失、凭据/规则未注入、入口不存在 | 修复配置或补前置，不判业务失败 |
| `tool_invocation` | schema、参数、转义、PowerShell 参数拆分错误 | 修正调用方式后最多做一次最小复验 |
| `environment_prerequisite` | Docker、端口、数据库、旧代理、等待时间、测试数据未就绪 | 恢复环境后复验 |
| `ready_issue_config` | 验证选择器、路径范围或 Ready 契约失真 | 最小等价修正并记录，不判测试失败 |
| `retrieval_gap` | 图谱召回不足、索引覆盖缺失 | 走明确备用路径，不作不存在断言 |
| `quality_or_security` | 可稳定复现的测试、构建、契约、权限、安全、数据一致性失败 | 进入实现整改或阻塞裁决 |
| `unknown` | 证据冲突或不足 | 收集更强证据，禁止强行归因 |

兼容映射：AutoPilot 现有结构化分类 Schema 为保持运行态兼容，将 `powershell_parser_error` 编码为 `tool_config`；在本统一策略、CI Skill 和第 47 条主线效率度量中，参数、schema、转义和 PowerShell 参数拆分统一归一化为 `tool_invocation`。该报表归一化不得改写 AutoPilot 历史状态，也不得放宽其 fail-close 或重试策略。

面向用户按“分类 + 影响 + 已采取动作 + 结果”汇总；重复的同类错误细节不逐条转播。

## 6. 浏览器执行模板

1. 每个线程首次使用浏览器时只做一次能力与页面状态初始化，不猜测 API 名称或参数。
2. cgc-pms 本地验收依次核对后端 health、前端入口和 `http://localhost:5173/api/auth/dev-login?redirect=/dashboard`。
3. 任一入口不可用先归类为 `environment_prerequisite` 并走运行态刷新；稳定等待时间由当前项目规则或动态配置读取。
4. 回到 `/login` 且 `/api/*` 指向旧 `172.19.x.x:8080` 时，先刷新前端运行态，不先判断路由守卫或业务回退。
5. 同一浏览器参数错误不得原样重试；修正参数或切换到网络、日志、源码核验。
6. 并行或批量验收必须使用唯一输出目录，禁止共享截图名、测试报告目录和缓存目录。
7. 页面证据至少包含：访问入口、关键动作、实际结果、关联接口/日志、最终裁决。

## 7. 分层验证与证据复用

| 层级 | 触发时机 | 最小证据 |
| --- | --- | --- |
| L1 目标验证 | 每次具体修改后 | 目标测试或静态核对、变更文件空白检查 |
| L2 模块回归 | 跨文件或关键边界修改后 | 相关模块测试、类型检查或构建 |
| L3 批次验收 | 一组紧密相关修改完成后 | 复用有效 L1/L2，仅补裁决必需项 |
| L4 发布验证 | 提交后推送、PR 或合并前 | 适用 CI、权限/数据边界和发布门禁 |

证据只有在代码/diff、命令、测试选择器、环境、配置和适用的上下文身份均未发生影响结果的变化时才可复用。AutoPilot 使用既有 Context Base/Delta 与 Evidence v2 绑定，不为普通任务另建缓存或第二套状态系统；静态廉价检查及专项策略要求重跑的集成/浏览器证据不得错误复用。

## 8. CI 轮询与重试

1. 首次红灯先分类，疑似瞬时或环境前置问题只允许一次最小等价复验。
2. 轮询使用退避节奏；状态未变化时不重复 commentary。
3. 只在状态变化、超过预期、出现确定失败或需要用户决策时更新。
4. 本地与远端结果不一致时，以当前提交绑定的真实 checks、workflow 和分支保护为裁决依据。
5. 未完成分类前，不得把 CI 红灯定性为业务代码失败。

## 9. Git 生命周期

```text
授权功能分支 -> 实施/验证 -> 本地提交 -> 推送功能分支
             -> PR/CI/审查 -> 授权合并 -> 预览并清理已合并分支
```

- 任何文件或 Git 状态变化前核对当前分支与 `git status --short`；怀疑隔离冲突时再查 worktree。
- 禁止直接向 `master` 推送。创建/切换分支、提交、推送、PR、合并和清理分别按授权与门禁执行。
- 未获得提交授权时停在已验证 diff；未获得推送/合并授权时停在对应前一状态。
- 清理前必须预览并确认已合并、无未推送提交、无活动 worktree 占用；禁止盲删。

## 10. 事件驱动沟通

只在以下事件发送 commentary：

1. 开始工作，说明范围、模式和关键假设。
2. 发现会改变方案或裁决的关键事实/风险。
3. 从分析切换到实施、验收或发布。
4. 出现真实阻塞或需要用户决策。
5. 长任务超过 60 秒且有实质进展变化。

不要为每条命令、纯等待、相同 CI 状态、重复工具错误或无信息量的“继续处理中”单独播报。普通任务通常不超过 5 次过程播报；高风险或长任务可增加，但必须由状态变化驱动。最终答复必须包含完整结论，不能依赖已折叠 commentary。

## 11. 任务恢复胶囊

阶段切换、线程切换、长等待或中断前，按需生成：

```text
任务目标=
当前模式=
授权范围=
当前分支与工作区=
已完成变更=
已通过验证=
失败分类与阻塞=
剩余步骤=
禁止重复执行=
是否允许提交/推送/合并=
正式承接的后续项=
```

恢复时先核验分支、工作区、关键文件和证据有效性，再从首个未完成步骤继续。AutoPilot 优先读取 durable checkpoint；普通任务不新增重型持久化系统。

## 12. 收口与度量

最终回报至少包含：正式交付物、验收证据、临时产物、Git 状态、结论、阻塞、剩余风险、新增后续项、关闭后续项和后续项净变化。

效率度量只保留正式汇总：工具调用分类、原样重试、浏览器调用错误、过程播报数、中断返工、控制面耗时、证据复用和后续项净变化。原始会话日志、run id、截图名和临时路径不得写入长期规则或正式报告。

进入正式试运行的样本必须在结束时向唯一试运行报告追加最小样本胶囊；缺少工具调用总数或有效耗时分母的样本可以保留定性证据，但不得进入失败率或耗时占比裁决。工具调用总数按实际调用的方法次数计算；批量编排按实际子调用数计算，纯包装层不重复计数。

比例由 `scripts/codex-autopilot/measure-codex-task-efficiency.ps1` 统一计算；该入口只接收汇总计数并向 stdout 输出 JSON，不读取原始会话、不写文件、不保存临时标识，也不内置计划目标阈值。输入分类总数超过工具调用总数、原样重复超过调用错误、浏览器错误超过浏览器调用，或耗时分子分母不完整时必须拒绝计算。

统一策略、样本计算、Git 生命周期、工具路由、控制面指纹与主线计划结构的批次验收统一使用 `scripts/codex-autopilot/test-codex-task-policy-suite.ps1`。调用者必须显式传入计划路径和 profile；该薄包装只顺序复用现有测试、失败即停，不替代任何底层验证，也不写运行态。

```text
样本类型=
统计起止=
工具调用总数=
tool_config/tool_invocation/retrieval_gap/environment_prerequisite/quality_or_security=
原样重复调用=
浏览器调用总数/调用错误=
commentary 数=
控制面耗时/有效执行耗时=
中断后重复实施=
证据复用=
新增后续项/关闭后续项/净变化=
样本结论=
```

控制面耗时或有效执行耗时无法可靠拆分时写 `not_available`，不得估算；普通任务以目标、授权和验收均明确的低风险分析、修复或文档任务为准，事故、发布、高风险验收和长等待任务必须单独分组。

## 13. 维护边界

- 本文不保存模型、推理强度、超时、评分权重、回顾阈值和运行态地址等动态配置副本。
- AutoPilot 调度、fencing、状态迁移、恢复、评分、金丝雀和 closeout 以 `plugins/cgc-pms-autopilot/references/control-plane-policy.md`、配置和 Schema 为准。
- 运行态、CI、主线和其他专项 Skill 只引用本文，并保留其领域特有步骤。
- 本文或其索引发生行为性变化时，必须纳入 AutoPilot 控制面指纹覆盖，并按规则重新完成用户授权的单 Issue 金丝雀。
