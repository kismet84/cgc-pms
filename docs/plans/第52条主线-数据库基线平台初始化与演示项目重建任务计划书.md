# 第52条主线：数据库基线、平台初始化与演示项目重建任务计划书

状态：已完成（本地验收通过，冻结 `N=215`；生产未操作）。

**Goal:** 将 CGC-PMS 当前 V1—V215 的累积迁移链收敛为面向新环境的最终数据库基线，同时保持既有数据库可继续校验和升级；新建 dev/test/demo 数据库默认只包含最终表结构和必要系统基础数据，不再自动加载旧演示项目、物资、库存或业务单据，并通过安全、幂等的平台引导机制创建首个组织和超级管理员，后续再显式加载一套完整、可追溯、可重复构建的演示项目。

**Architecture:** 采用“Flyway 最终基线 + 兼容历史迁移链 + 配置模板 + 外部密钥 + 幂等初始化服务 + 独立演示数据包”六层结构。当前冻结候选为 MySQL/H2 `B215`；V1—V215 原文件保持内容和校验和不变，移动到兼容位置供既有数据库验证，新空库选择最新基线并忽略低版本迁移。基线只承载最终 schema 和白名单系统数据；首个管理员、根公司和根部门由应用启动后的事务服务创建；真实密码只由环境变量或 Secret 注入；演示业务只能由显式命令加载。禁止连接或重建生产数据库，禁止以删除 `flyway_schema_history`、修改历史迁移、`repair` 掩盖缺失迁移或整库数据 dump 代替正式基线。

## 一、背景与当前事实

### 1.1 当前仓库基线

截至本计划编写时：

- 当前分支：`master`。
- MySQL Flyway SQL 文件：212 个，最高版本 `V215__document_cost_subject_v2_schema.sql`。
- H2 Flyway SQL 文件：204 个，最高版本同为 V215。
- MySQL 与 H2 均配置 `validate-on-migrate: true`。
- dev/prod 当前迁移位置均为 `classpath:db/migration`；生产已设置 `clean-disabled: true`。
- 当前存在 16 个文件名明确包含 `demo` 或 `sample` 的旧演示迁移：V6、V98、V100—V111、V120、V134。
- 用户表使用 `(tenant_id, username)` 唯一键，角色表使用 `(tenant_id, role_code)` 唯一键，用户角色关系使用 `(user_id, role_id)` 唯一键。
- 密码由 Spring Security `BCryptPasswordEncoder` 编码。
- 当前 dev-login 默认账号仍为 `demo_dev_super_admin`，清除旧演示账号后必须同步调整。
- 当前组织初始化服务已有按公司/部门编码查重的幂等思路，但默认名称硬编码，并且空业务库不一定触发组织创建。

以上数量和版本是计划冻结候选。实施开始时若已有 V216 或更高迁移，必须重新冻结为 `N`，并将全文 `B215`、V1—V215 按 `BN`、V1—VN 更新；不得让基线遗漏已合入迁移。

### 1.2 当前问题

1. 新数据库需要按顺序执行两百余个迁移；中间包含“增加字段—修改字段—删除字段”等历史演进步骤，执行成本和理解成本持续增长。
2. 历史迁移混合了结构变更、系统种子、修复脚本和演示业务数据，无法直接作为干净的新环境初始化定义。
3. 将旧迁移直接删除会导致既有数据库 Flyway validation 出现 unresolved migration，不能用简单清目录解决。
4. 整库导出会把旧演示项目、物资、库存、审批和业务单据一起带入新基线，不满足“业务空白环境”目标。
5. 把管理员密码写入迁移 SQL 或仓库 YAML 会形成长期凭据泄露风险。
6. 普通用户服务依赖当前登录用户和租户上下文，不适合尚未存在任何账号时的系统引导。
7. 旧演示数据由迁移自动加载，无法表达“一套完整演示项目”的版本、所有权、加载时机和清理边界。

## 二、目标、范围与非目标

### 2.1 本主线目标

1. 新空库从最终基线直接获得 V215 的最终数据库结构，不重放历史结构演进过程。
2. 既有 V1—V215 数据库继续通过 Flyway validation，并可执行 V216 及以后增量迁移。
3. 新建 dev/test/demo 数据库默认业务空白，不自动加载任何旧演示项目和业务事实。
4. 必要的系统角色、菜单权限、字典、成本科目、物资分类等仍可用。
5. 首个超级管理员、根公司和根部门由显式开启的 bootstrap 服务创建。
6. 管理员密码不进入 Git、迁移 SQL、日志或数据库明文字段。
7. bootstrap 可安全重跑、并发启动，不重复创建、不覆盖密码、不自动提权同名普通账号。
8. 完整演示项目成为独立、显式、受环境门禁控制的数据包，不再属于 Flyway 生命周期。

### 2.2 本主线范围

- 盘点和分类 V1—V215 MySQL/H2 SQL 迁移及 Java migration。
- 构建 MySQL/H2 最终基线迁移。
- 建立历史迁移兼容目录和双路径验证策略。
- 建立 bootstrap 配置模板、外部 Secret 接入、配置绑定和校验。
- 建立 bootstrap 状态表、执行器、事务服务、组织/管理员创建和角色绑定。
- 调整 dev-login 与新初始化管理员的关系。
- 从新基线排除旧演示数据。
- 建立完整演示项目的数据清单、显式加载入口、幂等键和验收脚本。
- 建立新空库、旧升级库、MySQL/H2、并发、安全和演示业务验证矩阵。
- 更新数据库初始化、部署和开发文档。

### 2.3 非目标

- 不连接、清空、重建或自动迁移生产数据库。
- 不修改 V1—V215 任一迁移文件内容或校验和。
- 不删除任何当前业务数据库中的项目、物资、库存、合同、成本、资金或审批事实。
- 不通过本主线建设完整租户注册/计费平台；第一版继续使用现有 `tenant_id=0` 模型。
- 不把管理员密码重置能力混入 bootstrap；密码重置继续走独立、可审计的现有运维或个人密码修改流程。
- 不在第一版新增“首次登录强制改密”字段和登录状态机；部署验收要求人工登录后立即修改临时密码并移除 Secret。
- 不将任意未来配置都塞入 bootstrap；没有明确数据库或运行时消费者的字段不进入模板。
- 不自动加载演示项目；“完整演示项目”只通过明确命令创建。


```yaml
spring:
```

预期行为：

| 数据库状态 | Flyway 行为 |
| --- | --- |
| 全新空库 | 选择最新 `B215`，忽略 V1—V215，再执行 V216+ |
| 已执行到 V180 | 忽略 `B215`，继续执行 V181—V215/V216+ |

### 4.2 必要系统数据白名单

基线至少保留：

- `SUPER_ADMIN`、`ADMIN` 等正式系统角色定义。
- 菜单、按钮权限、角色权限关系。
- 正式系统字典及状态枚举。
- 成本科目 V2 最终树、归集规则元数据和相关权限。
- 物资分类、计量单位等分类型主数据；不包含具体材料记录。
- 正式审批模板、审批节点、业务类型注册。
- 文档生成模板字段目录及必须的默认绑定；不包含业务生成记录。
- bootstrap 状态表及 `PLATFORM_ADMIN` 的 `PENDING` 控制行。

每一类都必须生成“表名、业务键、记录数、来源迁移、是否必需、责任人”的白名单。未进入白名单的数据默认不进入 B215。

### 4.3 明确排除内容

新基线不得包含：

- 旧演示项目及项目成员。
- 演示投标、中标、合同、目标成本和预算。
- 具体演示材料、供应商、分包商、客户和仓库。
- 库存余额、库存流水、收货、出库、领料和退料。
- 采购申请、采购订单、材料验收、结算、付款和发票。
- 施工进度、质量安全检查、整改、技术方案和 RFI。
- 工作流实例、任务、审批历史、抄送记录和消息。
- 财务流水、会计凭证、收入、回款、月结和预测。
- `demo_*` 用户及其业务角色关系。
- 旧科目清理审计行；新库从未发生旧科目迁移，不应伪造历史。

## 五、最终基线生成策略

### 5.1 冻结参考库

1. 从空 MySQL 数据库完整执行原 V1—V215。
2. 确认 Flyway 成功且应用测试通过。
3. 导出仅 schema，不导出业务数据。
4. 单独按白名单导出必要系统数据。
5. 记录参考库 schema 指纹和白名单数据指纹。

禁止直接使用包含数据的整库 dump 生成 B215。

### 5.2 基线内容

`B215__cgc_pms_baseline.sql` 必须包含 V215 的最终状态：

- 最终表结构，而非历史 `ALTER` 链。
- 最终索引、唯一约束、外键和检查约束。
- 最终字符集、排序规则、默认值、生成列和注释。
- 最终视图定义。
- 系统数据白名单的确定性插入。
- bootstrap 状态结构与初始控制行。

禁止包含：

- 已被后续迁移删除的字段、表和索引。
- 临时修复 SQL、历史回填和无意义中间状态。
- 任何具体环境密码、URL、主机名和令牌。
- 任何演示业务事实。

### 5.3 MySQL/H2 对称

- MySQL 是生产语义权威。
- H2 基线必须覆盖相同业务表、字段和约束意图。
- MySQL 专属生成列、JSON、索引和 DDL 差异通过现有 H2 兼容方式处理，不降低 MySQL 约束。
- Java migration 必须纳入版本唯一性和路径审计；不能只移动 SQL 而遗漏 Java 版本。

### 5.4 Schema 等价比较

以旧链参考库和 B215 新库做自动比较：

- 表、列、类型、长度、精度、可空、默认值。
- 主键、唯一键、普通索引、外键、检查约束。
- 生成列、视图、字符集、排序规则和注释。
- ORM 映射所需字段。
- 显式 SQL 引用字段。

允许差异只有：

- `flyway_schema_history` 内容不同。
- 新库不含历史业务数据和历史修复审计行。
- 经签认的 bootstrap 状态结构。

任何其他差异必须说明原因并签认，否则基线不通过。

## 六、Bootstrap 配置模板

### 6.1 文件与加载方式

建议：

```text
deploy/config/bootstrap.example.yml   # 提交 Git
deploy/config/bootstrap.yml           # 本地配置，不提交
deploy/.env.example                   # 提交变量模板
deploy/.env                           # 本地密钥，不提交
```

Docker/生产容器将外部配置只读挂载为 `/app/config/bootstrap.yml`，通过 Spring `spring.config.import` 或 `SPRING_CONFIG_ADDITIONAL_LOCATION` 加载。仓库内 application 配置只声明安全默认值，不保存真实密钥。

### 6.2 配置结构

```yaml
cgc-pms:
  bootstrap:
    enabled: ${CGCPMS_BOOTSTRAP_ENABLED:false}
    version: 1
    tenant-id: ${CGCPMS_BOOTSTRAP_TENANT_ID:0}

    organization:
      company-code: ROOT
      company-name: 总公司
      department-code: ROOT_DEPT
      department-name: 总部

    administrator:
      username: ${CGCPMS_BOOTSTRAP_ADMIN_USERNAME:admin}
      real-name: 平台管理员
      email: ${CGCPMS_BOOTSTRAP_ADMIN_EMAIL:}
      password: ${CGCPMS_BOOTSTRAP_ADMIN_PASSWORD:}
      role-code: SUPER_ADMIN

    platform:
      locale: zh-CN
      timezone: Asia/Shanghai
      currency: CNY
```

### 6.3 配置约束

- `enabled` 默认必须为 `false`。
- 启用时密码、用户名、公司编码和部门编码必须非空。
- 密码满足项目安全策略，不得使用仓库默认密码。
- `role-code` 第一版固定为 `SUPER_ADMIN`，不允许任意配置高权限角色。
- tenant 第一版固定使用现有受支持的 `tenant_id`；不添加不存在的租户名称持久化模型。
- 平台默认参数只有在存在明确读取或持久化位置时才生效；否则从模板删除。
- 配置对象的 `toString`、异常和日志必须屏蔽密码。

## 七、外部密钥方案

### 7.1 密钥来源

| 环境 | 来源 | 要求 |
| --- | --- | --- |
| local/dev | `deploy/.env` | 已被 Git 忽略，禁止共享真实值 |
| test/CI | CI Secret | 用后即销毁，不输出到日志 |
| demo | 外部 `.env` 或只读 Secret 文件 | 仅部署人员可读 |
| production | Docker/Kubernetes Secret 或 Vault 类系统 | 禁止烘焙进镜像 |

### 7.2 生命周期

1. 部署人员生成随机临时密码。
2. Secret 注入应用进程。
3. bootstrap 将密码 BCrypt 编码后写入 `sys_user.password`。
4. 初始化日志只记录用户名、租户、结果和时间，不记录密码。
5. 管理员首次登录后立即修改密码。
6. 删除/轮换 bootstrap Secret。
7. 将 `CGCPMS_BOOTSTRAP_ENABLED` 改回 `false`。

修改 Secret 后重启不得自动修改已有管理员密码。bootstrap 只负责首次创建，不承担密码同步或恢复。

## 八、幂等初始化服务

### 8.1 组件

建议新增：

```text
com.cgcpms.bootstrap
├─ config/PlatformBootstrapProperties.java
├─ runner/PlatformBootstrapRunner.java
├─ service/PlatformBootstrapService.java
├─ entity/BootstrapState.java
└─ mapper/BootstrapStateMapper.java
```

职责：

- `Properties`：配置绑定、格式校验、敏感字段脱敏。
- `Runner`：Flyway 完成后检查开关并触发服务。
- `Service`：事务、状态锁、组织、管理员、角色关系和完成状态。
- `StateMapper`：对固定状态行执行 `SELECT ... FOR UPDATE`。

### 8.2 状态表

最小结构：

```sql
CREATE TABLE sys_bootstrap_state (
    bootstrap_key VARCHAR(64) NOT NULL,
    bootstrap_version INT NOT NULL,
    status VARCHAR(20) NOT NULL,
    completed_at DATETIME NULL,
    PRIMARY KEY (bootstrap_key)
);
```

B215 预置：

```text
bootstrap_key = PLATFORM_ADMIN
bootstrap_version = 1
status = PENDING
```

### 8.3 执行顺序

```text
Flyway 完成
  ↓
检查 enabled
  ↓
校验配置和 Secret
  ↓
事务开始
  ↓
锁定 PLATFORM_ADMIN 状态行
  ↓
若 COMPLETED：退出
  ↓
检查 SUPER_ADMIN 角色存在
  ↓
按 tenant_id + company_code 查询/创建根公司
  ↓
按 tenant_id + department_code 查询/创建根部门
  ↓
按 tenant_id + username 查询/创建管理员
  ↓
绑定 SUPER_ADMIN
  ↓
状态改为 COMPLETED
  ↓
事务提交
```

### 8.4 幂等决策矩阵

| 发现状态 | 动作 |
| --- | --- |
| 状态为 `COMPLETED` | 直接退出，不更新任何业务数据 |
| 管理员不存在 | BCrypt 编码密码，创建用户并绑定角色 |
| 同名用户存在且已有 `SUPER_ADMIN` | 不修改密码；核对租户和状态后标记完成，日志注明 adopted |
| 同名用户存在但无 `SUPER_ADMIN` | 启动失败；禁止自动提权 |
| `SUPER_ADMIN` 角色不存在或停用 | 启动失败；判定基线不完整 |
| 公司/部门已存在且编码一致 | 复用；不重复创建 |
| 公司/部门编码存在但名称/归属冲突 | 启动失败，要求人工确认 |
| 任一步骤异常 | 整个事务回滚，状态保持 `PENDING`，下次允许重试 |
| 两个实例同时启动 | 状态行锁串行执行；第二个实例读取 `COMPLETED` 后退出 |

### 8.5 与现有服务的边界

- 不直接调用依赖 `UserContext` 的交互式 `SysUserService.create/assignRoles`。
- 复用统一 `PasswordEncoder`，不能自己实现哈希。
- 组织查重逻辑可从 `OrgInitService` 提取可配置的内部方法；保留其既有回填行为。
- 管理员同时写入现有兼容字段 `is_admin=1` 和 `SUPER_ADMIN` 用户角色关系；权限权威仍为角色关系。
- 不开放 HTTP bootstrap 接口，避免未认证远程创建超级管理员。
- bootstrap 成功、失败和 adopted 结果进入结构化安全日志；密码和摘要均不得输出。

### 8.6 dev-login 调整

旧 `demo_dev_super_admin` 被排除后：

- dev-login 默认用户名改为读取显式环境变量；可与 bootstrap 管理员用户名一致。
- dev-login 仍只允许 `dev/local` profile，production 保持禁用。
- 用户不存在时不得静默创建第二个超级管理员。
- 自动化测试覆盖默认用户名、显式用户名和不存在用户三种情况。

## 十、实施阶段

### Phase 0：计划确认与冻结

修改范围：仅计划、决策清单和只读盘点。

任务：

1. 确认主线范围、非目标和生产禁止边界。
2. 冻结最新迁移版本 N；当前候选 N=215。
3. 生成 MySQL/H2/Java migration 清单、checksum 和类型分类。
4. 确认 Flyway 当前版本对 baseline migration 的行为并建立最小验证用例。
5. 签认系统数据白名单与演示数据黑名单。

风险：实施期间继续新增迁移导致基线漂移。

验收：迁移 N、白名单、黑名单、兼容期和责任人全部确认；存在待确认项不得进入 Phase 1。

### Phase 1：最终基线与历史链分层

修改范围：Flyway 资源、配置、迁移完整性测试和数据库初始化文档。

任务：

1. 建立旧链参考 MySQL/H2 数据库并生成最终 schema 清单。
2. 构建 MySQL/H2 `BN`。
3. 原样移动 V1—VN 到 legacy 目录。
4. 配置新旧双路径。
5. 建立 schema 等价、历史 checksum、版本唯一和无演示数据测试。

风险：基线遗漏约束、视图、注释或系统种子；移动路径使旧库 validation 失败。

验收：新空库 BN 与旧链最终 schema 等价；旧 V180/VN 数据库 validation 和升级通过；V1—VN checksum 不变。

### Phase 2：配置模板、外部密钥与初始化服务

修改范围：bootstrap 配置、状态表、Java 服务、组织复用、部署模板和安全测试。

任务：

1. 建立 `bootstrap.example.yml` 和 `.env.example` 变量。
2. 建立配置绑定、脱敏和 fail-fast 校验。
3. 建立状态表、状态行锁和事务初始化服务。
4. 创建/复用根公司、根部门、管理员和 `SUPER_ADMIN` 关系。
5. 调整 dev-login 默认账号来源。
6. 建立禁用、首次执行、重复执行、并发、冲突和缺密钥测试。

风险：Secret 泄露、同名用户自动提权、多实例重复初始化、普通用户服务上下文不适用。

验收：密码不进入 Git/日志/明文库；初始化只成功一次；任何冲突 fail-close；production 默认禁用。

### Phase 3：旧演示数据退出

修改范围：BN 系统数据白名单、演示迁移分类报告和业务空白断言。

任务：

1. 对 V6、V98、V100—V111、V120、V134 逐文件拆分“正式系统配置”和“演示业务事实”。
2. 正式配置按白名单纳入 BN；演示事实全部排除。
3. 建立项目、合同、材料实例、仓库、库存、业务单据、审批实例零数据断言。
4. 保证系统字典、菜单权限、角色、科目和分类可正常使用。

风险：把正式权限或字典误判为演示数据，导致空库可启动但不可操作。

验收：新库业务事实为零；必要系统数据数量和业务键与白名单完全一致；管理员可登录并访问系统管理。

### Phase 4：完整演示项目数据包

修改范围：显式演示加载包、加载/验证入口和演示说明，不修改 Flyway 自动启动行为。

任务：

1. 定义演示项目名称、编码、规模、角色和时间轴。
2. 按端到端业务闭环准备数据。
3. 通过正式服务/API 或受控 fixture 入口加载。
4. 建立 package manifest、阶段 checkpoint 和幂等检查。
5. 建立经营指标、库存守恒、成本守恒、审批和追踪验收。

风险：直接 SQL 绕过业务规则；演示对象不完整；重复加载污染数据库。

验收：未执行加载命令时零演示业务；执行一次形成完整闭环；重复执行不产生重复事实；所有预期指标可解释。

### Phase 5：兼容、回归与运行态验收

修改范围：测试、部署配置、开发文档和正式验收报告。

任务：

1. MySQL 新空库 BN 验收。
2. MySQL 旧链升级与 validation 验收。
3. H2 新空库和旧链升级验收。
4. 后端全量测试、构建和迁移完整性门禁。
5. 本地运行态登录、菜单、权限、字典、科目和演示加载验收。
6. 检查镜像、日志和 Git diff 无 Secret。

风险：测试只覆盖空库，未覆盖存量库；dev-login 或角色权限回归。

验收：第十一节全部通过，正式质量报告给出通过/不通过结论；任一数据、安全或兼容阻塞未解决则不得合并。

### Phase 6：发布与历史链退役决策

修改范围：仅经单独授权的提交、推送、PR、部署和未来归档。

任务：

1. 新环境使用 BN。
2. 既有环境继续保留 legacy 验证和增量升级能力。
3. 维护环境清单：数据库、当前 Flyway 版本、是否已重建、最后验证时间。
4. 所有既有环境退役后，另立授权门决定是否把 legacy 移出运行包。

风险：过早移除 legacy 导致既有数据库 validation 失败。

验收：环境清单证明不存在仍依赖旧链的数据库前，不执行物理归档。

- 发现当前目标数据库不是 dev/test/demo。
- 数据库 host 不是 `localhost`/`127.0.0.1`，但操作涉及重置或删除数据。
- 缺少 `.codex-autopilot/ALLOW_TEST_DATA_RESET`，但操作涉及测试数据重置。

- BN schema 与旧链最终 schema 存在未签认差异。
- 系统数据白名单存在无法判断的业务数据。

- Secret 出现在 Git diff、日志或镜像层。
- 既有数据库 validation 失败。
- 完整演示项目加载出现库存、成本、资金或审批不守恒。

## 十三、回滚与恢复

### 13.1 基线改造回滚

- 合并前：恢复原 Flyway locations 和资源布局；V1—VN 内容始终未修改。
- 新空 dev/test/demo 库：在满足数据重置硬门禁后可删除并重新创建。
- 既有数据库：不执行 clean，不删除 `flyway_schema_history`，继续使用 legacy 链。
- BN 已在新环境执行后若发现问题：停止环境使用，修正 BN 后重建该空白环境；不得把错误 BN 用 `repair` 标记成功。

### 13.2 Bootstrap 回滚

- 事务失败自动回滚。
- 完成后不自动删除管理员；需要撤销时必须确认该环境尚无业务数据、无其他用户依赖，并使用专用受控脚本。
- 关闭 `enabled` 只停止后续初始化，不删除已创建组织或管理员。

### 13.3 演示数据恢复

- 默认不提供自动清理。
- 数据错误优先在新的空白 demo 库重新显式加载。
- 若必须清理，只允许 manifest 所有数据，并执行预览、引用检查、环境/host/marker 门禁和备份。
