# 第 10C 专项：角色菜单绑定编辑增强最小实现包任务计划书

## 计划状态

正式版。

本实现包由 `第10C专项-可编辑权限平台任务计划书` 派生。主负责人已基于 10C 正式版上下文冻结首包边界：只做现有角色菜单绑定链路的后端安全加固与最小审计补强，不做权限码本体编辑，不新做权限平台页面，不引入新权限事实源。

## 一、前置条件

以下前置条件现已冻结，可据此进入实现：

1. 首包仅允许“角色菜单绑定编辑增强”，不开放权限码本体编辑。
2. 前端默认零新增页面、零新增事实源，继续复用 `PermissionModal -> updateRoleMenus -> PUT /system/roles/{id}/menus`。
3. 后端继续复用 `SysRoleService.assignMenus`，在其上补最小安全门禁与最小审计。
4. 最小审计口径冻结为“独立快照表”，记录：
   - 操作者
   - 目标角色
   - 变更时间
   - `before_menu_ids`
   - `after_menu_ids`
   - 成功/失败结果
   - 错误摘要
5. 失败回滚 / 恢复路径冻结为：
   - 写操作事务内失败自动回滚；
   - 已落库成功但需恢复时，使用审计快照中的 `before_menu_ids` 通过现有角色菜单绑定链路回写；
   - 本首包不要求一键回滚页面或单独回滚接口。
6. 高危门禁冻结为三条硬规则：
   - `SUPER_ADMIN` 角色不允许通过本 MVP 编辑；
   - 当前用户持有的角色不允许被当前用户自己编辑；
   - 任一菜单 diff 触碰高危系统权限 `system:user:*` / `system:role:*` / `system:menu:*` 时，本 MVP 直接拒绝变更。

## 二、最小实现范围

### 仅允许范围

1. 前端：
   - 复用 `PermissionModal`
   - 默认零代码改动；若现有错误提示不足以承接后端拒绝语义，才允许补最小文案/提示
2. 后端：
   - 继续复用 `PUT /system/roles/{id}/menus`
   - 继续复用 `SysRoleService.assignMenus`
   - 只补最小安全门禁
   - 记录最小角色菜单变更审计快照
3. 数据：
   - 新增一张最小角色菜单变更审计快照表
   - MySQL / H2 对等 migration

### 明确不纳入

1. 不编辑 `sys_menu.perms`
2. 不开放权限码本体编辑
3. 不做动态权限 DSL
4. 不做全站权限平台
5. 不把项目级强隔离或入口注册并入

## 三、候选实现动作

1. 角色菜单绑定写操作的后端安全加固
2. 最小变更审计快照落库
3. 基于审计快照的最小恢复路径固化
4. 仅在必要时补前端最小提示，不默认改 UI

## 四、验收标准

1. 角色菜单绑定正向修改 1 条成功。
2. 无权限账号不可进入或不可提交。
3. 自我提权 / 自我锁死被阻止。
4. 审计记录可查。
5. `SUPER_ADMIN` 角色编辑被阻止。
6. 高危系统权限菜单 diff 被阻止。
7. 失败有恢复路径。
6. 不引入新权限事实源。

## 五、阻塞项

当前无主负责人层面的范围阻塞。剩余只允许出现以下执行期阻塞：

1. 现有 `SysRoleService.assignMenus` 链路无法在不扩散到权限码本体的前提下承接三条硬门禁；
2. 最小审计快照表若被证明不足以支撑恢复路径，需要补充字段，但仍不得扩成通用权限平台表；
3. 前端若无法用现有错误提示承接后端拒绝语义，才允许最小补提示。

## 六、当前建议

按本正式版直接分派首包实现：后端主责，数据库补最小快照表，前端默认待命，随后走安全、测试、运维、质量闭环。

## 七、主负责人裁决

1. 10C 首包只允许增强现有角色菜单绑定链路，不新建可编辑权限平台页面。
2. 10C 首包默认不改前端；只有后端拒绝语义在当前 `PermissionModal` 中不可理解时，才允许补最小提示。
3. 10C 首包的安全门禁固定为：
   - 禁改 `SUPER_ADMIN`
   - 禁改当前用户自己持有的角色
   - 禁改高危系统权限菜单 diff
4. 10C 首包必须新增最小审计快照表；仅靠现有 `sys_operation_audit_log` 不足以收口。
5. 10C 首包若执行过程中试图扩到权限码本体、动态权限 DSL、项目强隔离、入口注册或全站 RBAC，直接判越界。
