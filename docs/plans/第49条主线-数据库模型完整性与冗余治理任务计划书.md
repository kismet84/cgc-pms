# 第49条主线：数据库模型完整性与冗余治理任务计划书

**Goal:** 以 `master@5bac4980`、Flyway V194 为合并冻结基线，实施 `docs/database/database-structure-audit.md` 中 DBA-P0-001—008、DBA-P1-001—016、DBA-P2-001—005 的完整处置，优先消除会破坏数据正确性、租户隔离、金额守恒和幂等性的缺陷，并把无法从代码仓库确认的生产容量、外部消费者和历史数据问题转换为可执行的部署前门禁，而不是猜测或静默删表。
**Architecture:** 只新增 V195 及以后 MySQL/H2 对称迁移，禁止修改 V1—V194；以现有业务服务为唯一写入口，以非空生成列实现活动行唯一键，以事务行锁和数据库约束共同保护库存/成本/资金事实，以租户字段、父表外键和服务端同租户校验共同保护 RBAC；审批只保留一个运行时模板解析入口。采用“预检脚本/迁移前置断言 → 向后兼容迁移 → 服务适配 → 双数据库验证 → 运行态复核”的最小分阶段方案，不连接生产数据库，不自动删除无法证明无依赖的历史表，不建设第二套工作流、主数据或通用数据平台。

> 计划状态：已完成（2026-07-17）
> 计划类型：数据库高风险治理、跨模块数据一致性、正式审计收口
> 执行分支：`codex/database-model-remediation`
> 完成基线：MySQL/H2 V210；原审计基线 V180，合并实施基线 V194
> Git 边界：数据库实施阶段未自动提交；后续提交、push、PR 合并和分支清理由用户另行明确授权

## 二、冻结决策

### 2.1 数据正确性决策

| 决策 | 统一规则 |
| --- | --- |
| 活动行唯一键 | 对 V88 的 13 张失效表采用 `active_unique_token = CASE WHEN deleted_flag=0 THEN 0 ELSE id END` 的非空生成列；唯一键包含租户和业务键；`deleted_token` 进入兼容退役，不再作为权威约束 |
| 成本来源幂等 | `cost_item` 唯一键统一为租户、来源类型、来源头、来源行、成本类型、活动令牌；来源行为空时由业务服务使用来源头作为稳定行键或明确的固定哨兵，不依赖 NULL 唯一语义 |
| 租户关联 | 跨租户绑定必须在服务入口拒绝；数据库关联表保存 `tenant_id` 并通过父表组合唯一键/外键保护，不允许只靠全局 ID 假设 |
| 库存退料 | 锁定原始 `mat_stock_txn` 行后再统计并写入；退料状态支持确认和冲销，数量与价值必须守恒；主从、来源和主数据建立 FK |
| 银行入账 | 入账回单在同一事务创建/复用 `collection_record` 并回写 `bank_receipt.collection_record_id`，外部流水号为幂等键；付款方向与收款方向分流 |
| 审批路由 | `WorkflowCoreService` 是唯一运行时模板解析入口；`approval_routing_rule` 在该入口参与匹配，模板金额范围是兜底；未实现 JSON 配置禁止被管理接口接受或标记为生效 |

### 2.2 退役与兼容决策

- 不直接物理删除 `overhead_allocation_record`、状态冗余列、`collection_record.reversal_of_id`、`css_class` 或 `.bak` 文件；先提供依赖扫描和零数据断言，满足条件才由新增迁移/资源移动退役。
- `pm_project.approval_status` 选择补齐真实审批链，因为项目已是合同、计划、预算、收入和成本的根业务对象；不保留半实现状态。
- `md_material.category_id` 选择建设最小材料分类主表、FK 与服务校验；不继续保留无模型查询参数。
- `sys_user.org_id` 进入兼容期：审批 POSITION 解析改为明确的用户岗位关系；旧字段只读兼容并标注弃用。
- P2 的物理分区和生产容量策略没有当前生产证据，本轮交付可执行容量阈值、归档候选查询和决策门，不提前分区。
- V210 只退役 14 个已确认无数据、无索引、无运行时依赖且已由 `active_unique_token` 替代的 `deleted_token`；`overhead_allocation_record` 因仓库外消费者未知继续保留。

## 三、实施阶段

### M0：V187 基线刷新与迁移预检

1. 重算表、字段、索引、FK、唯一约束、JSON、注释和代码引用清单。
2. 对所有拟新增唯一键执行重复组扫描，对所有拟新增 FK 执行孤儿/错租户扫描。
3. 固化 MySQL/H2 迁移版本对称性和历史 checksum 门禁。
4. 生成部署前只读预检 SQL；检测到语义冲突数据必须停止迁移并输出具体键值，禁止自动猜测归并。

验收：预检在空库和本地开发库均可执行；所有清理规则可逆、可解释。

### M1：活动唯一性与成本事实（DBA-P0-005、008）

1. V195 为 13 张表新增活动令牌并重建唯一键，保留旧 `deleted_token` 兼容列。
2. 对 `cost_subject` 已知三组活动重复进行引用影响扫描；仅对可证明等价的种子做引用迁移和软删除，不等价科目改为新的唯一业务编码并记录映射。
3. V196 统一 `cost_item` 来源幂等键，删除 `idx_cost_item_tenant_source` 完全重复索引。
4. 增加 13 表“创建—删除—重建—再次删除—再次重建”测试和成本重复生成测试。

验收：重复活动键插入失败；历史删除行不冲突；合法的不同 `cost_type` 成本可共存；同类型重复事实失败。

### M2：RBAC 与组织租户边界（DBA-P0-002、P1-007、P2-002）

1. V197 为 `sys_user_role`、`sys_role_menu` 增加 `tenant_id`，从父表回填并在不一致时阻断。
2. 为 `sys_user`、`sys_role`、`sys_menu` 建立可被引用的 `(tenant_id,id)` 唯一键，关联表增加组合 FK 与组合唯一键。
3. `SysUserService.assignRoles`、`SysRoleService.assignMenus` 显式写租户；所有读取按租户过滤。
4. POSITION 审批人解析切换到 `org_user_position`/明确关系，不再以 `sys_user.org_id` 猜测公司或部门。

验收：跨租户用户—角色、角色—菜单写入在服务和 DB 两层均失败；权限、审计快照和菜单回归通过。

### M3：材料采购、领用与退料闭环（DBA-P0-003、004、006）

1. V198 为退料头、明细建立项目、合同、仓库、领料、领料明细、原库存流水、原成本、材料和主从 FK，并补唯一的“原流水+退料明细”事实约束。
2. `MaterialReturnService.confirm` 首先对原出库流水加 `FOR UPDATE` 行锁，再校验累计数量；补真实并发用例。
3. 退料由单一 `CONFIRMED` 扩为可审计的 `CONFIRMED/REVERSED`，冲销反向库存流水和成本事实必须幂等。
4. 采购申请/订单/收货/领料增加现有 `project_wbs_task`、目标成本科目/预算行的最小显式引用。
5. 建设供应商退货头/明细和不合格处置事实；不合格数量不得进入可用库存，退货必须引用原收货明细和原库存流水。

验收：孤儿、错租户、错误来源被拒；并发超退只能一个事务成功；库存数量、库存价值和成本冲销分币守恒；供应商退货可追溯。

### M4：收入、资金与银行回单闭环（DBA-P0-007、P1-004）

1. V199 明确 `bank_receipt` 收款方向与付款方向约束，并为回款转换建立唯一幂等关系。
2. `FinanceIntegrationService.autoMatchReceipt` 对入账回单调用收入域受控入口，事务内创建/复用回款并回写 `collection_record_id`；出账继续匹配付款记录。
3. Trace 同时支持 `bank_receipt → collection_record → cash_journal_entry/accounting_entry` 和反向查询。
4. `collection_reversal` 作为唯一冲销事实；兼容期读取不再使用 `collection_record.reversal_of_id`，零依赖后退役列。

验收：重复回调不重复入账；成功入账回单恰好一个回款；回款、现金日记、会计分录和冲销可双向追溯。

### M5：审批单一权威与状态模型（DBA-P0-001、P1-002、003、005）

1. V200 完善 `approval_routing_rule` 唯一性、范围合法性、优先级确定性和模板同租户约束。
2. `WorkflowCoreService.queryTemplate` 在唯一入口解析规则；无规则时回退模板金额范围，冲突时拒绝而非随机选取。
3. 对 `wf_template_node` 的 `approver_config`、`condition_config` 等 JSON 建立可执行字段白名单；未执行字段在 API 写入时拒绝。
4. 费用、结算状态明确权威状态机和兼容派生策略；项目审批补齐模板、handler、提交/通过/驳回测试。

验收：同一输入只产生一个模板；合同类型、费用分类和金额边界有用例；每个可编辑配置都能影响执行，或被明确拒绝。

### M6：模型可维护性与历史遗留治理（DBA-P1-001、006、008—016）

1. 材料分类最小主表、FK、查询和服务校验落地。
2. 对 LONGTEXT JSON 添加 `JSON_VALID`、版本和长度约束；真正需要 JSON 运算的字段评估改原生 JSON，MySQL/H2 保持等价。
3. 为 V157+ 表补表/字段 COMMENT；新增迁移静态检查禁止无注释表和无说明新增字段。
4. 为高风险 `SELECT *` 先覆盖资金、审批、库存、RBAC 核心写路径，改显式投影和类型化映射；其余按扫描结果逐条关闭或正式承接。
5. `finance_alert` 与 `alert_log` 明确一个驾驶舱权威读取源和同步契约。
6. 历史 `.bak` 从运行资源目录移到文档归档；证明无构建/运行依赖后执行。
7. 修正 local/dev `validate-on-migrate`，CI 同时跑 MySQL/H2 空库；更新数据库规范中的版本和软删除规则。

验收：静态门禁可阻止新增无注释结构、迁移缺镜像和核心 `SELECT *` 回归；运行资源不再混入旧 schema。

### M7：演进能力与最终收口（DBA-P2-001、003—005）

1. 建立版本化 `business_type/source_type` 注册清单和契约测试，不以注册表替代显式 FK。
2. 财务/收入共用审计事件、导入批次和对账问题只抽取明确复用接口/只读视图；没有重复运行逻辑的不物理合表。
3. 生成当前 schema 数据字典和 ER 关系视图，来源必须是 migration + INFORMATION_SCHEMA。
4. 定义数据量、时间跨度、查询退化三项归档/分区触发阈值；当前无生产证据时将“立即分区”有依据关闭。
5. 更新审计报告、项目地图和正式质量报告，逐项注销原 29 个后续项。

## 四、迁移与回滚策略

1. 每个版本先运行只读重复/孤儿扫描；发现冲突立即失败并输出表名、业务键和数量。
2. DDL 采用扩展—回填—双读/切换—约束—退役顺序；不在一个不可逆脚本中同时改语义和删兼容列。
3. MySQL 迁移提供上线前预估锁表 SQL；大表索引需记录行数和预计窗口。H2 Java 迁移仅处理 H2 无法可靠删除匿名唯一约束的差异。
4. 回滚以应用回退和兼容列保留为主；已写入的新事实表不物理删除。任何数据归并保留映射审计。
5. 本任务只操作本地 dev/test 数据库；生产迁移须另经备份、只读预检、变更窗口和人工批准。

## 七、风险与停止条件

- 若本地数据出现无法仅凭仓库事实判断的同编码不同语义记录，停止自动归并，保留预检结果并把迁移设计为阻断式；不删除、不猜测。
- 若新组合 FK 暴露历史错租户，先修业务写入口，再提供明确映射脚本；不能用 `FOREIGN_KEY_CHECKS=0` 作为最终方案。
- 若当前工作区既有改动与本任务同文件冲突，保留用户改动并缩小 patch；无法安全隔离时报告具体冲突。
- 若 MySQL/H2 无法给出等价约束，优先保证 MySQL 生产语义，并用 H2 契约测试显式记录差异；不得用弱约束冒充等价。
