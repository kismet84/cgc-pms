# 第39条主线：项目知识图谱采集机制任务计划书

**Goal:** 在现有 Neo4j、增量文档采集器和 `cgc-pms-knowledge-graph` MCP 基础上，形成覆盖正式文档、backlog、计划、报告、Git、Codex 结构化会话摘要及运行日志摘要的可追溯采集闭环，并以幂等、版本保留、失败恢复、来源覆盖率和禁止区零读取作为验收方向。  
**Architecture:** 采用“事件驱动即时采集 + 定时对账补漏”的本地混合架构；仓库与 Git 保持事实源，Neo4j 只作为关联索引；确定性元数据和显式引用可直接入图，规则抽取关系必须绑定来源，模型语义抽取不进入本期最小闭环。复用现有 Node 22、Neo4j Driver、MCP Server 和本机 Neo4j Community，不引入消息队列、独立调度平台、生产服务或原始私有会话扫描。

**Tech Stack:** Node.js 22、Neo4j Community 5.26、Neo4j JavaScript Driver、Cypher、Model Context Protocol、PowerShell、Git、Node Test Runner。

## 1. 当前基线

现有 `tools/knowledge-graph` 已具备：

- `Project`、`Artifact`、`Section`、`GitCommit`、`Episode`、`Entity` 节点约束及全文索引。
- `docs/` 与 `plugins/cgc-pms-autopilot/artifacts/` 白名单采集。
- 私有目录和历史封存目录禁止读取。
- 基于文件 SHA-256 的未变内容跳过。
- Markdown 章节解析和 `REFERENCES` 显式引用关系。
- 最近 250 个 Git 提交采集。
- MCP 查询、全文检索、关系遍历和受控 `kg_record_episode` 写入。

当前缺口：

- 只保存 `Artifact` 最新内容，缺少 `ArtifactVersion` 历史版本。
- 缺少 `Source`、`SourceCursor`、`CollectionRun`，无法证明采集位置和失败恢复状态。
- 每轮仍全量枚举文件并重建全部 `REFERENCES`，未做到文件级关系增量。
- Git 只读取固定最近 250 条，未按游标持续推进，也未建立 `CHANGES` 文件关系。
- 日志与会话依赖人工调用 `kg_record_episode`，缺少统一结构化载荷和收口入口。
- 缺少脱敏、失败队列、采集指标和自动对账触发。

## 2. 范围与非目标

### 2.1 本期范围

- 正式文档、计划、质量报告、backlog、迭代记录和插件正式产物。
- Git 提交及其变更文件关系。
- Codex 任务收口后的结构化会话摘要。
- 测试、构建、CI、本地运行结束后的结构化日志摘要。
- 文件版本、来源游标、采集运行、失败恢复和基础指标。
- 本地手动触发、任务收口触发和 Windows 定时对账方案。

### 2.2 非目标

- 不扫描 Codex 全局私有历史目录，不保存原始推理过程。
- 不读取项目规则明确禁止的私有目录。
- 不将原始大日志、截图、缓存或调试输出正文写入 Neo4j。
- 不建设独立 Web 管理后台。
- 不引入 Kafka、RabbitMQ、Redis Stream 等消息基础设施。
- 不使用 LLM 自动创建正式事实关系；后续如立项，只能先写候选关系并保留证据。
- 不发布生产、不连接生产数据库、不自动 push。

## 3. 数据模型

### 3.1 新增节点

| 节点 | 唯一键 | 用途 |
| --- | --- | --- |
| `Source` | `key` | 表示文档、Git、Codex 收口、日志摘要等采集源 |
| `SourceCursor` | `sourceKey` | 保存最后成功游标、最后成功时间和失败位置 |
| `CollectionRun` | `id` | 记录一次采集的开始、结束、状态和指标 |
| `ArtifactVersion` | `id = artifactId + sha256` | 保存文档历史版本和版本级正文摘要 |
| `Evidence` | `id` | 表示命令、测试、构建、审计或接口验收证据 |
| `Decision` | `id` | 表示有明确来源的正式决策或裁决 |

### 3.2 新增关系

```text
(ArtifactVersion)-[:VERSION_OF]->(Artifact)
(ArtifactVersion)-[:COLLECTED_IN]->(CollectionRun)
(ArtifactVersion)-[:DERIVED_FROM]->(Source)
(GitCommit)-[:CHANGES]->(Artifact)
(Episode)-[:DERIVED_FROM]->(Source)
(Evidence)-[:DERIVED_FROM]->(Episode|ArtifactVersion)
(Decision)-[:SUPPORTED_BY]->(Evidence)
(Decision)-[:SUPERSEDES]->(Decision)
(CollectionRun)-[:USED_CURSOR]->(SourceCursor)
```

所有语义事实关系至少包含：

```text
sourceRef
extractedAt
extractor
confidence
reviewStatus
```

确定性链接的 `extractor=deterministic`、`confidence=1.0`；缺少来源时不得创建正式关系。

```cypher
MATCH (r:CollectionRun) RETURN r.status, count(*) ORDER BY count(*) DESC;
MATCH (a:Artifact)-[:CURRENT_VERSION]->(v:ArtifactVersion)
RETURN count(a), count(DISTINCT v);
MATCH (c:GitCommit)-[:CHANGES]->(a:Artifact) RETURN count(c), count(a);
MATCH (n) WHERE n.sourceRef IS NULL AND (n:Decision OR n:Evidence)
```

## 5. 上线与回滚

### 5.1 本地启用顺序

1. 备份 Neo4j 命名卷或导出当前图谱。
2. 应用兼容 Schema。
3. 执行一次迁移采集并核对节点数量。
4. 启用任务收口入口。
5. 稳定观察后再安装定时对账任务。

### 5.2 回滚方式

- 停止或卸载本地定时任务。
- Codex 中禁用或移除 `cgc-pms-knowledge-graph` MCP。
- 保留仓库事实源，不受 Neo4j 回滚影响。
- 新增节点均带 Schema 版本和 CollectionRun，可按版本清理；任何清理前必须先备份和预览。
- 不回滚或覆盖仓库计划、报告、backlog 和 Git 历史。

## 6. 风险与控制

| 风险 | 等级 | 控制措施 |
| --- | --- | --- |
| 私有目录或敏感信息误采集 | 高 | 白名单优先、禁止区哨兵测试、入图前脱敏 |
| 游标错误导致数据遗漏 | 高 | Source 独立游标、事务成功后推进、定时对账补漏 |
| 语义抽取产生虚假事实 | 高 | 本期不落正式 LLM 关系，只采确定性和规则证据 |
| 文档变化导致版本膨胀 | 中 | SHA 幂等、版本保留策略后续单独立项，不盲删 |
| Neo4j 不可用影响项目工作 | 中 | 图谱为旁路索引，采集失败不得阻断仓库事实源 |
| 定时任务与手动采集并发 | 中 | 单实例锁、短事务、运行状态可查询 |
| MCP 任意写入扩大攻击面 | 高 | 保持只读查询，仅开放固定 Schema 的受控写入 |

## 7. 完成定义

只有同时满足以下条件，主线才可判定通过：

- Schema、采集器、触发器和 MCP 回归全部通过。
- 禁止目录读取次数为 0，敏感数据检索命中为 0。
- 所有 Decision 和 Evidence 均有来源。
- 增量、删除、重复、失败恢复和断点续采验证通过。
- 正式验收报告写入 `docs/quality/`，且不包含临时日志、截图或缓存。
- `git diff --check` 通过，工作区范围与本主线一致。
- 不发布生产、不连接生产库、不自动 push。

**计划裁决：** 当前方案可进入实现阶段，但定时任务安装必须在核心增量、脱敏、失败恢复和边界测试通过后执行。现有未提交的知识图谱工具实现属于本计划基线，实施前应先确认其 Git 归属与提交策略。
