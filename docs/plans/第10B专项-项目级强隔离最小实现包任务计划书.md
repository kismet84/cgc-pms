# 第 10B 专项：项目级强隔离最小实现包任务计划书

## 计划状态

正式版。

本实现包由 `第10B专项-项目级强隔离任务计划书` 派生。主负责人已基于 10B 正式版上下文冻结首批实现边界：只做同租户项目边界最小落地，不扩成全业务、全租户、全 `data_scope` 隔离改造。

## 一、前置条件

以下前置条件现已冻结，可据此进入实现：

1. 项目访问语义冻结为：
   - `ADMIN/SUPER_ADMIN`：放行；
   - `ALL`：放行同租户项目；
   - `SELF`：仅允许访问 `project.created_by = currentUserId` 的项目；
   - `DEPT/CUSTOM/未知 data_scope`：一律 `fail-close`；
   - `pm_project_member` / 项目角色：本实现包不纳入授权判定主链。
2. 首批验收只覆盖最小链路，不扩成全业务隔离大包。
3. 第二租户本包不做完整样本矩阵；以“最小跨租户负向后端证据或现有可得硬证”为通过门槛，不单独扩 seed。

## 二、最小实现范围

### 首批链路

1. 一个项目访问校验入口：
   - `ProjectAccessChecker.checkAccess`
2. 一条 workflow 带 `projectId` 链：
   - `WorkflowQueryService.getInstanceDetail`
3. 一类目标业务详情：
   - 首批固定为 `合同详情`：
     - `CtContractController`
     - `CtContractService.getById`

### 首批不纳入

1. 不做全业务模块隔离。
2. 不做复杂 `data_scope` 全矩阵。
3. 不做第二租户完整矩阵。
4. 不做全站列表/导出/统计一并改造。
5. 不做权限字典、可编辑权限平台、入口注册体系联动改造。

## 三、候选实现动作

1. 后端：
   - 维持 `ProjectAccessChecker` 当前最小语义
   - 补齐 `合同详情` 链路的项目访问校验
   - 保证 workflow 详情 `projectId` 校验与合同详情口径一致
   - 对未支持的 `data_scope` 维持 `fail-close`
2. 数据：
   - 默认零 migration
   - 仅在测试无法形成最小同租户正负样本时，才允许评估最小 seed
3. 前端：
   - 不承担隔离逻辑，只承接错误态 / 空态 / 深链负向表现

## 四、验收标准

1. 同租户有项目权限账号正向可用。
2. 同租户无项目权限账号对首批链路被拒绝或返回空结果。
3. workflow 带 `projectId` 的详情边界与目标业务详情一致。
4. 跨租户至少提供最小负向后端证据或现有可得硬证。
4. 第 8 / 第 9 条基线不回退。

## 五、阻塞项

当前无主负责人层面的范围阻塞。剩余仅允许出现以下执行期阻塞：

1. 现有 demo 数据无法形成同租户有权限 / 无权限最小对照样本；
2. 首批合同详情链路触达面超出最小包边界；
3. 运行态刷新后无法形成最小跨租户负向证据。

## 六、当前建议

按本正式版直接分派最小实现：后端主责，数据库按零迁移默认待命，前端仅在错误态/空态缺口时补最小变更，随后走安全、测试、运维、质量闭环。

## 七、主负责人裁决

1. 10B 首包只做 `ProjectAccessChecker` 语义落地一致性，不引入 `pm_project_member` 或项目角色授权模型。
2. 10B 首包只验三段链路：
   - `ProjectAccessChecker.checkAccess`
   - `WorkflowQueryService.getInstanceDetail`
   - `CtContractService.getById`
3. 10B 首包默认零 migration、零第二租户样本扩建、零复杂 `data_scope` 扩项。
4. 若执行过程中发现必须改动付款申请、采购申请、分包计量、入口注册、权限字典、可编辑权限平台，均判定为越界，不纳入本包。
