# 第 10D 专项：全站入口注册体系首批实现包任务计划书

## 计划状态

正式版。

本实现包由 `第10D专项-全站入口注册体系任务计划书` 派生。当前正式冻结为 10D 首批最小实现包，只允许围绕审批中心已通过第 8 / 第 9 条验收的三类入口做代码侧 registry 收口与验证，不扩成全站平台。

主负责人裁决：首批实现允许进入执行；但若前端只读核对后确认第 9 条 `workflowDisplay.ts` 已完全满足本包目标，则允许以“零代码实现 + 补充验证/归档”收口，不得为了 10D 制造无意义重构。

## 一、前置条件

以下条件已冻结满足，作为本包执行边界：

1. 已完成全站入口事实源盘点。
2. 已冻结 registry 只承载导航与展示元数据，不作为授权边界。
3. 已冻结第一批业务范围：仅覆盖审批中心既有三类业务入口：
   - 合同审批：`CONTRACT` / `CONTRACT_APPROVAL`
   - 采购申请：`PURCHASE_REQUEST`
   - 分包计量：`SUB_MEASURE`
4. 已冻结验收方式为“核心矩阵 + 抽样矩阵”。
5. 已明确第 8 / 第 9 条基线回归范围：管理员三类正向、workflow-only 三类负向、cc-readonly 轻量只读、non-participant 轻量不可见或 `data=null`。

## 二、最小实现范围

### 仅允许范围

1. 继续使用代码侧轻量 registry。
2. 首批业务仅限审批中心三类已冻结入口，不新增付款申请，不新增更多业务类型。
3. 只做前端 registry/helper 收口，后端默认零新增授权聚合。
4. registry 字段最小集固定为：
   - `businessType`
   - `displayName`
   - `permissionCode`
   - `targetRoute`
   - `openMode`
   - `forbiddenPolicy`
5. 若现有 `frontend-admin/src/pages/approval/workflowDisplay.ts` 已满足上述字段与调用口径，可以保持原文件不拆分；只有在能减少真实分叉或测试成本时，才允许做轻量整理。

### 明确不纳入

1. 不新增入口注册表。
2. 不数据库化。
3. 不插件化。
4. 不新增 workflow 授权聚合。
5. 不新增目标业务权限预判接口。
6. 不默认接入付款申请和更多审批类型。
7. 不把 `workflowDisplay.ts` 直接外扩成全站插件框架。
8. 不改目标业务接口鉴权，不让 registry 返回目标业务详情字段。
9. 不改第 10B 项目级强隔离或第 10C 角色菜单绑定编辑链路。

## 三、候选实现动作

1. 前端只读核对首批三类入口事实源：
   - `frontend-admin/src/pages/approval/workflowDisplay.ts`
   - `frontend-admin/src/pages/approval/todo.vue`
   - `frontend-admin/src/pages/approval/detail.vue`
   - 目标路由：合同详情、采购申请详情、分包计量详情
2. 判断现有 registry/helper 是否已满足首批字段、权限码、路由与无权限策略。
3. 若存在真实分叉或测试缺口，做最小前端整理与定向测试；若不存在真实缺口，按零代码实现收口。
4. 补充或复用最小前端验证：
   - 三类业务映射存在且字段完整。
   - 未知业务类型不误显入口。
   - 无权限时入口保持 disabled 或等价禁用策略。
   - `ADMIN/SUPER_ADMIN` 旁路与第 8 / 第 9 条基线一致。
5. 前端如有改动，执行定向单测和 `pnpm build`；如零代码，则至少回传只读核对证据与是否需要运行态刷新判断。

## 四、验收标准

1. 首批三类入口均由同一 registry/helper 口径提供展示名、权限码、目标路由、打开方式与禁用策略。
2. 管理员三类正向入口不回退：
   - 合同审批可进入合同详情。
   - 采购申请可进入采购申请详情。
   - 分包计量可进入分包计量详情。
3. `demo_workflow_only` 三类负向不回退：
   - workflow 详情可见。
   - 目标业务入口 disabled 或等价不可点。
   - 不命中目标业务详情接口。
   - 不泄露合同、采购申请、分包计量目标业务字段。
4. `demo_cc_readonly` 轻量回归：抄送详情只读，无同意/驳回/撤回/重提动作误显。
5. `demo_non_participant` 轻量回归：无关实例不可见或详情探针保持 `code=0,data=null` 等价拒绝。
6. 不引入数据库注册表、后端授权聚合、目标业务权限预判接口。
7. 不接入付款申请和更多业务类型。

## 五、阻塞项

当前无主负责人层面的实现阻塞。执行中若命中以下任一情况，必须停止实现并回报主线程裁决：

1. 前端提出需要新增后端授权聚合或目标业务权限预判接口。
2. 前端提出需要新增入口注册表、migration、seed 或数据库化 registry。
3. 实现需要默认接入付款申请或更多业务类型。
4. 实现会改变第 8 / 第 9 条审批入口安全基线。
5. 实现会把 registry 用作真实授权边界，或让 workflow 详情聚合目标业务详情字段。

## 六、当前建议

按“前端主责、后端/数据库零改动待命、安全复核、测试验收、运维按需刷新、质量归档”的顺序推进：

1. 前端先执行只读核对与最小实现判断。
2. 若前端有代码改动，完成定向测试和构建后转安全复核。
3. 若前端无代码改动，直接转安全/测试按零代码实现包验收。
4. 后端和数据库默认只做零改动边界复核。
5. 运维仅在前端运行态需要刷新时介入，不默认重启后端。
