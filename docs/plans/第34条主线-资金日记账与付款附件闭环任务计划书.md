# 第34条主线-资金日记账与付款附件闭环任务计划书

> **For agentic workers:** 实施本计划时必须使用 `superpowers:subagent-driven-development`（推荐）或 `superpowers:executing-plans`，严格按任务顺序执行；每个实现任务都必须遵循 TDD，先证明失败，再做最小实现。

**Goal:** 在现有付款、文件、审计和预警能力上新增企业级现金/银行统一资金日记账，使手工收支与付款成功流水都必须经财务上传附件并归档后才算闭环；超过 24 小时未归档的记录自动进入预警中心，并允许超级管理员通过受控撤销归档流程修正数据。

**Architecture:** 新增独立 `cashbook` 业务域，核心业务模型仅包含 `fund_account` 与 `cash_journal_entry`；附件复用 `sys_file`，预警复用 `alert_log`、现有通知分发器与预警页面，普通操作审计复用 `sys_operation_audit_log`。为满足已归档数据修正的前后快照要求，新增一张只追加的 `cash_journal_change_log` 辅助审计表；不改造会计分录语义，不把 `pay_record` 扩展成通用收支表，不建设银行导入、自动对账、收款平台或凭证中心。

**Tech Stack:** Java 21、Spring Boot 3、Spring Security、MyBatis-Plus、Flyway、MySQL 8 / H2、Vue 3、TypeScript、Vite、Ant Design Vue、Vitest。

---

## 1. 已确认业务决策

1. 模块范围为“现金 + 银行存款统一资金日记账”。
2. 流水来源包括：
   - 财务手工登记收入或支出；
   - 现有付款记录成功回写后自动生成支出流水。
3. 手工流水采用“草稿 → 财务上传附件 → 财务确认归档”的闭环。
4. 付款流水采用“付款成功 → 自动生成待归档流水 → 财务选择资金账户并上传付款凭证 → 财务确认归档”的闭环。
5. 每条流水至少有 1 个有效附件才允许归档；仅付款成功或仅上传附件均不算闭环。
6. 归档后普通财务不可修改、不可删除，只能红冲。
7. 超级管理员可以修正已归档记录，但必须执行“填写原因 → 撤销归档 → 修改/补附件 → 重新归档”，并保留修改前后快照。
8. 流水创建或重新打开后超过 24 小时仍未归档，进入预警中心。
9. 预警接收人是当前租户内具备 `payment:record:writeback` 或 `cashbook:journal:maintain` 权限的有效用户；不维护固定人员名单。
10. 账户属于租户级企业资金账户，流水可选关联项目和合同；不为每个项目复制资金账户。
11. 首期只提供期初余额、收支流水、实时余额、筛选、详情和导出，不做银行对账。
12. 不补历史付款流水；系统启用前的历史资金结果由账户期初日期和期初余额承接。

## 2. 全局约束

1. 只允许在本地 dev/test 环境实施和验收；不连接生产数据库，不发布生产。
2. 当前已应用 migration 最高版本为 `V135`；本主线新增 MySQL 与 H2 对应的 `V136`，不得修改任何既有 migration。
3. 金额统一使用 `DECIMAL(18,2)` / `BigDecimal`，禁止使用浮点数。
4. 所有账户、流水、附件、预警和审计查询必须先限定 `tenant_id`。
5. 项目和合同为可选维度；一旦填写，必须校验项目访问权以及合同与项目关系。
6. 付款来源流水必须以 `(tenant_id, source_type, source_id)` 唯一约束保证幂等；`source_type=PAY_RECORD` 时禁止人工修改方向、金额、来源和业务日期。
7. 余额只计算期初余额以及状态为 `ARCHIVED`、`REVERSED` 的已生效流水；`DRAFT`、`PENDING_ARCHIVE` 不计入余额。
8. 红冲必须生成方向相反、金额相同的新流水；原记录保留并标记 `REVERSED`，计算余额时原记录和红冲记录同时参与，净影响为零。
9. 停用账户不可新增或归档流水，但历史记录仍可查询。
10. 归档前不允许账户余额变成负数；首期不新增透支开关。
11. 银行账号接口默认返回掩码值，只有资金账户维护接口向具备维护权限的用户返回完整值。
12. 不新增第三方依赖，不新建附件系统、提醒系统或余额缓存表。
13. 自动提交仅允许本地 commit，`autoPush=false`；未经用户明确指令不得 push。

## 3. 非目标

本主线明确不包含：

1. 银行流水文件导入、银企直连和自动匹配。
2. 手工或自动银行对账。
3. 完整收款管理平台、应收核销或客户回款计划。
4. 会计凭证生成、总账过账或科目余额表改造。
5. 账户间调拨、现金盘点和多币种。
6. 历史付款流水回填。
7. 修改既有付款审批、合同余额、发票或会计分录业务口径。
8. 允许普通财务直接修改已归档数据。

## 4. 现有能力与复用边界

### 4.1 付款唯一写入入口

- 现有付款记录唯一权威入口是 `PayRecordService.writeback(PayRecord input)`。
- 该入口已经通过 `externalTxnNo` 幂等，并在同一事务中生成 `pay_record`、更新合同累计付款和付款申请状态。
- 日记账自动流水必须在该事务中调用 `CashJournalService.createPendingFromPayRecord(...)`；失败时整体回滚，不能出现“付款成功但日记账缺失”。

### 4.2 文件能力

- 复用 `/files/upload`、`/files`、`/files/{id}/url`、`/files/{id}`。
- 在 `BusinessObjectAuthorizer` 注册 `CASH_JOURNAL`，以日记账流水为附件业务对象。
- 已归档或已红冲记录只允许读取附件；删除附件前必须确认流水仍未归档。

### 4.3 预警能力

- 复用 `alert_log`、`AlertNotificationDispatcher` 和 `/alert` 页面。
- 新增预警域 `FINANCE`、规则类型 `CASH_JOURNAL_ARCHIVE_OVERDUE`、分类 `CASH_JOURNAL_CLOSURE`、来源类型 `CASH_JOURNAL`。
- 去重键固定为 `CASH_JOURNAL_ARCHIVE_OVERDUE:{tenantId}:{entryId}`。
- 预警默认严重度为 `MEDIUM`；流水归档后对应 OPEN 预警自动转 `ARCHIVED`。

### 4.4 审计能力

- 普通接口调用继续使用 `@AuditedOperation` 写入 `sys_operation_audit_log`。
- 由于现有操作审计表不保存请求体和前后值，超级管理员撤销归档及后续修正另写不可变 `cash_journal_change_log`。

## 6. 状态机与不变量

```text
手工：DRAFT --[字段完整 + >=1附件 + 归档]--> ARCHIVED
付款：PENDING_ARCHIVE --[选择账户 + >=1附件 + 归档]--> ARCHIVED
归档：ARCHIVED --[红冲]--> REVERSED + 新建 ARCHIVED 反向流水
归档：ARCHIVED --[SUPER_ADMIN + 原因]--> 原来源状态(DRAFT/PENDING_ARCHIVE)
重新打开：DRAFT/PENDING_ARCHIVE --[修正 + >=1附件 + 重新归档]--> ARCHIVED
```

强制不变量：

1. `ARCHIVED/REVERSED` 才影响余额。
2. 归档操作必须在后端实时统计 `sys_file` 中 `business_type=CASH_JOURNAL` 且 `business_id=entry.id` 的有效附件，数量小于 1 返回 `CASH_JOURNAL_ATTACHMENT_REQUIRED`。
3. `PAY_RECORD` 流水不允许修改 `direction/amount/business_date/source_type/source_id/project_id/contract_id`；只允许财务补账户、摘要、往来单位和附件。
4. 归档前锁定资金账户行并重新计算余额；归档支出后余额小于 0 返回 `FUND_ACCOUNT_INSUFFICIENT_BALANCE`。
5. 红冲操作与原记录状态更新、新反向流水插入、变更日志写入必须同一事务完成。
6. 重新打开时立即移出余额计算，`closure_due_at` 重置为当前时间 + 24 小时；重新归档前按未闭环处理。
7. 已红冲记录不可再次红冲、不可撤销归档。

## 7. API 契约

## 9. 阶段与任务拆分

## M1：数据库、领域模型与权限骨架

**任务性质：** 后端实现 / 数据一致性 / 权限。

**目标：** 建立三张新增表、两个核心实体、变更日志实体、菜单和最小权限，不接业务写入。

**实施步骤：**

- [ ] 在 MySQL 和 H2 的 `V136` 中创建三张表、索引和唯一约束。
- [ ] 在同一 migration 中注册“资金日记账”菜单，并创建 `cashbook:journal:query`、`cashbook:journal:maintain`、`cashbook:journal:export`、`cashbook:account:manage` 权限。
- [ ] 向财务维护角色授予 query/maintain/export 及现有 `file:upload/file:query/file:delete`；账户管理只授予管理员；撤销归档不注册普通权限，后端仅允许 `SUPER_ADMIN`。
- [ ] 新增实体、Mapper 和基础枚举常量；禁止用字符串散落重复定义状态。

## M3：付款成功自动生成待归档流水

**任务性质：** 后端实现 / 跨模块 / 财务一致性。

**目标：** 在付款权威回写事务中幂等生成 `PENDING_ARCHIVE` 支出流水。

**关键接口：**

```java
CashJournalEntry createPendingFromPayRecord(PayRecord record);
```

字段映射固定为：

## 12. 风险、回滚与上线边界

### 12.1 主要风险

1. **付款与日记账双写不一致：** 必须同事务，并用来源唯一约束兜底。
2. **并发支出突破余额：** 归档支出前锁账户行并重新计算余额。
3. **附件被归档后删除：** 文件业务授权必须检查流水状态。
4. **超级管理员修正破坏审计：** 强制原因、前后快照和重新归档，不提供直接 update 旁路。
5. **租户级财务预警被项目权限过滤：** `FINANCE` 域使用权限感知查询分支，不复用项目成员收件人。
6. **银行账号泄露：** 默认掩码展示，完整值只在账户维护权限下返回；日志和导出不得输出完整账号。
7. **上线产生历史预警风暴：** 不回填历史付款，以期初余额承接。

### 12.2 回滚策略

1. 应用回滚时先关闭“资金日记账”菜单和 `CASH_JOURNAL_ARCHIVE_OVERDUE` 规则，再回退应用版本。
2. 新增表和已产生财务流水不得自动删除；数据库回滚只允许停用入口，不执行破坏性 DROP。
3. 若付款接线存在缺陷，回退 `PayRecordService` 调用前必须先核对是否已生成孤立日记账流水；不得用脚本盲删。
4. 预警规则可通过配置停用；已生成预警按正式状态归档，不直接删除。

### 12.3 上线前人工确认

1. 每个启用资金账户的账户编码、期初日期和期初余额由财务复核。
2. 财务角色具备日记账维护及文件权限，业务只读角色仅具备 query。
3. 超级管理员受控修正路径完成一次真实演练。
4. 不存在未解释的跨租户、重复流水、负余额或无附件已归档记录。

## 13. 最终验收口径

第34主线只有同时满足以下条件才可判定通过：

1. 手工收入、手工支出和付款成功支出均能生成正确流水。
2. 付款重复回调只生成一条日记账流水，失败时付款与日记账整体回滚。
3. 无附件不能归档，未归档流水不影响余额。
4. 财务上传附件并归档后，余额正确变化，流水与附件可追溯。
5. 超过 24 小时未归档只向具备指定维护权限的租户内用户生成通知，并在预警中心可见。
6. 同一流水不产生重复有效预警，归档后预警自动关闭。
7. 普通财务不能修改归档记录；超级管理员只能按受控流程修正，前后快照完整。
8. 红冲后原记录、红冲记录和账户净余额正确。
9. 银行账号默认掩码，导出和日志不泄露完整账号。
10. 租户、项目、合同、附件、预警和权限测试无越权。
11. 定向后端测试、前端测试、类型检查、构建、`git diff --check` 和真实页面验收通过。
12. 正式质量报告、iteration 报告、backlog 状态和剩余风险完成同步。

任一金额一致性、租户隔离、附件越权、重复记账、归档篡改或审计快照缺失问题均为上线阻塞。
