# 第32条主线-既有未解决问题统一治理任务计划书

**Goal:** 把 `cgc-pms` 截至 `2026-07-09` 已有报告中确认通过、非阻塞、待补验收、待专项归因和长期未启动的平台化事项，统一归并到一条新的治理主线中，先完成状态裁决、风险分层和执行顺序冻结，再按“短期收口 / 中期专项 / 长期储备”三层推进，避免 Ready 队列、质量报告、真实角色验收和产品路线彼此脱节。

**Architecture:** 严格复用现有 `docs/backlog`、`docs/quality`、`docs/plans` 和既有主线结论，不新建额外台账系统，不重写历史报告，不把一次性运行日志、截图名或临时路径写入长期资产；执行策略采用“先裁决文档状态，再归因全量红灯，再补真实角色验收，最后分拆中长期能力项”的最小治理路径，不把本主线扩展成一次性全仓重测、完整平台化重构或生产集成实施。

## 1. 主线结论与边界

### 1.1 为什么需要第32主线

当前仓库存在以下已确认但尚未统一收口的问题：

1. `docs/backlog/ready-issues.md` 中 `ISSUE-008-008`、`ISSUE-008-009` 仍为 `Ready`，但对应质量报告和 `done-issues.md` 已写明“通过 / 非阻塞 / 已完成”，文档状态不一致。
2. `ISSUE-008-006/007/008/009` 的正式报告都提到 `.\mvnw.cmd test` 全量红灯，失败集中在 `dashboard`、`invoice`、`migration`、`purchase`、`revenue`、`workflow` 等既有测试域，当前缺少统一归因和专项治理入口。
3. 第30主线正式验收已明确：浏览器验收使用 `dev-login` 超管账号，只能证明主链路可达，不能替代真实采购、生产、财务角色抽样复核。
4. Mockito 动态 agent 未来兼容性告警、Spring Boot generated password 提示已多次出现，但仍未归并到独立构建兼容性治理。
5. 报表中心、规则治理中心、通知平台、WBS/甘特图/进度计划、供应商评分/采购增强虽已完成局部最小回归，但仍未达到完整平台化；`P3` 现场日报/移动端、财务集成/月结锁账、技术域/总工程师驾驶舱、BIM/IFC、AI 辅助也仍在储备态。
6. UI 一致性、可访问性、登录页品牌、`440px` 移动端布局、覆盖率与 E2E CI 仍缺统一复验和治理入口。

### 1.2 本主线目标

第32主线不是直接做业务实现，而是完成以下四件事：

1. 统一裁决“哪些事项已完成但状态未同步”。
2. 统一拆解“哪些红灯属于既有测试债，需要专项治理”。
3. 统一补齐“哪些验收结论因为账号口径或运行口径不完整，仍需真实角色复核”。
4. 统一冻结“哪些能力项进入近期治理，哪些继续作为中长期储备，不混入当前 Ready 执行队列”。

### 1.3 非目标

本主线明确不包含以下动作：

1. 不在本计划书阶段直接修改业务代码、脚本或运行环境。
2. 不把 `P3` 现场日报/移动端、财务集成/月结锁账、技术域、BIM、AI 直接推进为当前迭代实现。
3. 不把后端全量测试红灯一次性扩展成“所有失败一锅端”大改。
4. 不把 UI/可访问性治理直接扩展为全站重设计。
5. 不重写既有质量报告，只基于现有报告做裁决、归并和后续派工冻结。

## 3. 总体治理策略

### 3.1 分层治理原则

本主线统一按三层处理：

1. `L1 状态收口层`：只解决 backlog / done / quality 之间的状态不一致。
2. `L2 质量专项层`：只解决后端全量测试红灯、真实角色验收缺口、构建兼容性技术债这三类会影响后续裁决可信度的问题。
3. `L3 路线储备层`：只冻结产品能力和 UI/质量遗留的优先级、前置条件和进入标准，不直接假装已完成。

### 3.2 执行顺序

1. 先做文档状态裁决。
2. 再做后端全量测试红灯归因。
3. 再做真实角色抽样复核。
4. 再做构建兼容性技术债治理。
5. 再做产品能力项归并和阶段冻结。
6. 最后做 UI / 可访问性 / 覆盖率与 E2E CI 的统一复验计划。

## 4. 阶段拆分

## M1：backlog 状态同步裁决

### 目标

把“已完成但仍挂在 Ready 队列”的事项先裁决干净，恢复 backlog 与 quality 的一致性。

### 覆盖问题

1. `ISSUE-008-008`：`ready-issues.md` 仍为 `Ready`，但质量报告、`done-issues.md` 已写通过。
2. `ISSUE-008-009`：`ready-issues.md` 仍为 `Ready`，但质量报告、`done-issues.md` 已写通过 / 非阻塞。

### 预期产出

1. 明确这两个 Issue 是否正式迁入 `done`。
2. 若仍保留 `Ready`，必须给出明确反证依据；否则按已通过结论同步状态。
3. 同步记录“通过但后端全量仍有既有无关红灯”不等于本 Issue 未完成。

### 验收标准

1. backlog 状态与对应 `docs/quality` 结论一致。
2. 不再出现“Ready 队列仍挂着已完成事项”的文档假阻塞。
3. 裁决说明明确区分“本 Issue 通过”和“全量测试仍有历史红灯”两件事。

### 风险点

1. 若不先裁决，后续 AutoPilot 可能重复选到已完成任务。
2. 若把全量红灯误判为本 Issue 未完成，会污染 backlog 可信度。

## M2：后端全量测试红灯专项归因

### 目标

把当前全量测试红灯从“散落在多份报告里的备注”升级为一个明确的治理专项。

### 覆盖问题

以下报告均提到 `.\mvnw.cmd test` 全量失败，需要统一归并：

1. `ISSUE-008-006` 规则治理中心最小回归
2. `ISSUE-008-007` 通知平台最小回归
3. `ISSUE-008-008` WBS/进度计划与甘特图最小回归
4. `ISSUE-008-009` 供应商评分与采购增强最小回归

### 归因范围

按当前证据，失败集中在：

1. `dashboard`
2. `invoice`
3. `migration`
4. `purchase`
5. `revenue`
6. `workflow`

### 预期产出

1. 一条独立测试专项主线或成组 Ready Issue。
2. 每个失败域的失败分类：测试夹具债、断言漂移、迁移兼容性、业务逻辑真实缺陷、环境前置或工具调用问题。
3. 一份统一“先修哪些、后修哪些”的优先级顺序。

### 验收标准

1. 不再只在各个质量报告末尾重复写“全量测试仍有红灯”。
2. 至少能把失败域按模块分组，而不是维持笼统备注。
3. 明确哪些红灯会阻塞后续上线裁决，哪些只是既有测试债。

### 风险点

1. 不做统一归因，会持续导致“定向通过、全量红灯”并存但无人负责。
2. 若不先分类，容易把命令调用、测试夹具和真实业务问题混为一类。

## M3：真实角色抽样复核

### 目标

补齐第30主线和相关浏览器验收使用 `dev-login` 超管账号带来的角色口径缺口。

### 覆盖问题

1. 采购角色抽样复核。
2. 生产角色抽样复核。
3. 财务角色抽样复核。

### 复核重点

1. 关键页面能否进入。
2. 驾驶舱是否展示正确角色语义。
3. 默认筛选、默认域、按钮可见性是否符合角色边界。
4. 不得用超管可见结果替代真实角色体验结论。

### 验收标准

1. 至少每类角色留下一次正式抽样结论。
2. 明确通过 / 不通过 / 非阻塞观察项。
3. 若缺真实账号或环境前置，应明确归类为前置阻塞，而不是直接宣告功能通过。

### 风险点

1. 若继续只用 `dev-login`，后续很多“通过”结论只证明系统可达，不证明角色体验正确。
2. 采购、生产、财务三个角色若长期不补验，权限与默认筛选问题会持续潜伏。

## M4：构建兼容性技术债治理

### 目标

把当前已知但尚未收口的构建兼容性提示从“报告尾注”提升为独立治理项。

### 覆盖问题

1. Mockito 动态 agent 未来兼容性告警。
2. Spring Boot generated password 开发提示。

### 预期产出

1. 一条构建兼容性 / 本地运行提示治理任务。
2. 明确哪些提示只是开发噪音，哪些可能在未来 JDK / 安全策略升级时转为真实阻塞。
3. 形成最小处理顺序：先 agent 兼容，再收敛默认密码提示。

### 验收标准

1. 两类提示都被纳入正式任务，而不是继续作为“后续再说”的悬空项。
2. 明确是否阻塞当前发布；当前依据应维持为非阻塞。
3. 后续治理不误扩展为完整安全重构或登录体系重做。

### 风险点

1. 若不单独治理，未来升级 JDK 或测试插件时可能从警告变成门禁失败。
2. 默认 generated password 提示若长期不收敛，会持续污染开发和验收日志判断。

## M5：产品能力未完成项分层冻结

### 目标

把“已做过最小回归”和“尚未真正平台化”区分清楚，把“仍未进入当前执行”的长期能力明确冻结到储备层。

### A 类：已做局部最小回归，但未完整平台化

1. 报表中心
2. 规则治理中心
3. 通知平台
4. WBS / 进度计划 / 甘特图
5. 供应商评分 / 采购增强

### B 类：仍未进入当前执行

1. `P3` 现场日报 / 移动端
2. 财务集成 / 月结锁账
3. 技术域 / 总工程师驾驶舱
4. BIM / IFC
5. AI 辅助

### 预期产出

1. A 类事项标注为“已完成最小回归，但未平台化完成”。
2. B 类事项标注为“长期储备 / 不进入当前 Ready 执行”。
3. 每类事项明确前置条件，防止后续随意穿插进短周期收口任务。

### 验收标准

1. 不再把“已做最小回归”误写成“已完整建设”。
2. 不再把长期储备项混进当前可执行 Ready 队列。
3. 每个能力项都能落到“短期 / 中期 / 长期”之一。

### 风险点

1. 若不分层，主线会持续被“平台化幻觉”拖大。
2. 若不冻结长期储备项，Ready 队列会被不具备前置条件的任务污染。

## M6：UI 与质量遗留统一复验计划

### 目标

把零散的 UI / 可访问性 / 移动端 / 覆盖率 / E2E CI 遗留统一成一条质量增强路线，而不是散点补丁。

### 覆盖问题

1. UI 一致性
2. 可访问性
3. 登录页品牌
4. `440px` 移动端布局
5. 覆盖率
6. E2E CI

### 预期产出

1. 一条 UI 与质量增强主线，先做复验基线。
2. 先确定哪些属于设计一致性问题，哪些属于真实功能风险。
3. 覆盖率和 E2E CI 先补基线与门禁策略，再决定是否扩大自动化范围。

### 验收标准

1. 六类问题都进入正式治理清单。
2. 明确哪些先复验、哪些先整改、哪些需要独立 CI 任务。
3. 不把视觉一致性、可访问性、测试门禁混成一类无边界大任务。

### 风险点

1. 若没有统一复验基线，UI 和质量问题会长期以“观察项”形式漂浮。
2. 覆盖率与 E2E CI 若直接硬上门禁，容易在基线未清时制造新的假红灯。

## 5. 子任务类型建议

建议主线程后续按四类派工，不要混派：

1. `审计/归档型`：M1、M5 计划与状态裁决。
2. `测试/验收型`：M3 真实角色抽样复核、M6 UI 与质量复验。
3. `实现型`：M2 后端全量测试红灯治理、M4 构建兼容性治理。
4. `运维型`：仅在角色复核、浏览器复验或 CI 基线核查需要真实运行态时介入。

## 6. 最终验收口径

第32主线通过至少同时满足以下条件：

1. `ISSUE-008-008`、`ISSUE-008-009` 的 backlog 状态完成正式同步。
2. 后端全量测试红灯被拆成独立专项，不再只是分散备注。
3. 真实采购、生产、财务角色完成至少一轮正式抽样复核，或明确记录前置阻塞。
4. Mockito 动态 agent 与 generated password 提示进入独立治理项。
5. 已做最小回归的平台项与长期储备项完成分层冻结。
6. UI 一致性、可访问性、登录页品牌、`440px` 布局、覆盖率、E2E CI 形成统一复验与治理入口。

若以上任一项未完成，则第32主线不得宣告完全收口。

## 7. 一句话结论

第32主线的价值，不是再开一批松散待办，而是先把“状态不同步、全量红灯无人归因、真实角色未补验、技术债长期挂起、平台项和长期项混排、UI/质量遗留漂浮”这六类既有未解决问题统一拉回到一条可裁决、可分阶段推进的治理主线里。
