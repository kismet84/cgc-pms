## 一、专项目标

建立项目级强隔离的最小可验证边界，先裁决“项目访问权限如何判定”，再决定是否进入实现。

本专项不解决全量 RBAC，不处理权限字典历史码，不建设可编辑权限平台，不建设全站入口注册体系。

## 二、纳入范围

1. 只读盘点项目级访问链路：
   - `ProjectAccessChecker`
   - 项目详情 / 项目总览
   - workflow 详情中的 `projectId` 校验
   - 合同、采购申请、分包计量、付款申请等带 `projectId` 的核心业务详情接口
2. 明确当前项目隔离语义：
   - `ADMIN/SUPER_ADMIN`
   - `ALL`
   - `SELF`
   - `DEPT`
   - `CUSTOM`
   - 是否存在项目成员 / 项目角色型授权关系
3. 设计最小真实验收矩阵：
   - admin 正向
   - 同租户有目标项目权限账号正向
   - 同租户无目标项目权限账号负向
   - 非参与人 / 无项目权限直连详情负向
4. 裁决是否需要第二租户样本。

## 三、明确不纳入范围

1. 不做复杂 `data_scope` 全矩阵。
2. 不做全业务模块一次性强隔离改造。
3. 不做第二租户完整生产矩阵，除非安全/测试裁决为硬门禁。
4. 不改权限字典表，不新增权限中心表。
5. 不改可编辑权限平台。
6. 不改全站入口注册体系。
7. 不接入付款申请以外的更多业务类型。

## 四、串行小包

### 小包 1：项目隔离现状盘点

执行角色：

- 后端设计师 / 后端架构助手
- 数据库与迁移负责人
- 安全审计员
- 测试验收员

目标：

1. 梳理当前所有项目访问校验入口。
2. 标注哪些接口仅做 tenant 校验，哪些接口做 project 校验。
3. 梳理现有 demo 数据是否能支撑强隔离验收。

验收：

1. 形成接口清单。
2. 形成数据样本缺口清单。
3. 不改代码、不加 seed。

### 小包 2：隔离语义裁决

执行角色：

- 产品/需求负责人
- 安全审计员
- 后端设计师 / 后端架构助手
- 数据库与迁移负责人
- 测试验收员

目标：

1. 裁决项目访问应以什么为准：
   - 创建人
   - 角色 `data_scope`
   - 项目成员
   - 项目角色
   - 组合策略
2. 裁决 `DEPT/CUSTOM` 是继续 fail-close，还是进入实现。
3. 裁决是否补第二租户。

验收：

1. 输出语义裁决表。
2. 未裁决前不允许实现。

### 小包 3：最小实现包定义

执行角色：

- 主负责人裁决

目标：

1. 若允许实现，定义最小实现包。
2. 明确触达接口、seed、测试矩阵、运维路径。

验收：

1. 实现范围足够小。
2. 回滚与验收路径清楚。

## 五、通过 / 不通过标准

### 通过条件

1. 项目隔离现状清单完整。
2. 隔离语义已裁决。
3. 样本缺口已明确。
4. 是否进入实现已有清晰决策。

### 不通过条件

1. 未裁决隔离语义就直接实现。
2. 把复杂 `data_scope`、第二租户、全业务模块一次性并入。
3. 只补 seed 却不确认后端隔离语义。

## 六、当前建议

先做只读盘点和语义裁决。项目级强隔离不是 seed 问题，不能用补账号样本替代授权模型裁决。
