**Goal:** 在既有四态分类和知识图谱历史隔离基础上，对全部活动存量建立不遗漏、可追溯的处置计划：覆盖 10 个“仍适用”问题域、8 个“需要复验”风险域，以及 `blocked-issues.md` 中 3 条旧阻塞记录；M0 已完成状态归一化，后续按风险逐项复验、修复、验收和回写，直到每项都有“已解决、已否定、已阻塞或保留 Candidate/Frozen”的唯一当前状态。

## 4. 状态机与处置规则

每个条目只能处于以下一个当前状态：

1. `Candidate`：方向成立但尚未通过产品决策门。
2. `ReadyToSplit`：价值、边界、依赖和证据完整，可拆最小 Ready。
3. `Ready`：满足 Ready 字段和验证入口，可实施。
4. `InProgress`：已通过授权门并开始执行。
5. `VerifiedResolved`：当前同一 SHA 的实现与必要验证通过。
6. `VerifiedNotApplicable`：当前事实证明不适用，记录证据后关闭。
7. `Blocked`：缺少授权、环境、外部系统或关键数据，已写明解除条件。
8. `Frozen`：产品对象或业务决策前置不足，不实施、不伪装成缺陷。

“需要复验”不能直接写成“待修复”；只有复验失败并确认真实质量/安全问题后，才生成修复 Ready。

## 10. 总体验收标准

1. 10 个 A 域、8 个 V 域和 3 条旧 Blocked 记录均有唯一当前状态、证据、下一动作和载体，遗漏数为 0。
2. 3 条状态冲突完成归一化；历史段可保留，但活动段不得同时声明“阻塞”和“已全绿”。
3. 每个 V 域要么以当前证据判 `VerifiedResolved/VerifiedNotApplicable`，要么形成最小 Ready/Blocked；不得长期停留在报告备注。
4. 每个 A 域要么通过产品决策门进入 Candidate/Ready，要么明确 Frozen/否定依据；不得因竞品存在而自动实施。
5. 适用测试、构建、静态检查、真实角色、运行态、依赖/安全审计和 `git diff --check` 按 Issue 风险通过。
6. 涉及上线时，必须以同一待合并 SHA 的远端 required checks、分支保护和正式报告单独裁决。
7. 私有封存 Artifact 保持 0；知识图谱默认搜索继续隔离历史，引用未解析数保持 0。
