# 第22条主线：采购库存模块生产级增强方案

## 计划状态

正式版。

本计划书用于第22条主线“采购库存模块生产级增强”的任务分派、执行督办、验收裁决与质量归档。

**Goal:** 在不扩成 WMS 全量仓储系统、不重构采购审批流、不新增依赖、不改已应用 Flyway 的前提下，把采购库存模块收敛到桌面端可生产验收状态，重点完成契约复核、后端最小加固候选、必要的前端业务契约最小修正、运行态刷新、真实桌面验收与质量归档；忽略移动端相关调整与测试。

**Architecture:** 最小可行方案优先复用现有 `frontend-admin/src/pages/inventory/purchase-request.vue`、`frontend-admin/src/pages/purchase/order.vue`、`frontend-admin/src/pages/inventory/stock.vue`、现有 `inventory.ts`/`purchase.ts` API、既有列表页与详情能力，以及后端既有采购/库存控制器与服务。只处理采购库存生产级业务可用性缺口，前端仅允许做契约、跳转、只读查看态等最小修正；UI 相关任务不属于本主线；本主线不安排、不验收 UI 视觉对齐；不扩展到移动端，不引入新抽象。

---

## 一、已冻结边界

1. 本主线只覆盖采购库存模块生产级增强，不做 WMS 全量仓储系统。
2. 不新增条码、批次、库位、盘点、调拨能力。
3. 不重构采购审批流，不改工作流模板口径。
4. 不新增依赖，不替换现有表格/组件体系。
5. 不改已应用 Flyway；若复核证明确需数据库变更，只能另起新 migration 方案，由主线程单独裁决是否立项。
6. 忽略移动端相关调整与测试；本计划书、实现、验收均仅保留桌面端浏览器口径。
7. UI 相关任务不属于本主线；本主线不安排、不验收 UI 视觉对齐。

## 二、当前事实与轻量上下文

### 2.1 入口与模块现状

1. 路由导航已存在 `采购库存` 分组，包含：
   - `/inventory/purchase-request`
   - `/purchase/order`
   - `/inventory/stock`
2. 前端相关 API 已存在：
   - `frontend-admin/src/api/modules/inventory.ts`
   - `frontend-admin/src/api/modules/purchase.ts`
3. 采购申请页已采用与预警中心同类的列表页构件：
   - `useColumnSettings`
   - `ColumnSettingsButton`
   - KPI + 筛选 + 表格工具条布局
4. 后端采购订单、采购申请、库存链路已存在既有实体、控制器、服务与测试，不应先入为主地重造领域模型。

### 2.2 范围边界补充

1. UI 相关任务不属于本主线；本主线不安排、不验收 UI 视觉对齐。
2. 本主线允许的前端工作仅限业务可用性相关的契约、跳转、只读查看态、列表读取与详情链路修正。
3. 若复核发现问题仅属于 UI 范围，不作为本主线阻塞项。

### 2.3 本轮不纳入事实

1. 不写移动端适配，不写移动端卡片 fallback。
2. 不写移动端验收，不写移动端截图、视口、交互要求。
3. 不把“采购库存生产化增强”扩写为条码、盘点、调拨、库位、批次治理。

## 三、主负责人结论

第22条主线可以立项，但必须按“先复核、再最小实现、后验收归档”的顺序串行推进。

本主线的唯一目标是：**在既有采购申请、采购订单、库存台账页面与后端链路基础上，补齐生产级业务可用性、契约一致性与验收闭环**。

不允许出现以下失控扩包：

1. 借生产化名义重做采购审批流程。
2. 借库存增强名义演变成 WMS 全量建设。
3. 借 UI 优化名义重造新的页面框架或新依赖。
4. 把 UI 相关任务包装成本主线必交付内容。
5. 借测试名义追加移动端适配与移动端验收。

## 四、纳入范围

### 4.1 本阶段纳入范围

1. 采购库存契约与现状复核。
2. 库存后端链路最小加固候选。
3. 采购库存前端业务契约最小修正（仅限业务可用性）。
4. 实现后所需的运行态刷新。
5. 桌面真实验收。
6. 质量归档。

### 4.2 本阶段不作为交付项

1. 移动端相关调整与测试。
2. 条码、批次、库位、盘点、调拨。
3. 采购审批流重构、审批模板重配、审批中心联动扩展。
4. 新依赖引入、新状态库、新表格库、新 UI 体系。
5. 已应用 Flyway 修改。

## 五、建议执行顺序

1. `M22-1` 采购库存契约与现状复核（只读）。
2. `M22-2` 库存后端链路最小加固（仅在复核确认需要时执行）。
3. `M22-3` 采购库存前端业务契约最小修正。
4. `M22-4` 运行态刷新（仅实现后需要验收时执行，运维型）。
5. `M22-5` 桌面真实验收。
6. `M22-6` 质量归档。

## 六、子智能体模型分配表

| 任务/角色 | 分类 | model | thinking | reason |
| --- | --- | --- | --- | --- |
| M22-1 采购库存契约与现状复核 | 审计/归档型任务 | `gpt-5.4` | `medium` | 需要对照采购申请、采购订单、库存台账的现状与业务可用性边界，形成正式事实表，但仍以只读复核为主。 |
| M22-2 库存后端链路最小加固 | 实现型任务 | `gpt-5.5` | `high` | 涉及后端真实契约、库存数据正确性、权限/租户边界与可能的生产风险，不适合低档位。 |
| M22-3 采购库存前端业务契约最小修正 | 实现型任务 | `gpt-5.4` | `medium` | 仅处理业务可用性所需的前端契约、跳转、只读查看态与最小交互修正，边界明确。 |
| M22-4 运行态刷新 | 运维型任务 | `gpt-5.4` | `low` | 步骤固定，主要是环境刷新、端口与日志核对、证据回传。 |
| M22-5 桌面真实验收 | 验收型任务 | `gpt-5.5` | `medium` | 需要综合命令、接口、桌面浏览器与控制台证据，形成正式通过/不通过裁决。 |
| M22-6 质量归档 | 审计/归档型任务 | `gpt-5.4` | `medium` | 需要整理正式裁决、证据、阻塞项与分档复盘，但判断空间低于实现和验收。 |

### 6.1 分类说明

1. 实现型任务：`M22-2`、`M22-3`
2. 验收型任务：`M22-5`
3. 运维型任务：`M22-4`
4. 审计/归档型任务：`M22-1`、`M22-6`

### 6.2 分档经验记录

1. 后端库存链路只要涉及数据正确性、租户隔离、库存累计口径，就应升到 `gpt-5.5 + high`。
2. 前端若只做业务契约、跳转、只读查看态等最小修正，`gpt-5.4 + medium` 足够。
3. 运行态刷新不应与后端高风险改动统一档位。
4. 验收与归档不能混写成机械回传；验收保留 `gpt-5.5 + medium`，归档降到 `gpt-5.4 + medium` 更合理。

## 七、阶段任务与派工单

### M22-1 采购库存契约与现状复核（只读）

- 任务名称=M22-1 采购库存契约与现状复核
- 角色边界=你是被主线程明确派工的子智能体，不是主线程；在本派工范围内可以执行授权动作
- 目标=形成采购申请、采购订单、库存台账现状与业务可用性边界的对应表，明确必须改、可复用、暂不做。
- 范围=只读核对 `purchase-request`、`purchase/order`、`inventory/stock` 相关前后端文件与业务链路；不做代码修改。
- 禁止事项=不得改代码、配置、文档、测试、运行环境、Git 状态；不得扩扫私有目录；不得把移动端纳入复核项。
- model=`gpt-5.4`
- thinking=`medium`
- reason=需要在多个现有页面与既有业务链路之间做结构化复核，但不涉及实现落地。
- 验收输出=契约复核表、页面差异清单、必须改项、可延后项、是否触发 M22-2。

执行要点：

1. 核对采购申请、采购订单、库存台账的分页参数、筛选项、统计口径与详情入口。
2. 核对是否存在影响业务可用性的列表读取、跳转、只读查看态或等效详情能力缺口。
3. 核对哪些缺口属于前端生产化，哪些缺口需要后端补丁。
4. 明确本轮不纳入移动端、不纳入审批流重构。

### M22-2 库存后端链路最小加固（仅在需要时执行）

- 任务名称=M22-2 库存后端链路最小加固
- 角色边界=你是被主线程明确派工的子智能体，不是主线程；在本派工范围内可以执行授权动作
- 目标=仅在 M22-1 证实后端存在生产阻塞时，做库存相关后端链路最小加固。
- 范围=库存台账、采购申请、采购订单相关后端控制器/服务/测试；不扩到条码、批次、库位、盘点、调拨。
- 禁止事项=不得无证据启用；不得改已应用 Flyway；不得重构采购审批流；不得新增依赖。
- model=`gpt-5.5`
- thinking=`high`
- reason=库存后端问题一旦进入实现，往往同时涉及契约、数据正确性、权限和租户隔离，需要高档位。
- 验收输出=是否启用、改动文件、后端测试结果、阻塞项、回滚风险。

启用条件：

1. 后端分页/筛选契约与前端生产化目标不一致。
2. 库存 KPI、台账汇总、详情链路存在明确口径错误。
3. 库存相关接口存在租户/权限/项目范围风险。

### M22-3 采购库存前端业务契约最小修正

- 任务名称=M22-3 采购库存前端业务契约最小修正

- model=`gpt-5.4`
- thinking=`medium`

执行要点：

2. 仅在业务可用性必需时调整现有页面交互，不纳入 UI 相关任务。
3. 优先复用现有列表页布局、详情抽屉/弹窗和既有组件能力。
4. 明确排除移动端适配、移动端卡片和移动端验收逻辑。

### M22-5 桌面真实验收

## 八、执行原则

1. 先复核再实现，避免把前端问题误判成后端问题。
2. 优先复用现有页面与组件模式，不做额外抽象。
3. 只有 M22-1 证明后端缺口明确存在时，才允许启用 M22-2。
4. 以桌面端生产验收为唯一收口口径；移动端相关调整与测试明确排除。
5. UI 相关任务不属于本主线；本主线只处理业务可用性所需的最小前端修正。

## 九、验收标准

### 9.1 保留的验收项

1. 桌面端浏览器验收。
2. 后端相关测试。
3. 前端 `type-check`。
4. 前端 `build`。

### 9.2 桌面端通过条件

1. 桌面端核心筛选、分页、详情、状态展示、只读查看态和列表操作可用。
2. 必要的前后端契约、跳转与详情链路一致，且不存在影响业务闭环的明显断点。
3. 后端相关测试通过，或能明确证明失败项与本主线无关。
4. 前端 `type-check` 通过。
5. 前端 `build` 通过。
6. 浏览器控制台无本主线引入红错。
7. 相关接口无本主线引入失败请求。

### 9.3 不通过条件

1. 桌面端核心筛选、分页、详情、只读查看态或列表操作存在阻断，导致业务流程不可用。
2. 必要的前后端契约、跳转或详情链路断裂，导致桌面端无法完成业务闭环。
3. 后端相关测试失败且无法证明与本主线无关。
4. 前端 `type-check` 或 `build` 失败。
5. 浏览器控制台或接口请求出现本主线引入错误。

### 9.4 明确排除项

1. 不做移动端浏览器验收。
2. 不做移动端截图、视口、交互检查。
3. 不把移动端适配缺失记为本主线阻塞项。

## 十、风险点

1. 范围失控风险：
   - 容易从“采购库存生产化”扩成“仓储系统重构”。
2. 契约误判风险：
   - 若未先做 M22-1，可能把现有前端问题错误升级成后端改造。
3. 数据口径风险：
   - 库存 KPI、台账汇总、详情数值若口径不一致，会直接影响生产验收可信度。
4. 范围回流风险：
   - 若把 UI 相关任务重新带回本主线，会挤压后端契约与业务闭环修正资源。
5. 验收口径失真风险：
   - 若把移动端要求混入本主线，会打破已冻结边界。

## 十一、重新分档检查点

1. `M22-1` 完成后：
   - 若确认仅是前端业务契约、跳转、只读查看态缺口，维持 `M22-3` 的 `gpt-5.4 + medium`。
   - 若确认后端存在真实库存契约或数据口径问题，启用 `M22-2` 的 `gpt-5.5 + high`。
2. `M22-3` 完成后：
   - 从实现切换到验收前，显式升到 `M22-5` 的 `gpt-5.5 + medium`。
3. `M22-5` 首次回报“不通过”后：
   - 必须判断原因属于模型档位不足、推理强度不足、任务说明不清、验收标准缺失或执行边界冲突，再决定补派。
4. 若实现中途被证明只是环境陈旧：
   - 主动降到 `M22-4` 的 `gpt-5.4 + low`，不让高档位子智能体反复做机械刷新。

## 十二、上线裁决标准

### 12.1 通过 / 非阻塞 / 可上线

同时满足以下条件时，主线程可给出“通过 / 非阻塞 / 可上线”：

1. `M22-1` 已完成并明确本轮边界。
2. 必要时 `M22-2` 已完成且相关后端测试通过。
3. `M22-3` 已完成，采购申请、采购订单、库存台账桌面端业务链路满足本计划通过条件。
4. `M22-4` 已确认运行态基于最新产物。
5. `M22-5` 已完成桌面端真实验收，控制台与接口无本主线引入错误。
6. 前端 `type-check`、`build` 通过。
7. `M22-6` 已完成正式归档。

### 12.2 不通过 / 阻塞 / 不可上线

命中以下任一条即可判定“不通过 / 阻塞 / 不可上线”：

1. 后端库存/采购关键契约错误未修复。
2. 桌面端核心页面仍无法按生产流程完成筛选、查看、定位与只读业务操作。
3. 前端 `type-check` 或 `build` 失败。
4. 后端相关测试失败且与本主线强相关。
5. 浏览器控制台或接口存在本主线引入红错。

## 十三、暂不做

1. 暂不做移动端相关调整与测试。
2. 暂不做条码、批次、库位、盘点、调拨。
3. 暂不做采购审批流重构。
4. 暂不做新依赖引入或页面框架替换。
5. 暂不做已应用 Flyway 修改。
6. 暂不做 WMS 全量仓储系统建设。
