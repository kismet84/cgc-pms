# 第47条主线：Codex 主线执行效率与沟通降噪优化任务计划书

**Goal:** 在不降低授权门、验证门禁、Git 安全边界和正式裁决质量的前提下，系统性减少 cgc-pms 主线任务中的工具召回不足、无效工具重试、重复初始化、过密过程播报、上下文膨胀和中断后返工，使普通任务、连续迭代、验收及发布收口都具备可预测、可恢复、可度量的执行体验。
**Architecture:** 采用“轻量统一执行协议 + 确定性工具路由 + 分层验证 + 事件驱动沟通 + 可恢复任务胶囊”的最小可行架构；`AGENTS.override.md` 只保留最高级边界和索引，通用执行策略集中到单一规范文档，专项细节留在对应 Skill。不得新增重型编排系统，不得用降噪掩盖真实错误，也不得扩大自动提交、推送、合并或环境变更权限。

> 计划状态：In Progress / M3 Trial Window 2026-07-15–2026-07-21
> 计划类型：工程治理、执行链路优化
> 适用范围：Codex 桌面主线程、普通交互任务、主线/Backlog 工作流、AutoPilot 连续迭代及其验收收口
> 非实施声明：本计划书的创建不代表已授权修改规则、Skill、脚本、运行环境或 Git 状态。

## 一、背景与问题定义

近期会话复盘显示，主要成本并非集中在业务实现本身，而是集中在执行控制面：工具能力探测反复发生、检索路由不稳定、同类失败多次重试、浏览器调用参数猜测、验证重复、状态播报过密，以及长线程中断后恢复成本过高。

最近一周人工抽样数据如下。该组数据仅用于说明问题规模，当前缺少完整采集口径与可复现来源，**不得直接作为 M3 验收基线**；M0 必须按 2.3 的指标口径重新采集并冻结正式基线，未完成前不得进入 M1。

| 指标 | 基线 |
|---|---:|
| cgc-pms 相关会话 | 10 个 |
| 累计轮次 | 125 轮 |
| 累计会话时长 | 约 1830 分钟，包含等待时间 |
| commentary 更新 | 1332 次 |
| 可识别工具调用 | 1121 次 |
| 失败工具调用 | 108 次，约 9.6% |
| `node_repl.js` 相关失败 | 81 次 |
| `codebase-memory-mcp.search_code` | 16 次 |
| `search_graph` | 9 次 |

上述数据用于建立治理基线，不直接等同于业务质量问题。失败必须先分类：工具配置类、命令调用类、环境前置类、召回不足类、真实质量/安全类。未经分类，不得把工具红字或一次性失败直接定性为代码缺陷。

## 二、目标与量化指标

### 2.1 核心目标

1. 让同类任务使用稳定、可解释的工具路由，减少试错调用。
2. 让主线程只在状态发生实质变化时播报，降低沟通噪声。
3. 让长任务可从结构化状态恢复，避免中断后重复实现和重复验证。
4. 让测试、CI、浏览器和 Git 收口按风险分层，减少无意义全量重复。
5. 让规则有唯一权威来源，避免 `AGENTS`、Skill、脚本和会话提示词之间漂移。

### 2.2 一周试运行目标

| 指标 | 目标值 |
|---|---:|
| 工具调用失败率 | `<= 3%` |
| 浏览器工具调用错误 | 每个有效浏览器样本 `<= 1 次`，且一周总数 `<= 10 次` |
| 普通任务过程播报 | 通常 `<= 5 次` |
| 同一查询路径无变化重试 | `0 次` |
| 直接向 `master` 推送 | `0 次` |
| 中断后重复实施已完成工作 | `0 次` |
| 控制面耗时占总执行时长 | `<= 20%` |
| 每轮后续项净变化 | `<= 0` 为默认目标 |

指标不作为规避验证或隐藏错误的理由。高风险任务允许增加证据和播报，但必须说明增加原因。

### 2.3 指标口径与证据要求

试运行窗口从首个合格样本开始，连续 7 个自然日，时区固定为 `Asia/Shanghai`。只统计目标、授权边界、开始/结束状态和裁决均可识别的 cgc-pms 任务；同一任务跨线程恢复仍按一个样本计数，用户等待和外部 CI 排队时间不计入控制面耗时。

| 指标 | 计算口径 | 正式证据要求 |
|---|---|---|
| 工具调用失败率 | `(tool_config + tool_invocation) / 全部工具调用`；`retrieval_gap`、环境未就绪和真实质量失败分别统计，不计入工具调用失败 | M0 记录采集窗口、样本数量、分类规则、汇总命令或人工复核步骤 |
| 浏览器工具调用错误 | 每个有效浏览器样本中的 `tool_invocation` 次数，同时报告周总数；页面业务失败不计为工具调用错误 | 每个样本保留入口、关键动作、调用总数、错误分类和最终裁决摘要 |
| 普通任务过程播报 | 从开始到最终答复的 commentary 数；不含用户消息和最终答复 | 明确普通任务定义；长等待、事故、发布和高风险验收单独分组，不用其稀释普通任务结果 |
| 原样重复调用 | 工具、参数、目标和前置状态均无变化的重复调用次数 | 必须为 0；修正参数、恢复环境、出现新证据后的复跑不计原样重复 |
| 控制面耗时占比 | `(授权/路由/checkpoint/恢复/工具调用纠错耗时) / (总执行耗时 - 用户等待 - 外部排队)` | M0 明确起止点与归类规则；无法可靠拆分的样本不得用于该指标裁决 |
| 中断后重复实施 | 已有有效 diff、checkpoint 或验证证据仍被无依据重做的次数 | 恢复前后胶囊、Git 事实和证据有效性摘要 |
| 后续项净变化 | `新增正式后续项 - 本轮关闭后续项` | 只统计唯一正式载体中的条目，不统计会话建议和临时观察 |

M0 只保留支撑裁决的汇总口径和必要摘要；原始会话日志、临时截图、run id 与一次性路径不进入版本管理。若 7 日内未覆盖 Task 7 的全部样本类型，M3 结论必须为“部分完成/需要确认”，不得用模拟数量或放宽样本定义补齐。

## 三、关键决策与非目标

### 3.1 关键决策

- 建立一套统一任务状态机，普通任务与连续迭代共享状态语义，但分别遵守各自授权和 Ready 门禁。
- 建立“首选工具 + 默认最多一个有信息增益的备用路径”的调用预算；高优先级规则要求的源码交叉核验、跨层补充查询和高风险必要复核不计作备用路径，失败后先分类和改变策略，不做原样重试。
- 将图谱定位为检索加速器和关系导航器，当前分支源码、配置、日志及正式治理载体仍是最终事实源。
- 将 commentary 改为事件驱动，不按命令数量播报。
- 将中断恢复所需信息压缩为任务胶囊，线程过长时主动分段。
- 将普通验证、批次验证、发布验证分层，允许复用仍然有效的证据。

### 3.2 非目标

- 不建设新的重型任务编排平台。
- 不把所有职责机械拆成多个子智能体。
- 不取消授权门、stop/pause checkpoint、数据重置边界或发布门禁。
- 不把工具失败静默吞掉；用户可见性改为归类后的摘要。
- 不追求源码图谱覆盖 CSS、SQL、YAML、JSON、日志、Git 和运行态等全部事实。
- 不在长期规则中固化易变化的模型型号、工具 schema 或临时统计值。

### 3.3 实施授权与 Git 前置

- M0 的只读盘点不自动授权文件修改、创建分支、提交、推送或运行环境变更。
- M1 进入写入前必须再次核对分支与工作区，并获得创建 `codex/*` 功能分支及修改候选文件的明确授权；不得直接在 `master` 上实施本主线规则变更。
- 本地分阶段提交、推送、PR、合并和分支清理分别属于独立 Git 状态变化。若未取得本地提交授权，阶段只能形成待审 diff，不得宣称具备提交级回滚，也不得判定对应阶段完整收口。
- AutoPilot 控制面行为是否变化必须在实施 diff 形成后据实际配置、Schema 和行为契约判定，不得仅凭文件名推断。

## 四、统一执行协议

### 4.1 任务状态机

```mermaid
flowchart LR
    A["接收请求"] --> B["识别授权与模式"]
    B --> C["只读前置检查"]
    C --> D["实施或分析"]
    D --> E["分层验证"]
    E --> F["审查与失败分类"]
    F -->|"需补修"| D
    F -->|"通过"| G["归档与 Git 收口"]
    G --> H["最终裁决"]
```

每个阶段只维护一个当前状态，不允许同时把多个步骤标为进行中。任务状态至少包含：目标、授权范围、当前阶段、已完成证据、剩余动作、阻塞、工作区与分支、是否允许提交/推送/合并。

### 4.2 短指令固定语义

| 用户指令 | 默认语义 | 不自动包含 |
|---|---|---|
| `继续` | 从当前任务胶囊的下一未完成步骤继续 | 扩大范围、重新规划已完成部分 |
| `执行` | 在已明确方案和授权边界内实施 | 提交、推送、合并 |
| `修复` | 修复当前已确认失败并做必要复验 | 顺带重构、处理无关建议 |
| `推送` | 核对分支、状态和门禁后推送当前授权分支 | 直接推送 `master` |
| `合并` | 核对验证和目标分支后执行授权的合并流程 | 绕过 PR、保护分支或未通过门禁 |
| `清理分支` | 先预览，仅清理已合并且明确在范围内的分支 | 删除未合并分支、远端批量清理 |
| `启动迭代-N` | 启动最多 N 个合格迭代单元，并在上限或停止条件处总验收 | 为凑数量放宽 Ready 标准 |

若同一短指令在当前任务中有明确上下文，以最近的有效任务胶囊为准；没有可靠上下文时只做安全的只读核验并请求必要确认。

### 4.3 粘性执行模式

每个线程显式保持一种模式，除非用户指令或任务阶段改变：

- `分析模式`：只读调查、方案、评审，不实施变更。
- `执行模式`：实施已授权变更并验证，不默认发布。
- `验收模式`：复核证据并给出通过/不通过裁决，不顺带扩展实现。
- `连续迭代模式`：严格遵守 Ready、checkpoint、逐 Issue 收口和上限。
- `发布收口模式`：提交、推送、PR、合并、分支清理等 Git 生命周期动作。

模式切换时只播报一次“从什么模式切换到什么模式、原因是什么”。

## 五、确定性工具路由

### 5.1 路由表

| 查询目标 | 首选路径 | 备用/核验路径 |
|---|---|---|
| 已知文件、字符串、配置键、错误文本 | `rg` + 直接读取 | Git diff / 相关配置 |
| Java、TypeScript、Vue 未知符号和局部引用 | CodeGraph，一次有界查询 | `rg` + 直接文件读取 |
| 跨前后端、跨语言、多跳调用链、架构边界 | `codebase-memory-mcp` | `rg` + 源码核验 |
| PowerShell 定义、引用、调用链 | `codebase-memory-mcp` | `rg` + 直接读取；禁止 CodeGraph |
| CSS、SQL、YAML、JSON、日志、Git、CI | 对应直接工具或文本检索 | 当前配置/日志/运行态 |
| 存量问题、优先级、关联导航 | 项目知识图谱 | `sourceRefs` 对应正式载体与当前分支 |
| 页面行为和交互结果 | 内置浏览器 | 服务日志、网络响应、源码 |

### 5.2 调用预算

1. 每个问题先按查询对象选择一个首选路径；“已知文件、字符串”仅指明确文件、配置、错误文本或精确文本定位，不覆盖未知源码符号及关系分析。
2. 首选路径失败或召回不足时，默认最多切换一个有信息增益的备用路径。
3. `AGENTS.override.md` 强制要求的源码交叉核验、跨层/跨语言补充查询、PowerShell 核验和高风险必要复核属于必需证据，不计入备用路径预算。
4. 若必需证据仍不足，可以说明风险与信息增益后增加核验路径；不得为了遵守调用数量而作出缺证据裁决。
5. 备用路径仍失败时，输出失败分类、已获得证据、缺失前置和下一决策；不得原参数连续重试。
6. 只有出现新证据、参数已纠正、环境已恢复或明确的瞬时故障时，才允许复跑。
7. 图谱未命中已知文件、符号或字段，一律标记为“工具召回不足”，不得据此断言代码不存在。

### 5.3 失败分类与用户可见输出

| 分类 | 处理原则 | 用户可见摘要 |
|---|---|---|
| `tool_config` | 修配置、补索引、确认能力是否加载 | 工具不可用及采用的降级路径 |
| `tool_invocation` | 修正 schema、参数、转义或命令入口 | 首次失败原因和已纠正方式 |
| `environment_prerequisite` | 恢复服务、端口、Docker、数据库等前置 | 环境阻塞及复验结果 |
| `retrieval_gap` | 用 `rg`/直接读取交叉核验 | 图谱召回不足，不等于代码不存在 |
| `quality_or_security` | 进入实现整改或阻塞裁决 | 真实失败证据和影响范围 |

内部错误细节保留在必要证据中；面向用户按“分类 + 影响 + 已采取动作 + 结果”汇总，避免逐条转播相同错误。

## 六、浏览器、测试、CI 与 Git 优化

### 6.1 浏览器执行策略

- 每个线程首次使用浏览器时只做一次能力与页面状态初始化。
- 不猜测 API 名称、参数或返回结构；能力不明确时先读取可用工具说明。
- 同一浏览器调用因参数问题失败一次后，必须修正调用方式或切换到源码/日志核验，不原样重试。
- cgc-pms 前端验收优先使用规定的 dev-login 入口；回到 `/login` 且代理指向旧 backend 时，先归类运行态问题。
- 同一输出目录、截图名、测试报告目录不得并行复用，避免互相覆盖。
- 页面验收至少记录：访问入口、关键操作、实际结果、关联接口或日志、最终裁决。

### 6.2 分层测试

| 层级 | 触发时机 | 最小证据 |
|---|---|---|
| L1 目标验证 | 每次具体修改后 | 目标测试/静态核对、`git diff --check` |
| L2 模块回归 | 跨文件或关键边界修改后 | 相关模块测试、类型/构建检查 |
| L3 批次验收 | 一组紧密相关修改完成后 | 复用有效 L1/L2 证据，仅补裁决必需项 |
| L4 发布验证 | 推送、PR 或合并前 | 适用 CI、权限/数据边界、发布门禁 |

证据复用必须满足：代码未发生影响该证据有效性的变化、环境与配置未改变、测试选择器真实存在、结果可定位。否则必须重跑。

### 6.3 CI 轮询

- 首次失败先分类；疑似波动或前置状态问题只允许一次最小等价复验。
- CI 状态采用退避轮询，状态未变化时不重复向用户播报。
- 只在状态变化、超过预期时长、出现确定失败或需要用户决策时更新。
- CI 红灯在分类前不得直接判为业务代码失败。
- GitHub 整份 job 日志或 artifact 出现 EOF、Schannel 或稳定超时时，不把仓库 Git remote 切到 SSH，也不无界切换下载端点；优先从已登录 job 页面定位失败 step 的临时日志入口，确认支持 Range 后默认只取末段 `256 KB`，最终摘要不完整时至多扩大一次。临时签名 URL 和 run id 不写入长期规则。

### 6.4 Git 发布生命周期

```mermaid
flowchart LR
    A["授权分支"] --> B["实施与验证"]
    B --> C["本地提交"]
    C --> D["推送功能分支"]
    D --> E["创建或更新 PR"]
    E --> F["CI/审查通过"]
    F --> G["授权合并"]
    G --> H["预览并清理已合并分支"]
```

- 禁止直接向 `master` 推送。
- `提交`、`推送`、`创建 PR`、`合并`、`清理分支` 分别视为独立状态变化，按用户授权和适用门禁执行。
- 清理前必须预览并确认分支已合并、无未推送提交、无活动 worktree 占用。
- 普通实现完成但未获发布授权时，停在已验证的工作区或本地提交状态，并明确说明。

## 七、沟通与上下文恢复

### 7.1 事件驱动 commentary

只在以下事件播报：

1. 开始工作并说明范围、模式和关键假设。
2. 发现影响方案的关键事实或风险。
3. 状态从分析切到实施、从实施切到验收或发布。
4. 出现真实阻塞或需要用户决策。
5. 长任务超过 60 秒且有实质进展变化。
6. 完成并给出验收结论。

下列情况不单独播报：每条只读命令、相同 CI 状态、同类工具错误的重复明细、纯等待、无信息增益的“继续处理中”。

### 7.2 线程拆分阈值

满足任一条件时建议新开线程或生成恢复胶囊后继续：

- 单线程超过 20–25 个有效轮次。
- 工具调用超过约 200 次且任务目标已发生阶段切换。
- 从实现切换为独立验收、发布或事故复盘。
- 当前上下文中已出现两次以上对已完成步骤的重复确认。
- 长时间等待会使业务实现上下文失去价值。

阈值是治理信号，不是强制中断正在安全收口的任务。

### 7.3 任务恢复胶囊

恢复胶囊固定包含：

```text
任务目标=
当前模式=
授权范围=
当前分支与工作区=
已完成变更=
已通过验证=
失败分类与阻塞=
剩余步骤=
禁止重复执行=
是否允许提交/推送/合并=
正式承接的后续项=
```

恢复时先核验分支、工作区和关键文件事实，再从“剩余步骤”继续；不得仅凭会话记忆重新实施已完成部分。

### 7.4 主线程与侧线程隔离

- 侧线程默认只做问答和轻量只读探索，不接管主线程执行状态。
- 若用户在侧线程明确授权变更，必须最小化范围并先核对分支/工作区，避免干扰主线程。
- 涉及并行实现时必须使用独立分支或 worktree，并明确成果归属和回收方式。

## 八、规则与文档落地结构

采用单一权威来源，避免多处复制同一规则：

| 载体 | 职责 |
|---|---|
| `AGENTS.override.md` | 最高级授权、安全、事实源和规则索引 |
| `AGENTS.md` | 项目角色、通用协作约定及稳定入口 |
| `docs/standards/codex-task-execution-policy.md` | 本计划落地后的统一执行协议权威正文 |
| 专项 Skill | CI、运行态刷新、主线工作流等具体操作细节 |
| AutoPilot reference/script | 连续迭代特有 checkpoint、状态迁移和确定性校验 |

同一条动态阈值或流程只保留一个权威定义；其他文件只引用，不复制全文。规则更新必须检查冲突、失效入口和重复描述。

## 九、实施任务

### Task 1：建立统一执行策略文档

**目标：** 创建 `docs/standards/codex-task-execution-policy.md`，固化状态机、短指令语义、粘性模式、工具调用预算、失败分类和沟通协议。

**实施要点：**

- 从现有规则中提取稳定的通用部分，不复制 AutoPilot 专项细节。
- 在 `AGENTS.override.md` 或 `AGENTS.md` 增加最小索引与优先级说明。
- 对重复或冲突条款建立迁移清单，逐条标记为“已存在/需补强/新增/不改”，再决定保留、引用、合并或删除。

**验收：** 规则有唯一权威正文；短指令和任务状态均可无歧义解释；全文检索不存在互相冲突的重复定义。

### Task 2：收窄图谱与检索 Skill 触发

**目标：** 将“何时调用哪个图谱、何时直接 `rg`、何时停止重试”变成确定性规则。

**实施要点：**

- 调整主线 Skill 中的检索决策表和失败降级路径。
- 明确 CodeGraph、`codebase-memory-mcp`、项目知识图谱的职责与不可覆盖范围。
- 对 PowerShell、跨语言边界和已知符号检索增加强制路由测试样例。

**验收：** 代表性检索场景首次选路正确；图谱未命中不会被误判为代码不存在；同一路径无变化重试为零。

### Task 3：建立浏览器调用模板

**目标：** 减少能力探测、参数猜测、登录态和代理旧状态导致的重复失败。

**实施要点：**

- 固化一次性初始化、dev-login、健康检查、页面动作和证据记录模板。
- 固化错误分类和源码/日志降级路径。
- 为并行或批量验收分配唯一输出目录。

**验收：** 代表性页面验收一次初始化后完成；参数错误不发生原样重试；旧代理状态能被正确分类并恢复。

### Task 4：实施测试与 CI 分层

**目标：** 减少无意义重复验证，同时保留正式裁决所需证据。

**实施要点：**

- 定义 L1–L4 的触发条件和证据有效期。
- 校验测试选择器、脚本入口和 Maven/PowerShell 参数格式。
- 在 CI Skill 中加入失败分类、一次最小复验、退避轮询和失败 step 日志尾段分片取证规则。

**验收：** 同一未变化代码不重复跑全量测试；一次性前置失败不会被误判为代码失败；整份日志稳定超时时可用已登录 job 页面和 Range 尾段取得失败摘要；发布门禁仍完整。

### Task 5：固化 Git 发布与分支清理生命周期

**目标：** 统一提交、推送、PR、合并和清理分支的授权与门禁。

**实施要点：**

- 在主线工作流中明确禁止直接推送 `master`。
- 为每个 Git 状态变化增加最小前置核验与结果回报。
- 建立分支清理预览和 worktree 占用核验模板。

**验收：** 代表性发布流程只推送功能分支；未授权合并时停在 PR 或已推送分支；清理不删除未合并成果。

### Task 6：落地事件驱动沟通与恢复胶囊

**目标：** 控制过程播报数量，并保证长任务中断后可恢复。

**实施要点：**

- 将 commentary 触发条件写入统一策略和主线 Skill。
- 在阶段收口、线程切换和中断前生成任务恢复胶囊。
- 最终回报统一使用“交付物、证据、Git 状态、结论、阻塞、风险、后续项净变化”。

**验收：** 普通任务过程播报通常不超过 5 次；中断后不重复实施；最终结论不依赖折叠的 commentary。

### Task 7：开展一周试运行与复盘

**目标：** 用真实任务验证规则是否降噪且不降低质量。

**实施要点：**

- 选择普通修复、跨层分析、浏览器验收、CI 排障、连续迭代、发布收口各至少一个样本。
- 每个样本结束时写入统一策略定义的最小样本胶囊；没有工具调用总数或有效耗时分母的样本只作定性证据，不进入比例达标计算。
- 使用无持久化样本计算器统一计算工具失败率、控制面耗时占比与后续项净变化；计算器不复制目标阈值，并拒绝不守恒计数或不完整耗时分母。
- 只记录支撑指标与裁决的摘要，不提交原始会话日志或临时截图。
- 试运行结束后对指标、误判、漏检、额外成本和规则冲突做复盘。
- 从 M0 开始维护一份正式试运行报告，记录指标口径、冻结基线、样本覆盖、阶段结论和最终复盘；缺少任一样本类型时按 2.3 判定为部分完成，不得用模拟数据替代真实任务。

**验收：** 每个比例指标都有可复核分子与分母并达到核心目标；未发生安全门禁绕过；未达标项有明确根因、责任载体和下一步，不形成悬空建议。

## 十二、验收矩阵

| 场景 | 预期路由与行为 | 通过标准 |
|---|---|---|
| 已知配置键定位 | `rg` + 直接读取 | 不先调用图谱；结果与当前分支一致 |
| Java 未知符号影响范围 | CodeGraph 后源码核验 | 一次有界查询命中或明确标记召回不足 |
| 跨前后端权限链 | `codebase-memory-mcp` + 源码核验 | 多跳关系与当前实现交叉一致 |
| PowerShell 调用链 | memory 图谱 + `rg` | 全程不调用 CodeGraph |
| CSS/SQL/YAML/日志问题 | 直接文本、配置或日志工具 | 不为覆盖不足反复调用源码图谱 |
| 浏览器回到登录页 | 先查代理/运行态 | 正确分类并在恢复后复验 |
| Maven 参数解析失败 | `tool_invocation` | 修正单参数字符串后复跑，不判业务失败 |
| CI 首次红灯 | 分类 + 一次最小复验 | 波动与真实失败被区分 |
| 用户说“推送” | 推送授权功能分支 | 不直接推送 `master` |
| 用户说“清理分支” | 预览已合并分支 | 不删除未合并或被 worktree 占用分支 |
| 长线程中断后继续 | 读取恢复胶囊并核验状态 | 不重复修改和验证已完成部分 |
| 连续迭代达到 N | 完成当前安全收口和总验收 | 不启动第 N+1 个 Issue |

## 十三、风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 降噪导致真实错误不可见 | 误判通过 | 保留分类摘要、关键证据和阻塞，不吞错误 |
| 调用预算过严导致证据不足 | 漏检 | 高风险任务允许说明原因后增加核验路径 |
| 规则迁移产生冲突 | 路由不一致 | 单一权威正文、全文冲突检查、分阶段上线 |
| 证据复用失效 | 错误裁决 | 校验代码、环境、配置和选择器是否变化 |
| 短指令语义与用户意图不符 | 越权或停滞 | 结合最近任务胶囊；高风险动作仍需明确授权 |
| 线程拆分丢失上下文 | 返工 | 强制恢复胶囊和分支/工作区核验 |
| 优化本身变成重型治理 | 控制面成本升高 | 限制为文档、Skill 和少量确定性校验，不新建编排平台 |
| 控制面行为变化未进入指纹 | 连续迭代使用旧金丝雀证据 | M2 强制判定行为变化；触发时更新指纹覆盖并执行用户授权的单 Issue 金丝雀 |
| 指标采集不可复现 | 无法裁决是否降噪 | M0 冻结指标字典、采集窗口和汇总方法，缺证据样本不进入正式统计 |

## 十四、回滚方案

1. 所有规则和 Skill 修改必须在获得授权后使用独立 `codex/*` 功能分支实施，不直接在 `master` 上实施或推送。
2. 获得本地提交授权后，每个阶段形成独立、可审查提交，M1、M2、M3 可分别回滚；未获提交授权时只保留待审 diff，不宣称具备提交级回滚。
3. 若新路由造成漏检或阻塞率上升，先回退相应 Skill/策略引用，不回退授权门和安全规则。
4. 若指标采集成本超过收益，停止采集非关键细项，仅保留失败率、播报数、中断返工、直接主分支推送和后续项净变化。
5. 回滚后重新运行受影响的代表性验收场景，并更新唯一正式承接载体。
