**Goal:** 在不改造后端数据模型、不引入全站 i18n 体系、不新增依赖和不扩大到“所有英文文案清零”的前提下，完成 cgc-pms 全站“用户可见业务枚举码不再直出英文”的治理计划，形成可执行的路由清单、浏览器巡检、静态源码审计、最小修复、复扫验收和报告归档闭环；验收方向为“用户在真实页面 DOM 中看到的业务枚举值，优先显示中文 label 或明确可读的既有中文态”。 **Architecture:** 前端继续沿用 Vue 3 + TypeScript + Ant Design Vue 与现有页面组织方式，优先复用既有字典/标签能力，如 `getDictDataByCode(dictCode)`、`getDictLabelSync`、成本/项目等已有 label helper 与本地 fallback；本主线只治理用户可见业务枚举展示，不承诺所有英文消失，明确排除项目编号、单据编号、权限码、接口路径、技术日志、品牌/英文专名，不新增全局抽象、不引入 i18n、不推动后端联动重构，坚持最小可行方案。

# 第25条主线-全站业务枚举中文展示治理任务计划书

## 一、结论与执行建议

- 建议立项：通过。
- 主线性质：前端展示治理主线，核心是“真实页面中不再把业务枚举码直接展示给用户”。
- 当前已知边界：已有一次小修仅验证 `/project/list` 与 `/cost/ledger`，全站治理尚未执行；因此当前不能以局部通过替代全站收口。
- 推荐策略：按“先审计建账、再最小修复、后复扫验收”的顺序推进，避免一上来扩成全量 i18n 或后端字典治理。
- 风险等级：中。主要风险在于页面数量多、同一枚举存在多套展示路径、表格列/详情抽屉/弹窗/统计卡片可能分别遗漏。

## 二、治理目标与非目标

### 2.1 本主线治理目标

1. 只治理用户在浏览器真实页面中可见的业务枚举码直出问题。
2. 治理对象包括但不限于列表列值、详情页字段值、抽屉/弹窗只读字段、统计卡片中的业务状态、筛选项可见标签、步骤流中的业务状态显示。
3. 治理结果以真实 DOM 文本为准，而不是仅以源码搜索结果或接口返回值为准。
4. 优先复用已有字典与标签 helper，保持现有接口契约和业务逻辑不变。

### 2.2 明确非目标

以下内容不纳入本主线承诺范围：

1. 项目编号、合同编号、付款单号、审批单号、出入库单号等业务编号。
2. 权限码、角色码、菜单码、接口路径、路由路径、技术日志、调试信息。
3. 品牌名、英文专名、第三方产品名、法律/合同中的原始英文专有词。
4. 后端数据治理、数据库字典补数、接口结构重写、全站 i18n 改造。
5. “所有英文都消失”的泛化目标。

## 三、范围定义

### 3.1 纳入治理的问题类型

1. 后端返回 `DRAFT`、`APPROVED`、`ENABLE` 等业务枚举值，前端原样直出。
2. 页面已有字典能力，但只在筛选项使用中文，在表格/详情里仍显示英文码。
3. 同一业务状态在列表、详情、弹窗、统计卡片上展示不一致，部分中文、部分英文。
4. 页面已存在某类 label helper，但没有在全部展示点复用。

### 3.2 不纳入治理的问题类型

1. 枚举值只存在于接口调试、控制台日志、网络面板，不在用户可见 DOM 中。
2. 用户本就需要看到原码的技术性标识。
3. 因需求定义要求保留英文原词，且已有明确产品口径。

## 四、现状判断

### 4.1 当前已知事实

1. 现有小范围修复仅覆盖 `/project/list` 与 `/cost/ledger`，不能代表其他页面已合规。
2. 仓库已有字典集成模式：前端可通过 `getDictDataByCode(dictCode)`、`getDictLabelSync` 及本地 fallback 做兼容展示。
3. 真实验收必须以浏览器页面实际渲染的 DOM 文本为准，不能只看接口或源码。

### 4.2 风险判断

1. 同一枚举可能同时出现在表格列、详情抽屉、Tag、描述列表、表单回显和统计卡片中，单点修复容易漏。
2. 某些页面已有页面私有 label helper，若再造一套公共抽象，成本高且容易扩散。
3. 若直接做全站字符串替换，容易误伤编号、权限码、日志和真实英文专名。

## 五、执行总原则

1. 最小可行方案优先，不引入 i18n，不新增依赖。
2. 先确认“是不是用户可见业务枚举”，再决定是否治理。
3. 已有 helper 优先复用，优先顺序为：
   - `getDictLabelSync`
   - `getDictDataByCode(dictCode)` 配合当前页面既有处理
   - 成本、项目、采购、审批等模块已有 label helper
   - 页面内本地 fallback 映射
4. 同一页面内优先修共享展示入口，避免在每个列渲染器重复补丁。
5. 能在现有页面私有层收口的，不上提新全局抽象。

## 六、分阶段任务

## 阶段 M1：路由清单与治理台账

目标：先明确“要看哪些页面、按什么优先级看、哪些是已知风险高点”。

任务：

1. 基于现有前端路由和业务模块，整理全站业务页面路由清单。
2. 标注优先级：
   - P0：项目、成本、合同、付款、结算、采购、库存、审批、预警
   - P1：主数据、系统管理中用户可见业务状态页
   - P2：辅助页面、历史遗留低频页
3. 把已验证的 `/project/list`、`/cost/ledger` 记录为“已有局部样例，不代表全站通过”。
4. 为后续浏览器巡检建立页面台账，字段至少包含：路由、页面文件、业务域、是否已有字典 helper、巡检状态、问题摘要、证据链接。

阶段产出：

- 全站路由巡检清单
- 按业务域分组的问题台账
- 首批高优先级页面名单

## 阶段 M2：浏览器巡检

目标：以真实 DOM 为准，发现用户实际能看到的英文业务枚举直出问题。

任务：

1. 按 M1 路由清单逐页巡检，不只看首屏，还要看：
   - 表格列
   - 详情抽屉/详情页
   - 弹窗只读区
   - Tag/Badge/状态点
   - 统计卡片
   - 筛选栏已选值回显
2. 采证方式以浏览器截图或 DOM 文本摘录为准。
3. 每个问题都要记录“页面路径 + 位置 + 当前直出值 + 预期中文含义 + 是否已有 helper 可复用”。
4. 对非治理项当场排除并注明原因，避免误入编号、权限码、接口路径、技术文本。

阶段产出：

- 真实页面问题清单
- 首轮浏览器证据
- 排除项说明

## 阶段 M3：静态源码审计

目标：对浏览器已发现问题追根到源码，并顺带补出浏览器暂未触达但高概率同类的问题点。

任务：

1. 以浏览器发现的问题为锚点，回溯对应页面、列配置、详情组件、状态 tag helper、页面私有映射函数。
2. 搜索典型高频业务枚举码和高频 helper 调用，确认同页或同模块是否存在漏网展示点。
3. 明确每个问题的最小修复路径：
   - 直接复用 `getDictLabelSync`
   - 接入既有 `getDictDataByCode(dictCode)`
   - 复用模块已有 label helper
   - 页面内补一个本地 fallback
4. 对需要后端补数或字典缺项的场景单独标记为“依赖外部条件”，但本主线不扩写成后端治理计划。

阶段产出：

- 问题到源码位置映射表
- 最小修复建议清单
- 外部依赖清单

## 阶段 M4：最小修复

目标：在不扩架构的前提下，逐页消除用户可见业务枚举英文直出。

任务：

1. 优先修共享展示入口，例如同页复用的 `renderStatus`、`statusTag`、列格式化函数、详情字段 formatter。
2. 没有共享入口时，在页面内做局部最小修复。
3. 仅当现有 helper 不足以覆盖时，才在同模块内补轻量本地 fallback。
4. 修复时同步检查同页的列表、详情、弹窗、统计卡片是否一致。

阶段产出：

- 按页面分批次完成的最小修复包
- 每批次对应的问题关闭记录

## 阶段 M5：复扫验收

目标：确认修复后真实页面不再把业务枚举码直出给用户。

任务：

1. 回到原问题页面做浏览器复扫，逐项对照原证据位置。
2. 对同模块做相邻页面抽样复扫，验证没有“列表好了、详情还漏”的回归。
3. 对已排除项做一次口径复核，确保没有把本应治理项误排除。
4. 给出阶段性结论：通过/不通过、阻塞/非阻塞。

阶段产出：

- 复扫证据
- 通过/不通过判定
- 阻塞项清单

## 阶段 M6：报告归档

目标：把范围、证据、结论、剩余风险和未覆盖项形成正式归档。

任务：

1. 输出正式质量/验收报告。
2. 记录：
   - 已治理页面
   - 已确认排除项
   - 未覆盖页面
   - 仍依赖后续字典或产品口径确认的问题
3. 若存在非阻塞遗留，明确后续触发条件和建议主线承接关系。

阶段产出：

- 正式验收报告
- 遗留问题清单
- 收口结论

## 七、优先业务域建议

建议按以下顺序推进：

1. 项目、成本、合同、付款、结算
2. 采购、库存、材料验收、变更签证、分包
3. 审批中心、预警中心、流程详情
4. 主数据与系统管理中会被业务用户直接看到状态值的页面

理由：

- 上述页面业务状态展示更密集，且用户感知最强。
- 已知 `/project/list`、`/cost/ledger` 有先例，便于复用治理方式。

## 八、模型分配表

| 任务/角色 | 任务类型 | model | thinking | reason |
| --- | --- | --- | --- | --- |
| 路由清单整理与治理台账建立 | 审计型 | gpt-5.4 | low | 主要是结构化整理、分类和记录，判断空间有限 |
| 浏览器逐页巡检与证据采集 | 验收型 | gpt-5.5 | medium | 需要基于真实 DOM 做页面对照、排除误报并汇总证据 |
| 静态源码审计与修复路径判定 | 审计型 | gpt-5.5 | medium | 需要把页面现象映射到具体展示链路并判断最小修复点 |
| 页面最小修复执行 | 实现型 | gpt-5.4 | medium | 以前端单页/同模块最小改动为主，优先复用现有 helper |
| 复扫验收与通过性裁决 | 验收型 | gpt-5.5 | medium | 需要跨页面复核、对照前后证据并给出通过/阻塞结论 |
| 报告归档与收口文档整理 | 归档型 | gpt-5.4 | medium | 以结构化归档和正式结论整理为主 |

### 8.1 任务分类说明

- 审计型任务：路由清单整理、静态源码审计。
- 实现型任务：页面最小修复执行。
- 验收型任务：浏览器巡检、复扫验收与通过性裁决。
- 归档型任务：报告归档与收口文档整理。

### 8.2 分档经验约束

1. 浏览器巡检与复扫验收不应降到 `low`，因为需要做“真实页面证据 + 排除误报”判断。
2. 页面最小修复不默认升到 `gpt-5.5 + high`，除非中途升级为跨模块复杂状态链路问题。
3. 台账与归档任务不应与实现/验收任务统一同档，避免分档偷懒。

## 九、验收标准

### 9.1 通过标准

同时满足以下条件方可判定通过：

1. 纳入治理范围的页面中，用户可见业务枚举码不再直接以英文原码展示。
2. 复扫以真实浏览器 DOM 文本为准，原问题位置已显示中文 label 或明确可读的既有中文态。
3. 修复优先复用现有 helper，未引入 i18n、新依赖或无关抽象。
4. 排除项有明确理由，且与编号、权限码、接口路径、技术文本、英文专名边界一致。

### 9.2 不通过标准

出现以下任一情况即判定不通过：

1. 高优先级业务页仍存在用户可见英文业务枚举直出。
2. 只修表格未修详情/弹窗/统计卡片，导致同页展示不一致。
3. 通过源码搜索声称已完成，但拿不出真实页面证据。
4. 为解决展示问题引入超出本主线边界的架构扩张。

### 9.3 阻塞/非阻塞判定

- 阻塞：
  1. P0 页面仍有真实用户可见英文枚举直出。
  2. 关键枚举缺少中文来源，且无现有 fallback 可安全复用。
  3. 同一枚举在主流程页面中前后展示冲突，影响用户判断。
- 非阻塞：
  1. 低频页面未纳入本轮但已在台账中登记。
  2. 个别排除项需产品口径确认，但当前不影响主流程页面结论。
  3. 品牌/英文专名按口径保留。

### 9.4 证据要求

1. 每个判定为“已治理”的问题至少有一条浏览器证据，形式可以是截图或 DOM 文本摘录。
2. 每个判定为“排除”的问题必须写清排除原因。
3. 每个判定为“阻塞”的问题必须写清页面、位置、现象、原因和建议归属。

## 十、风险点与控制方案

### 10.1 主要风险

1. 页面巡检覆盖不全，导致“局部通过、全站漏扫”。
2. 误把编号、权限码或英文专名当成治理对象，造成误改。
3. 字典数据不完整，导致 helper 接入后仍只能回落原码。
4. 同模块有多个私有 formatter，修了一处漏另一处。

### 10.2 控制方案

1. 先有路由清单与台账，再做浏览器巡检。
2. 浏览器证据与源码位置一一对应，避免空口结论。
3. 复用优先，不新造全局体系。
4. 对“无字典、无 helper、无 fallback”的场景单列依赖，避免被动扩主线。

## 十三、收口判定

本主线的收口条件不是“站内所有英文彻底消失”，而是：

1. 用户可见业务枚举码直出问题已按范围完成治理。

3. 通过结论基于真实浏览器证据，而非仅基于源码猜测。
4. 剩余未治理项、阻塞项、外部依赖项均已入账并有后续去向。
