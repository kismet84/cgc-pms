# 第54条主线：25项存量问题分层治理与发布门禁收口任务计划书

**Goal:** 对当前知识图谱台账中的25项问题完成逐项核验、唯一裁决、最小修复、分层验证与正式收口；优先解除3项发布门禁，完成A-01父项及9个兼容性子项治理，并使所有产品深化、工程治理、冻结项和运维观察均具有可执行且不重复的最终状态。验收方向不是机械实现25个功能，而是确保每项只能落入“修复并复验”“具备完整前置的正式承接”或“证据充分关闭”之一。

**Architecture:** 复用现有Spring Boot、Vue、MySQL/Flyway、GitHub Actions、项目知识图谱和Backlog载体，不新建第二套问题系统、发布平台、通知框架或AutoPilot控制面。M0事实冻结后分为两条独立执行线：M1A只做发布门禁准备，不触碰目标环境；M2～M5依次完成接口兼容性、产品深化、工程治理及P2观察/冻结。两线在M6汇合，取得专项授权后执行目标环境动作、同SHA复验和总收口。A-01父项只汇总9个子项，不重复实施。涉及生产、密钥、远端分支保护、数据库、权限、金额、租户和控制面时保持fail-close并设置独立授权点。最小可行方案优先，证据不足项不强行编码。

## 1. 总体裁决

本主线覆盖25个台账节点，但实际治理结构为16个根问题和A-01下属9个子项。当前3项`RELEASE_GATE`是发布阻塞；其余22项不得无差别打包开发。

完成标准：

1. 25项全部有唯一结果、证据、责任边界和验收结论。
2. A-01只在9个子项全部裁决后关闭，父项不重复计入实现数量。
3. 3项发布门禁只能由同一待发布SHA和实际目标环境证据解除。
4. `NEEDS_CONFIRMATION`先完成调用方、兼容性和数据边界核验，再决定修复、保留或废弃。
5. `OBSERVATION`只有稳定复现和明确收益时升级Ready；否则保留观察或有依据关闭。
6. `FROZEN`只有前置齐备并重新通过产品决策门后解冻。
7. 收口报告给出新增后续项、关闭后续项、后续项净变化；不得留下口头后续。

完成结论分为两级，不得混用：

- **治理收口完成：** 25项均已修复并复验、正式承接或有依据关闭；允许仍有明确责任、解除条件和验收标准的阻塞/冻结项，但不得据此声明“25项全部修复”或“可上线”。
- **修复及发布收口完成：** 3项发布门禁全部解除，纳入实施项复验通过，首次非Draft PR前同一HEAD SHA等价CI及目标环境证据完整；只有达到本级才可声明本主线修复完成并进入上线裁决。

## 2. 范围与非目标

### 2.1 范围

- P0：A-01及3项发布门禁。
- P1：A-01的9个接口兼容性子项、A-02～A-04、2项AutoPilot效率观察、分支保护治理。

- 每批变更的相关验证、独立复核、Git收口与回滚证明。

### 2.2 非目标

- 不因“25项”一次性建设报表平台、通知平台、PWA、BIM、外部财务或完整计划软件。
- 不在本计划阶段连接生产、读取生产密钥、修改远端保护规则、推送、建PR或发布。
- 不修改已应用Flyway迁移；结构变化只能新增版本化migration。
- 不将前端未调用直接等同于接口可删除。
- 不读取或清理项目禁止目录。
- 不把上一轮代码审计尚未登记的4项发现混入25项计数；但必须在M0逐项去重，并正式承接或有依据关闭，禁止悬空。
- 不为了降低复杂度引入单实现接口、工厂、第二套调度器或泛化配置层。

## 5. M1A准备与M6执行：3项发布门禁

M1A仅完成范围、对象、责任人、维护窗口、回滚和证据模板准备，不读取或变更目标环境。以下任务中的目标环境“实施”统一在M6执行，且必须分别取得密钥、扫描、迁移/发布专项授权。M1A未完成不阻塞M2～M5；M1A与M2～M5均完成后才能进入M6。

### Task 1.1：REL-CREDENTIAL-ROTATION

M1A准备：确认环境身份、资产范围、维护窗口、密钥保管人、回滚密钥版本和双人复核人；不读取密钥值。

M6授权执行：

1. 为MySQL root/应用账号、Redis、MinIO、JWT、Jasypt及适用TLS材料生成互不复用的新凭据。
2. 通过目标环境受控Secret机制注入，不写入仓库、日志、报告或命令历史。
3. 逐服务切换并验证新凭据；旧凭据在回滚窗口结束后失效。
4. 验证旧JWT/旧凭据失败、新登录与关键业务健康。
5. 仅记录脱敏版本号、保管位置和双人签认，不记录密钥值。

验收：新凭据生效、旧凭据失效、服务健康、关键链路通过、双人脱敏复核完成。任一项缺失则保持`RELEASE_GATE`。

回滚：在旧凭据尚未销毁的受控窗口恢复上一Secret版本；恢复后重新验证并记录失败原因。不得将旧凭据重新写入仓库。

### Task 1.2：REL-FILE-RESCAN

M1A准备：明确目标对象存储、扫描器、对象总数、失败重试上限、隔离桶和审计字段；不连接目标对象存储。

M6授权执行：

1. 从业务文件表与对象存储双向形成对象清单，识别缺记录、缺对象和重复引用。
2. 逐对象读取真实内容并调用病毒扫描器；禁止SQL批量改`CLEAN`。
3. 感染、对象读取失败、扫描器不可用或结果不确定均fail-close。
4. 感染对象进入受控隔离；不得直接物理删除。
5. 输出总数、成功、感染、失败、缺失和重试结果的脱敏汇总。

验收：对象守恒，全部对象有真实扫描结果；异常对象均被阻断或隔离；抽样下载门禁验证通过。

回滚：仅回滚扫描状态写入和隔离引用；不恢复已确认感染对象到可下载状态。

### Task 1.3：REL-TARGET-SHA-REVALIDATION

M1A准备：确定待发布SHA冻结规则、制品摘要规则、required checks、目标环境验证清单及证据失效条件，不运行目标环境验证。

M6授权执行：

1. 冻结唯一待发布SHA和制品摘要。
2. 完成同SHA功能分支push事件上的完整前置CI。
3. 在目标环境执行Flyway验证、最小权限迁移、健康检查和真实角色正负样本。
4. 复核备份、恢复、回滚SQL/制品、密钥版本、文件扫描和监控告警。
5. 任一后续提交使旧证据失效，必须重新绑定新SHA。

验收：required checks、Flyway、角色、租户、金额、文件、回滚证据全部绑定同一SHA；否则禁止上线。

## 10. 验证矩阵

### 10.0 失败分类

所有验证失败先按通用执行协议归类为`tool_config`、`tool_invocation`、`environment_prerequisite`、`ready_issue_config`、`retrieval_gap`、`quality_or_security`或`unknown`。未分类前不得判定为业务代码失败；代码、测试、迁移或基线不同步导致的CI失败统一按`quality_or_security/DELIVERY_GATE_OMISSION`处理。

| 变更类型 | L1目标验证 | L2模块回归 | L3批次验收 | L4发布验证 |
|---|---|---|---|---|
| 纯裁决/台账 | 25项守恒、sourceRefs核验、重复项检查 | 项目地图/Current Focus/报告一致性 | 独立Reviewer逐项复核 | 不单独触发发布 |
| 后端接口 | 目标Service/Controller测试 | 相关模块测试、权限/租户/状态正负样本 | API契约与调用方回归 | 同SHA完整CI |
| 前端入口 | 目标组件/页面测试 | lint、test、type-check、build | 真实角色浏览器和接口联动 | V2门禁、E2E、完整CI |
| 数据库 | 新migration测试、H2/MySQL语义核验 | fresh baseline、legacy upgrade、最小权限迁移 | 数据守恒、回滚演练 | 同SHA MySQL CI与目标环境Flyway |
| 金额/排程 | 边界值、并发、状态机测试 | 业务闭环集成测试 | 故障注入与审计链 | 目标环境真实角色正负样本 |
| AutoPilot | 目标PowerShell契约测试 | 控制面策略套件 | 用户授权单Issue金丝雀 | 不以普通CI替代金丝雀 |
| 远端治理 | 配置diff与schema校验 | API复读 | 测试分支允许/拒绝路径 | 正式仓库规则复核 |
| 发布门禁 | 单项预检 | 目标环境模块验证 | 回滚演练、双人复核 | 同一SHA全部required checks |

### 10.1 首次非Draft PR前置

若本主线进入代码实施并准备首次非Draft PR，必须在同一HEAD SHA的功能分支push事件上完成：后端全量测试、测试顺序复验、MySQL最小权限迁移、前端lint/test/type-check/build、安全扫描、V2门禁、E2E和`build-summary`，再运行既有pre-PR核验入口。任一证据缺失均不得声明可提PR。

## 11. 风险控制

| 风险 | 影响 | 控制 |
|---|---|---|
| 将25项理解为25个功能 | 范围失控、重复建设 | 父子去重；先裁决再编码；Candidate/Ready门禁 |
| 接口误删 | 外部或旧客户端中断 | 调用方调查、兼容期、契约测试、可回滚路由 |
| 生产凭据泄漏 | 全系统失陷 | 目标环境授权、Secret注入、脱敏双人复核、不记录值 |
| 文件误判CLEAN | 恶意文件可下载 | 逐对象真扫描、fail-close、隔离而非删除 |
| 数据/金额错误 | 财务口径破坏 | 事务、锁、守恒、并发、故障注入和审计验证 |
| AutoPilot过度优化 | 控制面不稳定 | 有效样本、稳定根因、指纹、单Issue金丝雀 |
| 远端保护误配 | 团队无法推送或绕过审查 | 最小diff、测试分支、旧配置回滚 |
| ACL清理越界 | 用户文件损失 | 精确绝对路径、只读预览、用户授权、禁止递归广域目标 |
| 冷缓存规避扫描 | 漏洞门禁失效 | 只优化缓存，不降低HIGH/CRITICAL扫描标准 |

## 12. Git与交付边界

1. 实施前创建`codex/`前缀功能分支；禁止直接向`master`推送。
2. 每个批次保持外科手术式diff；不混入无关格式化、历史归档或临时产物。
3. 未获提交授权时停在已验证diff；提交、推送、PR、合并分别独立过门。
4. 计划书本身不授权生产、远端治理、提交、推送、PR或合并。
5. 所有临时日志、截图、缓存和测试产物不入库。
