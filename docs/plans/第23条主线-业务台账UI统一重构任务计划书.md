**Goal:** 在不改变既有业务流程、接口契约、权限模型和路由结构的前提下，对业务台账类页面做一次最小可行的 UI 统一重构，覆盖预警中心、合同台账、项目列表/项目管理、付款申请、结算管理、材料验收/验收明细、采购订单、库存台账/出入库记录，形成一致的页面骨架、文案语气、统计卡片、工具栏、列表列设置、右侧辅助/详情联动和移动端降级策略，验收方向为“同一业务域列表页视觉与交互语言统一、差异仅保留业务语义差异”。 **Architecture:** 前端保持 Vue 3 + TypeScript + Ant Design Vue 既有技术栈，不新增第二套组件体系，不做全站一次性重写；统一复用 `frontend-admin/src/assets/styles/global.css` 中现有 `lg-*` 体系、Ant Design Vue 基础组件、`ColumnSettingsButton`、`useColumnSettings`，并以预警中心右侧详情联动模式和合同/项目/采购等已落地的 `lg-grid + lg-table-wrap + lg-analysis-rail` 结构作为主基线，仅对目标页面做外科手术式收敛，避免跨模块抽象扩张。

## 四、统一 UI 语言与骨架标准

### 4.1 页面骨架标准

所有目标列表页统一对齐以下结构：

1. `lg-page` / `lg-list-page` 页面容器
2. `lg-page-head` 页面头，包含面包屑、标题、必要副说明
3. `lg-search-bar` 查询筛选区
4. KPI 区，统一 4 列或 5 列弹性布局
5. `lg-grid` 主工作区
6. 左侧 `lg-table-wrap` 列表区
7. 工具栏统一使用 `lg-toolbar`、`lg-toolbar-left`、`lg-toolbar-right`
8. 列表列设置统一通过 `ColumnSettingsButton`
9. 右侧统一为两种模式之一：  
   - 分析辅助模式：`lg-analysis-rail`  
   - 详情联动模式：参考预警中心的“左表右详情”
10. 统一分页、空态、错态、加载态和批量操作反馈

### 4.2 统一文案语言

- 只使用中文，不做中英文混排状态词。
- 操作文案统一简洁：`搜索`、`重置`、`刷新`、`导出`、`列设置`、`保存处理结果`、`标记已读`。
- 错误提示统一中性表达：`加载列表失败，请稍后重试`、`保存失败，请稍后重试`、`暂无数据`。
- 统计卡片口径统一使用“数量 + 当前说明”，避免同义词混用。

### 4.3 视觉与组件标准

- 色彩、字号、阴影、边框、圆角以 `docs/00-UI-Design-Baselines-and-Code-Specifications.md` 为唯一设计基准。
- 基础组件优先复用 Ant Design Vue。
- 图标继续优先使用 Ant Design Icons。
- 不新增新前端框架、不引入新样式方案、不复制出新的 `app-*` / `pt-*` 变体。

## 十、风险与回滚

### 10.1 主要风险

1. 共享样式调整导致已完成页面意外变形。
2. 页面间列设置、表格列宽、右侧面板宽度收敛后出现信息密度冲突。
3. 材料验收与采购订单、付款申请与结算管理之间存在不同业务文案历史包袱，统一时容易误伤语义。
4. 出入库记录若直接强套统一骨架，可能带出隐藏的老样式耦合。

### 10.2 风险控制

1. 先以高完成度页面为基线，不重造新骨架。
2. 共享文件改动优先最小化，页面特征优先在页面内收敛。
3. 每阶段结束做一次页面簇验收，不跨阶段堆积问题。
4. 文案统一只改非业务口径部分，涉及业务含义时先保守处理。

### 10.3 回滚策略

- 以页面簇为最小回滚单元，不做大面积一次提交后不可拆回。
- 若共享样式改动引发跨页回归，优先回滚共享文件，再逐页保留已确认安全的局部改动。
- 若某个页面统一成本明显超出预期，可退回到“结构统一、细节延后”的收口版本，优先保证主流程稳定。

## 十一、子智能体执行分档建议

### 11.1 实现型任务

   `model=gpt-5.4`

## 十二、实施顺序建议

建议执行顺序如下：

1. 先确定统一基线页面：合同台账、项目列表、采购订单
2. 以预警中心沉淀右侧详情联动标准
3. 统一付款申请、结算管理、材料验收
4. 最后补库存台账与出入库记录，并完成项目管理域收口
5. 每完成一个阶段即做一次验收，不等到最后一次性看结果

## 十三、非目标

- 不追求本轮把所有业务页面全部改成完全同一个模板。
- 不解决页面背后的业务规则缺陷、统计口径争议或接口慢问题。
- 不在本轮引入新的设计系统、低代码方案或前端基础设施改造。
- 不因统一 UI 顺手重写详情页、审批页、表单页全部结构。
- 不做“视觉好看优先于业务稳定”的激进调整。

## 十四、收口判定

本计划的收口条件不是“所有页面像同一个截图”，而是：

- 用户在目标页面间切换时，能感知为同一系统、同一台账产品语言；
- 页面保留业务差异，但不再保留实现风格差异；
- 改造范围可控、验证可执行、回滚可拆分；
- 不需要额外扩展成全站级重构才能交付。

满足上述条件后，可进入实施派工与阶段验收。
