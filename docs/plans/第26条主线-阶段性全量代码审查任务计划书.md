**Goal:** 在不修改业务代码、不改变运行环境、不提前进入整改实现的前提下，为 cgc-pms 建立一套阶段性全量代码审查计划，用于尽早识别重复性问题、结构性隐患、构建与发布包污染、权限/数据/回滚风险，并形成可派工、可验收、可分级、可归档的审查闭环；验收方向为“能稳定产出按风险分级的问题清单、明确阻塞项、明确整改归属，并在后续主线中减少重复返工”。 **Architecture:** 审查继续沿用仓库现有技术与交付边界，后端以 Spring Boot 分层、Flyway migration、Maven/CI 为主，前端以 Vue 3 + TypeScript + Vite + Ant Design Vue 与既有 `lg-*` 体系为主，部署与发布以 `deploy/`、Docker/Compose、现有脚本和文档规范为主；本主线只定义最小可行的“阶段性全量审查机制”，不引入新审计平台、不新增依赖、不发明新质量体系，不把计划书扩写成整改设计文档，坚持“先审查建账，再按风险派工整改，再做复核收口”。

# 第26条主线-阶段性全量代码审查任务计划书

## 一、结论与阶段定位

- 建议立项：通过。
- 主线性质：审计/归档型主线，目标不是立即改代码，而是提前建立阶段性全量代码审查门禁。
- 当前定位：作为后续阶段性审查、专项整改、上线前复核的统一入口，用于提前发现共性问题，避免问题拖到联调、上线或回归阶段再集中返工。
- 推荐策略：按“审查门禁定义 -> 分阶段审查 -> 风险分级派工 -> 复核归档”的顺序推进，不把一次审查扩成长期常驻治理平台。
- 风险等级：高。不是因为本计划本身高风险，而是其覆盖对象包含权限、安全、数据库、构建、发布包、回滚与可观测，一旦漏审会直接放大后续返工成本。

## 二、总体目标与不做范围

### 2.1 本主线目标

1. 建立适用于 cgc-pms 当前阶段的全量代码审查范围、阶段划分、门禁清单和验收口径。
2. 把代码规范审查、代码结构与可维护性审查、构建审查、构建与发布包洁净度审查纳入统一主线，而不是零散处理。
3. 同时覆盖后端、前端、安全权限、数据库、测试/CI、运维配置、日志可观测与回滚八大维度。
4. 输出可直接派给子智能体或责任人的任务清单，且每类任务都有明确模型建议、验收输出和风险分级规则。
5. 明确“阻塞/非阻塞”“通过/不通过”的判定依据，避免审查结果只停留在泛泛建议。

### 2.2 明确非目标

1. 本计划书不替代正式代码审计报告，不直接给出仓库当前已确认的问题结论。
2. 本计划书不直接触发代码整改、配置变更、测试执行、构建执行或部署执行。
3. 本计划书不引入新的质量平台、扫描平台、规范系统或通用流程引擎。
4. 本计划书不承诺“一次审查解决所有历史问题”，而是强调阶段性审查、风险分级和闭环节奏。

## 三、审查范围与不做范围

### 3.1 纳入审查范围

1. 后端生产代码：接口层、服务层、权限控制、租户隔离、金额/状态流转、异常处理、文件与审批相关链路。
2. 前端代码：页面、组件、状态流、路由权限、请求封装、字典显示、列表/详情/弹窗一致性、构建产物引用。
3. 安全与权限：认证、鉴权、对象级授权、租户边界、敏感配置、默认账号、调试入口、弱口令与泄漏风险。
4. 数据库与迁移：实体与表结构一致性、Flyway 迁移策略、种子数据、幂等性、回滚难度与数据修复风险。
5. 测试与 CI：测试是否能拦住真实问题、是否存在假绿、E2E 状态污染、CI 门禁与构建顺序是否可靠。
6. 运维与发布：Dockerfile、Compose、环境变量、健康检查、备份恢复、发布包是否混入无关文件或遗漏必需文件。
7. 日志可观测与回滚：错误日志是否足够定位、关键链路是否可追踪、回滚入口是否清晰、故障时是否能止损。

### 3.2 明确不做范围

1. 其他智能体私有目录、历史归档禁区和用户未明确授权的外部仓库。
2. 与当前交付无关的纯历史文档美化、旧草稿整理、非生产关键的格式性微调。
3. 没有进入版本管理或不属于当前主仓库交付面的个人临时文件。
4. 仅凭猜测扩展出的“也许有问题”长期规划，不纳入本轮阶段性全量审查承诺。

## 四、审查门禁清单

以下门禁满足任一项失败，即不得宣称该阶段审查“通过”：

1. 存在 P0/P1 级安全、权限、租户隔离、金额一致性或审批状态机问题未入账。
2. 存在构建脚本、测试脚本、CI 作业假绿，导致失败不能真实暴露。
3. 存在发布包污染、必需静态资源缺失、测试状态文件或开发配置进入生产交付面的情况。
4. 存在数据库迁移不可重复执行、不可审计、不可回滚或修改既有已应用 migration 的情况。
5. 存在关键故障场景下日志无法定位、健康检查失真、备份失败被吞掉或回滚入口不清的问题。
6. 审查结论无法对应到文件、路径、类别、风险级别和整改归属，只剩口头判断。

## 五、分阶段执行计划

## 阶段 M1：范围冻结与审查建账

目标：先把本轮到底审什么、按什么分级、交给谁改讲清楚。

执行任务：

1. 冻结本轮审查对象清单：后端、前端、数据库、测试/CI、运维配置、日志可观测与回滚。
2. 建立问题台账字段：模块、文件/路径、问题类别、风险级别、复现/证据、建议整改归属、是否阻塞。
3. 明确风险分级口径：P0、P1、P2、P3。
4. 明确输出分类：实现型整改任务、验收型复核任务、运维型修正任务、审计/归档型任务。

阶段产出：

- 审查范围冻结清单
- 风险分级口径
- 审查问题台账模板

## 阶段 M2：规范与结构审查

目标：先发现最容易反复出现、且返工代价高的代码层共性问题。

执行任务：

1. 代码规范审查：
   - 后端是否遵循既有分层、异常处理、DTO/实体边界、权限校验入口。
   - 前端是否遵循既有页面结构、`lg-*` 布局、公用请求封装和组件复用边界。
   - 是否存在硬编码、重复逻辑、魔法值、隐式协议、临时补丁长期固化。
2. 代码结构与可维护性审查：
   - 是否存在超长文件、超长方法、职责混杂、跨层调用、组件过重、状态链路失控。
   - 是否存在同类问题在多个模块重复出现，但没有收敛到共享入口或明确边界。
   - 是否存在“能局部收口却被扩成大重构”的过度设计迹象。
3. 输出每类问题的最小整改建议，明确应优先修共享入口还是局部页面/局部服务。

阶段产出：

- 代码规范问题清单
- 结构与可维护性问题清单
- 最小整改建议清单

## 阶段 M3：安全、数据与构建发布审查

目标：覆盖高风险但最容易在后期引发返工或上线阻断的问题。

执行任务：

1. 安全权限审查：
   - 默认 dev 能力、调试入口、越权访问、对象级授权缺失、租户过滤绕过、敏感配置暴露。
2. 数据库审查：
   - Flyway 迁移是否规范、是否修改已应用脚本、是否有不可重复执行/不可追踪变更。
   - 数据模型与业务口径是否一致，是否存在跨项目、跨租户、跨状态错写风险。
3. 构建审查：
   - Maven/Vite/脚本/CI 是否真实失败即失败，是否存在 lint、test、build 假绿。
   - 是否存在本地构建与容器构建不一致、必要资源漏拷贝或环境差异。
4. 构建与发布包洁净度审查：
   - 发布包是否夹带测试状态、开发 token、临时文件、无关产物或历史缓存。
   - 发布包是否遗漏 favicon、静态资源、配置模板、健康检查依赖等必需内容。

阶段产出：

- 安全权限问题清单
- 数据库与迁移问题清单
- 构建问题清单
- 发布包洁净度问题清单

## 阶段 M4：测试、运维、可观测与回滚审查

目标：确认项目在“出问题时能不能尽快发现、定位、止损、回退”。

执行任务：

1. 测试与 CI 审查：
   - 测试是否覆盖关键链路，是否存在被缓存 token、预置状态、错误跳过掩盖的问题。
   - CI 是否真正串起类型检查、单测、构建、打包和必要门禁。
2. 运维配置审查：
   - Dockerfile、Compose、`.env.example`、备份脚本、健康检查是否自洽。
   - 是否存在“容器实际上正常但健康检查持续失败”或“失败被吞掉”的情况。
3. 日志可观测审查：
   - 关键异常是否有足够上下文，是否能区分租户、对象、流程阶段和外部依赖失败点。
4. 回滚审查：
   - 配置回滚、镜像回滚、数据库回滚/数据修复、静态资源回退路径是否明确。
   - 若不能直接回滚，是否至少有停用开关、隔离手段或人工补救路径。
5. 内置浏览器自动化测试审查：
   - 该任务归类为验收型任务，默认使用内置浏览器，不默认使用外部浏览器；测试入口优先使用 `http://localhost:5173/api/auth/dev-login?redirect=/dashboard`。
   - 覆盖关键页面：首页驾驶舱、预警中心、审批中心、项目台账、成本管理、采购库存，以及导出、上传、提交、审批等高风险操作。
   - 对关键路径记录访问路径、操作步骤、截图，以及控制台错误、网络错误、接口错误证据。
   - 重点检查白屏、路由权限错误、接口异常无提示、重复提交未拦截、关键表单无法保存、控制台出现影响功能的错误。

阶段产出：

- 测试与 CI 风险清单
- 运维配置问题清单
- 日志可观测问题清单
- 回滚风险与补救建议清单
- 浏览器自动化测试报告

## 阶段 M5：分级派工、复核与归档

目标：让审查真正进入闭环，而不是停留在报告。

执行任务：

1. 按风险级别与任务性质拆分派工单。
2. 对阻塞项明确整改前置、责任归属和复核要求。
3. 对非阻塞项明确进入哪条后续主线、何时复查、何时升级为阻塞。
4. 形成正式审查报告、整改台账和阶段性收口结论。

阶段产出：

- 分级派工清单
- 阻塞项整改前置条件
- 审查报告与后续复核入口

## 六、子智能体模型分配建议

| 任务/角色 | 任务类型 | model | thinking | reason |
| --- | --- | --- | --- | --- |
| 范围冻结与台账建账 | 审计/归档型 | gpt-5.4 | low | 以整理、分类、字段定义为主，判断空间有限 |
| 代码规范与结构可维护性审查 | 审计型 | gpt-5.5 | medium | 需要跨模块识别重复问题、抽象边界和最小整改点 |
| 安全权限与数据库审查 | 审计型 | gpt-5.5 | high | 涉及权限模型、租户隔离、迁移规范、数据一致性和正式风险裁决 |
| 构建、CI 与发布包洁净度审查 | 审计型 | gpt-5.5 | medium | 需要结合脚本、Docker、CI、构建产物判断真假失败与制品边界 |
| 运维配置、可观测与回滚审查 | 运维/审计型 | gpt-5.4 | medium | 以配置一致性、故障定位与回滚路径审查为主 |
| 内置浏览器自动化测试验收 | 验收型 | gpt-5.5 | medium | 需要真实浏览器证据汇总，但不涉及代码实现，重点是页面与高风险操作的通过性裁决 |
| 整改执行 | 实现型 | 按问题升级单独分配 | 按问题升级单独分配 | 不允许把所有整改统一成同档，需按模块与风险重新分档 |
| 复核验收与最终归档 | 验收/归档型 | gpt-5.5 | medium | 需要汇总多类证据并给出通过/不通过、阻塞/非阻塞结论 |

### 6.1 分类说明

- 实现型任务：代码整改、配置修正、脚本修正。
- 验收型任务：整改后复核、通过性裁决、阻塞项复判。
- 运维型任务：构建链路、发布包、健康检查、备份回滚相关核查或修正。
- 审计/归档型任务：范围定义、问题建账、正式报告落盘、阶段性收口。

### 6.2 分档约束

1. 安全权限、数据库迁移、租户隔离审查不得降到 `gpt-5.4 / low`。
2. 建账、归档、证据摘录类任务不得滥用 `gpt-5.5 / high`。
3. 审查任务与整改任务不得默认使用同一档位，必须在派工前重新分档。

## 七、重点审查项

### 7.1 代码规范审查

1. 是否遵循仓库既有后端/前端规范，而不是各自为政。
2. 是否存在通用逻辑散落、命名失真、异常处理不一致、魔法值和重复映射。
3. 是否存在“临时修复长期化”的问题。

### 7.2 代码结构与可维护性审查

1. 超长文件、超长方法、页面组件过重、服务职责混杂。
2. 跨层直接调用、页面直接拼业务规则、Mapper/SQL 绕过统一边界。
3. 同类逻辑未复用现有入口，导致每次修复都重复补丁。

### 7.3 构建审查

1. `lint`/`test`/`build` 是否真实失败即失败。
2. CI 是否存在跳过、缓存污染、前后端构建顺序错乱或本地与容器结果不一致。
3. 打包链路是否覆盖必须验证项。

### 7.4 构建与发布包洁净度审查

1. 发布包是否混入测试状态、开发密钥、历史缓存、无关构建产物、临时日志。
2. 生产镜像/制品是否遗漏必须静态资源、配置模板、依赖工具或健康检查依赖。
3. `.env.example`、Dockerfile、Compose 与实际制品边界是否一致。

### 7.5 安全权限审查

1. 默认账号、dev-login、调试开关、认证绕过。
2. 菜单权限之外的对象级授权、文件访问授权、租户数据隔离。
3. 敏感信息泄露、弱口令、错误默认配置。

### 7.6 数据库审查

1. Flyway migration 是否版本化、可追踪、可重复执行。
2. 是否修改既有已执行 migration。
3. 金额、状态、外键归属、跨项目/跨租户错写风险。

### 7.7 测试与 CI 审查

1. 是否存在假绿、预置 token、跳过真实登录、错误被吞。
2. 是否覆盖关键正反向路径。
3. CI 失败是否能阻断发布。

### 7.8 运维配置、日志可观测与回滚审查

1. 健康检查是否可信，备份脚本是否真失败即失败。
2. 日志是否能支撑定位关键故障。
3. 回滚和数据修复路径是否明确、是否需要额外人工干预。

### 7.9 内置浏览器自动化测试审查

1. 默认使用内置浏览器，入口优先为 `http://localhost:5173/api/auth/dev-login?redirect=/dashboard`，用于跳过登录并直接进入首页驾驶舱。
2. 覆盖关键页面：首页驾驶舱、预警中心、审批中心、项目台账、成本管理、采购库存。
3. 覆盖高风险操作：导出、上传、提交、审批，以及关键表单保存与重复提交拦截。
4. 报告必须包含访问路径、操作步骤、截图、控制台错误、网络错误、接口错误证据，以及每个页面/操作的通过或不通过结论。
5. 任一出现以下情况即判不通过：页面白屏、路由权限错误、接口异常无提示、重复提交未拦截、关键表单无法保存、控制台出现影响功能的错误。

## 八、验收标准

### 8.1 计划通过标准

1. 审查范围、阶段、门禁、分级、派工口径已完整覆盖本主线要求。
2. 计划中明确包含代码规范、代码结构与可维护性、构建审查、构建与发布包洁净度审查。
3. 计划中同时覆盖后端、前端、安全权限、数据库、测试/CI、运维配置、日志可观测与回滚。
4. M4 已明确纳入基于内置浏览器的浏览器自动化测试任务，且写明测试入口、页面范围、高风险操作、证据要求与不通过标准。
5. 每个阶段都能产出可落盘、可派工、可验收的结果，而不是泛化建议。

### 8.2 审查执行通过标准

1. 每个发现的问题都能落到明确路径、明确类别、明确风险级别、明确整改归属。
2. P0/P1 问题必须给出阻塞判定和复核前置条件。
3. 非阻塞问题必须给出进入后续主线或后续阶段复查的去向。
4. 浏览器自动化测试必须产出正式测试报告，至少包含访问路径、操作步骤、截图/控制台错误/网络错误/接口错误证据，以及通过/不通过结论。
5. 最终能形成正式审查报告与整改台账。

### 8.3 不通过标准

1. 审查覆盖不全，缺失关键维度。
2. 只有问题清单，没有风险级别、整改归属和复核要求。
3. 把计划写成长期理想体系，无法直接派工。
4. 以局部抽查替代阶段性全量审查，却没有明确说明抽样边界。
5. 未明确内置浏览器入口、关键页面、高风险操作、证据要求或不通过标准，导致浏览器自动化测试无法直接派工验收。

## 九、风险分级与处理规则

| 级别 | 定义 | 默认处理 |
| --- | --- | --- |
| P0 | 可直接导致越权、跨租户、严重数据错误、发布阻断、不可回滚 | 立即阻塞，优先派工整改，整改后必须复核 |
| P1 | 高概率造成生产风险、财务口径错误、关键功能失真、构建/发布假绿 | 视同阻塞上线前项，需在近期阶段内完成整改 |
| P2 | 会持续放大维护成本、复发概率高、但短期不直接阻断发布 | 入账并安排后续主线或阶段整改 |
| P3 | 低风险规范性问题或局部一致性问题 | 可批量处理，不单独阻塞 |

处理规则：

1. P0/P1 必须有复核人和复核结论。
2. 同类 P2 若跨多个模块重复出现，应升级为结构性问题处理，不得长期按单点问题拆散。
3. P3 不得挤占高风险整改节奏，但应统一入账，避免无限积压。
