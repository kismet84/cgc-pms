# 产品价值导向的代码审查方案

## 目标

审查不是打分，是**保护用户的数据和业务**。每一项发现必须回答：**如果这个问题不修，用户的哪个业务会出什么事？**

## 审查维度（按用户影响排序）

### P0 — 数据安全与业务阻断

**审查什么：**
- **金额计算正确性**：成本汇总、收入确认、预付款、质保金、结算金额 — 任何一处算错，财务报表全错
- **并发写入安全**：合同变更、付款写回、成本生成的并发场景 — 数据覆盖、重复扣款、超付
- **审批闭环完整性**：审批通过后是否真正执行了业务动作（生成成本、更新状态、触发下游）
- **租户数据隔离**：跨租户数据泄露 = 安全事故

**审查方法：**
- 追踪每一条金额字段从源头（合同金额）到终点（成本汇总/财务报表）的全链路
- 找出所有 `@Transactional` + 写操作的并发窗口
- 画出 11 个 WorkflowBusinessHandler 的 onApproved 调用链，检查是否有"审批通过但什么都没发生"的 handler
- 检查所有 Mapper 查询的 tenant_id 条件完整性

### P1 — 业务流程可用性

**审查什么：**
- **工作流可用性**：提交→审批→驳回→撤回 链路是否畅通，每个节点都有审批人吗
- **编号生成防重**：合同编号、变更编号、发票编号 — 重复编号会导致业务无法继续
- **状态流转完整性**：DRAFT→PENDING→APPROVED→REJECTED→WITHDRAWN 每个转换是否合法
- **硬删除 vs 软删除一致性**：物理删除了但关联数据还在 = 孤儿数据

**审查方法：**
- 画出所有实体的状态机，检查是否有不可达状态或死循环
- 检查编号生成策略（CodeGenerationService + 各模块的 selectLastCodeByPrefix）
- 检查 @TableLogic 覆盖与数据库唯一约束的对应关系

### P2 — 用户体验与数据可追溯

**审查什么：**
- **错误提示有没有用**：用户看到 "SYSTEM_ERROR" 还是 "合同金额不能超过项目预算"
- **操作日志完整性**：关键操作（删除、审批、金额变更）有没有 @OperationLog
- **数据可追溯**：cost_item 的 sourceType/sourceId 能反向追溯到源头吗
- **前端防呆**：表单提交后按钮禁用、加载状态、重复点击防护

**审查方法：**
- 遍历 GlobalExceptionHandler 和所有 BusinessException 的 code/message
- 检查 @OperationLog 注解覆盖范围
- 检查前后端交互中的 loading 状态和防重复提交

## 审查输出标准

每一项发现必须包含：

```
### 发现编号 — 严重级别

**用户影响**：[用一句话说清楚用户会遇到什么问题]

**触发条件**：[什么操作/数据组合会触发]

**技术定位**：文件:行号

**证据**：[代码片段或调用链]

**修复建议**：[最简修复方案]
```
