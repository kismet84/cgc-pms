## 2. 故障根因分析

### 2.1 直接原因

**后端 `CostSubjectService.buildTreeNode()` 方法因数据循环引用导致 StackOverflowError（栈溢出），抛出 HTTP 500 错误。**

### 2.2 根本原因

#### 2.2.1 代码缺陷 —— 缺少环检测

`CostSubjectService.java`（第 54-73 行）在递归构建科目树时，没有对循环引用做保护性检测：

```java
private CostSubjectTreeNodeVO buildTreeNode(CostSubject subject, 
        Map<Long, List<CostSubject>> parentMap) {
    // ...
    List<CostSubject> children = parentMap.getOrDefault(subject.getId(), new ArrayList<>());
    node.setChildren(children.stream()
            .map(child -> buildTreeNode(child, parentMap))  // ← 无环检测，无限递归
            .collect(Collectors.toList()));
    return node;
}
```

#### 2.2.2 数据缺陷 —— 数据库存在循环引用

`cost_subject` 表中的以下三条记录构成了**循环引用链**：

```
┌─────────────────────────────────────────────────────┐
│                   循环引用链条                          │
│                                                       │
│  id=0     (subject_code='5001.01', parent_id=900001)  │
│    ↑                                                    │
│    └── id=900001  (subject_code='5401', parent_id=0)    │
│          ↑                                               │
│          └── id=900000  (subject_code='COST_ROOT',       │
│                          parent_id=0)                    │
│                                                         │
│  parent_id=0 的根节点有：900001                          │
│  → 查找 900001 的子节点：id=0                            │
│  → 查找 id=0 的子节点（parent_id=0）：又回到 900001      │
│  → 无限循环！                                            │
└─────────────────────────────────────────────────────┘
```

其中关键异常数据：
- **id=0, parent_id=900001** — 这是一条主键为 0 的僵尸数据，其 parent_id 指向了一个常规的根节点 900001

## 3. 故障复现步骤

1. 打开后端日志：`docker logs cgc-pms-backend`
2. 访问前端页面 `/dashboard`
3. 当仪表盘加载成本科目树时，触发 `getTree()` 方法
4. `buildTreeNode()` 从 parent_id=0 的根节点（id=900001）开始递归
5. 找到其子节点 id=0（parent_id=900001），再找 id=0 的子节点（parent_id=0）
6. 由于 parentMap 中 parent_id=0 包含 900001，再次回到 900001
7. **形成死循环，栈溢出报错**

---

## 4. 修复方案

### 4.1 紧急修复（已执行）

**删除造成循环引用的异常数据：**

```sql
DELETE FROM cost_subject WHERE id = 0;
```

> ⚠️ 本条数据（id=0, subject_code='5001.01'）经审查确认是早期迁移脚本产生的孤立僵尸数据，无任何业务关联记录，删除安全。

### 4.2 长期修复 —— 代码加固

**在 `buildTreeNode()` 中添加 visited set 环检测，防止任何循环引用导致栈溢出：**

```java
// 新增带 visited 集合的重载方法
private CostSubjectTreeNodeVO buildTreeNode(CostSubject subject, 
        Map<Long, List<CostSubject>> parentMap, 
        Set<Long> visited) {
    if (visited.contains(subject.getId())) {
        log.warn("检测到循环引用：subject_id={}，跳过以避免无限递归", subject.getId());
        return null;
    }
    visited.add(subject.getId());
    
    CostSubjectTreeNodeVO node = new CostSubjectTreeNodeVO();
    node.setId(subject.getId() != null ? subject.getId().toString() : null);
    // ... 其余字段赋值 ...
    
    List<CostSubject> children = parentMap.getOrDefault(subject.getId(), new ArrayList<>());
    node.setChildren(children.stream()
            .map(child -> buildTreeNode(child, parentMap, new HashSet<>(visited)))
            .filter(Objects::nonNull)
            .collect(Collectors.toList()));
    
    return node;
}
```

同时建议添加数据库约束级别的保护：
- **应用层约束：** Service 层的 create/update 方法增加 `parent_id` 合法性校验，防止设置为 `id=0` 或自身 id
