# 02 系统架构

## 架构形态

模块化单体架构：

```
Vue 3 管理后台
  → Nginx / Vite Dev Proxy
  → Spring Boot API (/api)
  → MyBatis-Plus
  → MySQL / H2
  → Redis / MinIO
```

> 当前阶段采用模块化单体是正确选择——业务模块多、事务边界紧、审批和成本生成需要强一致。后续团队和业务规模成熟后再考虑拆分。

## 后端分层

```
Controller  →  参数绑定、权限注解、统一响应
    ↓
Service     →  业务规则、事务、状态流转、跨表写入
    ↓
Mapper      →  SQL、MyBatis-Plus 查询、批量操作
    ↓
Database    →  MySQL / H2
```

规则：

1. Controller 不写业务流程逻辑
2. 多表写操作放在同一 `@Transactional` Service 方法内
3. Mapper 不处理权限和业务状态判断
4. Entity 映射数据库表，VO/DTO 用于接口输出和复杂请求
5. 跨模块状态变化集中到 Service 层，避免 Controller 串联多个业务写操作

## 数据架构

| 组件 | 说明 |
|------|------|
| MySQL 8.0 | 生产/开发默认数据库 |
| H2（MySQL 模式） | local/test 使用，迁移在 `db/migration-h2/` |
| Flyway | 表结构版本管理 |
| MyBatis-Plus | 雪花 ID、逻辑删除、分页、基础 CRUD |
| Redis | 缓存、Session、SSE 发布订阅 |
| MinIO | 文件存储（合同附件、发票、验收等） |

关键约定：

- 主键：`BIGINT` + MyBatis-Plus `ASSIGN_ID`（雪花算法）
- JSON 输出中的 ID 必须为字符串（避免 JS 精度丢失）
- 金额：数据库 `DECIMAL(18,2)`，Java `BigDecimal`，JSON 字符串
- 逻辑删除字段：`deleted_flag`（0 未删，1 已删）

## 认证与会话

JWT + HttpOnly Cookie：

- 前端不保存明文 Token
- Cookie 自动携带（`withCredentials`）
- 后端通过 `UserContext` 获取当前用户、租户、角色
- Axios 统一处理 401 刷新和登录失效跳转

## 部署架构

开发环境使用 Docker Compose 全栈容器化（`deploy/docker-compose.dev.yml`）。

生产环境：

```
Browser → Nginx HTTPS → Spring Boot → MySQL / Redis / MinIO
```

端口映射见 `deploy/docker-compose.prod.yml`。
