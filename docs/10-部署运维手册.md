## 本地开发

### 全容器化（推荐）

```bash
scripts/start-dev.bat
```

启动 5 个容器：MySQL、Redis、MinIO、Backend、Frontend。宿主机无需 JDK/Node。

### 手动启动

```bash
# 基础设施
cd deploy && docker compose up -d

# 后端
cd backend
./mvnw spring-boot:run -Dspring-boot.run.profiles=dev

# 前端
cd frontend-admin
pnpm install && pnpm dev
```

## 生产部署

```bash
cd deploy
cp .env.example .env
# 编辑 .env：修改密码、JWT secret、CORS 域名、MinIO 密钥
# 生产机补充：chmod 600 .env && chown root:root .env
mkdir -p ssl
docker compose -f docker-compose.prod.yml up -d
```

生产 `.env` 必须修改：数据库密码、Redis 密码、MinIO 密钥、JWT secret、CORS 域名、HTTPS 证书路径。

### 生产密钥文件约束

1. `deploy/.env` 是**本地/目标主机专用文件**，只能从 `deploy/.env.example` 初始化，禁止从聊天记录、仓库快照、压缩包或旧主机直接复制历史密钥。
2. `deploy/.env`、`deploy/ssl/server.key` 当前已被 `.gitignore` 命中，但这只能防止**再次提交**，不能替代密钥轮换。
3. 如任一密钥曾离开目标主机或被误传到制品、工单、CI 日志、聊天工具，必须按“已泄露”处理并轮换：`JWT_SECRET`、`JASYPT_ENCRYPTOR_PASSWORD`、`MYSQL_ROOT_PASSWORD`、`MYSQL_PASSWORD`、`REDIS_PASSWORD`、`MINIO_ROOT_PASSWORD`、TLS 证书/私钥。
4. 建议把真实生产密钥放在主机级 secret 管理（systemd `EnvironmentFile`、CI/CD secret、Vault/KMS），仓库中仅保留 `deploy/.env.example` 模板。

### Git 历史泄露应急核查

当前仓库只读核查建议至少执行一次：

```bash
git log --all -- deploy/.env deploy/ssl/server.key deploy/ssl/server.crt
git rev-list --all --objects | grep -E "deploy/\\.env$|deploy/ssl/server\\.(key|crt)$"
git check-ignore -v deploy/.env deploy/ssl/server.key deploy/ssl/server.crt
```

处理口径：

1. 当前可达历史**查不到**敏感文件，不等于所有分叉仓库、旧 clone、CI artifact、制品缓存都安全。
2. 当前可达历史**查得到**敏感文件时，本仓库内禁止直接重写历史；应先完成密钥轮换，再由代码托管平台管理员处理缓存、fork、release artifact、镜像层与备份清理。
3. 无论历史核查结果如何，只要旧审计或事件记录表明存在泄露嫌疑，都要保留轮换记录和平台侧处置记录。

### 备份调度（systemd timer）

`scripts/backup-scheduler.sh` 统一调度 MySQL + MinIO 每日备份，可通过 systemd timer 或 cron 定时执行。

**systemd timer 配置**（推荐）：

创建 `/etc/systemd/system/cgc-pms-backup.service`：

```ini
[Unit]
Description=CGC-PMS Daily Backup
Requires=docker.service
After=docker.service

[Service]
Type=oneshot
User=root
EnvironmentFile=/opt/cgc-pms/deploy/.env
ExecStart=/opt/cgc-pms/scripts/backup-scheduler.sh all
StandardOutput=append:/var/log/cgc-pms-backup.log
StandardError=append:/var/log/cgc-pms-backup.log
```

创建 `/etc/systemd/system/cgc-pms-backup.timer`：

```ini
[Unit]
Description=CGC-PMS Daily Backup Timer
Requires=cgc-pms-backup.service

[Timer]
OnCalendar=daily
Persistent=true
RandomizedDelaySec=300

[Install]
WantedBy=timers.target
```

启用并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now cgc-pms-backup.timer

# 查看状态
sudo systemctl status cgc-pms-backup.timer
sudo systemctl list-timers --all | grep cgc-pms

# 手动触发一次
sudo systemctl start cgc-pms-backup.service
```

**cron 替代方案**（不推荐，但可用于简单环境）：

```bash
# crontab -e
0 2 * * * /opt/cgc-pms/scripts/backup-scheduler.sh all >> /var/log/cgc-pms-backup.log 2>&1
```

### MySQL binlog 与 PITR（Point-In-Time Recovery）

启用 MySQL binlog 可以在全量备份基础上实现任意时间点恢复。

**推荐配置**：

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| `log-bin` | `mysql-bin` | binlog 文件名前缀 |
| `binlog-format` | `ROW` | 行级复制格式，最安全 |
| `expire_logs_days` | `7` | binlog 保留 7 天 |
| `binlog-row-image` | `FULL` | 记录完整行数据（含前后镜像） |
| `sync-binlog` | `1` | 每次事务刷盘（生产必须） |

**Docker Compose 启用方式**（`deploy/docker-compose.prod.yml` 或 `deploy/docker-compose.dev.yml`）：

```yaml
services:
  mysql:
    image: mysql:8.0
    command: >
      --log-bin=mysql-bin
      --binlog-format=ROW
      --expire-logs-days=7
      --binlog-row-image=FULL
      --sync-binlog=1
```

**PITR 恢复流程**：

```bash
# 1. 停止应用
docker stop cgc-pms-backend-dev

# 2. 恢复全量备份
./scripts/restore-mysql.sh /opt/cgc-pms/backups/mysql/cgc_pms_full_20260620_020000.sql.gz

# 3. 应用 binlog 到指定时间点
docker exec -i cgc-pms-mysql mysqlbinlog \
  --start-datetime="2026-06-20 02:00:00" \
  --stop-datetime="2026-06-20 14:30:00" \
  /var/lib/mysql/mysql-bin.* \
  | docker exec -i cgc-pms-mysql mysql -u root -p"${MYSQL_PASSWORD}" cgc_pms

# 4. 启动应用
docker start cgc-pms-backend-dev
```

### MySQL 恢复

使用 `scripts/restore-mysql.sh` 从 mysqldump 备份恢复数据库：

```bash
# 查看用法
./scripts/restore-mysql.sh

# 恢复压缩备份
export MYSQL_PASSWORD="$(grep MYSQL_ROOT_PASSWORD deploy/.env | cut -d= -f2)"
./scripts/restore-mysql.sh /opt/cgc-pms/backups/mysql/cgc_pms_full_20260620_020000.sql.gz

# 恢复明文 SQL 备份
./scripts/restore-mysql.sh /opt/cgc-pms/backups/mysql/cgc_pms_full_20260620_020000.sql
```

恢复后 Flyway 会在下次后端启动时自动执行迁移，无需手动干预。重启后端并验证健康检查：

```bash
docker restart cgc-pms-backend-dev
curl http://localhost:8080/api/actuator/health
```

### MinIO 备份注意事项

`mc mirror` 备份 MinIO 时，**必须使用 `--overwrite` 而不是 `--remove`**：

| 参数 | 行为 | 风险 |
|------|------|------|
| `--overwrite` | 覆盖目标中已存在的同名文件 | ✅ 安全 — 不会删除额外文件 |
| `--remove` | 删除目标中存在但源中不存在的文件 | ❌ 危险 — 如果源端误删或配置错误，目标文件会被永久删除 |

`scripts/backup-minio-mirror.sh` 已采用 `--overwrite`（不含 `--remove`）。如需清理目标端过期文件，使用 MinIO bucket 生命周期策略（`mc ilm`）或独立的过期清理脚本。

## MinIO 健康检查

后端提供了 Actuator HealthIndicator 用于 MinIO 连通性监控。

健康检查端点：`GET /api/actuator/health`

MinIO 组件响应示例：

```json
{
  "status": "UP",
  "components": {
    "minio": {
      "status": "UP",
      "details": {
        "endpoint": "reachable"
      }
    }
  }
}
```

故障时：
```json
{
  "status": "DOWN",
  "components": {
    "minio": {
      "status": "DOWN",
      "details": {
        "error": "IOException: Connection refused"
      }
    }
  }
}
```

**注意**：当 `minio.enabled=false` 时，`MinioHealthIndicator` Bean 不会被创建（`@ConditionalOnBean(MinioClient.class)`），`/actuator/health` 中不包含 `minio` 组件。

### MySQL 备份与恢复

#### 全量备份

使用 `scripts/mysql-backup.ps1` 执行 MySQL 全量备份（通过 Docker `mysqldump`，无需本地安装 MySQL 客户端）：

```powershell
# 设置环境变量
$env:MYSQL_ROOT_PASSWORD = "your-root-password"

# 执行备份
powershell -ExecutionPolicy Bypass -File scripts\mysql-backup.ps1

# 指定备份目录
powershell -ExecutionPolicy Bypass -File scripts\mysql-backup.ps1 -BackupDir "D:\backups\mysql"

# 自定义保留天数（默认 7 天）
powershell -ExecutionPolicy Bypass -File scripts\mysql-backup.ps1 -RetentionDays 30
```

备份命名规则：`{database}_full_{yyyyMMdd_HHmmss}.sql.gz`

#### 恢复到测试库

```powershell
# 恢复到默认测试库 cgc_pms_restore_test（安全：不覆盖生产库）
powershell -ExecutionPolicy Bypass -File scripts\mysql-restore.ps1 -BackupFile "backups\mysql\cgc_pms_full_20260624_020000.sql.gz"

# 恢复到指定数据库（生产恢复需显式输入库名确认）
powershell -ExecutionPolicy Bypass -File scripts\mysql-restore.ps1 -BackupFile "backup.sql.gz" -TargetDatabase "cgc_pms"
```

恢复后通过 `SELECT COUNT(*) FROM information_schema.tables` 做最小健康校验。

#### 备份调度（Windows Task Scheduler）

创建每日备份任务：

```powershell
$action = New-ScheduledTaskAction -Execute 'pwsh.exe' `
  -Argument '-ExecutionPolicy Bypass -File D:\projects-test\cgc-pms\scripts\mysql-backup.ps1 -BackupDir D:\backups\mysql'
$trigger = New-ScheduledTaskTrigger -Daily -At 2am
Register-ScheduledTask -TaskName 'CGC-PMS-MySQL-Backup' -Action $action -Trigger $trigger -Description 'Daily MySQL backup'
```

Linux 环境参考上述 systemd timer 章节。

## 常见故障

| 症状 | 常见原因 | 处理 |
|------|----------|------|
| 接口 404 `No static resource` | 漏了 `/api` | 检查路径 |
| 页面 401 | Token 过期 | 重新登录 |
| 页面 403 | 权限不足 | 检查角色/权限码 |
| Duplicate entry | 逻辑删除唯一索引冲突 | 检查索引含 `deleted_flag` |
| 目标成本不存在 | Long ID 精度丢失 | ID JSON 必须字符串 |
| Docker 重启后接口失败 | 服务未完全启动 | 等待 ~2 分钟后重试 |
| Flyway checksum mismatch | 迁移脚本被修改 | 不修改已执行脚本 |
| 退出登录后 Token 仍可用 | Redis 不可用，黑名单未生效 | 检查 Redis 连接，见下方 |

## TokenBlacklistService 与 Redis 强依赖

### 概述

`TokenBlacklistService` 负责在用户退出登录时将 JWT 令牌加入黑名单（Redis key，TTL = 令牌剩余有效期），防止已注销的令牌被重复使用。**该功能强依赖 Redis**。

### 依赖关系

| 场景 | Redis 状态 | TokenBlacklistService | 行为 |
|------|-----------|----------------------|------|
| 正常 | 可用 | 已注入 | 黑名单写入生效，已注销令牌被拒绝 |
| Redis 未配置/不可用 | 不可用 | Bean 未创建（`@ConditionalOnBean`） | 过滤器检测到 Service 为空，日志输出 `BLACKLIST_UNAVAILABLE`，请求正常放行，**退出登录保护失效** |
| Redis 运行时故障 | 可用→不可达 | Bean 存在但连接失败 | 写入黑名单失败日志 `TOKEN_BLACKLIST_WRITE_FAILED`；检查黑名单时 fail-close（拒绝令牌），日志 `TOKEN_BLACKLIST_CHECK_FAILED` |

### 关键点

- **Redis 是 Token 黑名单的必需依赖，不是可选项。** 虽然代码使用了 `@ConditionalOnBean` + `ObjectProvider.getIfAvailable()` 的"可选注入"模式保证系统在无 Redis 时不崩溃，但这只是一个降级容错设计，**不应视为生产环境的正常运行状态**。
- 如果生产环境出现 `BLACKLIST_UNAVAILABLE` 日志，意味着用户退出登录后 Token 仍然有效，任何持有该 Token 的人都可以继续访问系统，**退出登录保护已完全失效**。
- 同样，Redis 运行时连接故障（`TOKEN_BLACKLIST_WRITE_FAILED` / `TOKEN_BLACKLIST_CHECK_FAILED`）也会导致黑名单功能异常。

### 监控建议

**必须监控以下日志模式**（建议配置告警，阈值 > 0 即触发）：

| 日志模式 | 级别 | 含义 | 严重程度 |
|----------|------|------|----------|
| `BLACKLIST_UNAVAILABLE` | WARN | TokenBlacklistService Bean 未创建，Redis 未配置或不可用 | 🔴 严重 |
| `TOKEN_BLACKLIST_WRITE_FAILED` | WARN | Redis 连接故障，黑名单写入丢失 | 🟡 警告 |
| `TOKEN_BLACKLIST_CHECK_FAILED` | WARN | Redis 连接故障，黑名单检查失败（fail-close，拒绝访问） | 🟡 警告 |
| `BLACKLISTED_TOKEN` | WARN | 检测到已黑名单令牌尝试访问 | ℹ️ 安全事件 |

**生产环境检查清单**：
1. Redis 服务已部署并正常运行
2. 后端 `application-prod.yml` 中 Redis 连接配置正确，`spring.data.redis` 完整
3. 启动日志中 `TokenBlacklistService` 被正常加载（无 `BLACKLIST_UNAVAILABLE`）
4. 运维监控系统已配置上述 4 个日志模式的告警规则
