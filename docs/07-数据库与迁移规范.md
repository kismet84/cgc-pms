# 07 数据库与迁移规范

## 角色入口

主负责人：数据库与迁移负责人。补充阅读：[09 测试规范](09-测试规范.md)。任何数据库结构变化必须新增版本化 migration，不修改已应用脚本。

## 环境对应

| 环境 | 数据库 | 迁移目录 |
|------|--------|----------|
| dev / prod | MySQL 8.0 | `backend/src/main/resources/db/migration/` |
| local / test | H2 MySQL 模式 | `backend/src/main/resources/db/migration-h2/` |

## Flyway 规则

1. 已执行的 `V*.sql` 不得修改
2. 新结构/数据修复创建新的 `V{next}__description.sql`
3. 文件编码 UTF-8
4. MySQL 和 H2 差异大时分别维护两套迁移
5. 生产环境 `validate-on-migrate=true`
6. 使用 `scripts/check-flyway-immutability.sh`（或 `.bat`）在提交前检查迁移文件不可变性

## 主键与审计字段

```sql
id BIGINT NOT NULL PRIMARY KEY
```

通用审计字段（按项目实际情况）：

| 字段 | 说明 |
|------|------|
| `tenant_id` | 租户 ID |
| `created_by` | 创建人 |
| `created_at` | 创建时间 |
| `updated_by` | 更新人 |
| `updated_at` | 更新时间 |
| `deleted_flag` | 0 未删，1 已删 |
| `remark` | 备注 |

## 逻辑删除与唯一索引

推荐唯一索引含 `deleted_flag`：

```sql
UNIQUE KEY uk_tenant_code (tenant_id, code, deleted_flag)
```

允许删除后复用编码。

## 字段类型

| 场景 | 类型 |
|------|------|
| 金额 | `DECIMAL(18,2)` |
| 状态/类型 | `VARCHAR(32)` |
| 备注 | `VARCHAR(500)` |
| 长文本 | `TEXT` |
| JSON 结构 | `JSON`（MySQL 8） |

## 索引规范

| 场景 | 建议索引 |
|------|----------|
| 租户查询 | `(tenant_id)` 或组合业务键 |
| 项目列表 | `(tenant_id, project_id)` |
| 合同业务 | `(tenant_id, contract_id)` |
| 审批任务 | `(tenant_id, approver_id, task_status)` |
| 来源幂等 | `(tenant_id, source_type, source_id)` |

不盲目加过多单列索引。高频列表查询必须有匹配索引。

## 禁止事项

1. 生产手工改表不补迁移
2. 修改已执行迁移脚本
3. 迁移中写入乱码
4. 浮点类型存储金额
5. 无 WHERE 条件的批量 update/delete
6. 绕过租户字段全表查询

### 预提交钩子：Flyway 不可变性检查

项目提供了两个预提交脚本，用于警告已提交迁移文件的修改：

| 脚本 | 平台 | 用法 |
|------|------|------|
| `scripts/check-flyway-immutability.sh` | Linux/macOS/Git Bash | `bash scripts/check-flyway-immutability.sh` |
| `scripts/check-flyway-immutability.bat` | Windows CMD | `scripts\check-flyway-immutability.bat` |

脚本检查 `git diff --cached --diff-filter=M` 中 `backend/src/main/resources/db/migration/V*.sql` 文件，发现已提交迁移被修改时打印警告，**但不阻止提交**（退出码 0）。

> 建议：在 CI 流程或本地 `.git/hooks/pre-commit` 中集成此脚本。

### 未包含 `deleted_flag` 的唯一约束清单（R-31）

以下 8 张公共/字典表建有唯一约束，但约束中**不包含 `deleted_flag`**（或表本身无软删字段）。
删除记录后无法用相同键值重建，需评估是否需要软删唯一性：

| # | 表名 | 模块 | 唯一约束 | 约束列 | 是否有 deleted_flag 列 | 说明 |
|---|------|------|----------|--------|------------------------|------|
| 1 | `sys_user_role` | 系统 | `uk_sys_user_role` | `(user_id, role_id)` | ❌ 无 | 用户-角色关联表，物理删除即可 |
| 2 | `sys_role_menu` | 系统 | `uk_sys_role_menu` | `(role_id, menu_id)` | ❌ 无 | 角色-菜单关联表，物理删除即可 |
| 3 | `sys_dict_type` | 系统 | `uk_sys_dict_type_code` | `(tenant_id, dict_code)` | ❌ 无 | 字典类型表，通过 `status` 字段控制启用/禁用 |
| 4 | `sys_dict_data` | 系统 | `uk_sys_dict_data` | `(dict_type_id, dict_value)` | ❌ 无 | 字典数据表，通过 `status` 字段控制启用/禁用 |
| 5 | `wf_template_node` | 审批 | `uk_wf_template_node_code` | `(template_id, node_code)` | ✅ 有（未用） | 模板节点表，有 `deleted_flag` 列但未包含在唯一约束中 |
| 6 | `wf_idempotency` | 审批 | `uk_wf_idempotency` | `(tenant_id, user_id, idempotency_key)` | ❌ 无 | 幂等表，通过 `expired_at` 过期清理，无需软删 |
| 7 | `pm_project_member` | 项目 | `uk_ppm_project_user` | `(project_id, user_id)` | ✅ 有（未用） | 项目成员表，通过 `status` 字段控制在职/离职 |
| 8 | `cost_item` | 成本 | `uk_cost_source_item` | `(source_type, source_id, source_item_id, cost_type)` | ✅ 有（未用） | V4 原约束不含 `deleted_flag`；V87 已替换为含 `deleted_flag` 的新约束 `uk_cost_source` |

> **建议**：
> - 关联表（#1、#2、#6）采用物理删除或过期清理，无需软删唯一性
> - 字典表（#3、#4）通过 `status` 字段管理而非软删，当前方案已合理
> - 业务表（#5、#7、#8）已有 `deleted_flag` 列，建议将约束改为包含 `deleted_flag` 或迁移至 `deleted_token`

### H2 与 MySQL 迁移不对称

- MySQL 迁移链: 89 个版本
- H2 迁移链: 82 个 SQL 迁移 + 5 个 Java 迁移
- V88 (Token-Based 唯一约束) 仅在 MySQL 侧存在
- 本地开发使用 H2 profile 时，软删行为与生产环境不一致
