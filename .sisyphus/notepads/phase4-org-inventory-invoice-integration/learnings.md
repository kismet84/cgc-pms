# Learnings - Task 15: Purchase Request Backend

## V35 Column Naming Pattern
- V35 inventory tables use `created_time`/`updated_time` (not `created_at`/`updated_at`)
- Entities extending BaseEntity must:
  1. Add `@TableField(value = "created_time", fill = FieldFill.INSERT)` on `createdTime` field
  2. Add `@TableField(value = "updated_time", fill = FieldFill.INSERT_UPDATE)` on `updatedTime` field
  3. Mark BaseEntity's `createdAt`/`updatedAt` with `@TableField(exist = false)`
- This pattern is used by MatWarehouse entity as well

## Workflow Handler Pattern
- Implement WorkflowBusinessHandler interface
- supportBusinessType() returns constant from WorkflowBusinessTypes
- isCritical() returns true for transactional consistency (exceptions cause rollback)
- onApproved() receives WorkflowContext which contains WfInstance (with businessId)
- NEVER read UserContext in handler callbacks - use context from WorkflowEngine
- Business ID is resolved via instance.getBusinessId()

## H2 Schema Template ID Conflicts
- V39 MySQL migration uses template_id=50010 for PURCHASE_REQUEST
- H2 schema-phase23.sql already used 50010 for VAR_ORDER
- H2 uses business_type for template matching (not template_id), so ID is just a convention
- Used template_id=50012 in H2 to avoid conflict with existing 50010 (VAR_ORDER)

## Service Pattern
- Page queries return PageResult<T> (not IPage<T>) to be consistent with WarehouseService
- VO objects use all String fields for H2+MySQL compatibility
- createdTime/updatedTime are LocalDateTime in entity, formatted to String in VO
- Batch prefetch pattern: collect IDs from records -> selectBatchIds -> build Map -> use in convert
- Tenant isolation: LambdaQueryWrapper.eq(Entity::getTenantId, UserContext.getCurrentTenantId())
- Post-check: selectById then verify tenantId matches for single-record operations

## T27: Inventory Frontend Pages (2026-06-12)

### API Endpoints (Backend)
- Warehouse: GET/POST `/api/inventory/warehouses`, GET/PUT `/api/inventory/warehouses/{id}`, PUT `/api/inventory/warehouses/{id}/status`
- Stock: POST `/api/inventory/stock/in`, POST `/api/inventory/stock/out`, GET `/api/inventory/stock/ledger`
- Purchase Request: GET/POST `/purchase-requests`, GET/PUT/DELETE `/purchase-requests/{id}`, POST `/{id}/submit`, GET `/{id}/items`, POST `/{id}/items/batch`
- Backend VO: All IDs are String (H2+MySQL compatible), dates String

### Frontend Patterns Used
- warehouse.vue: simple CRUD table (add/edit/delete), follows project/partner page pattern
- stock.vue: ledger table with stock balance card + paginated transaction history, warehouse+material filter
- transaction.vue: IN/OUT tabbed form, warehouse select + material select + quantity input
- purchase-request.vue: CRUD table + items editor modal + submitForApproval button (follows purchase/order.vue pattern)

### Key Decisions
- Backend uses `pageNo` for warehouse queries but `pageNum` for purchase-request queries (different controllers)
- Warehouse uses `createdAt`/`updatedAt` in VO; Stock Txn uses `createdTime`/`updatedTime` (V35 naming)
- `ApprovalStatusTag` component reused from contract types (same ApprovalStatus enum values)
- Warehouse code auto-generated by backend; disabled during edit
- Purchase request submit button only visible when approvalStatus === 'DRAFT'
- Stock ledger requires warehouseId; materialId is optional filter
