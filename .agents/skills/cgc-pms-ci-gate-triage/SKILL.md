# cgc-pms CI 门禁分诊

1. 先读仓库根 `AGENTS.override.md`、`AGENTS.md`，不要把所有红灯直接归因为业务代码失败。
2. 先收集最小事实：
   - 工作流名、job、step
   - 分支名、提交号
   - 失败关键词
   - 本地与远端 checks 是否一致
3. 按三类分诊：
   - `工具配置类`：workflow 触发、脚本入口、凭据、工具版本、规则文件加载
   - `环境前置类`：Docker/WSL/数据库/端口/测试数据/等待时间/旧代理
   - `真实质量安全类`：测试、构建、类型、契约、权限、安全、数据一致性
4. 最小处理顺序固定为：先分类 -> 配置修复或环境恢复 -> 再复验 -> 仍失败才进入代码整改或阻塞裁决。
5. 需要核对远端状态时，优先使用真实远端 checks、分支保护规则和最新 workflow 结果，不只看本地缓存状态。
6. 回报时明确：
   - 失败分类
   - 当前是否阻塞
   - 下一步该派给实现型、运维型还是验收型子智能体

## 最小回报骨架

```text
失败任务=
失败步骤=
失败分类=
关键证据=
当前处理=
下一步=
是否阻塞=
```
