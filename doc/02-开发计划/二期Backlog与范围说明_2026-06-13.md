## 1. 财务接口集成

### 1.1 当前状态

| 维度 | 现状 |
|------|------|
| **付款申请** | `pay_request` 表完整，支持 MATERIAL/SUBCONTRACT/EQUIPMENT/LABOR 等多付款类型，审批通过后状态变为 WAIT_PAY |
| **付款回写** | `pay_record` 表存在，`PayRecordService.writeback()` 方法已实现。当前通过 API 手动回写（调用 `/api/pay-records`），无外部财务系统自动推送机制 |
| **发票管理** | `pay_invoice` 表完整，支持创建→登记→核验三态。当前发票数据完全由系统内手工录入，无外部财务系统发票同步 |
| **供应商主数据** | `md_partner` 表管理合作方信息，但无外部供应商主数据校验。当前供应商信息自维护，不校验财务系统 |
| **对账机制** | 无自动对账能力。付款申请 vs 实际付款的差异需人工比对 |

### 1.2 Gap 分析

| Gap | 严重度 | 说明 |
|-----|--------|------|
| **G1.1 付款推送缺失** | P0 | 付款申请审批通过后，缺少向财务系统自动推送付款指令的接口。当前仅更新 `pay_status=WAIT_PAY`，需人工导出再录入财务系统 |
| **G1.2 付款回写单向** | P0 | `pay_record` 仅支持本系统 API 写入。财务系统实际付款后无法自动回写。需要财务系统 → CGC-PMS 的回写通道 |
| **G1.3 发票同步缺失** | P0 | `pay_invoice` 数据完全由本系统手工维护。财务系统开具的发票（发票号码、金额、税额、开票日期）无法自动同步，存在双系统数据不一致风险 |
| **G1.4 供应商校验缺失** | P1 | 新建/编辑 `md_partner` 时不校验财务系统供应商主数据。可能存在合作方已在财务系统停用但仍在本系统生效的风险 |
| **G1.5 对账能力缺失** | P1 | 无自动对账规则引擎。付款申请金额 vs 实际付款金额 vs 发票金额的三方差异无法自动检出和报告 |
| **G1.6 付款状态回写滞后** | P1 | `pay_record.writeback()` 调用后虽触发 `updateContractPaidAmount` 和 `updatePayStatus`，但无财务系统确认的回调通知。当前依赖人工调用 API |

#### 1.3.2 付款回写接口（财务系统 → CGC-PMS）

**接口方向**: 财务系统回调 / CGC-PMS 被动接收  
**触发时机**: 财务系统完成实际付款后  
**接收数据**:

```json
{
  "callbackType": "PAYMENT_WRITEBACK",
  "traceId": "uuid",
  "timestamp": "2026-06-13T15:30:00+08:00",
  "data": {
    "requestCode": "PAY-20260613-001",
    "paymentCode": "FIN-PAY-2026-00123",
    "paymentAmount": "500000.00",
    "paymentDate": "2026-06-13",
    "paymentMethod": "BANK_TRANSFER",
    "bankAccount": "6222021234567890",
    "voucherNo": "PZ-2026-06-00123",
    "sourceSystem": "ERP_V5.2",
    "status": "SUCCESS"
  }
}
```

**CGC-PMS 处理逻辑**:
1. 根据 `requestCode` 查找 `pay_request`
2. 创建 `pay_record` 记录（`sync_status=SYNCED`, `source_system=ERP_V5.2`）
3. 更新 `pay_request.pay_status` → `PAID` / `PARTIAL`
4. 更新 `ct_contract.paid_amount`（合同累计已付金额）
5. 更新 `cost_summary.paid_amount`（成本汇总已付金额）
6. 若金额不匹配，生成对账差异记录

**响应契约**:

```json
{
  "code": "0",
  "message": "WRITEBACK_ACCEPTED",
  "data": {
    "recordId": "2065453640270249986"
  }
}
```

**幂等保障**: 使用 `(requestCode, paymentCode)` 作为唯一键，重复推送返回已存在的 `recordId` 而非报错。

**Backlog Item**: `FEAT-FIN-002` — 实现财务系统付款回写回调接口（含幂等防重、金额差异检测、状态联动）

#### 1.3.3 发票同步接口（财务系统 → CGC-PMS）

**接口方向**: 财务系统推送 / CGC-PMS 被动接收  
**触发时机**: 财务系统开具发票后  
**接收数据**:

```json
{
  "syncType": "INVOICE_SYNC",
  "traceId": "uuid",
  "data": {
    "invoiceNo": "FP-2026-06-00123",
    "invoiceType": "VAT_SPECIAL",
    "invoiceAmount": "500000.00",
    "taxAmount": "65000.00",
    "amountExcludingTax": "435000.00",
    "invoiceDate": "2026-06-13",
    "partnerCode": "P-20001",
    "relatedRequestCode": "PAY-20260613-001",
    "verifyUrl": "https://tax.example.com/verify/xxx",
    "attachmentUrl": "https://erp.example.com/invoice/xxx.pdf"
  }
}
```

**CGC-PMS 处理逻辑**:
1. 根据 `invoiceNo` 检查 `pay_invoice` 是否已存在（幂等）
2. 不存在则创建 `pay_invoice` 记录，`verify_status=PENDING`
3. 若 `relatedRequestCode` 匹配到 `pay_request`，自动关联
4. 若发票金额与付款申请金额不匹配，标记 `verify_status=DISCREPANCY`

**响应契约**:

```json
{
  "code": "0",
  "message": "SYNCED",
  "data": { "invoiceId": "20001" }
}
```

**Backlog Item**: `FEAT-FIN-003` — 实现财务系统发票同步接口（含幂等防重、自动关联付款申请、金额差异标记）

#### 1.3.5 对账异常处理

**对账维度**:

| 对比维度 | 数据源 A | 数据源 B | 差异类型 |
|----------|---------|----------|----------|
| 申请 vs 支付 | `pay_request.approved_amount` | `pay_record.payment_amount` (SUM) | 金额偏差、未支付 |
| 发票 vs 支付 | `pay_invoice.invoice_amount` (SUM) | `pay_record.payment_amount` (SUM) | 发票金额≠支付金额 |
| 发票 vs 申请 | `pay_invoice.invoice_amount` (SUM) | `pay_request.approved_amount` | 已开票≠已审批 |

**自动对账规则**:
1. **金额容差**: 差异 ≤ 0.01 元视为匹配（分币舍入误差）
2. **时间窗口**: 付款后 3 个工作日内发票应到位
3. **预警触发**: 付款后 7 天无发票 → 生成对账异常预警
4. **差异报告**: 按项目+合同+合作方维度汇总差异，支持导出 Excel

**对账差异报告 JSON 结构**:

### 1.4 验收标准

| 编号 | 验收项 | 判定标准 |
|------|--------|----------|
| FIN-V1 | 付款推送 | 付款申请审批通过后，财务系统在 30s 内收到推送数据（正常网络），推送失败触发重试（最多 3 次） |
| FIN-V2 | 付款回写 | 财务系统回写后，`pay_record` 创建成功，`pay_request.pay_status` 正确联动，合同 `paid_amount` 更新 |
| FIN-V3 | 回写幂等 | 同一 `(requestCode, paymentCode)` 重复推送返回已存在的 `recordId`，不重复创建 |
| FIN-V4 | 发票同步 | 财务系统推送发票后，`pay_invoice` 自动创建，自动关联匹配的 `pay_request` |
| FIN-V5 | 供应商校验 | 新建合作方时校验财务系统供应商主数据，已停用供应商拒绝保存 |
| FIN-V6 | 对账差异报告 | 可生成按项目/合同的差异报告，差异项风险等级正确着色（HIGH/MEDIUM/LOW） |
| FIN-V7 | 对账预警 | 付款后 7 天无发票自动生成预警，对账差异 > 合同金额 5% 触发预警 |
| FIN-V8 | 回退兼容 | 财务系统不可达时，系统不崩溃；支持手动回写（保留现有 `PayRecordService.writeback()` 能力） |
| FIN-V9 | 审计追踪 | 所有接口调用记录日志（请求/响应 traceId、时间戳、结果），支持按 traceId 检索 |

## 2. 设备租赁管理

### 2.1 当前状态

| 维度 | 现状 |
|------|------|
| **合同支持** | `ct_contract` 表支持 `contract_type='LEASE'`（租赁合同），可记录设备租赁合同基本信息 |
| **成本归集** | `cost_item.source_type` 枚举预留 `EQ_LEASE` 值，但成本归集策略 `EquipmentLeaseCostStrategy` **未实现** |
| **付款申请** | `pay_request_basis.basis_type` 枚举预留 `EQ_LEASE`，但无可用的设备租赁单据作为付款依据 |
| **数据表** | 无 `eq_*` 前缀表。设备台账、进退场、台班记录、维保记录均未建表 |
| **业务闭环** | 设备租赁在业务主线图中标注但无完整闭环设计。当前无法走通"租赁合同 → 进退场 → 台班 → 成本"链路 |

### 2.2 Gap 分析

| Gap | 严重度 | 说明 |
|-----|--------|------|
| **G2.1 无设备台账** | P0 | 缺少设备主数据表（设备编号、名称、规格、归属、状态）。无法管理租赁设备清单 |
| **G2.2 无进退场记录** | P0 | 缺少设备进场/退场的时间线记录。无法确定设备使用周期和租赁天数 |
| **G2.3 无台班记录** | P0 | 缺少台班工时记录。无法按"工时 × 单价"计算租赁费用，也无法生成租赁成本 |
| **G2.4 无维保管理** | P1 | 缺少设备维修保养记录。维修费用无法归集到项目成本 |
| **G2.5 成本归集未实现** | P0 | `EquipmentLeaseCostStrategy` 未实现。设备租赁费用无法自动归集到 `cost_item` |
| **G2.6 租赁结算缺失** | P1 | 结算模块预留了"租赁结算"类型但无数据来源。无法生成设备租赁结算单 |

#### 2.3.2 成本归集策略

**新增 `EquipmentLeaseCostStrategy`** 实现 `CostGenerationStrategy` 接口:

- **触发事件**: `eq_shift_record` 审批通过（`SHIFT_APPROVED`）
- **source_type**: `EQ_LEASE`
- **source_id**: `eq_lease_record.id`（进退场记录）
- **source_item_id**: `eq_shift_record.id`（台班明细）
- **cost_type**: `EQUIPMENT`
- **cost_subject_id**: 映射到成本科目树中的"机械成本"科目

**维保费用归集**（可选）:
- **触发事件**: `eq_maintenance` 确认且 `charge_to_project=1`
- **source_type**: `EQ_MAINTENANCE`
- **cost_type**: `EQUIPMENT`

| **G5.2 无自动归档索引** | P1 | 文件上传后不会自动归入对应归档目录。需人工在归档目录中手动建立文件引用 |
| **G5.3 无归档清单模板** | P1 | 缺少标准归档清单（按合同类型定义必须归档的资料类型）。不同合同类型（总包/分包/采购/租赁）应归档的资料不同 |
| **G5.4 无资料缺失预警** | P0 | 结算提交前无自动检查"归档清单中哪些资料缺失"。当前依赖人工核对 |
| **G5.5 无归档状态追踪** | P1 | 文件上传后无"已归档"标记。无法区分"已上传但未归档"和"已正式归档" |
| **G5.6 无审计日志检索** | P2 | 缺少专门的审计检索界面。审计人员需要一站式查看合同→执行→成本→付款→结算的完整链路 |

## 6. 附录：跨模块约束

### 6.1 成本来源统一

所有二期新增模块的成本归集必须遵循一期成本体系：

| 模块 | source_type | cost_type | cost_subject |
|------|------------|-----------|-------------|
| 财务接口 | 不生成 cost_item（付款走资金流） | — | — |
| 设备租赁 | `EQ_LEASE` | `EQUIPMENT` | 机械成本 |
| 设备维保 | `EQ_MAINTENANCE` | `EQUIPMENT` | 机械成本 |
| 劳务管理 | `LAB_PAYROLL` | `LABOR` | 人工成本 |
| BI 分析 | 不生成 cost_item（只读分析） | — | — |
| 审计归档 | 不生成 cost_item（文件管理） | — | — |

### 6.2 多租户约束

所有二期新增表必须包含 `tenant_id BIGINT NOT NULL DEFAULT 0` 列，且唯一索引包含 `tenant_id`：

```sql
UNIQUE KEY uk_eq_equipment_code (tenant_id, equipment_code)
UNIQUE KEY uk_lab_team_code (tenant_id, team_code)
UNIQUE KEY uk_lab_attendance (tenant_id, worker_id, attendance_date)
```

### 6.3 审批集成约束

所有需要审批的二期业务模块，必须遵循一期审批模式：

- 实现 `WorkflowBusinessHandler` 接口
- `isCritical()` 返回 `true` 时异常会触发审批回滚
- `onApproved()` 中触发成本生成/状态联动
- `onRejected()` 中恢复业务实体为可编辑状态
- 审批模板在 Flyway 迁移脚本中插入（`INSERT INTO wf_template ...`）

### 6.4 API 契约约束

所有二期 API 遵循一期契约模式：

- **统一前缀**: `/api/v1/`（二期建议统一使用 `/api/v1/` 替代一期混用的 `/api/`）
- **字段命名**: `lowerCamelCase` JSON → `snake_case` DB
- **ID 字段**: 统一字符串类型（避免 JS 大整数精度丢失）
- **金额字段**: 统一字符串类型（BigDecimal 精度保护）
- **响应结构**: `{ code: "0", message: "success", traceId: "...", data: {...} }`
- **分页结构**: `{ code: "0", data: { pageNo, pageSize, total, records: [...] } }`

### 6.5 优先级排序

二期 5 个模块的建议实施顺序：

| 优先级 | 模块 | 理由 |
|--------|------|------|
| **P0-1** | 财务接口集成 | 打通与财务系统的双向数据流，是付款闭环的必要条件。一期付款回写依赖人工，自动化是刚需 |
| **P0-2** | 设备租赁管理 | 填补业务主线缺口。一期业务主线图标注了"设备租赁"但无实现。设备成本是成本构成的重要维度 |
| **P1-3** | 劳务管理 | 填补人工成本归集缺口。一期 `LAB_PAYROLL` 仅预留枚举值。劳务成本是成本构成的另一重要维度 |
| **P1-4** | BI 分析 | 在设备/劳务数据完备后，BI 分析才有充足的数据源。依赖前两个模块的数据输入 |
| **P2-5** | 审计归档 | 在业务数据完整、BI 分析就绪后，归档才有完整内容。相对独立，可与其他模块并行 |
