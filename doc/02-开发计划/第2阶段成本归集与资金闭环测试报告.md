## 5. 已知限制

| 限制 | 影响 | 说明 |
|------|------|------|
| 仅 H2 local profile 验证 | 集成测试仅在 H2 内存库执行, 未在 MySQL 8.0 环境下验证 | MySQL 8.0 全栈验证 (审计项 #4) 延后至生产环境部署阶段 |
| 无前端 E2E 测试 | 前端所有页面仅通过构建验证 (vue-tsc + vite build), 无 Playwright/Cypress 自动化测试 | 前端页面交互逻辑需手动验证 |
| 签证变更未覆盖集成测试 | 测试仅覆盖 9 个核心场景, VAR_ORDER 审批 + COST 方向成本调整未纳入 | VarOrderWorkflowHandler 和 VarOrderCostStrategy 已实现但未在集成测试中验证 |
| 分包计量审批未覆盖 | SubMeasureWorkflowHandler 已实现但未在集成测试中覆盖审批通过→成本生成链路 | 与场景3 验收审批模式完全一致, 风险较低 |
| cost_summary 定时刷新 | cost_summary 表通过定时任务刷新, 集成测试中 cover 但非实时验证 | 存在短期数据不一致窗口 (取决于刷新频率) |
| 无并发测试 | 付款回写、成本生成等操作未验证高并发场景 (如多用户同时回写同一申请) | 乐观锁/唯一键约束提供基本保护, 但并发正确性未经严格验证 |

## 6. 结论

第 2 阶段成本归集与资金闭环全部 9 个全链路集成测试在 H2 local profile 下通过 (Tests run: 9, Failures: 0, Errors: 0)。

**核心能力**:
- ✅ 采购订单全栈 CRUD + 审批闭环 (PURCHASE_ORDER, PurchaseOrderWorkflowHandler, Flyway V13)
- ✅ 物料字典管理 (md_material, 启用/禁用)
- ✅ 材料验收全栈 CRUD + 审批 + 自动成本生成 (MAT_RECEIPT, MaterialReceiptWorkflowHandler, Flyway V14)
- ✅ 分包任务/计量 CRUD + 审批 + 自动成本生成 (SUB_MEASURE, SubMeasureWorkflowHandler)
- ✅ 成本科目树形管理 (cost_subject, 递归树, level 自动计算)
- ✅ 成本台账 (cost_ledger, 来源追溯 + 多维度筛选)
- ✅ 动态成本汇总 (cost_summary, project_id + subject_id 维度聚合)
- ✅ 通用 CostGenerationService 策略模式 (4 种 source_type 自动注册, 幂等机制)
- ✅ 付款申请 CRUD + 金额校验 + 审批闭环 (PAY_REQUEST, PayRequestWorkflowHandler, Flyway V15)
- ✅ 财务回写 pay_record + 合同联动 + cost_summary 联动 (PayRecordWorkflowHandler, PARTIALLY_PAID/PAID 状态机)
- ✅ 签证变更 CRUD + 审批 + 成本调整 (VAR_ORDER, COST direction 过滤, Flyway V17)
- ✅ Flyway 迁移 V12~V20 (8 个迁移脚本, 含 V19 索引修复, V20 ct_contract.cost_generated_flag)
- ✅ 前端: 采购订单/验收单/成本台账/成本汇总/付款申请/签证变更页面

**全链路业务流程**:
```
采购订单(DRAFT) → 审批通过(APPROVED)
     ↓
材料验收(DRAFT) → 提交审批(APPROVING) → 审批通过(APPROVED) → 成本自动生成(cost_item)
     ↓
付款申请(DRAFT) → 提交审批(金额校验) → 审批通过 → 待付款(PENDING)
     ↓
付款回写(pay_record) → PARTIALLY_PAID → PAID
     ├── 合同已付金额联动更新 (contract.paid_amount)
     └── 成本汇总联动刷新 (cost_summary)
```
