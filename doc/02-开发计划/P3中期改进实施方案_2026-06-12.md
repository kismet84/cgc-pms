## P3-1: H2 Schema 自动从 Flyway 生成

### 1.1 现状

**当前 H2 初始化链路（test/local profile）**：

```
Spring Boot 启动
  → spring.flyway.enabled: false     ← Flyway 被禁用
  → spring.sql.init.mode: always      ← 激话 SQL 初始化
  → 加载 db/h2/schema.sql           ← 18 张表（V1~V2 时代）
  → 加载 db/h2/schema-phase23.sql   ← 39 张表（V3~V40 时代）
  → 加载 db/h2/data.sql            ← 种子数据
```

**核心问题**：
- `schema.sql` + `schema-phase23.sql` 共 57 张表，手工维护
- Flyway migration 定义 55 张表，分布在 40 个 V1~V40 脚本中
- 每次新增 migration 必须在 H2 schema 文件中手动同步
- V31 修复 `JSON → TEXT` 时，必须同时在 H2 schema-phase23.sql 中手动改写
- H2 schema 与 MySQL 之间存在已知漂移（如 `wf_idempotency` 缺失 UNIQUE 约束）

**关键发现**：
- 所有 17 个测试类使用 `@ActiveProfiles("local")`，由 `src/test/resources/application-local.yml` 提供 H2 配置
- main 的 `application-local.yml` 实际指向 MySQL（非 H2），H2 仅用于测试
- 无自定义 Flyway Java 配置类，全部通过 YAML 控制

### 1.2 MySQL-Specific 语法阻碍直接 Flyway-on-H2

Flyway 40 个迁移脚本中的 MySQL 专用语法（H2 不兼容）：

| 类别 | 语法 | 出现次数 | H2 MySQL 模式兼容？ |
|------|------|---------|-------------------|
| 表选项 | `ENGINE=InnoDB` | 55 处 | ❌ 不兼容 |
| 表选项 | `DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci` | 55 处 | ❌ 不兼容 |
| 列注释 | `COMMENT '...'` | ~500+ 处 | ❌ H2 2.2.224+ 部分支持 |
| 时间戳 | `ON UPDATE CURRENT_TIMESTAMP` | ~30 处 | ❌ 不兼容 |
| 数据类型 | `JSON` 列类型 | ~15 处 | ❌ H2 用 `TEXT` 替代 |
| 数据类型 | `TINYINT` | ~20 处 | ❌ H2 用 `SMALLINT` |
| 函数 | `JSON_OBJECT(...)` | ~10 处 | ⚠️ H2 2.x 支持但语义有差异 |
| 索引 | `KEY` / `UNIQUE KEY` 语法 | ~40 处 | ⚠️ 部分兼容 |
| 会话 | `SET NAMES utf8mb4` | 3 处 | ❌ 不兼容 |
| 会话 | `SET FOREIGN_KEY_CHECKS = 0/1` | 4 处 | ❌ 不兼容 |
| 插入 | `INSERT IGNORE ... SELECT` | ~20 处 | ⚠️ H2 不支持 IGNORE 关键字 |

**结论**：直接启用 Flyway 到 H2 上不可行——55 个 `ENGINE=InnoDB` 就会全部失败。

### 1.3 目标架构

**方案 A：双路 Flyway（推荐）**

```
Spring Boot 启动
  → 检测 active profile
  → 若 profile=local (H2):
      → Flyway 启用
      → locations: db/migration-h2/    ← H2 兼容的迁移脚本
      → spring.sql.init.mode: never   ← 不再使用 schema.sql
  → 若 profile=dev/prod (MySQL):
      → Flyway 启用（现有逻辑不变）
      → locations: db/migration/
```

**方案 B：Flyway 自动方言适配（备选）**

创建一个 `BeforeMigrate` Java callback，在运行时将 MySQL migration 转换为 H2 兼容 SQL，然后执行。复杂度过高。

**方案 C：Flyway 占位符（否决）**

用 Flyway 占位符 `${{engine}}` 切换 `ENGINE=InnoDB` / `ENGINE=''`，但无法处理 JSON 类型、ON UPDATE、COMMENT 等结构性差异。

### 1.4 实施步骤（方案 A）

#### 第一步：验证可行性——测试单个迁移脚本 H2 兼容性

**目标**：确认 H2 2.2.224+ MySQL 兼容模式实际能容忍多少 MySQL 语法。

```bash
# 创建一个最小 H2 连接，逐个测试每个 migration 脚本
# 重点验证：
#   1. ENGINE=InnoDB 是否被忽略（H2 MySQL mode 已知可忽略）
#   2. COMMENT '...' 是否被接受（H2 2.2.224 新增支持）
#   3. ON UPDATE CURRENT_TIMESTAMP 是否被忽略
#   4. JSON_OBJECT 是否在 H2 中可用
```

**预计结果**：H2 MySQL 模式下 `ENGINE=InnoDB` 和列级 `COMMENT` 可能已被接受，但 `ON UPDATE CURRENT_TIMESTAMP`、`JSON` 类型、表级 `COMMENT`、`SET` 语句肯定失败。

#### 第二步：创建 H2 兼容迁移脚本目录

```
backend/src/main/resources/db/migration-h2/
  ├── V1__init_system_tables.sql
  ├── V2__init_business_tables.sql
  ├── V3__init_workflow_tables.sql
  ├── ...
  └── V40__fix_phase4_permission_codes.sql
```

**生成策略**：
- 批量脚本：对每个 `V*` 文件执行自动转换（Python/shell 脚本）
- 手动修补：V31（JSON→TEXT）、V3（依赖 JSON_OBJECT 的函数）等需人工审查
- 保持版本号一致，确保 Flyway 版本链完整

**自动转换规则**：

| MySQL 原语法 | H2 替换 |
|-------------|---------|
| `ENGINE=InnoDB` | 删除整行 |
| `DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_0900_ai_ci` | 删除 |
| 列级 `COMMENT '...'` | 删除（或替换为 `COMMENT ON COLUMN` 独立语句） |
| 表级 `COMMENT='...'` | 删除 |
| `ON UPDATE CURRENT_TIMESTAMP` | 删除 |
| `JSON` | `TEXT`（如 V31 已示范） |
| `TINYINT` | `SMALLINT` |
| `SET NAMES utf8mb4` | 删除 |
| `SET FOREIGN_KEY_CHECKS = 0/1` | 删除 |
| `INSERT IGNORE INTO` | `INSERT INTO`（配合测试数据幂等性处理） |
| `` `backtick` `` | 保留（H2 支持） |

#### 第三步：配置 Flyway H2 profile

修改 `src/test/resources/application-local.yml`：

```yaml
spring:
  flyway:
    enabled: true              # 从 false → true
    locations: classpath:db/migration-h2  # 指向 H2 兼容脚本
    baseline-on-migrate: true
    baseline-version: 0
    clean-disabled: false      # 测试环境允许 clean
  sql:
    init:
      mode: never              # 不再使用 spring.sql.init
```

删除或归档 `spring.sql.init.schema-locations` 和 `data-locations` 配置。

#### 第四步：迁移种子数据

当前 `db/h2/data.sql` 的种子数据（admin 用户、角色、菜单、演示数据等）作为测试 fixture 而非 schema 的一部分。处理方式：
- 移入 `db/migration-h2/V41__seed_test_data.sql`（或用 `@Sql` 注解在测试类中加载）
- 或者在 `@BeforeEach` 中通过 Repository 插入（推荐，更灵活）

#### 第五步：验证

1. 删除 `db/h2/schema.sql`、`db/h2/schema-phase23.sql`、`db/h2/data.sql`
2. 运行 `./mvnw test -Dspring.profiles.active=local -Dtest=WorkflowEngineIntegrationTest`
3. 确认所有 16 个测试通过（预期 test09 和 test15 因 §5 P1/P2 修复已通过）
4. 运行全量测试 `./mvnw test -Dspring.profiles.active=local`，确认 162/162 通过

### 1.5 风险

| 风险 | 缓解 |
|------|------|
| H2 MySQL 模式不完全兼容 | 第一步验证先行，确认实际阻断点 |
| JSON_OBJECT 插入失败 | V6、V9 的模板数据改用普通 INSERT |
| H2 schema 中存在 Flyway migration 不存在的表 | 检查 schema.sql 和 schema-phase23.sql 中是否有手动添加但 migration 中无对应的表 |
| 新 migration 必须同时维护两份 | 建立 CI 检查：`src/test` 必须能通过 H2 profile，即新 migration 若只改了 MySQL 版本则测试会失败，倒逼同步维护 migration-h2 |

### 1.6 验收标准

- [ ] `db/h2/schema.sql`、`db/h2/schema-phase23.sql`、`db/h2/data.sql` 已删除或归档
- [ ] `db/migration-h2/` 目录存在，含 V1~V40 的 H2 兼容脚本
- [ ] `application-local.yml` 中 `flyway.enabled=true`，`sql.init.mode=never`
- [ ] 全量 162 个测试在 H2 local profile 下通过
- [ ] 新增 migration 时 CI 可检测 H2 兼容性（即不会出现单向漂移）

## P3-2: 统一 sys_notification.biz_id 语义

### 2.1 现状

`NotificationService.create(Long tenantId, Long userId, String title, String content, String bizType, Long bizId)` 的第 6 个参数 `bizId` 在不同调用点传入不同语义的 ID：

| 操作 | 文件:行 | bizId 值 | 语义 |
|------|---------|---------|------|
| SUBMIT | WorkflowEngine.java:124 | `instance.getId()` | **Instance ID** |
| APPROVE | WorkflowEngine.java:176 | `instanceForNotify.getId()` | **Instance ID** |
| REJECT | WorkflowEngine.java:272 | `instance.getId()` | **Instance ID** |
| WITHDRAW | WorkflowEngine.java:304 | `instanceId` | **Instance ID** |
| TRANSFER | WorkflowEngine.java:420 | `task.getBusinessId()` | **Business ID** ❗ |
| ADD SIGN | WorkflowEngine.java:499 | `task.getBusinessId()` | **Business ID** ❗ |
| CC | WfCcService.java:86 | `instance.getBusinessId()` | **Business ID** |
| ALERT | AlertEvaluationService.java:467 | `alert.getId()` | **Alert ID** |

**不一致的地方**：同一个 `WorkflowEngine.java` 内，SUBMIT/APPROVE/REJECT/WITHDRAW 用 instance ID，TRANSFER/ADD_SIGN 用 business ID。

### 2.2 影响面分析

**生产代码**：
- 无任何生产 Service/Controller 按 `bizId` 查询通知
- `NotificationService.getPage()` 仅按 `tenantId + userId + isRead` 过滤
- 数据库有 `idx_sn_biz (biz_type, biz_id)` 索引但从未在生产查询中使用

**前端**：
- `NotificationBell.vue` 接收 `bizId` 但仅存储，不做路由/跳转/判断
- `bizId` 对前端纯展示数据，无业务含义依赖

**测试代码**：
- `WorkflowEngineIntegrationTest.java`：7 处按 `bizId` 断言（修复后）
- `Phase3IntegrationTest.java`：1 处按 `bizId` 断言

**结论**：生产无影响——`bizId` 仅用于测试断言和数据追溯。修改的 blast radius 极低。

### 2.3 推荐方案：统一为 instance ID

**理由**：
1. 修改量最小（仅 TRANSFER 和 ADD_SIGN 两处，从 `task.getBusinessId()` 改为 `instance.getId()`）
2. 与 SUBMIT/APPROVE/REJECT/WITHDRAW 保持一致
3. `instance.getId()` 在 TRANSFER（line 410）和 ADD_SIGN（line 448）处均已加载可用
4. 测试断言只需修改两处，而非四处

## P3-3: 测试数据隔离强化

### 3.1 现状

`WorkflowEngineIntegrationTest` 在项目中的独特性：

| 特征 | WorkflowEngineIntegrationTest | 其他 5 个集成测试 |
|------|------------------------------|-------------------|
| `@TestInstance` | `PER_CLASS` | 默认 `PER_METHOD` |
| 事务管理 | **无 `@Transactional`** | **`@Transactional`（自动回滚）** |
| 数据清理 | `@AfterAll` 批量清理 9 个表 | 无需清理（事务回滚） |
| 测试间依赖 | Tests 1-4 共享 `testInstanceId` | 无依赖 |
| ID 生成 | 共享 `RUN_ID = System.currentTimeMillis()` | 各测试独立 |
| 幂等键清理 | LIKE 模式匹配（脆） | 事务回滚 |

**为什么它不用 @Transactional？**
- Tests 1-4 是顺序依赖链：test01 创建实例 → test02 审批同一个实例 → test03 加签 → test04 驳回
- `@Transactional` 下 Spring 管理的 JPA/MyBatis-Plus 事务会回滚，但 `WorkflowEngine` 内部使用 `jdbcTemplate` 直接操作，事务回滚后跨测试的状态会丢失
- 测试意图是模拟"真实事务提交后"的持久化行为

**RUN_ID 共享的问题**：
- `RUN_ID` 是 `static final`，值在类加载时确定
- 16 个测试用了 21 个 `RUN_ID + X`（`+1` ~ `+21`），无重叠
- 但若未来新增测试，容易重复或越界
- `cleanupTestData` 用固定范围 `RUN_ID+1 ~ RUN_ID+21`，新增测试需手动扩展

### 3.2 目标架构

完全按测试方法隔离：

```
每个 @Test 方法:
  → @BeforeAll / constructor: 生成 per-test 唯一 base ID
  → 测试内所有 business_id = baseId + 1, baseId + 2 ...
  → @AfterEach: DELETE ... WHERE business_id BETWEEN baseId AND baseId+N
  → 无跨测试共享 mutable state
```

### 3.3 推荐方案：保持 PER_CLASS + RUN_ID，但引入类型安全的清理范围

考虑到 tests 1-4 的跨方法依赖是刻意设计的状态机测试（审批流程需要多个步骤），强行拆散会改变测试语义，**不建议**改成 `PER_METHOD`。

**替代方案——结构化清理范围**：

```java
// 将 RUN_ID 范围显式定义为常量，新增测试时一目了然
private static final long RUN_ID = System.currentTimeMillis();
private static final long BID_START = RUN_ID + 1;   // 第一个 business ID
private static final long BID_END   = RUN_ID + 21;  // 最后一个 business ID

// 每个测试声明自己使用的 business ID 范围
// 示例：test15 使用 RUN_ID+15 ~ RUN_ID+19
//       test16 使用 RUN_ID+20 ~ RUN_ID+21

// 幂等键统一用 per-test key 模式，并在 cleanup 中按 business_id 范围清理
// 不再用脆弱的 LIKE 模式
```

同时在清理中统一用 `BID_START` / `BID_END` 而不是写死 `RUN_ID + 1` / `RUN_ID + 21`。

### 3.4 更彻底的方案（长期）

如果未来要完全去除跨测试依赖（tests 1-4）：

```
test01: 创建实例 + 审批 + 加签 → 合并为一个长测试
test02: 创建实例 + 驳回 → 独立
test03: 创建实例 + 撤回 → 独立
test04: 转办 → 独立
...
```

然后所有测试统一改为 `@Transactional`（匹配其他集成测试的模式），无需 `@AfterAll` 清理。

但这种重构会改变测试的组织方式和可读性（一个长测试 vs 4 个独立步骤），需要业务方确认。

### 3.5 实施步骤（保守方案）

#### 步骤 1：定义清理范围常量

```java
private static final long BID_FIRST = RUN_ID + 1;
private static final long BID_LAST  = RUN_ID + 21;
```

替换 `cleanupTestData` 中的硬编码 `startBid = RUN_ID + 1; endBid = RUN_ID + 21;`。

#### 步骤 2：消除幂等键 LIKE 模式清理

当前：
```java
jdbcTemplate.update("DELETE FROM wf_idempotency WHERE idempotency_key LIKE 'tenant-idem-%' OR idempotency_key LIKE 'test13-%' OR idempotency_key LIKE 'test14-%' OR idempotency_key LIKE 'test15-%'");
```

改为：
```java
// 按 business_id 范围清理（覆盖所有通过 workflowEngine 创建的幂等记录）
jdbcTemplate.update("DELETE FROM wf_idempotency WHERE business_id BETWEEN ? AND ?", BID_FIRST, BID_LAST);
```

**前提**：确认所有测试的幂等记录 `business_id` 都在 `BID_FIRST ~ BID_LAST` 范围内。

#### 步骤 3：P3-2 统一 bizId 后简化通知清理

如 §P3-2 步骤 4 所述，统一 instance ID 后通知清理简化为一条子查询。
