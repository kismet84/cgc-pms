## 1. 阶段目标

第 1 阶段打通了"合同 → 审批 → 生效"的第一条业务闭环，并落地了 `WorkflowBusinessHandler` 回调机制与 `cost_item` 锁定成本生成。

第 2 阶段的总目标：

```text
把"业务事实"接入审批引擎，让材料验收、分包计量、签证变更在审批通过的瞬间自动归集成本，
再由成本驱动付款申请，最终由财务回写形成"业务 → 成本 → 资金"的完整闭环。
```

一句话总结：

```text
第 1 阶段证明了"审批通过 → 回调 → 生成业务结果"这个模式可行；
第 2 阶段把这个模式复制到采购、材料、分包、签证、付款五条链路上，并让成本与资金真正流动起来。
```

核心成果验收：

```text
材料验收审批通过 → 自动生成材料成本（MAT_RECEIPT）
分包计量审批通过 → 自动生成分包成本（SUB_MEASURE）
签证变更审批通过 → 自动调整成本/收入（VAR_ORDER）
成本台账可按来源追溯、可按科目/项目汇总
付款申请关联业务依据（验收/计量）并受合同余额校验
财务回写更新付款状态、合同已付金额、动态成本
```


### 4.2 成本生成统一规则（所有链路遵守）

```text
幂等键：uk_cost_source_item (source_type, source_id, source_item_id, cost_type)
生成时机：业务单据审批通过的 onApproved() 回调，同事务
失败策略：isCritical()=true，成本生成失败则审批事务整体回滚
防重：先查 cost_generated_flag，再 try-insert + catch DuplicateKeyException 兜底
```

| source_type | source_id | source_item_id | cost_type | amount 来源 |
|---|---|---|---|---|
| `MAT_RECEIPT` | mat_receipt.id | mat_receipt_item.id | 材料 | receipt_item.amount |
| `SUB_MEASURE` | sub_measure.id | sub_measure_item.id | 分包 | measure.net_amount |
| `VAR_ORDER` | var_order.id | var_order_item.id | 签证 | var_order_item.amount |

### 4.3 付款金额校验规则（来自 02.md 5.7.4）

```text
本次申请金额 ≤ 合同可用余额
本次申请金额 ≤ 合同付款比例允许金额 - 已审批付款金额
材料付款：本次申请 ≤ 已验收金额 - 已申请金额
分包付款：本次申请 ≤ 已计量金额 - 已申请金额
```

## 7. 验收标准

### 7.1 功能验收

```text
□ 采购订单可新建、提交审批、审批通过后生效（PURCHASE_ORDER）
□ 有合同采购强制关联合同，订单金额不超合同可用余额
□ 材料验收可新建、提交审批，审批通过后自动生成材料成本（MAT_RECEIPT）
□ 分包计量可新建、提交审批，审批通过后自动生成分包成本（SUB_MEASURE）
□ 成本台账可按 source_type 追溯到来源单据
□ 成本台账可按项目、科目、成本类型筛选
□ 动态成本汇总展示目标/锁定/已发生/已付款/偏差
□ 付款申请可关联多个验收单/计量单作为依据
□ 付款金额校验生效（超合同余额/超已验收金额被拦截）
□ 财务回写后合同已付金额、付款状态、动态成本同步更新
□ 签证变更审批通过后调整动态成本（VAR_ORDER，P2）
□ 六个新 WorkflowBusinessHandler 全部经审批引擎正确回调
```

### 7.2 一致性与幂等验收（重点）

```text
□ 材料验收重复审批/重复回调不产生重复成本（uk_cost_source_item）
□ 分包计量重复触发不产生重复成本
□ 成本生成失败时审批事务整体回滚（isCritical 验证）
□ 付款申请审批通过确认不生成 cost_item（付款≠成本）
□ 并发提交付款申请，金额校验不被绕过
```

## 9. 风险与控制

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 多条成本生成链路幂等不一致 | 成本数据错误/重复 | 统一走 CostGenerationService + uk_cost_source_item + isCritical 回滚 |
| 付款表命名分歧（pay_application vs pay_request） | 前后端联调 404、文档代码不一致 | W0 先裁定，全程统一，注释标注别名 |
| 付款金额校验逻辑复杂易错 | 超付/重复付款 | W0 请 Oracle 评审校验算法，D2 专项边界测试 |
| 采购→验收→付款链路长，集成测试难 | 缺陷晚发现 | 每条链路独立集成测试，F 阶段端到端串联 |
| cost_summary 实时性与性能矛盾 | 汇总慢或不准 | W0 定为定时+手动刷新，避免实时触发 |
| 财务系统真实接口未定 | 回写无法对接 | 本阶段用模拟回写接口，预留 sync_status 字段 |
| 模块数量多，11 张新表 | DDL 与实体不一致 | 实体继承 BaseEntity 时，建表 DDL 必须含 created_by/updated_by/remark |

> ⚠️ **特别提醒**：第 1 阶段踩过 `ct_contract_item` 缺 `created_by` 列的坑（实体继承 BaseEntity 但建表漏列）。本阶段所有新表 DDL 必须包含完整审计列（created_by/created_at/updated_by/updated_at/deleted_flag/remark），否则 MyBatis-Plus insert 必报 `Unknown column`。

## 10. 完成定义（Definition of Done）

第 2 阶段完成后，应满足：

```text
1. 采购订单、材料验收、分包计量、付款申请四类业务单据可全流程提交审批
2. 材料验收/分包计量审批通过自动生成可追溯、不重复的成本
3. 成本台账可按来源/科目/项目查询，动态成本汇总可展示
4. 付款申请受合同余额与业务依据校验，财务回写形成资金闭环
5. 付款申请审批通过不生成成本（付款≠成本原则得到验证）
6. 六个新业务 Handler 经审批引擎统一回调，复用第 1 阶段机制
7. 成本生成幂等与事务一致性经专项测试验证
8. 全链路集成测试（采购→验收→成本→付款→回写）通过
9. H2 + MySQL 双环境测试基线保持绿色
10. 签证变更成本调整链路打通（P2，时间允许时）
```

下一阶段（第 3 阶段）预告：库存出入库、设备租赁、劳务工资、合同变更、签证收入向、资料归档。

## 11. 执行建议

```text
先打通最短的成本闭环：采购订单 → 材料验收 → 材料成本
再并行分包链路：分包任务 → 分包计量 → 分包成本
成本归集稳定后再做付款：付款申请 → 校验 → 财务回写
签证变更（P2）和移动端基础工程作为并行副线
驾驶舱、结算、库存等留待第 3-5 阶段
```

一句话总结：

```text
第 2 阶段的核心是把第 1 阶段验证过的"审批回调生成业务结果"模式，
规模化复制到五条业务链路，让成本和资金真正在系统里流动起来。
```
