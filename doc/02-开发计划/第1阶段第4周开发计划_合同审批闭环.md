## 1. 本周目标

把合同从"草稿"打通到"提交审批 → 审批流转 → 审批通过 → 合同生效 → 锁定成本生成"的第一条完整业务闭环，并让审批业务回调机制（`WorkflowBusinessHandler`）首次落地为可复用的真实实现。

一句话总结：

```text
本周不是再做一个新模块，而是把已经各自就绪的"合同模块"和"审批引擎"用业务回调机制焊接成一条闭环。
```

对应 Backlog 任务：CT-008、CT-009、CT-010、COST-001、COST-002、COST-003、WF-017（落地）。

## 4. 关键设计决策

### 4.1 业务类型与状态机

约定合同审批业务类型字符串（前后端 + 模板 seed 必须一致）：

```text
businessType = "CONTRACT_APPROVAL"
```

合同状态机（建议以 DB 字典 `contract_status` / `approval_status` 既有值为准，前端向其对齐）：

```text
approvalStatus 流转：
  DRAFT ──submit──▶ APPROVING ──审批通过──▶ APPROVED
                       │                        │
                       ├──驳回──▶ REJECTED       └──(合同生效)
                       └──撤回──▶ DRAFT

contractStatus 流转：
  DRAFT ──审批通过──▶ PERFORMING（履约中）──▶ SETTLED ──▶ TERMINATED
```

> T0 决策点：前端当前用 `EXECUTING/COMPLETED/SUBMITTED`，与 DB 字典不符。**推荐方案：前端对齐 DB 字典（PERFORMING/SETTLED，审批中间态用 APPROVING 而非 SUBMITTED）**，避免改动数据库与字典 seed。最终以 T0 会议结论为准并记入 notepad。

### 4.2 回调事务一致性（本周最高风险，必须解决）

`WorkflowEngine.notifyHandler()` 会吞掉 Handler 异常（catch + log，不回滚）。若把"合同状态变更 + 成本生成"直接放进 `onApproved`，一旦成本生成失败，会出现**审批通过但合同未生效/成本缺失**的脏数据。

本周必须在 T4 选定一种处理策略，并在 notepad 记录决策：

- **方案 A（推荐，改动小）**：在 `ContractWorkflowHandler.onApproved` 内部对关键写入（合同状态 + 成本）做 try-catch，失败时**主动抛出 `RuntimeException` 并标记需要人工介入**，同时评估是否需要让引擎在关键业务回调失败时回滚——即在引擎 `notifyHandler` 对"关键回调"不吞异常。
- **方案 B**：引擎层增加"关键回调"标识，关键业务回调异常向上抛出以触发 `@Transactional` 回滚（需改 `WorkflowEngine`，影响面较大，需 oracle 评审）。

> 该决策涉及审批引擎核心事务边界，T4 实施前应由架构/Oracle 评审确认，不可由实施者擅自决定。

### 4.3 成本生成幂等

锁定成本生成必须依赖 `uk_cost_source_item` 唯一键保证幂等：

```text
source_type    = "CT_CONTRACT"
source_id      = ct_contract.id
source_item_id = ct_contract_item.id   （一条清单一条成本）
cost_type      = 合同清单对应的成本类型（如无清单分类，先用统一值，T0 一并确认）
```

实现要点：成本生成服务必须捕获唯一键冲突（或先查后插 + DB 唯一键兜底），重复审批/重复回调不得产生重复成本。

## 5. 验收标准

### 5.1 功能验收（对应 Backlog 15.4 节）

```text
□ 合同可从草稿提交审批（POST /contracts/{id}/submit），approvalStatus: DRAFT → APPROVING
□ 审批中的合同不可被编辑（update 守卫生效）
□ 审批通过后 approvalStatus → APPROVED，contractStatus → PERFORMING（履约中）
□ 审批驳回后 approvalStatus → REJECTED；撤回后回到 DRAFT
□ 审批通过后自动生成合同锁定成本（cost_item，source_type=CT_CONTRACT）
□ 重复提交/重复回调不产生重复成本（uk_cost_source_item 幂等验证）
□ 合同审批记录可追溯（GET /contracts/{id}/approval-records 返回完整时间轴）
□ 前端详情页可提交审批、可查看审批记录与状态标签
□ WorkflowBusinessHandler 机制被合同模块真实复用（首个生产 Handler 落地）
```

### 5.2 测试验收

```text
□ 合同提交审批 → 多节点流转 → 通过：合同生效 + 成本生成（集成测试）
□ 合同提交审批 → 驳回：合同回到可编辑态，无成本生成
□ 合同提交审批 → 撤回：回到 DRAFT，无成本生成
□ 回调中成本生成失败时的事务一致性（按 4.2 选定方案验证：要么整体回滚，要么明确标记人工介入）
□ 成本幂等：对同一合同重复触发回调，cost_item 不重复
□ H2 + MySQL 双环境跑通（沿用 Week 2 测试基线）
```

| 风险 | 影响 | 控制措施 |

| 回调异常被引擎吞掉导致脏数据 | 审批通过但合同未生效/成本缺失 | 4.2 方案选定 + Oracle 评审 + T7 专项测试 |
