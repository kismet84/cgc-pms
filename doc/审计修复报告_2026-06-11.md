## 二、第一优先级修复详情（P1-P10）

### P1 — 后端方法级 RBAC 权限控制

**问题**: 所有 11 个 Controller 无任何方法级授权检查。任何已登录用户可执行所有 API 操作，RBAC 只在前端菜单显示层存在。

**修复方案**: JWT token 嵌入 `roleCodes` 和 `permissions` 声明 → Filter 构建 `SimpleGrantedAuthority` → `@EnableMethodSecurity` + 全量 `@PreAuthorize`

**修改文件**:

| 文件 | 改动 |
|------|------|
| `auth/util/JwtUtils.java` | 新增 `CLAIM_ROLES` / `CLAIM_PERMISSIONS` 常量；`generateToken` 写入角色码和权限码 claims；access token 加 `tokenType=access` claim |
| `auth/service/AuthService.java` | login 时把 `roleCodes` 和 `permCodes` 传入 `jwtUtils.generateToken()` |
| `auth/filter/JwtAuthenticationFilter.java` | 从 JWT claims 读取角色和权限；移除 `AuthorityUtils.NO_AUTHORITIES`；构建 `ROLE_xxx` + 直接权限码的 `SimpleGrantedAuthority` 列表注入 `UsernamePasswordAuthenticationToken` |
| `auth/config/SecurityConfig.java` | 类上加 `@EnableMethodSecurity(prePostEnabled = true)` |
| 10 个 Controller | 每个 GET 方法加 `@PreAuthorize("hasRole('ADMIN') or hasAuthority('xxx:query')")`；POST/PUT/DELETE/PATCH 分别加 `:add/:edit/:delete/:assign` 权限约束；WorkflowController 的审批动作用 `isAuthenticated()` 放行（权限由 WorkflowEngine 内部的任务归属校验保证） |

**权限码约定**（与 `sys_menu.perms` 一致）: `domain:resource:action`

### P2 — 幂等性 TOCTOU 竞态条件

**问题**: `WorkflowEngine.checkIdempotency()` 使用 check-then-insert 模式。两个并发请求可同时通过 count=0 检查，导致 DuplicateKeyException 绕过 BusinessException 返回 500。

**修复**: 改为 insert-first 策略——直接插入，靠 `uk_wf_idempotency` 唯一约束原子检测重复，catch `DuplicateKeyException` 转换回 `BusinessException("DUPLICATE_REQUEST")`。

**修改文件**: `workflow/service/WorkflowEngine.java` — `checkIdempotency()` 方法重构；新增 `import org.springframework.dao.DuplicateKeyException`。

### P3 — 合同列表 N+1 查询

**修复**: 在 `getPage` 中用 `selectBatchIds` 批量预取所有 projectId 和 partnerId 的 Map → 重载 `toVO` 接受预取 Map。降为 3 条 SQL（1 主查询 + 1 批量项目 + 1 批量合作方）。

**修改文件**: `contract/service/CtContractService.java` — `getPage` 增加批量预取逻辑；`toVO` 拆分为 `buildBaseVO`（共享字段映射）+ 两个重载（单条/批量）。

### P13 — Refresh Token + Redis 令牌黑名单

**问题**: 单 access token 24 小时有效期，无 refresh 机制，logout 不使 token 失效。

**修复架构**:

```
登录 → 返回 access token (含 roleCodes+permissions) + refresh token
API 调用 → JwtAuthenticationFilter 验证 access token，检查黑名单
refresh token 用作换取新 token 的凭证（X-Refresh-Token 头）
logout → 将 access token 加入 Redis 黑名单（TTL = token 自然过期时间）
/refresh → 旧 refresh token 加入黑名单 → 发新一对 token（Token 轮换）
```

**修改文件**:

| 文件 | 改动 |
|------|------|
| `auth/config/JwtProperties.java` | 新增 `refreshExpiration` 字段（默认 604800000ms = 7天） |
| `auth/util/JwtUtils.java` | 新增 `generateRefreshToken`（仅含 userId + tokenType=refresh，不含权限）；新增 `isRefreshToken`、`getRemainingTtlMillis` |
| `auth/service/TokenBlacklistService.java` | **新建**，`@ConditionalOnBean(StringRedisTemplate.class)`。用 token 末尾 32 字符（JWT 签名部分）作为 Redis key，TTL = token 剩余有效期。local profile 无 Redis 时 Bean 不创建 |
| `auth/dto/LoginResponse.java` | 新增 `refreshToken` 字段 |
| `auth/service/AuthService.java` | `login()` 同时生成 refresh token；新增 `loginById()` 供 refresh 时复用（从 DB 重载角色/权限，保证权限变更实时生效） |
| `auth/controller/AuthController.java` | `POST /auth/logout` 将 access token 加入黑名单；新增 `POST /auth/refresh` 实现 token 轮换；使用 `ObjectProvider<TokenBlacklistService>` 兼容 local profile |
| `auth/filter/JwtAuthenticationFilter.java` | `doFilterInternal` 增加黑名单检查 + 拒绝 refresh token 直接访问 API（只能用于 /auth/refresh）；同样使用 `ObjectProvider` 兼容 |
| `auth/config/SecurityConfig.java` | 白名单加 `/auth/refresh` |
| 4 个 application yml | 添加 `jwt.refresh-expiration: 604800000`（prod 用 `${JWT_REFRESH_EXPIRATION}`） |

**关键设计决策**:

- **local profile 兼容**: `TokenBlacklistService` 通过 `@ConditionalOnBean` 在 local profile 不创建；controller/filter 通过 `ObjectProvider.getIfAvailable()` 安全跳过，logout 不报错、黑名单始终返回 false
- **Token 轮换**: 每次 `/auth/refresh` 将旧 refresh token 加入黑名单，阻止重放
- **权限实时性**: refresh 时从 DB 重新加载角色和权限，权限变更即时生效

### P14 — 前端集成 Refresh Token + 自动静默刷新

**问题**: 前端无 refresh token 概念，401 强制登出。

**修复**:

| 文件 | 改动 |
|------|------|
| `types/user.ts` | `LoginResult` 添加 `refreshToken: string` |
| `stores/user.ts` | 新增 `refreshToken` ref 和 `setRefreshToken()`；refresh token 存 `sessionStorage`（关闭标签页后自动清除，比 localStorage 更安全）；`logout` 同步清理 |
| `api/modules/auth.ts` | 新增 `refreshTokenApi(refreshToken)`，`X-Refresh-Token` 请求头 |
| `api/request.ts` | 401 拦截器实现 **静默刷新**: 有 refresh token 时自动换 access token 并重放原请求；并发 401 请求排队等待，只触发一次刷新；无 refresh token 才强制登出 |
| `pages/login/index.vue` | 登录成功后调用 `userStore.setRefreshToken(refreshToken)`；移除 mock 登录注释代码 |

**刷新流程**:
```
请求返回 401 → 检查 refresh token 是否存在
  ├─ 不存在 → 登出 → 跳转登录页
  └─ 存在 → 调用 /auth/refresh
       ├─ 成功 → 更新 access token + refresh token → 重放原请求
       │         同时释放排队中的所有请求
       └─ 失败 → 登出 → 跳转登录页
```

## 六、风险提示

1. **P13 Refresh Token** — 需要 Redis 服务运行。local profile 已自动跳过（`@ConditionalOnBean`），dev/test 若 Redis 不可用启动可能阻塞。建议 local profile 启动方式不变。

2. **P16 级联删除** — 删除项目时连锁逻辑删除合同及其子记录，属不可逆操作。建议后续增加"软删除确认"提示。

3. **P18 vite 升级** — vite 6 有 API 变更，需在联网环境中升级并完成全链路回归测试后再提交。
