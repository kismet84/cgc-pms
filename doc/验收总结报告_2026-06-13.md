**通过项（8/9）**：变更单创建（CC 编号）、字段完整性（beforeAmount/changeAmount/afterAmount）、提交审批、三级审批、生效标志（effectiveFlag=1）、合同 currentAmount 联动、变更成本自动生成、成本幂等保护（uk_cost_source_item + DuplicateKeyException）。

**有条件通过项（1/9）**：多重变更累加逻辑上正确（currentAmount = SUM changeAmount），但验收环境仅创建 1 次变更，未端到端验证多次变更。

#### 闭环 8: 审批异常路径专项 — ⚠️ 有条件通过

**通过项（6/8）**：
- 驳回→重新编辑→重新提交：完整链路验证，round 2 审批通过。
- 审批撤回：级联取消所有 pending task + resetActiveNodes + instance→WITHDRAWN。
- 审批转办：原 task→TRANSFERRED，新 task 创建给目标用户，目标可审批。
- 审批加签（COUNTERSIGN）：双方均需审批，重复加签被 exists check 阻止。
- 幂等提交保护：重复提交被状态检查阻止。
- 无权限用户审批：TASK_ALREADY_HANDLED 先于 NOT_TASK_OWNER 触发（行为正确）。

**不通过项（2/8）**：
- AuthorizationDeniedException → 500（应返回 403）：GlobalExceptionHandler 未处理该异常类型。
- SUPER_ADMIN vs ADMIN 角色不匹配：几乎所有业务端点受影响（通用问题）。

### 2.3 全局阻塞项说明

以下 2 个 P0 问题影响了多个闭环的验收，属于系统级缺陷，修复后所有受影响场景应重新验证：

1. **SUPER_ADMIN vs ADMIN 角色不匹配**（影响闭环 1/2/3/4/5/8）：DB 种子中的角色码为 `SUPER_ADMIN`，但所有 Controller 的 `@PreAuthorize` 注解检查 `hasRole('ADMIN')`。admin 用户虽有 SUPER_ADMIN 角色但无法通过 ADMIN 角色检查。临时绕过方式：手动 INSERT role_code='ADMIN' 并分配给 admin 用户。

2. **权限码系统性不一致**（影响所有非工作流端点）：DB 种子（V39 sys_menu）中的权限码（如 `contract:list`, `system:user:list`）与 Controller `@PreAuthorize` 中的权限码（如 `contract:query`, `system:user:query`）不一致。V40 迁移仅修复了 Phase 4 端点，系统级和业务级权限码仍未对齐。

## 3. 安全验收结果（T13-T15）

### 3.1 权限矩阵（T13） — ✅ 通过

22 个端点测试确认权限隔离生效：

| 端点类别 | Admin（SUPER_ADMIN+ADMIN） | 非 Admin（COMMON_USER/PROJECT_MANAGER/无角色） |
|---------|--------------------------|------------------------------------------|
| 系统管理（users/roles/menus） | ✅ 200 | ❌ 500（AuthorizationDeniedException） |
| 业务管理（contracts/projects/partners） | ✅ 200 | ❌ 500 |
| 财务管理（settlements/payments/cost） | ✅ 200* | ❌ 500 |
| 库存管理（warehouses/stock/invoices） | ✅ 200* | ❌ 500 |
| 仪表盘（dashboard） | ✅ 200 | ❌ 500 |
| 预警（alerts） | ✅ 200 | ❌ 500 |
| 审批工作流（workflow） | ✅ 200 | ✅ 200（isAuthenticated 放行） |

> \* 部分端点因业务逻辑问题（缺数据/NPE）返回 500，非鉴权问题。

**结论**：权限矩阵在 admin 与非 admin 角色之间实现了完整隔离。系统管理和业务管理 API 仅 admin 可访问，工作流 API 对所有已认证用户开放（由工作流引擎进行任务级别权限控制）。已知缺陷：AuthorizationDeniedException 返回 500 而非 403（参见已知问题清单）。

### 3.2 多租户数据隔离（T14） — ✅ 通过

9/9 IDOR 攻击测试全部通过：

| 测试类别 | 测试数 | 结果 |
|---------|--------|------|
| 列表查询数据隔离（projects/contracts/partners） | 3 | ✅ 其他租户列表为空 |
| IDOR 攻击（跨租户访问详情） | 4 | ✅ 统一返回 NOT_FOUND（防枚举） |

## 5. 性能基线结论（T20）

### 5.1 P95 达标情况

在近乎空数据库条件下（~300 行系统种子数据，35 张业务表为空），5 个核心端点 P95 全部远低于目标值：

| 端点 | P50 | P95 | 目标 | 余量 | 状态 |
|------|-----|-----|------|------|------|
| GET /api/contracts | 9.8ms | 12.2ms | < 500ms | 97.6% | ✅ PASS |
| GET /api/cost-ledger | 4.7ms | 6.1ms | < 1s | 99.4% | ✅ PASS |
| GET /api/dashboard/cost-manager | 14.8ms | 34.6ms | < 2s | 98.3% | ✅ PASS |
| POST /api/alerts/batch-evaluate | 4.4ms | 6.1ms | < 5s | 99.9% | ✅ PASS |
| GET /api/api/notifications | 4.9ms | 5.7ms | < 300ms | 98.1% | ✅ PASS |

### 5.2 关键局限

**本次基线不具备生产代表性**。原因如下：

- 数据库为极轻量状态：多数业务表行数为 0（ct_contract 仅 3 行），系统表共约 300 行。
- 无法评估数据量增长后的性能衰减曲线。
- 测试为串行（无并发负载），无法反映并发竞争和连接池压力下的响应时间。
- Dashboard 端点（cost-manager）响应波动最大（10.5ms~37.1ms），涉及多表聚合查询，数据量增长后可能成为首个性能瓶颈。

### 5.3 建议

- **P1**：使用数据生成器填充 1 万~10 万行业务数据后重新测试，获得生产代表性基线。
- **P2**：在 Dashboard 端点加载条件下进行并发测试。
- **P3**：修复 Phase 4 控制器双 /api 前缀问题，统一 URL 规范。
- **P3**：统一角色码（SUPER_ADMIN vs ADMIN），消除鉴权层面的间歇性 500 错误。

| 2. 库存并发出库 | 乐观锁 | ✅ PASS | 双线程并发出库 10 单位，仅一笔成功（余额 10→0），另一笔 INSUFFICIENT_STOCK，无超卖 |
| 3. 付款余额并发 | 余额校验 | ⛔ SKIPPED | 权限不足：SUPER_ADMIN ≠ ADMIN，无法访问 /api/pay-applications |
| 4. 重复成本约束 | 唯一约束 | ⛔ SKIPPED | 权限不足：无法访问 /api/cost-ledger 等端点。DB 约束 uk_cost_source_item 存在且集成测试已验证 |
| 5. 结算锁定 | 状态守卫 | ⛔ SKIPPED | 权限不足：无法访问 /api/settlements。锁定逻辑已于 T10 验收确认 |

### 6.2 关键发现

- **库存乐观锁有效**：`@Version` + 余额校验组合正确实现了并发出库的防超卖保护。
- **3/5 场景因权限问题无法测试**：系统性角色/权限码不匹配导致付款、成本、结算相关端点全部返回 500。
- **WorkflowController 缺失 PURCHASE_REQUEST case**：`getRequiredPermission()` switch 中缺少 `PURCHASE_REQUEST` 映射，导致 IllegalArgumentException。
- **@Valid 先于 @PreAuthorize 执行**：在结算 PUT 请求中确认了该信息泄露模式（P3 级别）。

### 6.3 结论

已测试的 2 个场景（库存并发出库、工作流幂等）均通过。跳过 3 个场景的原因明确且可复现，均源于已知的权限系统缺陷，非并发逻辑本身问题。修复权限问题后应重新执行全部 5 个场景。

## 8. 上线建议

### 8.1 综合判定：🟡 有条件上线

系统核心业务能力已完成并验证，基础架构稳定，安全机制到位。但存在 3 个 P0 阻断项和 4 个 P1 高优先级问题，**在 P0 问题全部修复前不建议直接上线生产环境**。

### 8.2 上线前置条件（MUST FIX）

上线前必须完成以下 3 项 P0 修复并重新验证：

| 序号 | 条件 | 验证方式 |
|------|------|---------|
| 1 | **修复重复结算 guard**（P0-01）：添加 contractId 唯一性校验 | 对同一合同重复 POST /settlements，确认第二次返回业务错误 |
| 2 | **修复角色匹配**（P0-02）：使 admin 用户的角色可通过所有 `@PreAuthorize` 检查 | admin 用户登录后访问全部 22 个端点，确认 0 个 500 |
| 3 | **修复权限码对齐**（P0-03）：统一 DB 种子与 Controller 注解的权限码 | 非 admin 角色携带对应权限码后，确认可访问目标端点 |

### 8.3 强烈建议修复（SHOULD FIX）

以下 P1 问题应在首个迭代内完成修复：

| 序号 | 条件 | 说明 |
|------|------|------|
| 4 | 修复质保金计算错误（P1-01） | 结算金额准确性直接影响财务对账 |
| 5 | 修复 AuthorizationDeniedException → 403（P1-02） | 影响前端错误处理和用户体验 |
| 6 | 修复采购 items/batch 端点（P1-03） | 随 P0-02 一并修复 |
| 7 | 添加 PURCHASE_REQUEST case（P1-04） | 采购申请审批流程的必要修复 |

### 8.4 可延后修复（COULD FIX）

P2 问题（VO 字段完整性、AlertLog 结构化字段、种子数据、来源追溯端点）不影响核心业务流程，可在上线后迭代中逐步完善。

### 8.5 上线前额外建议

1. **性能数据补充**：在 P0 修复完成后，使用数据生成器填充 1 万行业务数据，重新执行 T20 性能基线测试，获得生产代表性数据。
2. **并发测试补充**：在权限问题修复后，重新执行 T19 的全部 5 个并发场景。
3. **E2E 执行验证**：在后端和前端均运行后，执行全部 9 个 spec 文件（36 个测试用例），录制首轮 E2E 通过率。
4. **Flyway 验证**：确认生产环境 MySQL 8.0 实例可成功执行 V1-V40 全部迁移。

### 8.6 上线步骤建议（概要）

1. **修复阶段**（预计 1-2 天）：完成 P0-01/P0-02/P0-03 修复 + 回归测试
2. **验证阶段**（预计 0.5 天）：重新执行 T1-T21 中受影响的场景，确认修复有效
3. **演练阶段**（预计 0.5 天）：在生产同等配置环境执行一次完整部署演练（Docker Compose + Flyway + 冒烟测试）
4. **上线阶段**：执行数据库备份 → Flyway 迁移 → 后端部署 → 前端部署 → 冒烟验证 → 监控开启
