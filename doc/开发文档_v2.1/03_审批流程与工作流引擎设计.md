# 审批流程与工作流引擎设计

## 1. 设计目标

审批引擎是系统核心基础能力，应统一处理合同、采购、验收、计量、签证、付款、结算等业务审批。业务模块不允许各自实现一套审批逻辑。

核心目标：

```text
统一审批模板
统一审批实例
统一审批任务
统一审批记录
统一审批动作
统一审批轮次
统一可用操作
统一业务回调
统一并发与幂等控制
```

## 2. 审批状态模型

### 2.1 审批实例状态

```text
DRAFT       草稿
RUNNING     审批中
APPROVED    已通过
REJECTED    已驳回
WITHDRAWN   已撤回
TERMINATED  已终止
VOIDED      已作废
```

### 2.2 审批节点状态

```text
WAITING     未开始
RUNNING     处理中
PASSED      已通过
REJECTED    已驳回
SKIPPED     已跳过
CANCELED    已取消
```

### 2.3 审批任务状态

```text
WAITING         等待中
PENDING         待处理
PENDING_HOLD    挂起中
APPROVED        已同意
REJECTED        已驳回
TRANSFERRED     已转办
CANCELED        已取消
WITHDRAWN       已撤回
TIMEOUT         已超时
```

## 3. 审批动作

| 动作 | 编码 | 说明 |
|---|---|---|
| 提交 | `SUBMIT` | 发起审批 |
| 重新提交 | `RESUBMIT` | 驳回后新轮次提交 |
| 同意 | `APPROVE` | 当前任务通过 |
| 驳回 | `REJECT` | 退回发起人或指定节点 |
| 退回上一步 | `RETURN_PREVIOUS` | 退回上一个审批节点 |
| 退回发起人 | `RETURN_START` | 修改后可重提 |
| 撤回 | `WITHDRAW` | 发起人撤回 |
| 转办 | `TRANSFER` | 转给同权限人员 |
| 前加签 | `ADD_SIGN_BEFORE` | 当前任务挂起，先由加签人审批 |
| 后加签 | `ADD_SIGN_AFTER` | 当前人审批后增加加签任务 |
| 并行加签 | `ADD_SIGN_PARALLEL` | 增加并行审批人 |
| 作废 | `VOID` | 已通过但未发生后续业务时作废 |

## 4. 审批模式

| 模式 | 编码 | 通过规则 | 驳回规则 |
|---|---|---|---|
| 顺序审批 | `SEQUENTIAL` | 当前审批人同意后进入下一人 | 任一人驳回即节点驳回 |
| 会签 | `COUNTERSIGN` | ALL / COUNT / RATIO | ANY_REJECT / RATIO_REJECT |
| 或签 | `OR_SIGN` | 任一人同意即通过 | 全部驳回才驳回 |

## 5. 审批轮次设计

审批实例增加：

```text
current_round 当前审批轮次
resubmit_count 重新提交次数
```

节点、任务、记录均增加：

```text
round_no 所属轮次
```

规则：

```text
首次提交 current_round = 1
驳回后重新提交 current_round = current_round + 1
旧轮次节点、任务、记录全部保留
审批时间轴按 round_no 分组展示
当前有效流程以 instance.current_round 为准
```

## 6. 并发与幂等

### 6.1 乐观锁

`wf_task` 增加：

```text
version taskVersion
```

前端提交审批动作必须携带：

```json
{
  "taskId": "80001",
  "taskVersion": 3,
  "actionType": "APPROVE"
}
```

后端校验失败返回：

```json
{
  "code": "WF_TASK_VERSION_CONFLICT",
  "message": "当前任务已发生变化，请刷新后重试"
}
```

### 6.2 幂等键

所有提交类动作必须携带：

```text
idempotencyKey
```

后端按以下维度保证幂等：

```text
user_id + business_type + business_id + action_type + idempotency_key
```

## 7. 审批后端调用原则

### 7.1 禁止 HTTP 调自己

同一后端应用内部，审批引擎与业务模块之间不允许通过 HTTP 调用自身 Controller。

错误方式：

```java
outboxDao.save(out);
processOutbox(); // 发 HTTP POST 给自己
```

正确方式：

```text
事务内领域服务直接调用
事务提交后发布异步事件
Outbox 仅用于外部系统和移动端离线同步
```

### 7.2 审批处理事务边界

```java
@Transactional
public ApprovalResult approveTask(ApproveCommand cmd) {
    WfTask task = wfTaskService.lockAndValidate(cmd);
    wfRecordService.createApproveRecord(task, cmd);
    wfEngineService.completeTask(task, cmd);
    wfEngineService.moveNext(task.getInstanceId());

    workflowBusinessHandlerRegistry
        .get(task.getBusinessType())
        .onWorkflowChanged(task);

    domainEventPublisher.publishAfterCommit(new WorkflowHandledEvent(task));
    return ApprovalResult.success(task.getInstanceId());
}
```

## 8. 业务回调接口设计

### 8.1 接口定义

```java
public interface WorkflowBusinessHandler {

    BusinessType supportType();

    void beforeSubmit(WorkflowSubmitCommand command);

    void onRunning(WorkflowStateChangedCommand command);

    void onApproved(WorkflowStateChangedCommand command);

    void onRejected(WorkflowStateChangedCommand command);

    void onWithdrawn(WorkflowStateChangedCommand command);

    void onVoided(WorkflowStateChangedCommand command);
}
```

### 8.2 注册中心

```java
@Component
public class WorkflowBusinessHandlerRegistry {

    private final Map<BusinessType, WorkflowBusinessHandler> handlers;

    public WorkflowBusinessHandler get(BusinessType businessType) {
        WorkflowBusinessHandler handler = handlers.get(businessType);
        if (handler == null) {
            throw new BusinessException("未找到审批业务处理器");
        }
        return handler;
    }
}
```

### 8.3 合同审批回调

```java
@Component
public class ContractWorkflowHandler implements WorkflowBusinessHandler {

    @Override
    public BusinessType supportType() {
        return BusinessType.CONTRACT;
    }

    @Override
    @Transactional
    public void onApproved(WorkflowStateChangedCommand command) {
        Contract contract = contractRepository.getById(command.getBusinessId());
        contract.markApproved();
        contract.markSigned();
        paymentTermService.activateTerms(contract.getId());
        costService.generateLockedCostIfNeeded(contract);
    }
}
```

### 8.4 材料验收审批回调

```java
@Component
public class MaterialReceiptWorkflowHandler implements WorkflowBusinessHandler {

    @Override
    public BusinessType supportType() {
        return BusinessType.MATERIAL_RECEIPT;
    }

    @Override
    @Transactional
    public void onApproved(WorkflowStateChangedCommand command) {
        MatReceipt receipt = receiptRepository.getById(command.getBusinessId());
        receipt.markApproved();
        costItemService.generateFromReceipt(receipt); // 必须幂等
    }
}
```

## 9. 成本生成强一致原则

成本生成属于核心业务副作用，不建议只依赖普通异步事件。

| 场景 | 推荐处理 |
|---|---|
| 材料验收通过生成材料成本 | 同事务或强一致领域回调 |
| 分包计量通过生成分包成本 | 同事务或强一致领域回调 |
| 付款审批通过推送财务 | 事务提交后 Outbox |
| 审批完成发送消息通知 | 事务提交后异步事件 |
| 同步 BI / 数据仓库 | Outbox / MQ |

## 10. 审批引擎 POC 验收清单

第 1 阶段必须先完成审批引擎独立 POC，验收项如下：

| 验收项 | 验收标准 |
|---|---|
| 提交审批 | 生成 `wf_instance`、首个 `wf_node_instance`、`wf_task`、`wf_record` |
| 顺序审批 | 当前人通过后进入下一人 |
| 会签 | 多人全部同意后节点通过，任一驳回按规则处理 |
| 或签 | 任一人同意后节点通过，其他待办自动取消 |
| 驳回重提 | `current_round + 1`，旧记录保留，新任务生成 |
| 撤回 | 未处理任务取消，记录保留，业务单据回到可编辑状态 |
| 转办 | 原任务转办状态，新任务生成，记录留痕 |
| 加签 | 前加签、后加签、并行加签均能形成可追溯记录 |
| availableActions | 不同状态、角色、节点返回不同可操作按钮 |
| taskVersion | 两人同时处理同一任务时，一个成功，一个返回版本冲突 |
| idempotencyKey | 重复提交同一动作只执行一次 |
| 业务回调 | 合同审批通过可触发合同状态变化 |
| 成本幂等 | 验收单重复回调不重复生成成本 |

## 11. 审批记录保留规则

```text
审批记录原则上不物理删除
撤回、驳回、重新提交后旧记录保留
旧记录是否参与当前审批结果，由 round_no、record_status、instance_status 共同判断
转办、加签、系统取消、自动跳过均需要记录
```
