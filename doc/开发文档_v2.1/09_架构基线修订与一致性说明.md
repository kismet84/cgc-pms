# 架构基线修订与一致性说明

## 1. 修订目的

本文件用于固定项目架构基线，解决文档之间可能出现的技术栈摇摆、数据库选型不一致、审批调用方式不合理、移动端节奏过早等问题。

## 2. 最终定版结论

```text
1. 一期数据库统一采用 MySQL 8.0，不默认 PostgreSQL。
2. 审批规则、动态表单、节点规则使用 MySQL JSON 字段。
3. 后端审批引擎内部禁止 HTTP 调自己，采用事务内领域服务调用。
4. Outbox 只用于外部系统、消息通知、BI 同步和移动端离线同步。
5. 前端技术栈冻结为 Vue 3 + TypeScript + Vite + Ant Design Vue。
6. 参考 HTML 页面必须组件化重构，不作为正式工程代码。
7. 移动端正式业务开发后移，第 6 周做基础工程，第 8 周左右进入真实业务。
8. 成本生成采用审批业务回调 + 幂等约束，不允许重复生成。
9. 开发顺序按阻塞依赖链重排，审批引擎 POC 必须先完成。
10. 结算、驾驶舱、预警后置，依赖前面业务数据稳定。
```

## 3. 数据库基线修订

### 3.1 修订前问题

文档中曾出现 PostgreSQL 和 MySQL 混用倾向，数据库设计示例仍是 MySQL 风格，而部分技术描述提到 PostgreSQL JSONB。

### 3.2 修订后基线

| 项目 | 定版 |
|---|---|
| 默认数据库 | MySQL 8.0 |
| JSON 存储 | MySQL JSON |
| PostgreSQL | 仅作为后续可选部署 |
| DDL 风格 | MySQL 8.0 |
| Migration | Flyway |
| 运维手册 | 按 MySQL 编写 |

### 3.3 PostgreSQL 可选条件

只有满足以下任一条件时，才考虑 PostgreSQL：

```text
明确需要 PostGIS 地理能力
明确需要复杂 JSONB 索引和查询
企业已有成熟 PostgreSQL 运维能力
甲方指定 PostgreSQL
```

## 4. 审批调用基线修订

### 4.1 禁止方式

```java
processOutbox(); // HTTP POST 调自己的 Controller
```

### 4.2 正确方式

```text
校验任务
→ 写审批记录
→ 更新任务状态
→ 更新节点状态
→ 更新实例状态
→ 同事务内回调业务模块
→ 提交事务
→ 事务提交后发布异步事件
```

### 4.3 Outbox 边界

| 可以使用 | 不可使用 |
|---|---|
| 推送财务系统 | 审批服务调用自身 Controller |
| 企业微信/短信/站内信 | 同应用模块之间绕 HTTP |
| BI/数据仓库同步 | 审批内部状态流转 |
| 移动端离线同步 | 成本生成强一致链路 |

## 7. 成本生成基线修订

成本生成不允许作为普通通知随意异步处理。材料验收、分包计量、合同变更等审批通过后的成本生成必须满足：

```text
业务状态更新和成本生成在清晰事务边界内完成
重复回调不重复生成成本
成本可通过 source_type/source_id 反查来源
生成失败必须可重试并可审计
```

推荐唯一约束：

```sql
CREATE UNIQUE INDEX uk_cost_source_item
ON cost_item(source_type, source_id, source_item_id, cost_type);
```

## 10. 后续变更管理

任何涉及以下内容的变更，必须经过架构评审：

```text
数据库从 MySQL 改为其他数据库
审批引擎改为外部工作流引擎
前端技术栈从 Vue 改为 React
移动端从 uni-app 改为 Flutter / 原生
核心成本生成改为完全异步
模块化单体改为微服务
ID 策略从雪花 ID 改为自增 ID
```

变更必须输出：

```text
变更原因
影响范围
迁移成本
回滚方案
测试方案
上线窗口
负责人
```
