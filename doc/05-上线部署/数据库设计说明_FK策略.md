# CGC-PMS 数据库外键约束策略说明

## 当前状态

CGC-PMS 数据库中 **54 张表均无数据库级 FOREIGN KEY 约束**。所有引用完整性由应用层强制执行。

## 设计理由

1. **灵活性**: 软删除（`deleted_flag`）场景下，FK 约束会阻断跨表软删除操作
2. **迁移安全**: Flyway 迁移中无 FK 约束可避免因表创建顺序导致的迁移失败
3. **性能**: 高并发写入场景下，FK 约束校验增加额外开销
4. **运维便捷**: 数据修复/迁移时无需临时禁用外键检查

## 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 应用 bug 导致孤立记录 | Service 层逐条校验引用有效性（如 `requireProjectInTenant()`） |
| 直接数据库操作产生脏数据 | 生产环境仅应用账号可访问，DBA 操作需审批 |
| 多租户数据泄露 | Service 层 `LambdaQueryWrapper` 统一追加 `.eq(Entity::getTenantId, ...)` |

## 后续计划

二期可考虑对核心实体（如 `ct_contract`、`pm_project`）添加 FK 约束作为防御层，需配合 Flyway 迁移顺序调整和充分测试。
