# CGC-PMS 备份恢复方案

> **文档版本**: v1.0  
> **编制日期**: 2026-06-13  
> **适用环境**: Docker Compose 部署（MySQL 8.0 + Redis 7 + MinIO）  
> **RPO**: 24 小时（全量备份间隔）  
> **RTO**: 2 小时（从备份恢复到服务可用）

---

## 1. 概述

### 1.1 备份目标

保障 CGC-PMS 生产环境在以下灾难场景下的数据可恢复性：

| 场景 | 影响范围 | 恢复手段 |
|------|---------|---------|
| MySQL 数据损坏/误删 | 全部业务数据（合同、审批、成本、库存、发票等 55 张表） | 全量 mysqldump + binlog 时间点恢复 |
| MinIO 文件丢失 | 上传的附件、合同文件、发票扫描件 | mc mirror 增量同步恢复 |
| Redis 数据丢失 | JWT 黑名单、SSE 通知缓存 | **无需恢复**（缓存可重建，详见 §4） |
| Docker 卷误删 | 全部持久化数据 | 从备份文件重建卷并恢复数据 |

### 1.2 RPO / RTO 定义

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **RPO** (Recovery Point Objective) | ≤ 24h | 每日全量备份 + binlog 持续归档，最坏情况丢失 < 24h 增量数据 |
| **RTO** (Recovery Time Objective) | ≤ 2h | 包含：停止服务 → 恢复 MySQL → 恢复 MinIO → 验证 → 启动服务 |

### 1.3 环境信息

| 组件 | 容器名 | 端口映射 | 卷挂载 | 凭据 |
|------|--------|---------|--------|------|
| MySQL 8.0 | `cgc-pms-mysql` | 3307:3306 | `mysql-data:/var/lib/mysql` | root / `$MYSQL_ROOT_PASSWORD` |
| Redis 7 | `cgc-pms-redis` | 6379:6379 | `redis-data:/data` | 密码 `$REDIS_PASSWORD` |
| MinIO | `cgc-pms-minio` | 9000:9000, 9001:9001 | `minio-data:/data` | admin / `$MINIO_ROOT_PASSWORD` |

> **注意**: 上述凭据为开发环境默认值，**生产环境必须更换**（通过 `deploy/.env` 注入）。

### 1.4 备份存储策略

```
/data/backups/cgc-pms/
├── mysql/
│   ├── full/          # 每日全量备份，保留 30 天
│   └── binlog/        # binlog 增量备份，保留 7 天
├── minio/
│   └── mirror/        # MinIO 文件镜像，保留最近 3 个版本
└── logs/
    └── backup.log     # 备份操作日志
```

## 2. MySQL 备份策略

### 2.1 启用 binlog（前提条件）

MySQL 增量恢复依赖二进制日志（binlog）。在 Docker 环境中通过自定义配置文件启用。

**步骤 1：创建 MySQL 配置文件** `deploy/mysql/conf.d/custom.cnf`

```ini
[mysqld]
# ===== binlog 配置（增量备份基础） =====
server-id = 1
log-bin = /var/lib/mysql/mysql-bin
binlog_format = ROW
expire_logs_days = 7
max_binlog_size = 100M
sync_binlog = 1

# ===== 慢查询日志（可选，用于性能诊断） =====
slow_query_log = 1
slow_query_log_file = /var/lib/mysql/mysql-slow.log
long_query_time = 2
```

**步骤 2：重启 MySQL 容器使配置生效**

```bash
cd deploy
docker compose restart mysql
```

**步骤 3：验证 binlog 已启用**

```bash
docker exec cgc-pms-mysql mysql -u root -p$MYSQL_ROOT_PASSWORD -e "SHOW VARIABLES LIKE 'log_bin%';"
```

预期输出：

```
+---------------------------------+------------------------------------+
| Variable_name                   | Value                              |
+---------------------------------+------------------------------------+
| log_bin                         | ON                                 |
| log_bin_basename                | /var/lib/mysql/mysql-bin           |
+---------------------------------+------------------------------------+
```

### 2.2 每日全量备份（mysqldump）

#### 2.2.1 备份脚本 `scripts/backup-mysql-full.sh`

```bash
#!/bin/bash
# ============================================================
# CGC-PMS MySQL 每日全量备份脚本
# 用法: ./backup-mysql-full.sh
# crontab: 0 2 * * * /data/backups/cgc-pms/scripts/backup-mysql-full.sh
# ============================================================

set -euo pipefail

# === 配置 ===
CONTAINER_NAME="cgc-pms-mysql"
DB_NAME="cgc_pms"
DB_USER="root"
DB_PASS="$MYSQL_ROOT_PASSWORD"
BACKUP_DIR="/data/backups/cgc-pms/mysql/full"
RETENTION_DAYS=30
LOG_FILE="/data/backups/cgc-pms/logs/backup.log"

TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/${DB_NAME}_full_${TIMESTAMP}.sql.gz"

# === 创建备份目录 ===
mkdir -p "${BACKUP_DIR}" "$(dirname "${LOG_FILE}")"

# === 执行备份 ===
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Starting MySQL full backup..." | tee -a "${LOG_FILE}"

docker exec "${CONTAINER_NAME}" mysqldump \
    -u "${DB_USER}" -p"${DB_PASS}" \
    --single-transaction \
    --quick \
    --lock-tables=false \
    --routines \
    --triggers \
    --events \
    --set-gtid-purged=OFF \
    --databases "${DB_NAME}" \
    | gzip > "${BACKUP_FILE}"

# === 验证备份 ===
if [ ${PIPESTATUS[0]} -eq 0 ] && gzip -t "${BACKUP_FILE}"; then
    BACKUP_SIZE=$(du -h "${BACKUP_FILE}" | cut -f1)
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup SUCCESS: ${BACKUP_FILE} (${BACKUP_SIZE})" | tee -a "${LOG_FILE}"
else
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] Backup FAILED!" | tee -a "${LOG_FILE}"
    exit 1
fi

# === binlog 刷新（开始新日志文件） ===
docker exec "${CONTAINER_NAME}" mysql -u "${DB_USER}" -p"${DB_PASS}" \
    -e "FLUSH BINARY LOGS;" 2>/dev/null

# === 清理过期备份 ===
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Cleaning backups older than ${RETENTION_DAYS} days..." | tee -a "${LOG_FILE}"
find "${BACKUP_DIR}" -name "${DB_NAME}_full_*.sql.gz" -mtime +${RETENTION_DAYS} -delete 2>/dev/null || true
echo "[$(date '+%Y-%m-%d %H:%M:%S')] Cleanup done." | tee -a "${LOG_FILE}"
```

#### 2.2.2 关键参数说明

| 参数 | 作用 |
|------|------|
| `--single-transaction` | InnoDB 一致性快照，不锁表（适用于 InnoDB 表） |
| `--quick` | 逐行导出，避免大表撑爆内存 |
| `--lock-tables=false` | 与 `--single-transaction` 配合，避免锁表 |
| `--routines` | 导出存储过程和函数 |
| `--triggers` | 导出触发器 |
| `--events` | 导出定时事件 |
| `--set-gtid-purged=OFF` | 不包含 GTID 信息（避免恢复时冲突） |
| `FLUSH BINARY LOGS` | 备份后刷新 binlog，标记全量备份时间点 |

### 2.3 binlog 增量备份

#### 2.3.1 备份脚本 `scripts/backup-mysql-binlog.sh`

```bash
#!/bin/bash
# ============================================================
# CGC-PMS MySQL binlog 持续备份脚本
# 用法: ./backup-mysql-binlog.sh
# crontab: */30 * * * * /data/backups/cgc-pms/scripts/backup-mysql-binlog.sh
# ============================================================

set -euo pipefail

CONTAINER_NAME="cgc-pms-mysql"
BACKUP_DIR="/data/backups/cgc-pms/mysql/binlog"
RETENTION_DAYS=7
LOG_FILE="/data/backups/cgc-pms/logs/backup.log"

mkdir -p "${BACKUP_DIR}"

# === 复制 MySQL 容器内的 binlog 文件 ===
# 使用 docker cp 而非卷挂载，以便备份到独立存储
docker exec "${CONTAINER_NAME}" ls /var/lib/mysql/mysql-bin.[0-9]* 2>/dev/null | while read -r binlog; do
    binlog_name=$(basename "${binlog}")
    dest="${BACKUP_DIR}/${binlog_name}"

    # 跳过已备份的文件
    if [ -f "${dest}" ]; then
        continue
    fi

    docker cp "${CONTAINER_NAME}:${binlog}" "${dest}" 2>/dev/null && \
        echo "[$(date '+%Y-%m-%d %H:%M:%S')] Binlog copied: ${binlog_name}" | tee -a "${LOG_FILE}"
done

# === 清理过期 binlog ===
find "${BACKUP_DIR}" -name "mysql-bin.[0-9]*" -mtime +${RETENTION_DAYS} -delete 2>/dev/null || true
```

#### 2.3.2 备份策略

| 项目 | 配置 |
|------|------|
| 执行频率 | 每 30 分钟（crontab `*/30 * * * *`） |
| 保留周期 | 7 天 |
| 存储位置 | `/data/backups/cgc-pms/mysql/binlog/` |
| 全量 + binlog 配合 | 全量备份后刷新 binlog，恢复时：全量 → binlog 增量 |

## 3. MinIO 文件备份策略

### 3.1 环境准备

在备份服务器上安装 MinIO Client (`mc`)：

```bash
# Linux
wget https://dl.min.io/client/mc/release/linux-amd64/mc
chmod +x mc
sudo mv mc /usr/local/bin/

# 配置 MinIO 别名
mc alias set cgc-pms http://localhost:9000 admin $MINIO_ROOT_PASSWORD
```

### 3.2 增量镜像备份

#### 3.2.1 备份脚本 `scripts/backup-minio-mirror.sh`

```bash
#!/bin/bash
# ============================================================
# CGC-PMS MinIO 文件增量镜像备份
# 用法: ./backup-minio-mirror.sh
# crontab: 0 3 * * * /data/backups/cgc-pms/scripts/backup-minio-mirror.sh
# ============================================================

set -euo pipefail

MINIO_ALIAS="cgc-pms"
MINIO_BUCKET="cgc-pms-files"          # 根据实际 bucket 名称调整
BACKUP_BASE="/data/backups/cgc-pms/minio/mirror"
BACKUP_DIR="${BACKUP_BASE}/$(date +%Y%m%d_%H%M%S)"
LOG_FILE="/data/backups/cgc-pms/logs/backup.log"

mkdir -p "${BACKUP_DIR}"

echo "[$(date '+%Y-%m-%d %H:%M:%S')] Starting MinIO mirror backup..." | tee -a "${LOG_FILE}"

# === 增量镜像同步 ===
mc mirror "${MINIO_ALIAS}/${MINIO_BUCKET}" "${BACKUP_DIR}/${MINIO_BUCKET}" \
    --overwrite \
    --remove \
    2>&1 | tee -a "${LOG_FILE}"

if [ ${PIPESTATUS[0]} -eq 0 ]; then
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] MinIO backup SUCCESS: ${BACKUP_DIR}" | tee -a "${LOG_FILE}"
else
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] MinIO backup FAILED!" | tee -a "${LOG_FILE}"
    exit 1
fi

# === 保留最近 3 个备份版本 ===
ls -1dt "${BACKUP_BASE}"/*/ 2>/dev/null | tail -n +4 | xargs rm -rf 2>/dev/null || true

echo "[$(date '+%Y-%m-%d %H:%M:%S')] MinIO cleanup done." | tee -a "${LOG_FILE}"
```

#### 3.2.2 `mc mirror` 关键参数

| 参数 | 作用 |
|------|------|
| `--overwrite` | 覆盖目标端旧版本文件 |
| `--remove` | 删除目标端多余文件（源端已删除的文件在目标端也删除） |
| 无 `--watch` | 不使用持续同步模式（与 crontab 配合，定期执行） |

#### 3.2.3 备用方案：rclone

若 MinIO Client 不可用，可使用 rclone：

```bash
# 配置 rclone S3 兼容端点
rclone config create cgc-minio s3 \
    provider Minio \
    endpoint http://localhost:9000 \
    access_key_id admin \
    secret_access_key $MINIO_ROOT_PASSWORD

# 增量同步
rclone sync cgc-minio:cgc-pms-files /data/backups/cgc-pms/minio/mirror/
```

### 3.3 备份策略

| 项目 | 配置 |
|------|------|
| 执行频率 | 每日凌晨 3:00（crontab `0 3 * * *`） |
| 保留版本 | 最近 3 次备份（含校验的完整镜像） |
| 存储位置 | `/data/backups/cgc-pms/minio/mirror/YYYYmmdd_HHMMSS/` |
| 同步方式 | 增量镜像（仅传输变更文件） |

## 4. Redis 数据边界

### 4.1 Redis 使用的数据类型

| 用途 | 数据结构 | 不可恢复时的后果 | 备份必要性 |
|------|---------|----------------|-----------|
| JWT 黑名单（登出 token） | SET（TTL = token 过期时间） | 已登出 token 恢复有效（安全风险低：token 本身有短时效） | ❌ 不需 |
| Refresh Token 黑名单 | SET（TTL = 7 天） | 已轮换的 refresh token 恢复有效（低风险） | ❌ 不需 |
| SSE 通知推送状态 | PUB/SUB 频道 | SSE 连接断开，用户刷新页面即可重连 | ❌ 不需 |
| 业务缓存（如角色/菜单） | STRING/HASH（TTL ≤ 30 分钟） | 缓存未命中，自动从 MySQL 重新加载 | ❌ 不需 |
| Spring Session（如有） | — | 当前部署未使用 Spring Session | ❌ 不需 |

### 4.2 结论

**Redis 不需要备份**。理由：

1. 所有 Redis 数据均为**可重建的缓存数据**，源数据在 MySQL 中
2. JWT 黑名单和 Refresh Token 黑名单均有 TTL 自动过期
3. 当前 Docker Compose 已启用 AOF 持久化（`--appendonly yes`），Redis 容器重启后可自动恢复最近的缓存状态
4. 若 Redis 数据完全丢失，仅影响当前正使用的 token 黑名单（安全窗口极小），不影响业务数据完整性

### 4.3 极端场景下 Redis 恢复

若发生 Redis 卷完全损毁：

```bash
# 1. 删除并重建 Redis 卷
docker compose down redis
docker volume rm deploy_redis-data
docker compose up -d redis

# 2. 验证 Redis 可用
docker exec cgc-pms-redis redis-cli -a $REDIS_PASSWORD ping
# 预期: PONG

# 3. 无需数据导入 —— 应用启动后自动填充缓存
```

## 5. 恢复步骤

### 5.1 MySQL 恢复（全量 + binlog 时间点恢复）

#### 5.1.1 恢复到最新状态

```bash
#!/bin/bash
# ============================================================
# CGC-PMS MySQL 恢复到最新状态
# 用法: ./restore-mysql-latest.sh
# ============================================================

set -euo pipefail

CONTAINER_NAME="cgc-pms-mysql"
DB_NAME="cgc_pms"
DB_USER="root"
DB_PASS="$MYSQL_ROOT_PASSWORD"
BACKUP_DIR="/data/backups/cgc-pms/mysql"

# === 步骤 1: 找到最新的全量备份 ===
LATEST_FULL=$(ls -1t "${BACKUP_DIR}/full/${DB_NAME}_full_"*.sql.gz 2>/dev/null | head -1)

if [ -z "${LATEST_FULL}" ]; then
    echo "ERROR: No full backup found in ${BACKUP_DIR}/full/"
    exit 1
fi

echo "=== Using full backup: ${LATEST_FULL} ==="

# === 步骤 2: 停止后端服务（防止新数据写入） ===
# docker compose stop backend  # 如果后端在 docker-compose 中

# === 步骤 3: 恢复全量备份 ===
echo "=== Step 1/3: Restoring full backup... ==="

# 先删除旧数据库（可选，如果数据已损坏必须执行）
docker exec "${CONTAINER_NAME}" mysql -u "${DB_USER}" -p"${DB_PASS}" \
    -e "DROP DATABASE IF EXISTS ${DB_NAME}; CREATE DATABASE ${DB_NAME} DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;"

# 导入全量备份
gunzip -c "${LATEST_FULL}" | docker exec -i "${CONTAINER_NAME}" \
    mysql -u "${DB_USER}" -p"${DB_PASS}" "${DB_NAME}"

echo "=== Full backup restored. ==="

# === 步骤 4: 应用 binlog 增量（恢复到最新） ===

# 从全量备份中提取 binlog 位置
# 方式 A: 查看备份文件的 CHANGE MASTER 注释
BINLOG_FILE=$(gunzip -c "${LATEST_FULL}" | grep "^-- CHANGE MASTER TO MASTER_LOG_FILE" | head -1 | sed "s/.*='\(.*\)'.*/\1/")
BINLOG_POS=$(gunzip -c "${LATEST_FULL}" | grep "^-- CHANGE MASTER TO MASTER_LOG_POS" | head -1 | sed "s/.*=\([0-9]*\).*/\1/")

if [ -n "${BINLOG_FILE}" ] && [ -n "${BINLOG_POS}" ]; then
    echo "=== Step 2/3: Applying binlog from ${BINLOG_FILE}:${BINLOG_POS}... ==="

    # 生成 binlog 恢复 SQL
    docker exec "${CONTAINER_NAME}" mysqlbinlog \
        --start-position="${BINLOG_POS}" \
        /var/lib/mysql/"${BINLOG_FILE}" \
        /var/lib/mysql/mysql-bin.[0-9]* \
        | docker exec -i "${CONTAINER_NAME}" \
            mysql -u "${DB_USER}" -p"${DB_PASS}"

    echo "=== Binlog applied. ==="
else
    echo "=== Step 2/3: No binlog position found in backup, using backup-only recovery... ==="
fi

# === 步骤 5: 验证恢复 ===
echo "=== Step 3/3: Verifying recovery... ==="

# 检查关键表行数
TABLES_TO_CHECK=(
    "sys_user"
    "ct_contract"
    "wf_instance"
    "pm_project"
    "md_partner"
    "cost_ledger"
    "pay_invoice"
    "mat_stock"
    "sys_notification"
)

for table in "${TABLES_TO_CHECK[@]}"; do
    COUNT=$(docker exec "${CONTAINER_NAME}" mysql -u "${DB_USER}" -p"${DB_PASS}" \
        -N -e "SELECT COUNT(*) FROM ${DB_NAME}.${table};" 2>/dev/null || echo "ERROR")
    echo "  ${table}: ${COUNT} rows"
done

# 检查 Flyway 迁移历史
MIGRATION_COUNT=$(docker exec "${CONTAINER_NAME}" mysql -u "${DB_USER}" -p"${DB_PASS}" \
    -N -e "SELECT COUNT(*) FROM ${DB_NAME}.flyway_schema_history WHERE success=1;" 2>/dev/null || echo "ERROR")
echo "  flyway_schema_history: ${MIGRATION_COUNT} successful migrations"

echo "=== Recovery verification complete. ==="
```

#### 5.1.2 恢复到指定时间点（PITR）

```bash
#!/bin/bash
# ============================================================
# CGC-PMS MySQL 时间点恢复 (Point-in-Time Recovery)
# 用法: ./restore-mysql-pitr.sh "2026-06-13 14:30:00"
# ============================================================

set -euo pipefail

TARGET_TIME="${1}"
CONTAINER_NAME="cgc-pms-mysql"
DB_NAME="cgc_pms"
DB_USER="root"
DB_PASS="$MYSQL_ROOT_PASSWORD"
BACKUP_DIR="/data/backups/cgc-pms/mysql"

if [ -z "${TARGET_TIME}" ]; then
    echo "Usage: $0 'YYYY-MM-DD HH:MM:SS'"
    exit 1
fi

echo "=== Restoring ${DB_NAME} to point-in-time: ${TARGET_TIME} ==="

# 找到目标时间之前的最新全量备份
LATEST_FULL=$(ls -1t "${BACKUP_DIR}/full/${DB_NAME}_full_"*.sql.gz | while read -r f; do
    f_ts=$(basename "$f" | sed 's/.*_full_\([0-9]\{8\}_[0-9]\{6\}\).*/\1/')
    if [[ "${f_ts}" < "$(date -d "${TARGET_TIME}" +%Y%m%d_%H%M%S 2>/dev/null || echo 99999999_999999)" ]]; then
        echo "$f"
        break
    fi
done | head -1)

if [ -z "${LATEST_FULL}" ]; then
    echo "ERROR: No full backup found before ${TARGET_TIME}"
    exit 1
fi

echo "Using full backup: ${LATEST_FULL}"

# === 恢复全量备份 ===
docker exec "${CONTAINER_NAME}" mysql -u "${DB_USER}" -p"${DB_PASS}" \
    -e "DROP DATABASE IF EXISTS ${DB_NAME}; CREATE DATABASE ${DB_NAME} DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_0900_ai_ci;"

gunzip -c "${LATEST_FULL}" | docker exec -i "${CONTAINER_NAME}" \
    mysql -u "${DB_USER}" -p"${DB_PASS}" "${DB_NAME}"

# === 应用 binlog 到指定时间点 ===
BINLOG_FILE=$(gunzip -c "${LATEST_FULL}" | grep "^-- CHANGE MASTER TO MASTER_LOG_FILE" | head -1 | sed "s/.*='\(.*\)'.*/\1/")
BINLOG_POS=$(gunzip -c "${LATEST_FULL}" | grep "^-- CHANGE MASTER TO MASTER_LOG_POS" | head -1 | sed "s/.*=\([0-9]*\).*/\1/")

docker exec "${CONTAINER_NAME}" mysqlbinlog \
    --start-position="${BINLOG_POS}" \
    --stop-datetime="${TARGET_TIME}" \
    /var/lib/mysql/"${BINLOG_FILE}" \
    /var/lib/mysql/mysql-bin.[0-9]* \
    | docker exec -i "${CONTAINER_NAME}" mysql -u "${DB_USER}" -p"${DB_PASS}"

echo "=== PITR recovery to '${TARGET_TIME}' complete. ==="
```

### 5.2 MinIO 文件恢复

```bash
#!/bin/bash
# ============================================================
# CGC-PMS MinIO 文件恢复
# 用法: ./restore-minio.sh
# ============================================================

set -euo pipefail

MINIO_ALIAS="cgc-pms"
MINIO_BUCKET="cgc-pms-files"
BACKUP_BASE="/data/backups/cgc-pms/minio/mirror"

# === 步骤 1: 找到最新备份 ===
LATEST_BACKUP=$(ls -1dt "${BACKUP_BASE}"/*/ 2>/dev/null | head -1)

if [ -z "${LATEST_BACKUP}" ]; then
    echo "ERROR: No MinIO backup found in ${BACKUP_BASE}/"
    exit 1
fi

echo "=== Using backup: ${LATEST_BACKUP} ==="

# === 步骤 2: 确保 bucket 存在 ===
mc mb "${MINIO_ALIAS}/${MINIO_BUCKET}" 2>/dev/null || true

# === 步骤 3: 恢复文件（反向镜像） ===
echo "=== Restoring files to MinIO... ==="
mc mirror "${LATEST_BACKUP}${MINIO_BUCKET}" "${MINIO_ALIAS}/${MINIO_BUCKET}" \
    --overwrite

if [ $? -eq 0 ]; then
    echo "=== MinIO restore SUCCESS. ==="
else
    echo "=== MinIO restore FAILED! ==="
    exit 1
fi

# === 步骤 4: 验证文件数量和总大小 ===
echo "=== Verifying restored files... ==="
mc du "${MINIO_ALIAS}/${MINIO_BUCKET}"
```

### 5.3 恢复后验证步骤

无论哪种恢复方式，恢复完成后必须执行以下验证：

### 7.3 备份验证脚本 `scripts/verify-backup.sh`

```bash
#!/bin/bash
# ============================================================
# 每周备份完整性验证
# crontab: 0 4 * * 0
# ============================================================

set -euo pipefail

BACKUP_DIR="/data/backups/cgc-pms/mysql"
LOG_FILE="/data/backups/cgc-pms/logs/backup.log"

echo "[$(date)] ===== Backup Verification Started =====" | tee -a "${LOG_FILE}"

# 1. 检查最近 24h 是否有全量备份
RECENT_FULL=$(find "${BACKUP_DIR}/full" -name "*.sql.gz" -mtime -1 | wc -l)
if [ "${RECENT_FULL}" -gt 0 ]; then
    echo "  [PASS] Full backup exists (${RECENT_FULL} files in last 24h)" | tee -a "${LOG_FILE}"
else
    echo "  [FAIL] No full backup in last 24h!" | tee -a "${LOG_FILE}"
fi

# 2. 检查最近 1h 是否有 binlog 备份
RECENT_BINLOG=$(find "${BACKUP_DIR}/binlog" -name "mysql-bin.*" -mmin -60 | wc -l)
if [ "${RECENT_BINLOG}" -gt 0 ]; then
    echo "  [PASS] Binlog backup exists (${RECENT_BINLOG} files in last 1h)" | tee -a "${LOG_FILE}"
else
    echo "  [WARN] No binlog backup in last 1h" | tee -a "${LOG_FILE}"
fi

# 3. 验证最新全量备份可解压
LATEST=$(ls -1t "${BACKUP_DIR}/full/"*.sql.gz 2>/dev/null | head -1)
if [ -n "${LATEST}" ]; then
    if gzip -t "${LATEST}" 2>/dev/null; then
        SIZE=$(du -h "${LATEST}" | cut -f1)
        echo "  [PASS] Latest backup valid: ${LATEST} (${SIZE})" | tee -a "${LOG_FILE}"
    else
        echo "  [FAIL] Latest backup corrupted: ${LATEST}" | tee -a "${LOG_FILE}"
    fi
fi

echo "[$(date)] ===== Backup Verification Complete =====" | tee -a "${LOG_FILE}"
```

### 8.3 注意事项

1. **生产环境凭据**: 本文档中的凭据（`$MYSQL_ROOT_PASSWORD`, `$MINIO_ROOT_PASSWORD` 等）为开发环境默认值。**生产环境必须通过 `deploy/.env` 更换所有密码**。
2. **备份存储**: 备份文件不应与 MySQL 数据存储在同一物理磁盘上，避免单点故障。建议同步到 NAS 或云存储。
3. **binlog 格式**: 必须使用 `ROW` 格式（已在上方 `custom.cnf` 中配置），否则时间点恢复不精确。
4. **Docker volume 备份**: 除逻辑备份外，建议定期备份 Docker 卷数据（`docker run --rm -v deploy_mysql-data:/data -v /backup:/backup alpine tar czf /backup/mysql-volume.tar.gz -C /data .`）。此方法与 mysqldump 互补：数据文件恢复速度更快，但需停机且不可跨版本。
5. **恢复顺序**: 必须先恢复 MySQL，再恢复 MinIO，最后启动后端服务。Flyway 在应用启动时自动执行迁移，MySQL 恢复后的状态必须是完整且一致的。
6. **应用停机窗口**: 执行恢复操作前，必须停止后端应用（`docker compose stop backend`），防止恢复期间有新数据写入导致不一致。
7. **GTID**: 若生产环境启用了 GTID 复制，备份脚本中 `--set-gtid-purged=OFF` 参数需根据实际情况调整。当前单实例部署不影响。
8. **Windows 环境**: PowerShell 续行符为 `` ` ``（反引号），CMD 为 `^`。本文档示例以 Bash 为主要脚本语言，Windows 下请参考 §7.2 的 PowerShell 版本。
