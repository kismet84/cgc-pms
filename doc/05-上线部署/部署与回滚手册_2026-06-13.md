## 1. 前置条件

### 1.1 硬件要求

| 资源 | 最低配置 | 推荐配置 |
|------|---------|---------|
| CPU | 4 核 | 8 核 |
| 内存 | 8 GB | 16 GB |
| 磁盘 | 50 GB（不含业务数据增长） | 200 GB SSD |
| 网络 | 出站访问 Docker Hub / Maven Central | 同左 |

系统共运行 5 个容器：

| 容器 | 镜像 | 内存限制 |
|------|------|---------|
| MySQL 8.0 | `mysql:8.0` | 512 MB |
| Redis 7 | `redis:7-alpine` | 256 MB |
| MinIO | `minio/minio:RELEASE.2024-10-13T13-34-11Z` | 512 MB |
| Backend（Java 21） | 自构建 `cgc-pms-backend:latest` | 1 GB |
| Frontend（Nginx + Vue 3） | 自构建 `cgc-pms-frontend:latest` | 128 MB |

**总计**：运行时内存约 2.4 GB（不含操作系统开销）。建议宿主机可用内存不低于 6 GB。

### 1.2 软件要求

| 软件 | 最低版本 | 验证命令 |
|------|---------|---------|
| Docker Engine | 24.0+ | `docker --version` |
| Docker Compose | v2.0+（`docker compose` 子命令） | `docker compose version` |
| Git | 2.30+ | `git --version` |
| OpenSSL | 1.1+（生成自签证书用） | `openssl version` |

> Docker Compose v1（`docker-compose` 独立二进制）已停止维护。本手册所有命令使用 `docker compose`（v2 插件形式）。如果仍在使用 v1，请将命令中的 `docker compose` 替换为 `docker-compose`。

### 1.3 端口规划

| 端口 | 用途 | 对外暴露 |
|------|------|---------|
| 80 | HTTP 入口（自动 301 跳转 HTTPS） | 是 |
| 443 | HTTPS 入口（Nginx 反向代理） | 是 |
| 8080 | 后端 API（调试用，生产建议关闭） | 可选 |
| 3306 | MySQL（仅 Docker 内部网络） | 否（默认） |
| 6379 | Redis（仅 Docker 内部网络） | 否（默认） |
| 9000 | MinIO API（仅 Docker 内部网络） | 否（默认） |
| 9001 | MinIO Console（仅 Docker 内部网络） | 否（默认） |

生产部署时，MySQL、Redis、MinIO 端口**默认不对外暴露**。如需从宿主机直接访问（例如 DBA 工具连接），可在 `deploy/docker-compose.prod.yml` 中取消对应 `ports` 注释。

## 2. 环境准备

### 2.1 克隆代码仓库

**Bash / Linux / macOS：**

```bash
git clone https://github.com/your-org/cgc-pms.git
cd cgc-pms
git checkout main
```

**PowerShell / Windows：**

```powershell
git clone https://github.com/your-org/cgc-pms.git
Set-Location cgc-pms
git checkout main
```

### 2.2 生成 SSL 证书

生产环境应使用 CA 签发的正式证书（如 Let's Encrypt）。如果仅为内网测试或首次部署验证，可生成自签名证书：

**Bash / Linux / macOS：**

```bash
mkdir -p deploy/ssl
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout deploy/ssl/server.key \
  -out deploy/ssl/server.crt \
  -subj "/CN=your-domain.com"
```

**PowerShell / Windows：**

```powershell
New-Item -ItemType Directory -Force -Path deploy\ssl
openssl req -x509 -nodes -days 365 -newkey rsa:2048 `
  -keyout deploy\ssl\server.key `
  -out deploy\ssl\server.crt `
  -subj "/CN=your-domain.com"
```

> 将 `your-domain.com` 替换为实际域名。使用自签名证书时，浏览器会显示"不安全"警告。Nginx 配置中的 HSTS（`Strict-Transport-Security`）头部在自签名证书场景下建议临时注释掉（编辑 `frontend-admin/nginx.conf` 第 68 行）。

### 2.3 创建工作目录

确保以下目录结构存在（部分由 Git 仓库提供，部分需手动创建）：

```
cgc-pms/
├── deploy/
│   ├── docker-compose.prod.yml    # 生产编排文件（Git 仓库已有）
│   ├── .env                        # 环境变量（需从 .env.example 复制并填写）
│   ├── .env.example                # 环境变量模板（Git 仓库已有）
│   ├── ssl/                        # SSL 证书目录（需手动创建）
│   │   ├── server.crt
│   │   └── server.key
│   ├── mysql/
│   │   ├── conf.d/                 # MySQL 自定义配置（可选）
│   │   └── init/                   # MySQL 初始化脚本（可选）
├── backend/                        # 后端源码（Git 仓库已有，构建用）
└── frontend-admin/                 # 前端源码（Git 仓库已有，构建用）
```

### 3.3 完整环境变量清单

| 变量名 | 说明 | 默认值 | 生产建议值 | 必填 |
|--------|------|--------|-----------|------|
| `MYSQL_ROOT_PASSWORD` | MySQL root 密码 | 无 | 随机 32 位强密码 | ✅ |
| `MYSQL_DATABASE` | 数据库名 | `cgc_pms` | 保持默认 | 否 |
| `MYSQL_USER` | 应用数据库用户 | `cgc` | 保持默认 | 否 |
| `MYSQL_PASSWORD` | 应用数据库密码 | 无 | 随机 24 位强密码 | ✅ |
| `REDIS_PASSWORD` | Redis 认证密码 | 无 | 随机 24 位强密码 | ✅ |
| `MINIO_ROOT_USER` | MinIO 管理员用户名 | 无 | `minioadmin` 或自定义 | ✅ |
| `MINIO_ROOT_PASSWORD` | MinIO 管理员密码 | 无 | 随机 24 位强密码 | ✅ |
| `JWT_SECRET` | JWT 签名密钥 | 无 | 随机 64 位 Base64 字符串 | ✅ |

> **重要提示**：`JWT_SECRET` 在 `.env.example` 中未列出，但在 `docker-compose.prod.yml` 中被引用。请在 `.env` 末尾手动添加此行。生产环境必须使用高强度随机密钥，**切勿使用示例值或弱密码**。

生成随机密钥的命令：

**Bash / PowerShell：**

```bash
# 生成 64 字符的随机 JWT 密钥
openssl rand -base64 48
```

将输出完整粘贴为 `JWT_SECRET` 的值。

## 4. 构建镜像

### 4.1 构建命令

在项目根目录执行：

**Bash / PowerShell（命令相同）：**

```bash
docker compose -f deploy/docker-compose.prod.yml build
```

### 4.2 构建过程说明

构建分两个阶段：

**阶段 1：Backend 后端构建**

- 使用 `maven:3.9-eclipse-temurin-21` 镜像
- 先下载依赖（利用 Docker 层缓存加速后续构建）
- 执行 `mvnw clean package -DskipTests -B -q`（跳过测试以加快生产构建）
- 产出 `cgc-pms-backend.jar`（Spring Boot fat jar）

**阶段 2：Backend 运行时镜像**

- 使用 `eclipse-temurin:21-jre`（精简 JRE，不含 JDK）
- 创建非 root 用户 `appuser` 运行应用
- JVM 参数：
  - `-Xms512m`：初始堆 512 MB
  - `-Xmx1g`：最大堆 1 GB
  - `-XX:+UseContainerSupport`：尊重容器 cgroup 限制
  - `-XX:MaxRAMPercentage=75.0`：自适应堆上限
- 暴露端口 8080

**阶段 3：Frontend 前端构建**

- 使用 `node:20-alpine` 镜像
- 安装 pnpm 11.0.9
- 执行 `pnpm install --frozen-lockfile`（锁定依赖版本）
- 执行 `pnpm build`（含 vue-tsc 类型检查 + Vite 打包）

**阶段 4：Frontend 运行时镜像**

- 使用 `nginx:1.27-alpine`
- 将构建产物拷贝到 `/usr/share/nginx/html/`
- 挂载 Nginx 配置文件（反向代理 `/api/` 到 backend）
- 暴露端口 80 和 443

### 4.3 构建参数与优化

| 参数 | 说明 |
|------|------|
| `--no-cache` | 完全重新构建（忽略 Docker 层缓存），首次部署或依赖变更时使用 |
| `--parallel` | 并行构建 backend 和 frontend（Docker Compose v2 默认行为） |
| `--build-arg` | 传递构建参数（当前无需额外参数） |

强制无缓存重建示例：

```bash
docker compose -f deploy/docker-compose.prod.yml build --no-cache
```

### 4.4 镜像打标签（推荐）

为便于回滚和版本追溯，构建完成后为镜像打上日期或版本标签：

**Bash：**

```bash
VERSION=$(date +%Y%m%d-%H%M%S)
docker tag cgc-pms-backend:latest cgc-pms-backend:${VERSION}
docker tag cgc-pms-frontend:latest cgc-pms-frontend:${VERSION}
```

**PowerShell：**

```powershell
$VERSION = Get-Date -Format "yyyyMMdd-HHmmss"
docker tag cgc-pms-backend:latest cgc-pms-backend:${VERSION}
docker tag cgc-pms-frontend:latest cgc-pms-frontend:${VERSION}
```

## 5. 启动服务

### 5.1 启动命令

在项目根目录执行：

```bash
docker compose -f deploy/docker-compose.prod.yml up -d
```

`-d` 参数表示后台运行（detached mode）。

### 5.2 启动顺序

Docker Compose 按依赖关系编排启动顺序：

```
1. mysql ──────┐
2. redis ──────┼──→ 4. backend ──→ 5. frontend
3. minio ──────┘
```

具体行为：

| 阶段 | 服务 | 等待条件 | 预计耗时 |
|------|------|---------|---------|
| 1 | mysql | 无（首个启动） | 10-30 秒 |
| 1 | redis | 无（与 mysql 并行） | 3-5 秒 |
| 1 | minio | 无（与 mysql 并行） | 5-10 秒 |
| 2 | backend | mysql **且** redis **且** minio 均 `healthy` | 60-90 秒（含 Spring Boot 启动 + Flyway 迁移） |
| 3 | frontend | backend `healthy` | 5-10 秒 |

> Backend 设置了 `start_period: 60s` 健康检查宽限期，因为 Spring Boot 启动 + Flyway 自动执行 40 个数据库迁移脚本需要一定时间。在此期间即使健康检查失败也不会重启容器。

### 5.3 首次启动数据库迁移

Backend 首次启动时会自动执行 Flyway 数据库迁移（V1 至 V40，共 40 个脚本），创建 55 张业务表并初始化种子数据（管理员用户、角色、菜单权限、审批模板等）。无需手动执行任何 SQL。

### 5.4 查看启动日志

**Bash / PowerShell：**

```bash
# 查看所有容器日志（跟随模式）
docker compose -f deploy/docker-compose.prod.yml logs -f

# 仅查看 backend 日志
docker compose -f deploy/docker-compose.prod.yml logs -f backend

# 查看最近 100 行
docker compose -f deploy/docker-compose.prod.yml logs --tail 100
```

### 5.5 查看运行状态

```bash
docker compose -f deploy/docker-compose.prod.yml ps
```

预期输出：5 个服务状态均为 `Up`（healthy），无 `restarting` 或 `unhealthy`。

### 6.6 一键验证脚本

将以下内容保存为 `deploy/health-check.sh`（Linux/macOS）：

```bash
#!/bin/bash
set -e
echo "=== CGC-PMS Health Check ==="
echo ""

# Backend health
echo -n "Backend health: "
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/api/actuator/health)
if [ "$HTTP_CODE" = "200" ]; then
    echo "OK ($HTTP_CODE)"
else
    echo "FAIL ($HTTP_CODE)"
fi

# Frontend
echo -n "Frontend: "
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:80/)
if [ "$HTTP_CODE" = "301" ] || [ "$HTTP_CODE" = "200" ]; then
    echo "OK ($HTTP_CODE)"
else
    echo "FAIL ($HTTP_CODE)"
fi

# Container status
echo ""
echo "Container status:"
docker compose -f deploy/docker-compose.prod.yml ps
```

Windows PowerShell 版本（保存为 `deploy/health-check.ps1`）：

```powershell
Write-Host "=== CGC-PMS Health Check ===" -ForegroundColor Cyan
Write-Host ""

# Backend health
try {
    $backend = Invoke-RestMethod -Uri "http://localhost:8080/api/actuator/health" -TimeoutSec 10
    Write-Host "Backend health: OK ($($backend.status))" -ForegroundColor Green
} catch {
    Write-Host "Backend health: FAIL" -ForegroundColor Red
}

# Frontend
try {
    $frontend = Invoke-WebRequest -Uri "http://localhost:80/" -TimeoutSec 10 -MaximumRedirection 0 -ErrorAction SilentlyContinue
    Write-Host "Frontend: OK ($($frontend.StatusCode))" -ForegroundColor Green
} catch {
    Write-Host "Frontend: FAIL" -ForegroundColor Red
}

Write-Host ""
Write-Host "Container status:" -ForegroundColor Cyan
docker compose -f deploy/docker-compose.prod.yml ps
```

## 7. 回滚步骤

### 7.1 回滚策略概述

本项目采用**镜像 Tag 管理 + 快速切换**的回滚策略：

- 每次发布前，将当前运行的镜像打上备份标签
- 新版本上线后如果出现问题，停止当前服务，将镜像 Tag 切回旧版本，重新启动
- Docker Volume 数据（MySQL、Redis、MinIO）**不受回滚影响**，数据完整性由数据库自身保证

### 7.2 部署前备份（预防性操作）

**每次发布新版本前**，执行以下命令保存当前运行版本的镜像：

**Bash：**

```bash
# 记录当前版本 Tag
BACKUP_TAG="backup-$(date +%Y%m%d-%H%M%S)"

# 为当前镜像打备份标签
docker tag cgc-pms-backend:latest cgc-pms-backend:${BACKUP_TAG}
docker tag cgc-pms-frontend:latest cgc-pms-frontend:${BACKUP_TAG}

echo "Backup tags: cgc-pms-backend:${BACKUP_TAG}, cgc-pms-frontend:${BACKUP_TAG}"
```

**PowerShell：**

```powershell
$BACKUP_TAG = "backup-" + (Get-Date -Format "yyyyMMdd-HHmmss")

docker tag cgc-pms-backend:latest cgc-pms-backend:${BACKUP_TAG}
docker tag cgc-pms-frontend:latest cgc-pms-frontend:${BACKUP_TAG}

Write-Host "Backup tags: cgc-pms-backend:${BACKUP_TAG}, cgc-pms-frontend:${BACKUP_TAG}"
```

### 7.3 数据库备份（强烈建议）

在生产环境部署前，备份 MySQL 数据卷：

**Bash：**

```bash
# 导出数据库 SQL
docker exec cgc-pms-mysql mysqldump -u root -p"${MYSQL_ROOT_PASSWORD}" \
  --single-transaction --routines --triggers \
  cgc_pms > backup_$(date +%Y%m%d_%H%M%S).sql
```

也可以通过 Docker Volume 直接备份：

```bash
docker run --rm -v cgc-pms_mysql-data:/data -v $(pwd):/backup alpine \
  tar czf /backup/mysql-backup-$(date +%Y%m%d-%H%M%S).tar.gz -C /data .
```

### 7.4 回滚操作步骤

假设新版本 `latest` 部署后发现严重问题，需要回滚到备份版本 `backup-20260613-120000`：

**步骤 1：停止当前服务**

```bash
docker compose -f deploy/docker-compose.prod.yml down
```

**步骤 2：切换镜像 Tag**

**Bash：**

```bash
# 移除当前的 latest 标签
docker rmi cgc-pms-backend:latest cgc-pms-frontend:latest 2>/dev/null || true

# 从备份版本重新打 latest 标签
docker tag cgc-pms-backend:backup-20260613-120000 cgc-pms-backend:latest
docker tag cgc-pms-frontend:backup-20260613-120000 cgc-pms-frontend:latest
```

**PowerShell：**

```powershell
docker rmi cgc-pms-backend:latest cgc-pms-frontend:latest 2>$null
docker tag cgc-pms-backend:backup-20260613-120000 cgc-pms-backend:latest
docker tag cgc-pms-frontend:backup-20260613-120000 cgc-pms-frontend:latest
```

**步骤 3：重新启动服务**

```bash
docker compose -f deploy/docker-compose.prod.yml up -d
```

**步骤 4：验证回滚结果**

参照[第 6 章验证部署](#6-验证部署)执行完整的健康检查和冒烟测试。

### 7.5 Flyway 迁移回滚注意事项

Flyway 数据库迁移是**不可逆的**（除非迁移脚本本身包含 `UNDO` 部分）。回滚镜像版本时：

- 如果新版本引入了**新增表或新增列**的迁移脚本，回滚旧版本镜像不会导致问题（旧版本代码不使用新表/新列）。
- 如果新版本**修改或删除了已有列**，回滚可能导致旧版本代码无法正常工作。
- **数据本身**：Docker Volume 持久化在宿主机，不受容器重建影响。

建议在数据库中保留迁移历史（`flyway_schema_history` 表），便于追溯每次变更。

## 8. 附录：常见问题

### 8.1 数据库连接失败

**现象**：Backend 启动后持续报错 `CommunicationsException: Communications link failure`。

**排查步骤**：

1. 检查 MySQL 容器是否健康：
   ```bash
   docker compose -f deploy/docker-compose.prod.yml ps mysql
   ```

2. 查看 MySQL 日志：
   ```bash
   docker compose -f deploy/docker-compose.prod.yml logs mysql
   ```

3. 如果 MySQL 容器反复重启，检查磁盘空间是否充足：
   ```bash
   df -h    # Linux/macOS
   ```
   ```powershell
   Get-PSDrive C  # PowerShell
   ```

4. 确认 `.env` 中的 `MYSQL_ROOT_PASSWORD` 和 `MYSQL_PASSWORD` 已正确填写，且未被引号包裹（`.env` 文件中不要加引号）。

5. 如果启动时 MySQL 数据卷损坏，可重建数据卷：
   ```bash
   docker compose -f deploy/docker-compose.prod.yml down -v
   # ⚠️ 警告：-v 会删除所有数据卷，仅适用于全新环境
   docker compose -f deploy/docker-compose.prod.yml up -d
   ```

### 8.2 端口冲突

**现象**：启动时报错 `port is already allocated`。

**排查步骤**：

1. 检查端口占用情况：

   **Bash：**
   ```bash
   sudo lsof -i :80
   sudo lsof -i :443
   sudo lsof -i :8080
   ```

   **PowerShell：**
   ```powershell
   netstat -ano | findstr ":80 "
   netstat -ano | findstr ":443 "
   netstat -ano | findstr ":8080 "
   ```

2. 如果其他服务占用了 80/443 端口（如宿主机上的 Nginx、Apache、IIS），先停止该服务，或修改 `deploy/docker-compose.prod.yml` 中 frontend 的端口映射（例如改为 `"8081:80"` 和 `"8443:443"`）。

3. 如果 8080 端口被占用且生产环境不需要对外暴露，直接注释掉 `backend` 服务的 `ports` 配置（backend 通过 Docker 内部网络与 frontend 通信，无需对外暴露）。

### 8.3 磁盘空间不足

**现象**：`no space left on device` 或 MySQL/MinIO 写入失败。

**排查与处理**：

1. 查看 Docker 磁盘占用：
   ```bash
   docker system df
   ```

2. 清理未使用的 Docker 资源（谨慎操作）：
   ```bash
   docker system prune -a
   # 交互式确认后清理：停止的容器、未使用的网络、悬空镜像、构建缓存
   ```

3. 检查数据卷实际占用：
   ```bash
   docker system df -v | grep -A 5 "Local Volumes"
   ```

4. 对于长期运行的生产环境，建议：
   - 配置 MySQL binlog 过期策略（`expire_logs_days`）
   - 为 MinIO 设置存储桶生命周期策略
   - 定期清理应用日志文件（`/var/log/cgc-pms/`）

### 8.4 Nginx 502 Bad Gateway

**现象**：浏览器访问时显示 502 Bad Gateway。

**原因**：前端 Nginx 反向代理到后端失败，通常是因为 backend 容器尚未就绪或已崩溃。

**排查步骤**：

1. 检查 backend 容器状态：
   ```bash
   docker compose -f deploy/docker-compose.prod.yml ps backend
   ```

2. 如果 backend 状态为 `unhealthy` 或反复 `restarting`，查看日志：
   ```bash
   docker compose -f deploy/docker-compose.prod.yml logs backend --tail 200
   ```

3. 常见原因：
   - **JWT_SECRET 未设置**：日志中出现 `JWT secret key must not be null or empty`。在 `.env` 中添加 `JWT_SECRET` 并重启。
   - **MySQL 未就绪**：backend 等待 MySQL 健康检查通过（最多 5 次重试 × 10 秒）。如果 MySQL 启动超过 50 秒仍不健康，检查 MySQL 日志。
   - **内存不足**：backend 分配了 1 GB 内存限制，如果宿主机可用内存不足，容器会被 OOM Killer 终止。增加宿主机内存或降低 backend 内存限制（修改 `deploy/docker-compose.prod.yml` 中的 `memory` 值）。

4. 手动测试后端连通性（从 frontend 容器内）：
   ```bash
   docker exec cgc-pms-frontend wget -qO- http://backend:8080/api/actuator/health
   ```

### 8.5 无法登录 / Token 过期异常

**现象**：登录时提示"用户名或密码错误"，或登录后立即被强制退出。

**排查步骤**：

1. 确认使用了正确的默认管理员账号：`admin` / `admin123`。

2. 如果修改过密码后忘记，可通过数据库重置：
   ```bash
   # 进入 MySQL 容器
   docker exec -it cgc-pms-mysql mysql -u root -p
   ```
   ```sql
   USE cgc_pms;
   -- BCrypt 加密的新密码 "admin123"（需用后端工具生成正确的 BCrypt 哈希）
   -- 或者直接通过后端 API 修改（推荐）
   ```

3. 检查 `.env` 中的 `JWT_SECRET` 是否在**所有后端容器**中一致。如果在运行期间修改了 `JWT_SECRET` 并重启，所有旧 Token 将立即失效，用户需重新登录。

4. 检查 Redis 是否正常运行：
   ```bash
   docker exec cgc-pms-redis redis-cli -a "$REDIS_PASSWORD" ping
   ```
   如果 Redis 不可用，Token 黑名单（登出机制）和 Refresh Token 管理将失效。

### 8.6 文件上传失败

**现象**：上传文件时报错或返回异常。

**排查步骤**：

1. 检查 MinIO 容器状态：
   ```bash
   docker compose -f deploy/docker-compose.prod.yml ps minio
   ```

2. 测试 MinIO 连通性（从 backend 容器内）：
   ```bash
   docker exec cgc-pms-backend wget -qO- http://minio:9000/minio/health/live
   ```

3. 确认 `.env` 中的 `MINIO_ROOT_USER` 和 `MINIO_ROOT_PASSWORD` 与 backend 中配置的 `MINIO_ACCESS_KEY` / `MINIO_SECRET_KEY` 一致。

4. 确保上传文件不超过 50 MB 且文件扩展名在允许列表中（`.exe`、`.jsp`、`.sh` 等危险类型会被拒绝）。

### 8.7 Docker Compose 命令找不到

**现象**：执行 `docker compose` 提示 `unknown command`。

**解决方案**：

- 确认 Docker Compose v2 已安装：`docker compose version`
- 如果只有 v1（独立二进制）：将所有 `docker compose` 替换为 `docker-compose`
- 安装/升级 Docker Compose v2：参考 [Docker 官方文档](https://docs.docker.com/compose/install/)

### 8.8 构建时依赖下载超时

**现象**：`docker compose build` 时 Maven 或 pnpm 下载依赖卡住/超时。

**解决方案**：

- **Maven**：编辑 `backend/.mvn/settings.xml`（如不存在则创建），配置国内镜像：
  ```xml
  <mirrors>
    <mirror>
      <id>aliyun</id>
      <mirrorOf>central</mirrorOf>
      <url>https://maven.aliyun.com/repository/public</url>
    </mirror>
  </mirrors>
  ```

- **pnpm**：在构建前设置镜像环境变量，或在 Dockerfile 中添加：
  ```dockerfile
  RUN pnpm config set registry https://registry.npmmirror.com
  ```

- 或使用 Docker BuildKit 的 `--mount=type=cache` 缓存依赖（需修改 Dockerfile 和构建命令）。

### 8.9 .env 文件中的特殊字符

**问题**：密码中包含 `$`、`#`、`!` 等特殊字符时，Docker Compose 可能解析异常。

**解决方案**：

- 避免在密码中使用 `$` 字符（会被 shell 解释为变量）。
- 如果必须使用，在 `.env` 文件中**不要**给值加引号，直接写值。
- 推荐使用 `openssl rand -base64 24` 生成的纯字母数字密码。

### 8.10 忘记环境变量导致的安全漏洞

**常见错误**：

- `JWT_SECRET` 留空 → 后端使用默认空密钥 → 攻击者可伪造任意用户 Token
- `REDIS_PASSWORD` 留空 → Redis 无密码保护 → 缓存数据可被任意访问
- 所有服务使用相同的弱密码 → 单点泄露导致全栈沦陷

**检查清单**：

```bash
# 检查所有必填变量是否已设置（不应有空行输出）
grep -E '^[A-Z_]+=$' deploy/.env
```

## 附录 C：环境变量快速检查脚本

**Bash**（保存为 `deploy/check-env.sh`）：

```bash
#!/bin/bash
REQUIRED_VARS=("MYSQL_ROOT_PASSWORD" "MYSQL_PASSWORD" "REDIS_PASSWORD" "MINIO_ROOT_USER" "MINIO_ROOT_PASSWORD" "JWT_SECRET")
MISSING=0

for VAR in "${REQUIRED_VARS[@]}"; do
    VALUE=$(grep "^${VAR}=" deploy/.env 2>/dev/null | cut -d'=' -f2)
    if [ -z "$VALUE" ]; then
        echo "❌ MISSING: $VAR"
        MISSING=1
    else
        echo "✅ $VAR = *** (set)"
    fi
done

if [ $MISSING -eq 1 ]; then
    echo ""
    echo "请编辑 deploy/.env 填入所有必填变量后重新部署。"
    exit 1
else
    echo ""
    echo "所有必填环境变量已设置。"
fi
```

**PowerShell**（保存为 `deploy/check-env.ps1`）：

```powershell
$requiredVars = @("MYSQL_ROOT_PASSWORD", "MYSQL_PASSWORD", "REDIS_PASSWORD", "MINIO_ROOT_USER", "MINIO_ROOT_PASSWORD", "JWT_SECRET")
$missing = 0

foreach ($var in $requiredVars) {
    $match = Select-String -Path "deploy\.env" -Pattern "^${var}=(.+)$" -ErrorAction SilentlyContinue
    if (-not $match -or [string]::IsNullOrWhiteSpace($match.Matches.Groups[1].Value)) {
        Write-Host "❌ MISSING: $var" -ForegroundColor Red
        $missing++
    } else {
        Write-Host "✅ $var = *** (set)" -ForegroundColor Green
    }
}

if ($missing -gt 0) {
    Write-Host ""
    Write-Host "请编辑 deploy\.env 填入所有必填变量后重新部署。" -ForegroundColor Yellow
    exit 1
} else {
    Write-Host ""
    Write-Host "所有必填环境变量已设置。" -ForegroundColor Green
}
```
