## 正式版基线说明

本文件已按正式开发交付口径整理，并统一以下架构基线：

1. 一期数据库默认采用 **MySQL 8.0**，审批规则、动态表单和节点规则使用 MySQL JSON 字段。
2. 后端审批引擎内部采用**事务内领域服务直接调用**，不通过 HTTP 调用自身接口。
3. PC Web 是一期完整管理后台；移动端以 uni-app 为现场主端，企业微信和微信小程序作为轻入口。
4. 移动端正式业务开发节奏后移：第 6 周启动基础工程，第 8 周左右进入真实业务功能。
5. 所有业务数据以 `project_id` 为根，涉及合同的业务以 `contract_id` 为纲，成本以业务事实归集，付款与成本口径分离。

## 1. 优化目标

本次优化目标不是简单增加菜单和页面，而是将系统从“模块台账型系统”升级为“全过程业务闭环系统”。

核心目标如下：

1. 建立以 `project_id` 为根的数据主线，所有合同、采购、分包、成本、付款、结算、资料均归集到项目。
2. 建立以 `contract_id` 为纲的合同主数据中心，避免分包、材料、付款、结算模块重复维护合同信息。
3. 建立以业务事实为依据的成本归集机制，成本来源于验收、计量、签证、合同变更、设备使用、劳务记录等业务单据。
4. 建立付款申请与实际支付分离机制，付款申请由业务发起，实际付款由财务系统或财务人员确认。
5. 建立从合同到执行、从执行到成本、从成本到付款、从付款到结算的全过程闭环。
6. 建立统一审批引擎，实现金额分级、异常升级、状态联动和业务强校验。
7. 建立角色化驾驶舱，让项目经理、商务经理、成本经理、财务人员、公司管理层看到不同重点数据。

## 2. 总体优化原则

系统优化应遵循以下原则：

```text
项目为根
合同为纲
执行为据
成本为果
付款为流
结算为终
```

对应到系统设计中：

| 原则 | 系统落地方式 |
|---|---|
| 项目为根 | 所有核心业务表必须包含 `project_id` |
| 合同为纲 | 合同统一在合同管理模块维护，其他模块通过 `contract_id` 引用 |
| 执行为据 | 采购、验收、入库、出库、分包计量、签证等作为业务事实 |
| 成本为果 | 成本由业务事实自动生成，不作为重复录入口 |
| 付款为流 | 付款区分付款条件、付款申请、付款审批、实际支付 |
| 结算为终 | 结算从合同、计量、验收、签证、付款、成本自动汇总 |

## 3. 当前系统主要问题判断

### 3.1 模块边界容易重叠

当前系统已有合同管理、材料设备管理、分包管理、成本管理、付款管理、竣工结算管理等模块，但如果实现时不强制数据关系，容易出现以下问题：

- 分包管理中重新维护分包合同；
- 材料设备管理中重复维护采购合同金额；
- 成本管理中手工录入合同金额、采购金额、分包金额；
- 付款管理中把付款金额误认为成本发生金额；
- 结算管理中重新录入一套结算数据，无法追溯来源。

### 3.2 页面完整但业务闭环不足

合同台账、新建合同、首页驾驶舱页面完成度较高，但仍需加强：

- 首页指标需要按角色分层；
- 合同台账需要体现付款、变更、结算、归档、风险状态；
- 新建合同页面字段较多，需要分步录入；
- 页面按钮需要绑定审批、校验、状态流转，不能只是静态操作；
- 右侧图表应支持点击下钻到明细数据。

### 3.3 审批规则需要后端强校验

审批流程文档已较完整，但落地时不能只做“流程流转”，必须同时做业务校验：

- 未签合同不得付款；
- 材料未验收不得付款；
- 分包未计量不得付款；
- 超合同金额必须先走合同变更；
- 超付款比例必须自动升级审批；
- 质保金未到期不得释放；
- 黑名单合作方禁止提交合同或付款。

## 4. 总体业务架构优化

### 4.1 优化后的业务主线

```text
项目管理
  ↓
合同管理
  ↓
采购执行 / 分包执行 / 设备租赁 / 劳务管理
  ↓
验收 / 入库 / 出库 / 计量 / 签证
  ↓
成本自动归集
  ↓
付款申请
  ↓
付款审批
  ↓
财务实际支付
  ↓
结算汇总
  ↓
项目经营分析
```

### 4.2 核心数据流

```text
Project
  ↓
Contract
  ↓
PurchaseOrder / SubcontractMeasure / MaterialReceipt / VariationOrder
  ↓
CostItem
  ↓
PaymentRequest
  ↓
PaymentRecord
  ↓
Settlement
```

### 4.3 核心对象关系

| 对象 | 主模块 | 关键字段 | 说明 |
|---|---|---|---|
| Project | 项目管理 | `project_id` | 所有业务的根节点 |
| Contract | 合同管理 | `contract_id` | 合同金额、条款、付款条件的唯一依据 |
| Partner | 合作方管理 | `partner_id` | 统一管理供应商、分包商、租赁商、服务商 |
| PurchaseOrder | 材料设备 | `project_id`, `contract_id`, `partner_id` | 采购执行单据 |
| MaterialReceipt | 材料设备 | `order_id`, `project_id`, `contract_id` | 材料成本依据 |
| SubcontractMeasure | 分包管理 | `project_id`, `contract_id`, `partner_id` | 分包成本依据 |
| VariationOrder | 变更签证 | `project_id`, `contract_id` | 动态成本调整依据 |
| CostItem | 成本管理 | `source_type`, `source_id` | 成本归集核心表 |
| PaymentRequest | 付款管理 | `project_id`, `contract_id`, `basis_id` | 付款申请 |
| PaymentRecord | 财务接口 | `pay_request_id` | 实际付款记录 |
| Settlement | 结算管理 | `project_id`, `contract_id` | 最终结算收口 |

### 5.1.3 新增优化点

## 5.2 合同管理模块优化

### 5.2.1 定位

合同管理是系统合同数据中心，负责统一维护总包合同、分包合同、采购合同、租赁合同、服务合同及补充协议。

其他模块只能引用合同，不得重复创建合同主数据。

### 5.2.2 合同类型统一

建议统一使用字段：

```text
contract_type
```

枚举值：

```text
总包合同
分包合同
采购合同
租赁合同
服务合同
补充协议
其他合同
```

### 5.2.3 合同状态拆分

当前系统容易将合同状态和审批状态混用，建议拆分为两个字段。

| 字段 | 示例值 | 说明 |
|---|---|---|
| `approval_status` | 草稿、审批中、已通过、已驳回、已撤回 | 审批状态 |
| `contract_status` | 未签订、已签订、履约中、已结算、已归档、已作废 | 合同业务状态 |

### 5.2.4 合同台账优化

合同台账列表建议默认字段：

| 字段 | 是否默认显示 |
|---|---|
| 合同编号 | 是 |
| 合同名称 | 是 |
| 合同类型 | 是 |
| 项目名称 | 是 |
| 合作方 | 是 |
| 合同金额含税 | 是 |
| 变更金额 | 是 |
| 当前合同金额 | 是 |
| 已付款金额 | 是 |
| 付款比例 | 是 |
| 签订日期 | 是 |
| 合同状态 | 是 |
| 审批状态 | 是 |
| 结算状态 | 是 |
| 归档状态 | 是 |
| 风险等级 | 是 |

支持列设置，用户可自定义隐藏或显示字段。

### 5.2.5 合同详情页优化

合同详情页建议分为以下页签：

```text
基本信息
合同清单
付款条件
履约节点
合同变更
付款记录
成本影响
结算信息
附件资料
审批记录
操作日志
```

### 5.2.6 新建合同页面优化

当前新建合同页面字段完整，但一次性展示过多。建议改为分段式录入：

```text
第一步：基本信息
第二步：合同方信息
第三步：金额税率信息
第四步：合同清单
第五步：付款条件
第六步：履约节点
第七步：附件与提交审批
```

每一步保存草稿，最后统一提交审批。

### 5.2.7 合同金额自动计算

合同金额相关字段建议自动计算：

```text
税额 = 合同金额含税 - 合同金额不含税
合同金额不含税 = 合同金额含税 / (1 + 税率)
质保金金额 = 合同金额含税 × 质保金比例
当前合同金额 = 原合同金额 + 已生效合同变更金额
付款比例 = 已支付金额 / 当前合同金额
```

### 5.2.8 合同审批通过后动作

合同审批通过后系统自动执行：

1. 更新 `ct_contract.approval_status = 已通过`；
2. 更新 `ct_contract.contract_status = 已签订`；
3. 生成合同台账记录；
4. 锁定合同金额；
5. 生成付款条件；
6. 如果是成本类合同，生成合同锁定成本；
7. 允许采购订单、分包任务、付款申请引用该合同；
8. 生成合同归档待办。

## 5.3 合作方管理模块优化

### 5.3.1 定位

合作方管理统一维护供应商、分包商、租赁商、服务商、业主、监理等单位信息。

### 5.3.2 合作方类型

```text
建设单位
供应商
分包商
租赁商
服务商
劳务单位
其他
```

### 5.3.3 合作方风险控制

合作方应增加以下字段：

| 字段 | 说明 |
|---|---|
| `blacklist_flag` | 是否黑名单 |
| `risk_level` | 风险等级 |

- 已存在合同允许查看，但所有新增业务需公司领导特批。

## 5.4 材料设备管理模块优化

### 5.4.1 定位

材料设备管理负责采购执行、材料验收、入库、出库、库存、使用追溯，不维护采购合同主数据。

### 5.4.2 采购订单三类场景

| 场景 | 规则 |
|---|---|
| 有合同采购 | 必须关联采购合同 |
| 零星采购 | 可不关联合同，但必须标记为零星采购 |
| 框架协议采购 | 关联框架合同，订单记录本次采购明细 |

### 5.4.3 采购业务闭环

```text
材料需求计划
  ↓
采购申请
  ↓
采购申请审批
  ↓
采购订单
  ↓
采购订单审批
  ↓
材料到货
  ↓
材料验收
  ↓
入库
  ↓
出库
  ↓
使用部位追溯
  ↓
材料成本生成
  ↓
付款申请
  ↓
采购结算
```

### 5.4.4 采购订单控制规则

| 控制项 | 规则 |
|---|---|
| 项目 | 必须关联 `project_id` |
| 合同 | 有合同采购必须关联 `contract_id` |
| 合作方 | 必须关联 `partner_id` |
| 订单金额 | 不得超过合同剩余可用金额 |
| 物资数量 | 不得明显超过需求计划 |
| 零星采购 | 必须填写零星采购原因 |
| 框架协议 | 必须记录本次订单金额和累计使用金额 |

### 5.4.5 材料验收控制规则

| 控制项 | 规则 |
|---|---|
| 无采购订单 | 原则上不允许验收 |
| 到货数量大于订单数量 | 触发超量预警 |
| 质量不合格 | 禁止入库，进入退场或复检流程 |
| 资料不全 | 可暂存，不形成正式成本 |
| 验收通过 | 自动生成材料成本 |
| 已验收金额 | 作为付款申请上限之一 |

### 5.4.6 材料成本生成规则

材料验收审批通过后，系统自动生成 `cost_item`：

```text
cost_item.project_id = mat_receipt.project_id
cost_item.contract_id = mat_receipt.contract_id
cost_item.partner_id = mat_receipt.partner_id
cost_item.cost_type = 材料
cost_item.amount = mat_receipt.total_amount
cost_item.source_type = MAT_RECEIPT
cost_item.source_id = mat_receipt.id
cost_item.cost_status = 已确认
```

### 5.5.1 定位

分包管理负责分包履约过程，不重复维护分包合同主数据。

分包合同来源于合同管理模块，分包管理只通过 `contract_id` 引用。

### 5.5.2 分包业务闭环

```text
分包合同签订
  ↓
分包任务下达
  ↓
分包进场
  ↓
分包进度填报
  ↓
质量安全确认
  ↓
分包计量
  ↓
计量审批
  ↓
分包成本生成
  ↓
分包付款申请
  ↓
财务付款
  ↓
分包评价
  ↓
分包结算
```

### 5.5.3 分包任务控制规则

| 控制项 | 规则 |
|---|---|
| 分包合同 | 必须引用已签订分包合同 |
| 分包单位 | 从合同合作方自动带出 |
| 作业范围 | 不得超出合同清单范围 |
| 计划工期 | 不得明显超出合同履约期 |
| 状态 | 未开始、进行中、暂停、已完成、已关闭 |

### 5.5.4 分包计量控制规则

| 控制项 | 规则 |
|---|---|

```text
cost_item.contract_id = sub_measure.contract_id
cost_item.partner_id = sub_measure.partner_id
cost_item.cost_type = 分包
cost_item.amount = sub_measure.net_amount
cost_item.source_type = SUB_MEASURE
cost_item.source_id = sub_measure.id
cost_item.cost_status = 已确认
```

## 5.6 成本管理模块优化

### 5.6.1 定位

成本管理负责成本归集、动态成本、成本分析、利润测算、成本预警，不作为重复录入入口。

### 5.6.2 成本口径统一

| 成本口径 | 计算逻辑 |
|---|---|
| 目标成本 | 项目初始控制目标 |
| 合同锁定成本 | 已签订成本类合同金额 |
| 已发生成本 | 已验收、已计量、已确认的业务成本 |
| 已付款成本 | 财务已实际支付金额 |
| 预计待发生成本 | 尚未发生但预计会发生的成本 |
| 动态成本 | 已发生成本 + 预计待发生成本 |
| 成本偏差 | 动态成本 - 目标成本 |
| 预计利润 | 合同收入 - 动态成本 |

### 5.6.3 成本来源

成本明细 `cost_item` 只能由以下来源生成：

| 来源 | `source_type` |
|---|---|
| 材料验收 | `MAT_RECEIPT` |
| 材料出库 | `MAT_ISSUE` |
| 分包计量 | `SUB_MEASURE` |
| 设备租赁 | `EQ_LEASE` |
| 劳务工资 | `LAB_PAYROLL` |
| 现场签证 | `VAR_ORDER` |
| 合同变更 | `CT_CHANGE` |
| 手工调整 | `MANUAL_ADJUST` |

其中 `MANUAL_ADJUST` 必须走审批，且必须填写原因。

### 5.6.4 成本台账优化

成本台账应支持以下维度查询：

- 项目；
- 合同；
- 合作方；
- 成本科目；
- 成本类型；
- 来源单据；
- 成本状态；
- 成本发生日期；
- 是否已结算；
- 是否已付款。

### 5.6.5 动态成本计算

建议每日定时或实时计算 `cost_summary`：

```text
目标成本 = cost_target 当前生效版本金额

合同锁定成本 = 已签订成本类合同金额合计

已发生成本 = cost_item 中 cost_status = 已确认 的金额合计

已付款金额 = pay_record 实际付款金额合计

预计待发生成本 = 合同剩余未发生金额 + 已确认预计签证 + 管理费预测

动态成本 = 已发生成本 + 预计待发生成本

成本偏差 = 动态成本 - 目标成本

预计利润 = 总包合同收入 + 已确认收入变更 - 动态成本
```

## 5.7 付款管理模块优化

### 5.7.1 定位

付款管理负责付款申请、付款审批、付款进度、付款预警；实际付款由财务系统或财务接口回写。

### 5.7.2 付款三层结构

| 层级 | 所属模块 | 说明 |
|---|---|---|
| 付款条件 | 合同管理 | 合同约定的付款节点、比例、质保金 |
| 付款申请 | 业务模块/付款管理 | 按验收、计量、合同节点发起 |
| 实际支付 | 财务系统/财务接口 | 实际付款金额、日期、凭证 |

### 5.7.3 付款申请控制规则

| 控制项 | 规则 |
|---|---|
| 未签合同 | 禁止付款，零星采购除外 |
| 材料未验收 | 禁止材料付款 |
| 分包未计量 | 禁止分包付款 |
| 超合同金额 | 必须先合同变更 |
| 超付款比例 | 自动升级审批 |
| 发票缺失 | 可审批但不允许实际支付，按企业规则配置 |
| 质保金未到期 | 禁止释放或总经理特批 |
| 实际付款 | 以财务系统回写为准 |

### 5.7.4 付款申请金额校验

付款申请提交时执行：

```text
本次申请金额 <= 当前合同金额 - 已审批付款金额

本次申请金额 <= 合同付款比例允许金额 - 已审批付款金额

材料付款：
本次申请金额 <= 已验收金额 - 已申请金额

分包付款：
本次申请金额 <= 已计量金额 - 已申请金额

质保金释放：
当前日期 >= 质保金到期日
```

### 5.7.5 付款申请审批通过后动作

```text
pay_request.approval_status = 已通过
pay_request.pay_status = 待付款
生成财务付款待办
推送财务系统
更新合同付款申请累计金额
更新首页待办和预警
```

### 5.7.6 财务付款回写后动作

```text
生成 pay_record
更新 pay_request.pay_status
更新 ct_contract.paid_amount
更新 cost_summary.paid_amount
更新付款比例
关闭或部分关闭付款待办
```

## 5.8 竣工结算管理模块优化

### 5.8.1 定位

竣工结算管理负责项目最终收口，不重新录入前端业务数据，而是从系统已有数据中汇总形成结算依据。

### 5.8.2 结算类型

```text
总包结算
分包结算
采购结算
租赁结算
服务结算
```

### 5.8.3 结算数据来源

| 来源 | 用途 |
|---|---|
| 合同主表 | 原合同金额 |
| 合同清单 | 结算清单基础 |
| 合同变更 | 调整合同金额 |
| 分包计量 | 分包结算依据 |
| 材料验收 | 采购结算依据 |
| 现场签证 | 变更结算依据 |
| 付款记录 | 已付金额 |
| 成本明细 | 成本复核 |
| 附件资料 | 审计归档 |

### 5.8.4 结算金额计算

```text
最终结算金额 = 原合同金额 + 已确认变更金额 + 已确认签证金额 - 扣款金额

未付金额 = 最终结算金额 - 已付款金额 - 质保金金额

质保金金额 = 最终结算金额 × 质保金比例
```

### 5.8.5 结算审批控制规则

| 控制项 | 规则 |
|---|---|
| 合同未签订 | 不允许结算 |
| 存在未审批变更 | 不允许结算确认 |
| 存在未确认计量 | 不允许分包结算确认 |
| 存在未验收材料 | 不允许采购结算确认 |
| 付款金额大于结算金额 | 触发红色预警 |
| 资料缺失 | 不允许归档 |
| 结算确认后修改 | 必须走结算调整审批 |

## 10. 预警规则优化

## 10.1 合同预警

| 预警 | 规则 |
|---|---|
| 合同即将到期 | 距合同结束日期 ≤ 30 天 |
| 合同已逾期 | 当前日期 > 合同结束日期且未结算 |
| 合同金额超限 | 当前合同金额 > 原合同金额且无有效变更 |
| 合同付款超比例 | 已付款金额 / 当前合同金额 > 合同约定比例 |
| 合同未归档 | 合同签订后 N 天未上传盖章文件 |

## 10.2 成本预警

| 预警 | 规则 |
|---|---|
| 动态成本超目标 | 动态成本 > 目标成本 |
| 材料成本超预算 | 材料成本 > 材料目标成本 |
| 分包成本超合同 | 分包计量累计金额 > 分包合同金额 |
| 成本无来源 | cost_item 缺少 source_type 或 source_id |
| 签证未确认 | 签证发起后 N 天未确认 |

## 10.3 付款预警

| 预警 | 规则 |
|---|---|
| 未验收付款 | 材料付款无验收依据 |
| 未计量付款 | 分包付款无计量依据 |
| 超合同付款 | 累计付款 > 当前合同金额 |
| 超比例付款 | 累计付款 > 约定付款比例 |
| 质保金提前释放 | 当前日期 < 质保金到期日 |
| 长期未付款 | 付款申请审批通过后 N 天未付款 |

## 10.4 结算预警

| 预警 | 规则 |
|---|---|
| 分包结算逾期 | 分包任务完成后 N 天未结算 |
| 总包结算资料缺失 | 缺少结算关键资料 |
| 签证未闭合 | 已发生签证未确认或未归档 |
| 付款与结算不一致 | 已付款金额 > 最终结算金额 |
| 成本明细无来源 | 成本数据无法追溯 |

建议系统建设不要一次性铺开所有模块，而是先完成主线闭环。

```text
  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

```

第二优先级是：

```text
合同变更
现场签证
库存出入库

```

第三优先级是：

```text

经营分析

```

只要第一阶段主线打通，系统就能从“看起来完整”变成“业务真正能跑通”。后续再逐步增强过程精细化、成本分析和经营决策能力。

## 18. 结论

本系统优化的关键不是增加多少页面，而是建立四个中心：

```text
合同中心
成本归集中心
付款控制中心
结算收口中心
```

同时通过：

```text
project_id 统一项目主线

source_type + source_id 统一成本来源
pay_request + pay_record 区分申请与支付

```

最终形成建筑工程总包项目从项目立项、合同签订、采购分包执行、成本归集、付款控制、竣工结算到经营分析的全过程闭环管理系统。

## V1.2 增量：成本自动生成幂等规则

材料验收、分包计量、签证变更等业务审批通过后生成成本时，必须保证幂等。

```text
source_type 表示来源业务类型，例如 MAT_RECEIPT、SUB_MEASURE、VAR_ORDER。
source_id 表示来源单据主表 ID，例如 mat_receipt.id、sub_measure.id。
source_item_id 表示来源单据明细 ID；若该来源单据不按明细拆分成本，则统一写入 0。
同一 source_type + source_id + source_item_id + cost_type 不得重复生成成本。
```

### source_item_id 来源规则

| source_type | source_id 来源 | source_item_id 来源 | 说明 |
|---|---|---|---|
| `MAT_RECEIPT` | `mat_receipt.id` | `mat_receipt_item.id`；整单汇总时为 `0` | 材料验收生成材料成本 |
| `SUB_MEASURE` | `sub_measure.id` | `sub_measure_item.id`；整单汇总时为 `0` | 分包计量生成分包成本 |
| `VAR_ORDER` | `var_order.id` | `var_order_item.id`；无明细时为 `0` | 现场签证或变更生成动态成本 |
| `CT_CHANGE` | `ct_contract_change.id` | 合同变更明细 ID；无明细时为 `0` | 合同变更调整合同锁定成本或预计成本 |
| `MANUAL_ADJUST` | 手工调整单 ID | 调整明细 ID；无明细时为 `0` | 仅允许有权限人员发起，必须留痕 |

推荐统一使用以下幂等唯一约束。即使一张来源单据只生成一条成本记录，也将 `source_item_id` 写为 `0`，避免 MySQL 唯一索引中 `NULL` 导致重复数据无法拦截。

```sql
CREATE UNIQUE INDEX uk_cost_source_item
ON cost_item(source_type, source_id, source_item_id, cost_type);
```

## V1.2 增量：模块边界补充红线

| 模块 | 不允许行为 |
|---|---|
| 合同管理 | 不允许其他模块重复创建合同主表 |
| 材料设备 | 不允许重复维护采购合同金额 |
| 分包管理 | 不允许重新维护一套分包合同主数据 |
