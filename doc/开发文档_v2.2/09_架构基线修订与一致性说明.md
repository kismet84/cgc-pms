---

## 一、修订背景

在对《建筑工程总包项目全过程管理系统开发计划书（v2）》及相关数据库设计、审批流程设计、技术选型报告进行一致性复核后，发现当前文档存在三类需要修正的问题：

1. 数据库选型从 MySQL 切换到 PostgreSQL，但数据库设计方案和 DDL 仍保持 MySQL 语法，文档内部不一致。
2. 审批伪代码中存在后端通过 HTTP 调用自身接口的设计，不符合后端领域服务调用原则。
3. 移动端开发启动过早，与后端核心业务接口成熟节奏不匹配，容易造成前期低效开发和后续返工。

上述问题不属于普通表述优化，而是会影响系统落地、开发排期、运维交付和架构一致性的关键问题。因此，建议作为 v2 开发计划书的“架构基线修订项”统一调整。

---

## 二、修订结论

本次建议定版如下：

```text
1. 数据库：一期统一采用 MySQL 8.0，不默认使用 PostgreSQL。
2. 审批后端：删除“HTTP 调自己”的伪代码，改为事务内领域服务直接调用。
3. 移动端：正式业务开发节奏后移，第 6 周启动基础工程，第 8 周左右进入真实业务功能开发。
```

# 三、数据库选型修订

## 3.1 当前问题

当前文档存在明显不一致：

- 数据库设计方案中的 DDL 仍然是 MySQL 风格；
- v2 开发计划书中又将数据库改为 PostgreSQL；
- 技术选型报告原先推荐 MySQL 8.0；
- 目前没有明确的 PostgreSQL 特有能力需求。

数据库设计方案中的表结构示例存在以下 MySQL 特征：

```text
DATETIME
TINYINT
UNIQUE KEY
表内 INDEX
MySQL 风格 COMMENT
AUTO_INCREMENT / 自增设计倾向
```

需要说明的是，`BIGINT PRIMARY KEY` 本身不是 MySQL 专属，PostgreSQL 也支持。真正造成不一致的是 `TINYINT`、`DATETIME`、`UNIQUE KEY`、表内 `INDEX`、`AUTO_INCREMENT` 等语法和 DDL 风格。

## 3.2 不建议一期默认 PostgreSQL 的原因

本系统一期核心目标是打通：

```text
项目 → 合同 → 采购 / 分包 → 成本 → 付款 → 结算
```

系统重点在合同、审批、付款、成本、结算的业务闭环，不是空间地理、复杂 JSON 检索、实时分析型数据库或地理信息系统。

如果只是因为审批规则、动态表单、节点通过规则和驳回规则需要半结构化字段，例如：

```text
pass_rule_json
reject_rule_json
form_schema
condition_rule
node_config
```

MySQL 8.0 的 JSON 类型已经可以满足一期需要。

建筑工程总包行业的甲方 IT 和项目运维团队通常对 MySQL 更熟悉，默认采用 MySQL 可以降低部署、备份、迁移、排障和长期运维风险。

## 3.3 修订建议

将 v2 开发计划书中的数据库描述：

```text
数据库：PostgreSQL（支持 JSONB 优化审批规则、表单Schema等半结构化数据）
```

修改为：

```text
数据库：MySQL 8.0。审批规则、动态表单 Schema、节点通过/驳回规则等半结构化配置统一使用 MySQL JSON 字段存储。PostgreSQL 仅作为后续可选部署方案，在明确需要 PostGIS、复杂 JSONB 检索，或企业已有 PostgreSQL 运维能力时采用。
```

## 3.4 数据库字段补充建议

审批相关表中建议增加或明确以下 JSON 字段：

```sql
pass_rule_json JSON NULL COMMENT '节点通过规则',
reject_rule_json JSON NULL COMMENT '节点驳回规则',
form_schema JSON NULL COMMENT '动态表单结构',
condition_rule JSON NULL COMMENT '流程匹配条件',
node_config JSON NULL COMMENT '节点扩展配置'
```

## 3.5 数据库基线定版

一期数据库基线建议定为：

| 项目 | 定版建议 |
|---|---|
| 默认数据库 | MySQL 8.0 |
| JSON 配置 | 使用 MySQL JSON 类型 |
| DDL 风格 | 统一 MySQL 8.0 语法 |
| ORM 适配 | 以 MySQL 方言为默认 |
| PostgreSQL | 作为后续可选部署版本 |
| 运维手册 | 按 MySQL 8.0 编写 |
| 索引脚本 | 按 MySQL 8.0 编写 |
| 数据迁移 | 按 MySQL 8.0 优先设计 |

# 四、审批后端调用方式修订

## 4.1 当前问题

v2 文档中审批伪代码存在类似如下逻辑：

```java
void approveTask(...) {
    OutboxMessage out = new OutboxMessage(...);
    outboxDao.save(out);
    processOutbox(); // 发 HTTP POST 给自己
}
```

该设计不合理。

这是把移动端或离线客户端的 Outbox 模式错误搬到了后端内部。后端审批引擎与业务模块如果处于同一个应用或同一个模块化单体内，不应通过 HTTP 调用自身 Controller。

这种设计会带来以下问题：

1. 事务边界变复杂；
2. 调用链路绕远；
3. 错误处理困难；
4. 日志链路混乱；
5. 性能下降；
6. 本地方法调用被伪装成网络调用；
7. 容易形成重复提交和状态不一致；
8. 增加无意义的 HTTP 序列化和反序列化成本。

## 4.2 正确原则

后端审批引擎应采用：

```text
事务内领域服务直接调用
```

同一后端应用内部不通过 HTTP 调用自身接口。

审批动作应在一个清晰的事务边界内完成：

```text
校验任务
→ 写审批记录
→ 更新任务状态
→ 更新节点状态
→ 更新流程实例状态
→ 回调业务单据状态
→ 提交事务
→ 事务提交后发布异步事件
```

## 4.3 推荐审批处理伪代码

```java
@Transactional
public ApprovalResult approveTask(ApproveCommand cmd) {
    // 1. 校验任务状态、审批人、taskVersion、幂等键
    WfTask task = wfTaskService.lockAndValidate(cmd);

    // 2. 写审批记录
    wfRecordService.createApproveRecord(task, cmd);

    // 3. 更新当前任务、节点、实例状态
    wfEngineService.completeTask(task, cmd);

    // 4. 推进流程：下一节点 / 会签完成 / 或签取消其他任务
    wfEngineService.moveNext(task.getInstanceId());

    // 5. 同事务内回调业务单据状态
    businessApprovalService.onWorkflowChanged(
        task.getBusinessType(),
        task.getBusinessId()
    );

    // 6. 事务提交后再发布领域事件
    domainEventPublisher.publishAfterCommit(
        new WorkflowApprovedEvent(task)
    );

    return ApprovalResult.success(task.getInstanceId());
}
```

## 4.4 Outbox 模式保留范围

Outbox 模式可以保留，但必须限定适用边界。

### 适合使用 Outbox 的场景

```text
审批完成后推送财务系统
审批完成后发送短信 / 企业微信 / 站内信
审批完成后同步 BI / 数据仓库
审批完成后通知外部 ERP / OA / 财务系统
移动端离线草稿恢复后提交到后端
移动端弱网环境下的自动重试队列
```

### 不适合使用 Outbox 的场景

```text
审批服务调用自己的 Controller
同一个后端应用内部模块之间绕 HTTP 调用
事务内自己发 HTTP 再处理自己
审批引擎内部状态流转
业务模块与审批模块在同一应用内的同步联动
```

## 4.5 文档修订建议

建议将 v2 中关于审批伪代码的相关表述改为：

```text
审批引擎内部采用领域服务直接调用。
同一应用内不通过 HTTP 调用自身接口。
Outbox 模式仅用于跨系统异步集成、消息通知、移动端离线同步，不用于后端内部同步审批。
审批任务处理、审批记录写入、节点状态更新、实例状态更新和业务单据状态回调应处于同一事务边界内。
事务提交后再发布领域事件，用于通知、待办统计、消息推送、财务系统同步等异步场景。
```

# 五、移动端开发节奏修订

## 5.1 当前问题

v2 计划书中移动端开发启动过早，移动端从第 2 周就开始进入开发，但移动端依赖的后端业务接口并不会在第 2 周成熟。

移动端 P0 功能包括：

```text
我的待办
审批详情
同意 / 驳回 / 转办 / 加签
项目速览
材料验收单
拍照上传
现场签证
质量整改
扫码查询
离线草稿
自动同步
消息通知
```

这些功能高度依赖以下后端接口：

```text
登录认证接口
用户权限接口
审批任务接口
审批详情接口
审批动作接口
项目接口
合同接口
材料验收接口
附件上传接口
现场签证接口
质量整改接口
付款接口
计量接口
结算接口
消息通知接口
离线同步接口
```

如果第 2 周就全面启动移动端开发，前 6 周大概率只能完成：

```text
UI 壳
登录页
Mock 列表
空状态页面
基础组件
静态表单
本地假数据流程
```

这会导致：

1. 接口不稳定时反复调整字段；
2. 流程状态不稳定时反复重写页面逻辑；
3. 审批动作和按钮规则变化导致返工；
4. 附件上传和离线同步方案不成熟导致重构；
5. 移动端开发效率低；
6. 前后端联调周期被拉长。

## 5.2 修订原则

移动端不应完全提前于业务 API 开发。

建议采用：

```text
移动端原型先行
接口契约先行
基础工程适度提前
真实业务功能跟随后端核心接口
现场高频功能在业务链路稳定后集中开发
```

## 5.3 调整后的移动端排期

| 周期 | 移动端安排 | 目标 |
|---|---|---|
| 第 1-4 周 | 不启动正式移动端开发，只做移动端原型、交互规范、接口契约评审 | 明确移动端范围和交互，不写大量业务代码 |
| 第 5-6 周 | 启动移动端基础工程 | 完成登录、Token 刷新、项目上下文、附件上传组件、离线草稿框架 |
| 第 7-10 周 | 开发轻量移动审批 | 我的待办、审批详情、同意、驳回、转办、加签、项目速览 |
| 第 11-16 周 | 开发现场高频业务 | 材料验收、拍照上传、现场签证、质量整改、扫码查询 |
| 第 17-20 周 | 开发财务与经营查询 | 付款、结算、成本状态查询、消息通知 |
| 第 21 周以后 | 稳定性优化 | 离线同步压测、多端兼容、弱网恢复、UAT 优化 |

## 5.4 企业微信 / 轻审批入口

企业微信或轻审批入口可以略早于现场 App 主功能启动。

建议：

```text
第 5-6 周：评审企业微信入口方案；
第 7-10 周：接入我的待办、审批详情、审批动作；
第 11 周后：根据主流程稳定情况逐步补充消息提醒和轻表单。
```

## 5.5 现场 App / 小程序启动原则

现场 App 或小程序主功能不宜早于核心 API 稳定前全面开发。

建议判断条件：

```text
审批任务接口稳定；
审批动作接口稳定；
附件上传接口稳定；
项目上下文接口稳定；
材料验收接口完成第一版；
现场签证接口完成第一版；
质量整改接口完成第一版；
离线草稿数据结构完成第一版。
```

# 六、建议写入 v2 的“架构基线修订”章节

建议在 v2 开发计划书中新增如下章节。

## 架构基线修订

为保证开发计划书、数据库设计方案、技术选型报告、审批流程设计方案和后续 DDL 脚本保持一致，本项目一期架构基线修订如下：

1. 一期数据库统一采用 MySQL 8.0，JSON 类配置使用 MySQL JSON 类型。
2. PostgreSQL 不作为一期默认数据库，仅作为后续可选部署方案。
3. 仅在明确需要 PostGIS、复杂 JSONB 检索、或企业已有 PostgreSQL 运维能力时，才切换 PostgreSQL。
4. 数据库设计文档、技术选型报告、开发计划书、DDL 脚本、运维手册必须保持数据库选型一致。
5. 审批引擎内部采用事务内领域服务调用，不允许通过 HTTP 调用自身接口。
6. Outbox 模式仅用于移动端离线同步、跨系统异步推送、消息通知，不用于后端内部同步审批。
7. 审批动作处理应在事务内完成任务校验、记录写入、任务状态更新、节点状态更新、实例状态更新和业务单据状态回调。
8. 事务提交后再发布领域事件，用于消息通知、待办统计、财务系统推送、BI 同步等异步场景。
9. 移动端开发节奏后移：第 1-4 周仅参与原型和接口契约评审，第 5-6 周启动基础工程，第 7-10 周开发轻量审批，第 11 周后进入现场业务功能。
10. 企业微信轻审批入口可以略早建设，现场 App / 小程序主功能应在核心 API 稳定后进入全面开发。
