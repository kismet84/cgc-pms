## 1. 总体方案正文

建筑工程总包项目全过程管理系统更新方案

一、更新原则

为避免合同、成本、材料、分包等模块之间出现重复录入、数据不一致和业务边界不清的问题，本系统应按照“项目为根、合同为纲、执行为据、成本为果、付款为流、结算为终”的原则进行优化设计。

系统建设不应将各业务模块设计成相互独立的功能孤岛，而应围绕项目全过程管理形成统一的数据主线和业务闭环。

总体原则如下：

项目是所有业务数据的根节点，所有合同、采购、分包、成本、付款、结算均应关联 project_id。

合同是金额、条款、付款条件和责任主体的唯一依据，所有合同数据统一在合同管理模块维护。

材料设备管理和分包管理负责业务执行过程，不重复维护合同主数据。

成本管理负责成本归集、计算、分析和预警，不作为重复录入业务数据的入口。

付款应区分付款条件、付款申请、付款审批和实际支付，避免付款数据与成本数据混淆。

竣工结算管理应从合同、变更、签证、计量、验收、付款、成本等前端业务自动汇总形成结算依据。

二、更新后的总体业务关系

系统应形成如下业务主线：

项目管理
→ 合同管理
→ 材料设备管理 / 分包管理 / 设备租赁 / 劳务管理
→ 成本管理
→ 付款管理 / 财务接口
→ 竣工结算管理
→ 项目经营分析

核心业务逻辑为：

合同管理提供合同金额、合同类型、付款条件、履约条款和合同变更依据。

材料设备管理根据采购合同或采购需求执行采购申请、采购订单、材料验收、入库、出库和使用追溯。

分包管理根据分包合同执行分包任务、进度填报、计量确认、付款申请、质量安全评价和分包结算。

成本管理从合同、材料、分包、设备、劳务、签证等业务数据中自动归集成本，形成目标成本、动态成本、利润分析和成本预警。

竣工结算管理汇总总包合同、分包合同、采购合同、变更签证、付款记录、成本明细和项目资料，形成最终结算依据。

三、模块边界更新

1. 项目管理

项目管理作为系统的基础模块，负责项目主数据和项目组织关系维护。

主要包括：

项目基本信息；

项目编号；

项目阶段；

总包单位；

项目组织架构；

项目目标成本；

项目合同关系；

项目权限范围；

项目资料归档口径。

所有业务数据均必须关联项目，形成统一项目视图。

2. 合同管理

合同管理负责合同主数据、合同金额、合同条款和合同履约依据，是系统的合同数据中心。

主要包括：

总包合同；

分包合同；

采购合同；

租赁合同；

合同台账；

合同审批；

合同签订；

合同金额；

合同清单；

付款条件；

履约节点；

合同变更；

合同归档；

合同预警。

合同管理只维护一套合同数据，其他模块通过 contract_id 引用合同，不得重复建立合同主表。

合同类型建议统一通过 contract_type 区分：

总包合同；

分包合同；

采购合同；

租赁合同；

服务合同；

其他合同。

3. 材料设备管理

材料设备管理负责材料和设备采购执行过程，不重复维护采购合同金额。

主要包括：

材料需求计划；

采购申请；

采购订单；

供应商管理；

材料进场验收；

材料复检；

入库管理；

出库管理；

库存管理；

使用部位追溯；

设备进场；

设备退场；

设备维保；

供应商评价。

采购订单应支持三种业务场景：

有合同采购：采购订单必须关联采购合同；

零星采购：允许不关联合同，但必须标记为零星采购；

框架协议采购：订单关联框架合同，并记录具体订单明细。

材料设备管理产生的验收、入库、出库和使用数据，应自动传递至成本管理，形成材料成本和机械成本。

4. 分包管理

分包管理负责分包履约过程，不重复维护分包合同主数据。

主要包括：

分包单位台账；

分包单位准入；

分包任务；

分包进度；

分包计量；

分包付款申请；

分包质量评价；

分包安全评价；

分包结算；

分包履约预警。

分包合同主数据由合同管理维护，分包管理通过 contract_id 引用分包合同。

分包管理中的“分包合同”功能可以保留，但应作为合同管理中分包合同的引用、查看或跳转入口，不应重新创建一套分包合同数据。

分包计量确认后，应自动形成分包成本，并传递至成本管理。

5. 成本管理

成本管理负责成本归集、动态成本分析、利润测算和成本预警，不应成为重复录入业务数据的入口。

主要包括：

目标成本；

合同成本；

材料成本；

分包成本；

机械成本；

人工成本；

管理费用；

预计待发生成本；

动态成本；

成本偏差分析；

利润测算；

成本预警。

成本管理应从以下业务来源自动归集数据：

合同金额；

采购订单；

材料验收入库；

材料出库消耗；

分包计量；

设备租赁；

劳务记录；

现场签证；

财务付款记录。

6. 付款管理与财务接口

付款相关功能建议拆分为三个层级：

付款申请应关联：

project_id；

contract_id；

付款对象；

付款依据；

已计量金额；

已验收金额；

已申请金额；

已审批金额；

已支付金额。

系统应支持以下付款控制规则：

未签合同不得付款，零星采购除外；

未完成验收的材料不得付款；

未完成计量确认的分包不得付款；

超合同金额付款必须走合同变更审批；

超付款比例应触发预警；

质保金未到期不得释放；

实际付款数据应以财务系统为准。

7. 竣工结算管理

竣工结算管理负责项目最终结算收口，不重新录入前端业务数据，而是从系统已有数据中汇总形成结算依据。

主要包括：

总包结算；

分包结算；

采购结算；

租赁结算；

设计变更汇总；

现场签证汇总；

付款记录汇总；

成本明细汇总；

质保金管理；

审计资料归档；

竣工资料归档。

合同；

材料出库；

分包计量；

分包付款；

最终形成从合同到执行、从成本到付款、从付款到结算的全过程闭环。

七、权限与流程控制建议

系统应根据不同角色设置数据权限和流程权限。

建议角色包括：

公司管理层；

项目经理；

商务经理；

成本经理；

合同管理员；

材料员；

设备管理员；

分包管理员；

财务人员；

资料员；

审计人员。

权限控制建议：

项目人员只能查看和处理所属项目数据；

合同金额、付款条件等关键字段仅合同管理角色可维护；

成本管理人员可查看成本数据，但不应随意修改业务来源数据；

财务人员负责确认实际付款，不负责修改业务计量和验收数据；

分包管理员可发起计量和付款申请，但不能修改合同金额；

材料员可维护采购、验收、入库、出库数据，但不能修改采购合同金额；

管理层可查看驾驶舱、成本分析、付款预警和结算风险。

八、预警规则更新

系统应增加以下预警规则：

1. 合同预警

合同即将到期预警；

合同金额超限预警；

合同付款比例超限预警；

合同履约节点逾期预警；

合同未归档预警。

2. 成本预警

动态成本超过目标成本预警；

分包成本超合同金额预警；

材料成本超预算预警；

机械成本异常预警；

人工成本异常预警；

变更签证未及时确认预警。

3. 付款预警

未验收付款预警；

未计量付款预警；

超合同金额付款预警；

超付款比例付款预警；

质保金提前释放预警；

长期未付款预警。

4. 结算预警

分包结算逾期预警；

总包结算资料缺失预警；

签证未闭合预警；

付款与结算金额不一致预警；

成本明细无来源预警。

十、最终更新结论

更新后的系统方案应从原来的模块并列设计，调整为以项目为根节点、以合同为主线、以业务执行为依据、以成本归集为结果、以付款资金流为辅助、以竣工结算为收口的全过程管理体系。

系统最终应实现以下目标：

合同数据统一维护，避免重复建合同；

材料、设备、分包、劳务等执行数据与合同自动关联；

成本数据从业务过程自动归集，减少人工重复录入；

付款申请、审批和实际支付形成闭环控制；

动态成本、利润测算和成本预警实时生成；

竣工结算能够自动汇总合同、计量、验收、签证、付款和成本数据；

管理层能够通过驾驶舱掌握项目合同、成本、付款、结算和风险情况。

因此，系统建设主线建议最终确定为：

项目为根，合同为纲，执行为据，成本为果，付款为流，结算为终。

该设计既能保留合同、材料、分包、成本、结算等各业务模块的专业性，又能通过统一数据关系实现全过程协同，避免重复录入和数据不一致，提升建筑工程总包项目全过程管理的实用性和可落地性。

| 成本口径 | 含义 |
| 目标成本 | 项目初始成本控制目标 |
| 合同锁定成本 | 已签合同对应的成本 |
| 已发生成本 | 已验收、已计量、已确认的实际成本 |
| 已付款成本 | 财务已实际支付的金额 |
| 预计待发生成本 | 尚未发生但预计会发生的成本 |
| 动态成本 | 已发生成本 + 预计待发生成本 |
| 成本偏差 | 动态成本与目标成本的差额 |

| 层级 | 归属 | 说明 |
| 付款条件 | 合同管理 | 根据合同约定维护付款比例、节点、质保金等条件 |
| 付款申请 | 业务模块发起 | 分包付款由分包管理发起，材料付款可由材料或合同相关流程发起 |
| 实际支付 | 财务系统或财务接口 | 记录实际付款金额、付款日期、付款凭证 |

| 数据对象 | 主模块 | 关键关联 |

## V1.2 增量：开工前总体成功标准

## V1.2 增量：业务闭环红线

1. 未关联 `project_id` 的业务单据不得提交审批；
2. 涉及合同的业务单据必须关联 `contract_id`，零星采购除外；
3. 成本生成必须来自验收、计量、签证、合同变更等业务事实；
4. 付款审批不得被当作成本发生依据；
5. 结算不得重新录入一套孤立数据，必须能反查合同、执行、成本和付款来源。
