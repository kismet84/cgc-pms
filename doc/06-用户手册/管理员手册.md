# 第 4 阶段管理员手册

> 项目：建筑工程总包项目全过程管理系统 (cgc-pms)
> 版本：V1.0.0-SNAPSHOT
> 日期：2026-06-12
> 适用角色：系统管理员、运维人员

---

## 1. 概述

第 4 阶段新增了 7 个 Flyway 迁移脚本（V33~V39），涉及组织架构、项目成员、库存管理、发票管理、消息通知、审批抄送、菜单权限种子数据。本手册面向系统管理员，说明部署步骤、迁移清单、权限配置和审批模板配置。

---

## 2. 部署步骤

### 2.1 前置条件

| 组件 | 要求 |
|------|------|
| JDK | 21（`mvnw.cmd` 自动检测） |
| MySQL | 8.0+（数据库 `cgc_pms` 已创建，字符集 `utf8mb4`） |
| Redis | 7（消息 SSE 推送依赖） |
| Flyway | 版本 10.10.0（随 Spring Boot 自动执行） |

### 2.2 部署操作

```bash
# 1. 拉取最新代码（第 4 阶段分支）
git checkout master && git pull origin master

# 2. 确认 Flyway 迁移脚本在正确位置
#    路径：backend/src/main/resources/db/migration/
#    第 4 阶段新增：V33 ~ V39（共 7 个文件）

# 3. 启动数据库（如果尚未运行）
#    方式 A：Docker Compose
cd deploy && docker compose up -d mysql redis

#    方式 B：本地 MySQL 服务（Windows: net start MySQL80）

# 4. 启动后端（Flyway 自动执行迁移）
cd backend
./mvnw spring-boot:run -Dspring-boot.run.profiles=dev

#    观察启动日志，确认如下输出：
#    Migrating schema "CGC_PMS" to version "39 - init phase4 menu perms"
#    Successfully applied 39 migrations

# 5. 启动前端
cd frontend-admin
pnpm install && pnpm dev
```

### 2.3 验证部署

| 检查项 | 验证方式 |
|--------|----------|
| 数据库表是否创建 | 登录 MySQL 执行 `SHOW TABLES LIKE 'org_%'` / `'mat_%'` / `'pay_invoice'` / `'sys_notification'` / `'wf_cc'` / `'pm_project_member'` |
| 菜单是否出现 | 使用 `admin` 登录前端，检查侧边栏是否出现「组织架构」「库存管理」「发票管理」「消息中心」一级菜单 |
| 组织架构页面 | 访问 `/org`，确认公司/部门/岗位三个 Tab 正常加载 |
| 采购申请审批模板 | 检查 `wf_template` 表：`SELECT * FROM wf_template WHERE template_code = 'TPL-PURCHASE-REQUEST-001';` |

## 3. Flyway 迁移清单

第 4 阶段共 7 个迁移脚本，按执行顺序排列：

| 版本 | 文件名 | 内容 | 新增表 |
|------|--------|------|--------|
| V33 | `V33__init_org_tables.sql` | 组织架构基础表 | `org_company`、`org_department`、`org_position` |
| V34 | `V34__add_project_member_and_user_org.sql` | 项目成员表 + 用户组织关联 | `pm_project_member` + `sys_user` 增加 `org_id` 列 |
| V35 | `V35__init_inventory_tables.sql` | 库存与采购申请业务表 | `mat_warehouse`、`mat_stock`、`mat_stock_txn`、`mat_purchase_request`、`mat_purchase_request_item` |
| V36 | `V36__init_invoice_table.sql` | 发票管理表 | `pay_invoice` |
| V37 | `V37__init_notification_table.sql` | 站内消息通知表 | `sys_notification` |
| V38 | `V38__init_workflow_cc_table.sql` | 审批抄送表 | `wf_cc` |
| V39 | `V39__init_phase4_menu_perms.sql` | 菜单/权限/审批模板种子数据 | 无（DML 种子数据） |

### 3.1 V33 组织架构表

三个表均遵循统一的审计列约定（`created_time` / `updated_time` / `deleted_flag`）：

| 表 | 用途 | 关键约束 |
|----|------|----------|
| `org_company` | 公司信息 | `uk_oc_tenant_code`：同一租户内公司编码唯一 |
| `org_department` | 部门树形结构 | `parent_id` 自引用实现树，支持无限层级 |
| `org_position` | 岗位/HR 头衔 | `uk_op_tenant_code`：同一租户内岗位编码唯一 |

### 3.2 V34 项目成员 + 用户关联

| 表/列 | 用途 | 说明 |
|--------|------|------|
| `pm_project_member` | 项目成员分配 | `uk_ppm_project_user`：同一用户在同一项目中只能分配一次 |
| `sys_user.org_id` | 用户关联组织 | 使用 `information_schema` 守卫幂等添加，防止重复执行报错 |

项目角色枚举（`role_code`）：`PM`（项目经理）、`CM`（商务经理）、`CSTM`（成本经理）、`MAT`（材料员）、`SUBC`（分包管理员）、`FIN`（财务）、`OTH`（其他）。

### 3.3 V35 库存与采购申请表

| 表 | 用途 | 关键设计 |
|----|------|----------|
| `mat_warehouse` | 仓库信息 | 关联项目（`project_id`），状态启用/停用 |
| `mat_stock` | 库存余额（数量型台账） | `uk_ms_warehouse_material`：同一仓库同一物料唯一记录；`version` 列支持乐观锁并发控制 |
| `mat_stock_txn` | 库存流水（出入库记录） | `txn_type`：`IN`/`OUT`/`ADJUST`；`source_type` + `source_id` 追溯来源业务单据 |
| `mat_purchase_request` | 采购申请 | 编号自动生成 `PR-yyyyMMdd-XXX`；双状态字段 `approval_status` + `status` |
| `mat_purchase_request_item` | 采购申请明细 | 关联 `mat_purchase_request` 和 `md_material` |

### 3.4 V36 发票表

| 表 | 用途 | 关键约束 |
|----|------|----------|
| `pay_invoice` | 发票信息 | `uk_pi_tenant_invoice_no`：同一租户内发票号码唯一；关联 `pay_application_id` / `pay_record_id` |

发票类型：`VAT_SPECIAL`（专票）、`VAT_NORMAL`（普票）、`OTHER`（其他）。
核验状态：`PENDING` → `VERIFIED` / `ABNORMAL`，不走审批链。

### 3.5 V37 消息通知表

| 表 | 用途 | 关键设计 |
|----|------|----------|
| `sys_notification` | 站内消息通知 | `tenant_id` + `user_id` 显式冗余（非 UserContext），支持 SSE 推送/定时任务路径；索引 `idx_sn_tenant_user_read` 加速未读查询 |

### 3.6 V38 审批抄送表

| 表 | 用途 | 说明 |
|----|------|------|
| `wf_cc` | 审批抄送记录 | 独立 join 表实现抄送，不侵入 `wf_instance` / `wf_task` / `wf_record` 核心表；`cc_user_name` 冗余存储避免联表 |

### 3.7 V39 菜单权限种子数据

全量 `INSERT IGNORE`，幂等可重复执行。菜单 ID 区间 700~799，审批模板 ID 50010+。

## 5. 审批模板配置

### 5.1 采购申请审批模板 (50010)

V39 自动创建采购申请审批模板，配置如下：

| 属性 | 值 |
|------|-----|
| template_id | 50010 |
| template_code | `TPL-PURCHASE-REQUEST-001` |
| template_name | 采购申请审批 |
| business_type | `PURCHASE_REQUEST` |
| enabled | 1（启用） |
| amount_min / amount_max | 0.00 ~ 999,999,999.99（无金额门槛） |
| condition_rule | NULL（无条件限制） |
| form_schema | NULL（使用默认表单） |

### 5.2 审批节点配置

采购申请审批包含 3 个顺序审批节点：

| 节点 ID | 节点编码 | 节点名称 | 顺序 | 类型 | 审批人配置 | 可转办 | 可加签 | 超时 |
|---------|----------|----------|------|------|------------|--------|--------|------|
| 50011 | NODE_PR_001 | 项目经理 | 1 | APPROVAL | `{"type":"ROLE","roleCode":"PROJECT_MANAGER"}` | 是 | 是 | 48h |
| 50012 | NODE_PR_002 | 商务经理 | 2 | APPROVAL | `{"type":"ROLE","roleCode":"COMMERCIAL_MANAGER"}` | 是 | 是 | 48h |
| 50013 | NODE_PR_003 | 成本/采购 | 3 | APPROVAL | `{"type":"ROLE","roleCode":"COST_MANAGER"}` | 是 | 是 | 48h |

**审核模式**：`SEQUENTIAL`（顺序审批）。每个节点审批通过后流转到下一节点，全部通过后审批完成。

**审批人解析规则**：`approver_config` 为 JSON 格式：
```json
{
  "type": "ROLE",
  "roleCode": "PROJECT_MANAGER"
}
```
系统根据 `roleCode` 查找拥有对应角色的用户作为审批人。

### 5.3 手动管理审批模板

若需修改或新增审批模板，可通过 SQL 直接操作 `wf_template` 和 `wf_template_node` 表。所有种子数据使用 `INSERT IGNORE`，重复执行不会报错。

**修改审批节点配置示例**：

```sql
-- 修改节点超时时间
UPDATE wf_template_node
SET timeout_hours = 72
WHERE template_id = 50010 AND node_code = 'NODE_PR_001';

-- 禁用模板
UPDATE wf_template SET enabled = 0 WHERE template_id = 50010;
```

## 6. 常见运维操作

### 6.1 重置某用户的未读消息数

```sql
-- 查看用户未读消息
SELECT user_id, COUNT(*) AS unread_count
FROM sys_notification
WHERE tenant_id = 0 AND user_id = <用户ID> AND is_read = 0
GROUP BY user_id;

-- 全部标记已读
UPDATE sys_notification SET is_read = 1, read_time = NOW()
WHERE tenant_id = 0 AND user_id = <用户ID> AND is_read = 0;
```

### 6.2 清理过期通知数据

```sql
-- 查看 30 天前的已读通知
SELECT COUNT(*) FROM sys_notification
WHERE is_read = 1 AND read_time < DATE_SUB(NOW(), INTERVAL 30 DAY);

-- 删除过期数据（谨慎）
DELETE FROM sys_notification
WHERE is_read = 1 AND read_time < DATE_SUB(NOW(), INTERVAL 90 DAY);
```

### 6.3 仓库停用后恢复出入库

仓库状态改为 `DISABLE` 后前端不允许新增出入库操作。如需恢复：

```sql
UPDATE mat_warehouse SET status = 'ENABLE' WHERE id = <仓库ID>;
```

### 6.4 检查 Flyway 迁移状态

```sql
SELECT version, description, script, installed_on, success
FROM flyway_schema_history
WHERE version >= '33' AND version <= '39'
ORDER BY version;
```

预期输出 7 行，`success` 均为 1。

## 8. 故障排查

### 8.1 菜单未在侧边栏显示

**原因**：当前登录用户角色未授予对应菜单权限。

**解决**：
1. 确认 `sys_menu` 表中对应记录 `status = 'ENABLE'` 且 `visible = 1`。
2. 确认 `sys_role_menu` 表中已建立角色和菜单的关联。
3. 确认当前用户拥有该角色。

### 8.2 采购申请提交审批失败

**原因**：可能是审批模板未启用或审批人不存在。

**解决**：
1. 检查 `wf_template` 表 `enabled = 1`。
2. 检查用户是否在对应项目中分配了审批节点要求的角色（`PROJECT_MANAGER` 等）。
3. 检查 `wf_template_node` 中节点顺序是否正确。

### 8.3 SSE 消息推送不工作

**原因**：Redis 未启动或连接失败（SSE 推送依赖 Redis 发布/订阅）。

**解决**：
1. 确认 Redis 服务已启动：`redis-cli ping` 返回 `PONG`。
2. 检查 `application-dev.yml` 中 Redis 连接配置。
3. 检查前端控制台是否有 SSE 连接错误。

### 8.4 Flyway 迁移冲突

**原因**：数据库已有同名表或字段。

**解决**：
- 所有 `CREATE TABLE` 使用 `IF NOT EXISTS`，可安全重跑。
- V34 中 `ALTER TABLE sys_user ADD COLUMN org_id` 使用 `information_schema` 守卫幂等。
- V39 中所有 `INSERT` 使用 `INSERT IGNORE`。
- 若仍需修复：`DELETE FROM flyway_schema_history WHERE version = 'XX'` 后重启可重新执行对应版本。
