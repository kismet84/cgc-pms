## 二、🔴 阻断级问题（上线前必须修复）

### P0-1: V42 种子角色零权限
- **文件**: `V42__seed_material_warehouse_cost_subject.sql`
- **问题**: `MATERIAL_CLERK` (id=5) 和 `FINANCE` (id=6) 角色只创建了 `sys_role` 记录，**没有关联任何 `sys_role_menu` 权限**。用户分配到这些角色后，菜单为空，无任何功能访问权限。
- **修复**: 在 V42 中追加 `INSERT IGNORE INTO sys_role_menu` 语句，至少关联库存/付款相关菜单。

### P0-2: 生产 docker-compose 关闭 SSL
- **文件**: `deploy/docker-compose.prod.yml:132`
- **问题**: MySQL JDBC URL 包含 `useSSL=false`。`application-prod.yml` 正确配置了 `useSSL=true`，但 compose 环境变量覆盖为 false。数据库流量在 Docker 网络内明文传输。
- **修复**: 改为 `useSSL=true&ssl-mode=VERIFY_CA` 或至少 `useSSL=true`。

### P0-3: Nginx 缓冲阻断 SSE 实时推送
- **文件**: `frontend-admin/nginx.conf:98-118`
- **问题**: `/api/` location 使用默认 `proxy_buffering on`，会缓冲 SSE 事件流。NotificationBell 的实时通知推送将延迟或完全失败。
- **修复**: 在 `/api/` location 块中添加 `proxy_buffering off;`。

### P0-4: 输入校验大面积缺失（16+ 端点）
- **文件**: 
  - `backend/.../invoice/controller/InvoiceController.java` — create/update/verify/register 均缺 `@Valid`
  - `backend/.../payment/controller/PayApplicationController.java` — create/update/batchSaveBasis 均缺 `@Valid`
  - `backend/.../payment/controller/PayRecordController.java` — create/update/writeback 均缺 `@Valid`
- **根因**: `PayInvoice`、`PayApplication`、`PayRecord`、`PayApplicationBasis` 实体**零字段级校验注解**（无 `@NotNull`、`@NotBlank`、`@Positive`）。
- **修复**: 实体添加 Jakarta Validation 注解 + Controller 添加 `@Valid`。

### P0-5: Mass Assignment — @RequestBody 直接绑定 Entity
- **文件**: 同 P0-4
- **问题**: Controller 将 Entity 对象直接作为 `@RequestBody` 参数。攻击者可设置 `tenantId`、`createdBy`、`updatedBy`、`deletedFlag` 等应服务端控制的字段。MyBatis-Plus `FieldFill` 仅在字段为 null 时填充，手工传值可覆盖。
- **修复**: 引入 DTO 类（如 `PayInvoiceCreateDTO`），或在实体敏感字段上加 `@JsonIgnore` / `@TableField(fill = FieldFill.INSERT, insertStrategy = FieldStrategy.NEVER)`。
