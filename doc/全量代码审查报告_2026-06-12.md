# 全量代码审查报告（2026-06-12）

## 1. 审查概况

- 审查对象：`cgc-pms` 当前工作区代码
- 审查范围：后端 Spring Boot / MyBatis-Plus / Flyway，前端 Vue 3 / TypeScript / Vite，数据库迁移脚本
- 审查方式：静态代码审查 + 关键路径 grep + 构建/测试验证
- 审查重点：
  - Flyway 迁移可重复执行与空库初始化
  - 多租户隔离与权限边界
  - 工作流状态机一致性
  - 动态成本/驾驶舱金额口径
  - 前后端字段契约与展示口径

## 2. 总体结论

当前代码存在 **2 个 P0 阻断问题、2 个 P1 高风险问题、1 个 P2 用户可见展示问题**。

其中 Flyway 迁移问题会导致后端在已迁移环境或空库迁移环境中启动失败，属于合并/发布前必须修复的问题。工作流模块仍存在租户隔离和状态机绕过风险。前端成本偏差展示与后端字段语义不一致，会造成明显错误的业务展示。

## 3. 问题清单

### P0-1：Flyway 已执行迁移被改写，现有库启动失败

**位置**

- `backend/src/main/resources/db/migration/V21__add_submit_permissions.sql`
- `database/migration/V21__add_submit_permissions.sql`

**现象**

执行后端全量测试时，Spring Boot 在 Flyway 初始化阶段失败：

```text
Migration checksum mismatch for migration version 21
-> Applied to database : 140775814
-> Resolved locally    : -826127236
```

**影响**

Flyway 已执行过的版本化迁移被修改后，任何已经应用旧 V21 的数据库在启动时都会触发校验失败。该问题会阻断本地测试、预发环境和生产环境应用启动。

**修复建议**

- 恢复已发布/已执行的 V21 原始内容。
- 将新增权限码补丁改为新迁移，例如 `V32__add_submit_permissions.sql`。
- 如果数据库已经被错误迁移，需要在明确受控环境执行 `flyway repair`，并记录操作窗口和影响范围。

### P0-2：空库迁移会因重复添加 `cost_target_id` 失败

**位置**

- `backend/src/main/resources/db/migration/V24__enhance_settlement_alert_summary.sql`
- `backend/src/main/resources/db/migration/V26__add_cost_target_id_to_cost_summary.sql`

**现象**

`V24` 已执行：

```sql
ALTER TABLE cost_summary
    ADD COLUMN cost_target_id BIGINT NULL ...
```

`V26` 再次无条件执行：

```sql
ALTER TABLE cost_summary
    ADD COLUMN cost_target_id BIGINT NULL ...
```

**影响**

MySQL 空库从 V1 跑到 V31 时，会在 V26 因重复列报错，导致数据库无法完成初始化。README 和测试报告均声明 Flyway V1~V31 可完整执行，该问题与交付声明冲突。

**修复建议**

- 删除或改写 V26 中重复的 `ADD COLUMN cost_target_id`。
- 如果 V26 仍需保留版本号，可改为幂等检查或只补充索引/注释等真正未包含在 V24 的内容。
- 增加空库迁移验证作为发布前检查。

### P1-1：工作流记录与幂等键硬编码租户 `0L`

**位置**

- `backend/src/main/java/com/cgcpms/workflow/service/WorkflowEngine.java`
  - `writeRecord()` 中 `record.setTenantId(0L)`
  - `checkIdempotency()` 中 `idem.setTenantId(0L)`
- `backend/src/main/java/com/cgcpms/workflow/service/WorkflowQueryService.java`
  - `getMyTodos(Long userId, ...)`
  - `getInstanceDetail(Long instanceId, Long currentUserId)`

**现象**

工作流实例、节点、任务在提交时写入真实租户，但审批记录和幂等记录固定写入租户 `0`。查询服务也仅以 user/task/instance 维度查询，没有显式传入当前租户过滤。

**影响**

- 审计记录租户错误，后续按租户审计或清理不可信。
- 幂等键唯一约束为 `(tenant_id, user_id, idempotency_key)`，硬编码租户会造成跨租户幂等冲突。
- 工作流详情和待办查询缺少完整租户边界，存在跨租户数据暴露风险。

**修复建议**

- `WorkflowEngine` 的审批动作方法引入当前租户上下文，或根据 task/instance 反查租户并贯穿到 `writeRecord()`、`checkIdempotency()`。
- `WorkflowQueryService` 查询方法增加 `tenantId` 参数，并在 `wf_task`、`wf_instance`、`wf_node_instance`、`wf_record` 查询中带上租户过滤。
- 增加跨租户工作流查询和幂等键回归测试。

### P1-2：加签接口可对已处理任务继续创建待办

**位置**

- `backend/src/main/java/com/cgcpms/workflow/service/WorkflowEngine.java`
  - `addSign(Long taskId, List<Long> additionalUserIds, Long userId, ...)`

**现象**

`approve`、`reject`、`transfer` 均校验任务必须为 `TASK_PENDING`，但 `addSign` 只校验任务存在和审批人匹配，没有校验原任务仍处于待办状态。

**影响**

攻击者或异常客户端可以拿一个已审批、已驳回或已转办的历史任务调用加签接口，在已完成节点中重新创建 `PENDING` 任务，破坏流程状态机和待办统计。

**修复建议**

- 在 `addSign` 中补充：

```java
if (!WorkflowConstants.TASK_PENDING.equals(task.getTaskStatus())) {
    throw new BusinessException("TASK_ALREADY_HANDLED", "该任务已被处理");
}
```

- 同时校验所属实例仍为 `RUNNING`、节点仍为 `ACTIVE`。
- 增加“已处理任务不可加签”的回归测试。

### P2-1：成本偏差金额被前端按百分比展示

**位置**

- 后端金额计算：
  - `backend/src/main/java/com/cgcpms/cost/service/CostSummaryService.java`
- 前端展示：
  - `frontend-admin/src/pages/cost/summary.vue`
  - `frontend-admin/src/pages/dashboard/index.vue`

**现象**

后端 `costDeviation` 语义是金额：

```java
BigDecimal costDeviation = dynamicCost.subtract(targetCost);
```

数据库字段注释也是“成本偏差”，但前端 `fmtDeviation()` 将该值乘以 100 并加 `%`，界面列名写为“偏差率”。

**影响**

如果成本偏差金额为 `10000`，前端会展示为 `1000000.00%`，严重误导用户。

**修复建议**

- 方案 A：前端将 `costDeviation` 按金额展示，列名改为“成本偏差(万元)”。
- 方案 B：后端新增 `costDeviationRate` 字段，按 `(dynamicCost - targetCost) / targetCost` 计算百分比；前端百分比展示改用新字段。
- 推荐采用方案 B，保留金额与比例两个口径，避免字段语义混淆。

## 4. 验证结果

### 后端测试

命令：

```powershell
cd backend
.\mvnw.cmd test
```

结果：失败。

主要失败原因：

- `WorkflowEngineIntegrationTest` 启动阶段 Flyway 校验失败：`Migration checksum mismatch for migration version 21`
- 部分 H2 测试出现 `sys_user(id=1)` 重复插入，说明当前测试数据初始化也存在幂等性/上下文隔离问题

### 前端构建

命令：

```powershell
cd frontend-admin
pnpm build
```

结果：通过。

备注：Vite 输出 chunk 体积超过 500kB 的警告，该警告与本次审查发现的功能缺陷无直接关系。

## 5. 建议修复顺序

1. 修复 Flyway V21 checksum 问题：恢复旧迁移，新增 V32 权限补丁。
2. 修复 V24/V26 重复添加 `cost_target_id`，并重新验证空库迁移。
3. 修复工作流租户传递与查询过滤，补充跨租户回归测试。
4. 修复加签状态校验，补充状态机回归测试。
5. 修正成本偏差金额/比例字段契约，前后端同步改名或新增字段。
6. 重新执行后端全量测试、前端构建、空库 Flyway 初始化验证。

## 6. 当前合并建议

当前状态 **不建议合并或发布**。

阻断原因：

- 后端全量测试无法通过。
- Flyway 迁移存在现有库启动失败和空库初始化失败两类 P0 问题。
- 工作流模块仍存在多租户隔离和状态机一致性风险。
