## 3. 问题清单

### P0-1：`pay_invoice` 迁移使用 `created_time/updated_time`，实体写入 `created_at/updated_at`，发票功能不可用

**位置**

- `backend/src/main/resources/db/migration/V36__init_invoice_table.sql:13`
- `backend/src/main/resources/db/migration/V36__init_invoice_table.sql:26`
- `backend/src/main/resources/db/migration/V36__init_invoice_table.sql:28`
- `backend/src/main/java/com/cgcpms/invoice/entity/PayInvoice.java:17`
- `backend/src/main/java/com/cgcpms/common/entity/BaseEntity.java:17`
- `backend/src/main/java/com/cgcpms/invoice/service/InvoiceService.java:60`

**证据**

V36 创建发票表时使用：

```sql
created_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
updated_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

但 `PayInvoice extends BaseEntity`，`BaseEntity` 的字段按 MyBatis-Plus 默认映射为 `created_at`、`updated_at`。`InvoiceService.create()` 调用 `payInvoiceMapper.insert(invoice)` 时生成的 SQL 会包含 `created_at`、`updated_at`。

后端全量测试已复现该问题：

```text
Column "created_at" not found
INSERT INTO pay_invoice (... created_by, created_at, updated_by, updated_at ...)
```

受影响用例包括：

- `InvoiceServiceTest`：11 个测试中 10 个 error
- `Phase4IntegrationTest.test02_invoiceChain`

**影响**

这不是 H2 专属问题，生产 MySQL 迁移 V36 也创建的是 `created_time/updated_time`。因此发票新增、登记、更新、分页按 `createdAt` 排序等路径都会在真实库中失败或不可用。

**修复建议**

- 方案 A：保持 V36 字段名不变，在 `PayInvoice` 中像 `MatWarehouse` 一样显式映射 `createdTime/updatedTime`，并用 `@TableField(exist = false)` 屏蔽继承的 `createdAt/updatedAt`。
- 方案 B：新增迁移把 `pay_invoice.created_time/updated_time` 统一改为 `created_at/updated_at`，保持与 `BaseEntity` 一致。
- 同步修复 `InvoiceVO` 转换和排序字段，并补充发票创建、登记、分页排序的回归测试。

**位置**

- `backend/src/main/resources/application-test.yml:4`

**现象**

本次执行：

```powershell
cd backend
.\mvnw.cmd test
```

结果：失败。

主要失败来源：

- `InvoiceServiceTest`：`pay_invoice.created_at` 缺列。
- `Phase4IntegrationTest`：发票缺列、SSE completed、`sys_user.org_id` 缺列。
- `ProjectOverviewServiceTest`：`sys_user.org_id` 缺列。
- `WorkflowEngineIntegrationTest` 使用 `@ActiveProfiles("test")`，而 `application-test.yml` 指向 `jdbc:mysql://localhost:3306/cgc_pms_test`，当前失败为：

```text
Detected failed migration to version 39 (init phase4 menu perms).
```

同时多数其他测试使用 `@ActiveProfiles("local")`，实际走内嵌 H2/手写 schema。也就是说全量测试不是在同一类隔离测试数据库上运行。

**影响**

- “后端全量测试通过”的结论当前不可复现。
- `test` profile 依赖本机 MySQL 的历史状态，失败迁移会污染后续测试启动。
- `local` profile 依赖手写 H2 DDL，和 Flyway MySQL 迁移链持续漂移。

**修复建议**

- 统一测试 profile：全量单测/集成测试默认使用隔离 H2 或 Testcontainers MySQL，不依赖本机共享库。
- 如必须测试 MySQL Flyway，单独提供可清理、可重建的迁移完整性测试库。
- 将 `WorkflowEngineIntegrationTest` 从 `test` profile 调整为同一隔离 profile，或显式声明其外部依赖。
- 后端 CI 必须执行 `.\mvnw.cmd test` 并以 Surefire 报告为准。

### P1-3：SSE 推送只捕获 `IOException`，已完成 emitter 会让通知创建事务失败

**位置**

- `backend/src/main/java/com/cgcpms/notification/service/NotificationService.java:68`
- `backend/src/main/java/com/cgcpms/notification/service/NotificationService.java:203`
- `backend/src/main/java/com/cgcpms/notification/service/NotificationService.java:209`
- `backend/src/main/java/com/cgcpms/notification/service/NotificationService.java:212`

**证据**

`NotificationService.create()` 在事务内先插入通知，再调用 `pushToUser()`。`pushToUser()` 只捕获 `IOException`，但 `SseEmitter.send()` 在 emitter 已完成时会抛出 `IllegalStateException`。

本次全量测试复现：

```text
java.lang.IllegalStateException: ResponseBodyEmitter has already completed
at com.cgcpms.notification.service.NotificationService.pushToUser(NotificationService.java:209)
```

**影响**

一个已经完成或异常关闭的 SSE 连接，会导致通知创建接口抛错；因为方法有 `@Transactional`，还可能回滚已经插入的通知。通知推送这种旁路能力不应影响主业务写入。

**修复建议**

- `pushToUser()` 捕获 `IOException` 与 `IllegalStateException`，或捕获更窄的发送异常集合并移除失效 emitter。
- 建议在事务提交后异步推送，避免推送失败影响数据库写入。
- 增加“emitter 已 completed 时 create 仍成功”的回归测试。

### P1-4：权限种子与接口 `@PreAuthorize` 权限码不一致，非管理员用户会被错误拒绝

**位置**

- `backend/src/main/java/com/cgcpms/inventory/controller/MatWarehouseController.java:21`
- `backend/src/main/resources/db/migration/V39__init_phase4_menu_perms.sql:49`
- `backend/src/main/resources/db/migration/V39__init_phase4_menu_perms.sql:56`
- `backend/src/main/resources/db/migration/V39__init_phase4_menu_perms.sql:57`
- `backend/src/main/resources/db/migration/V39__init_phase4_menu_perms.sql:58`
- `backend/src/main/java/com/cgcpms/notification/controller/NotificationController.java:39`
- `backend/src/main/resources/db/migration/V39__init_phase4_menu_perms.sql:77`
- `backend/src/main/java/com/cgcpms/alert/controller/AlertController.java:26`

**证据**

仓库接口统一要求：

```java
hasAuthority('inventory:warehouse:*')
```

但 V39 种子授予的是：

```text
inventory:warehouse:list
inventory:warehouse:add
inventory:warehouse:edit
inventory:warehouse:delete
```

Spring Security 的 `hasAuthority` 不会把 `inventory:warehouse:add` 当作 `inventory:warehouse:*` 匹配。

消息接口要求 `notification:view` / `notification:edit`，但 V39 只种了 `notification:list`。预警接口要求 `alert:view` / `alert:edit`，迁移中未看到对应权限种子。

**影响**

非 `ADMIN` 用户即使被授予菜单权限，也会在接口层 403。库存仓库、消息中心、预警中心是第 4 阶段用户可见功能，该问题会直接影响验收。

**修复建议**

- 统一权限码规范：要么接口使用具体权限码，要么数据库种子授予接口实际检查的权限码。
- 不建议在 `hasAuthority` 中使用未实现匹配语义的 `*`。
- 为每个菜单权限增加最小权限用户的接口级测试。

### P2-1：测试手工构造 `UserContext` 未带 `roleCodes`，与真实 JWT 请求路径不一致

**位置**

- `backend/src/main/java/com/cgcpms/auth/context/UserContext.java:30`
- `backend/src/main/java/com/cgcpms/workflow/service/WorkflowQueryService.java:154`
- `backend/src/test/java/com/cgcpms/Phase4IntegrationTest.java:96`
- `backend/src/test/java/com/cgcpms/project/ProjectOverviewServiceTest.java:70`

**现象**

当前实现让 `UserContext` 从 JWT claims 读取 `roleCodes`，并提供 `hasRole()`。`WorkflowQueryService` 已使用 `UserContext.hasRole("ADMIN")` 做审批详情管理员绕过。

但多处测试手工构造 claims 时只设置 `userId`、`username`、`tenantId`，不设置 `roleCodes`。这会导致 service 直测路径中的管理员身份和真实 JWT 路径不一致。

**影响**

权限相关测试可能出现误报或漏报：真实请求中管理员可访问，测试里不可访问；或者测试绕过了 filter 构建 authorities 的路径。

**修复建议**

- 提供统一的测试 claims/helper，例如 `TestUserContext.admin(tenantId)`，默认写入 `roleCodes=["ADMIN"]` 和必要 permissions。
- 对依赖 Spring Security authorities 的接口补充 MockMvc/API 层测试，而不仅是 service 直测。
