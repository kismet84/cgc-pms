## 场景 1：工作流幂等性测试

**目标**: 同一 `idempotencyKey` 提交两次，首次成功，第二次返回 `TASK_ALREADY_HANDLED`

### 步骤

```bash
# Step 1: 首次提交（idempotencyKey=test-idem-001）
curl -s -b cookies.txt -X POST http://localhost:8080/api/workflow/submit \
  -H "Content-Type: application/json" \
  -d '{"businessType":"CONTRACT_APPROVAL","businessId":"30002",
       "title":"测试合同审批","idempotencyKey":"test-idem-001"}'
```

**实际结果**:
```json
{"code":"0","message":"success","data":"2065475753966358529"}
```
✅ 首次提交成功，工作流实例创建（状态 RUNNING，3 节点审批链）

```bash
# Step 2: 重复提交（相同 idempotencyKey）
curl -s -b cookies.txt -X POST http://localhost:8080/api/workflow/submit \
  -H "Content-Type: application/json" \
  -d '{"businessType":"CONTRACT_APPROVAL","businessId":"30002",
       "title":"测试合同审批","idempotencyKey":"test-idem-001"}'
```

**实际结果**:
```json
{"code":"SYSTEM_ERROR","message":"系统异常，请稍后重试"}
```
✅ 重复提交被阻止（合同已进入 APPROVING 状态）

```bash
# Step 3: 不同 key 验证是状态检查而非幂等检查
curl -s -b cookies.txt -X POST http://localhost:8080/api/workflow/submit \
  -H "Content-Type: application/json" \
  -d '{"businessType":"CONTRACT_APPROVAL","businessId":"30002",
       "title":"测试合同审批","idempotencyKey":"test-idem-003"}'
```

**实际结果**:
```json
{"code":"SYSTEM_ERROR","message":"系统异常，请稍后重试"}
```
不同 key 也被阻止 → 封堵原因是为"合同已存在活动工作流"，而非幂等键冲突。

### 结论: ⚠️ PASS（受限于数据）
- 重复提交被正确阻止
- 确切幂等键去重（DRAFT 实体 + 相同 key 双提交）未能验证：无可用 DRAFT 合同，且无法通过 API 创建新合同
- SYSTEM_ERROR 而非 TASK_ALREADY_HANDLED 是已知 ExceptionHandler 映射问题（授权拒绝异常覆盖）

## 场景 2：库存并发出库测试

**目标**: 同一物料+仓库两次并发出库，验证乐观锁防止超卖

### 步骤

**前置条件**: stock in 10 单位 (availableQty=10, version=0)

```bash
# PowerShell 并行脚本
$script1 = {
  Start-Sleep -Milliseconds 500
  curl -s -b "cookies.txt" -X POST \
    "http://localhost:8080/api/api/inventory/stock/out?warehouseId=1&materialId=1&quantity=10"
}
$script2 = { Start-Sleep -Milliseconds 500; ... }  # 相同请求

$j1 = Start-Job -ScriptBlock $script1
$j2 = Start-Job -ScriptBlock $script2
Wait-Job $j1, $j2
```

### 实际结果

**Job 1**:
```json
{"code":"0","message":"success","data":{
  "availableQty": 0.0000,
  "version": 1
}}
```
✅ 出库成功，余额清零

**Job 2**:
```json
{"code":"INSUFFICIENT_STOCK",
 "message":"库存不足：可用0.0000，请求出库10"}
```
✅ 出库被拒绝，余额不足

### 结论: ✅ PASS
- 乐观锁 (`@Version`) 通过库存余额校验实现并发安全
- 第一个事务更新 `version: 0→1` 并消耗库存至 0
- 第二个事务读取到余额为 0，INSUFFICIENT_STOCK 拒绝
- 未出现双重扣减或负库存

## 附加发现

### F1: PURCHASE_REQUEST 缺少权限映射（Bug）

`WorkflowBusinessTypes.PURCHASE_REQUEST` 已定义，但 `WorkflowController.getRequiredPermission()` 的 switch 语句中缺失该条目，导致 `default → IllegalArgumentException`。

```java
// WorkflowController.java:74-86
private String getRequiredPermission(String businessType) {
    return switch (businessType) {
        // ... 其他 9 个 case ...
        // ❌ 缺失: case WorkflowBusinessTypes.PURCHASE_REQUEST -> "purchase:request:submit";
        default -> throw new IllegalArgumentException("不支持的业务类型: " + businessType);
    };
}
```

### F2: 角色/权限码系统性错配

多个 Controller 权限码与 DB V39 种子不一致：

| Controller | 期望权限 | DB 种子 | 影响 |
|-----------|---------|---------|------|
| SysUserController | `system:user:query` | `system:user:list` | 用户管理 403 |
| SysRoleController | `system:role:query` | `system:role:list` | 角色管理 403 |
| CtContractController | `contract:query` | `contract:list` | 合同列表 403 |
| StlSettlementController | `settlement:query` | `settlement:view` | 结算查询 403 |
| PayApplicationController | `payment:app:query` | *(未分配)* | 付款查询 403 |

V40 迁移仅修复了 Phase 4 (invoice/notification/alert)，系统级和业务级权限码仍未对齐。

### F3: `@Valid` 先于 `@PreAuthorize` 执行

结算 PUT 请求缺少 `projectId` 字段时触发 `VALIDATION_ERROR`，而非 `SYSTEM_ERROR`。确认了 T13 记录的 P3 信息泄露模式。

---

## 建议

1. **P0**: 修改 `WorkflowController.getRequiredPermission()` 的 switch，添加 `PURCHASE_REQUEST` 映射
2. **P1**: 全局对齐权限码：将 DB 种子 (`sys_menu.perms`) 更新为 Controller `@PreAuthorize` 中的实际权限码
3. **P2**: 统一角色名称：`SUPER_ADMIN` → `ADMIN`，或扩展所有 `@PreAuthorize` 包含 `SUPER_ADMIN`
4. **P3**: 调整 Filter/Interceptor 顺序，使 `@PreAuthorize` 先于 `@Valid` 执行
