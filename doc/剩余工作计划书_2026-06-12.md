# 剩余工作计划书

计划日期：2026-06-12  
适用项目：建筑工程总包项目全过程管理系统（CGC-PMS）  
依据文档：

- `doc/开发文档_v2.3/01_项目总体方案与业务闭环设计.md`
- `doc/项目完整度分析报告_2026-06-12.md`

## 1. 计划定位

当前项目已进入“可交付候选 / 验收收口”阶段，项目完整度分析报告给出的综合完整度为 **92%**。剩余工作不应再按“新增主功能开发”推进，而应按“验收收口、上线保障、闭环补强、二期范围确认”推进。

本计划书的目标是把剩余 8% 工作拆解为可管理、可验收、可排期的工作包，确保项目从“功能完成”稳定进入“业务验收”和“生产上线准备”。

## 2. 总体原则

剩余工作应继续遵循总体方案中的核心主线：

> 项目为根，合同为纲，执行为据，成本为果，付款为流，结算为终。

具体执行原则：

- 先验收主闭环，再补齐边缘能力。
- 先保证干净环境可复现，再讨论生产上线。
- 先锁定一期范围，再决定移动端和外部系统集成深度。
- 生产上线工作不能只看功能可用，还必须覆盖数据、权限、安全、备份、监控和回滚。
- 所有新增或修复工作必须能落到验收标准，避免只形成描述性文档。

## 3. 范围假设

本计划默认一期交付范围为：

- PC 管理后台。
- Spring Boot 后端服务。
- MySQL 8.0 正式数据库。
- Redis、MinIO 基础依赖。
- Docker Compose 测试/准生产依赖环境。
- 合同、审批、成本、采购、库存、发票、通知、组织、预警、结算等已实现主模块。

本计划默认不把以下内容作为一期强制上线范围：

- uni-app 移动端正式业务功能。
- 企业微信、微信小程序入口。
- 与真实财务系统、BI 系统、OA 系统的生产级双向集成。
- 多项目大规模生产数据压测后的容量扩展。

如上述内容被纳入一期交付，需单独调整排期和验收标准。

## 4. 总体排期建议

建议将剩余工作拆成 4 个阶段推进：

| 阶段 | 建议周期 | 目标 | 结果 |
|------|----------|------|------|
| A. 验收复验 | 3-5 个工作日 | 证明当前系统在干净环境可运行、可测试、可演示 | 形成验收复验报告 |
| B. 业务闭环验收 | 5-8 个工作日 | 按真实业务流程走通项目全过程主链路 | 形成业务验收清单和问题闭环 |
| C. 上线工程保障 | 5-10 个工作日 | 补齐发布、备份、监控、日志、权限、安全 | 形成上线准备清单 |
| D. 二期范围决策 | 3-5 个工作日 | 明确移动端、财务接口、设备租赁、劳务、BI 等后续范围 | 形成二期 Backlog 和优先级 |

建议先完成 A+B，再决定是否进入生产上线；C 阶段可与 B 阶段部分并行，但不能省略。

## 5. 工作包一：干净环境复验

优先级：P0  
建议周期：3-5 个工作日  
目标：确认系统不依赖历史构建产物、历史数据库或本机残留状态。

### 5.1 后端全量测试复验

工作内容：

- 在干净工作区执行后端全量测试。
- 清理或隔离本机历史 MySQL、H2、target 产物影响。
- 记录测试命令、耗时、通过数、失败数。
- 对照现有 Surefire 报告中的 162/162 通过记录，确认最新结果一致。

建议命令：

```powershell
cd backend
.\mvnw.cmd clean test
```

验收标准：

- 后端全量测试通过。
- Surefire 报告可追溯。
- 若失败，必须形成失败清单、责任模块、修复建议和复验记录。

交付物：

- `doc/验收复验报告_YYYY-MM-DD.md`
- 后端测试日志摘要。

### 5.2 前端生产构建复验

工作内容：

- 清理或隔离历史 `dist` 产物影响。
- 安装依赖并执行前端类型检查和生产构建。
- 记录构建警告，尤其是大 chunk、类型错误、路由懒加载问题。

建议命令：

```powershell
cd frontend-admin
pnpm install
pnpm build
```

验收标准：

- `pnpm build` 成功。
- 无 TypeScript 阻断错误。
- 构建警告已记录并判断是否阻断上线。

交付物：

- 前端构建结果摘要。
- 构建警告处置意见。

### 5.3 MySQL 全新库迁移复验

工作内容：

- 新建空 MySQL 数据库。
- 使用 Flyway 从 V1 跑到 V40。
- 验证关键表、索引、唯一约束、权限种子、审批模板种子存在。
- 抽查 `wf_idempotency`、`sys_notification`、`pay_invoice`、`sys_user`、`stl_settlement` 等历史风险表。

验收标准：

- 全新库迁移一次成功。
- Flyway schema history 状态正常。
- 历史审查问题涉及字段和约束均存在。

交付物：

- MySQL 迁移验证记录。
- 关键表结构抽查记录。

### 5.4 Docker Compose 启动复验

工作内容：

- 基于 `deploy/.env.example` 创建测试 `.env`。
- 启动 MySQL、Redis、MinIO。
- 验证健康检查、端口、账号、桶初始化或文件上传路径。

建议命令：

```powershell
cd deploy
docker compose up -d
docker compose ps
```

验收标准：

- MySQL、Redis、MinIO 均处于 healthy 或可连接状态。
- 后端能连接依赖服务。
- 文件上传和预签名 URL 可用。

交付物：

- Docker Compose 启动验证记录。

## 6. 工作包二：业务主闭环验收

优先级：P0  
建议周期：5-8 个工作日  
目标：按真实业务过程验证系统是否满足总体方案中的闭环原则。

### 6.1 项目到合同闭环

工作内容：

- 新建项目。
- 新建合作方。
- 新建采购合同、分包合同、服务合同或其他合同。
- 维护合同清单、付款条件、附件。
- 提交合同审批并完成审批。
- 验证合同状态、审批记录、成本锁定结果。

验收标准：

- 所有业务数据均关联 `project_id`。
- 涉及合同业务均关联 `contract_id`。
- 审批中合同不可编辑关键字段。
- 审批通过后合同状态和成本生成结果一致。

交付物：

- 合同闭环验收记录。
- 问题清单和截图。

### 6.2 采购与材料闭环

工作内容：

- 基于采购合同创建采购申请或采购订单。
- 完成材料验收。
- 完成入库。
- 完成出库。
- 验证库存余额、库存流水、材料成本归集。
- 验证未验收付款限制或预警。

验收标准：

- 采购订单可追溯项目、合作方、合同。
- 验收入库能生成或影响材料成本。
- 出入库后库存余额正确。
- 成本来源能追溯至材料验收或出库事实。

交付物：

- 采购材料闭环验收记录。

### 6.3 分包与计量闭环

工作内容：

- 基于分包合同创建分包任务。
- 创建分包计量。
- 提交分包计量审批。
- 审批通过后验证分包成本归集。
- 发起分包付款申请，验证付款依据与计量数据一致。

验收标准：

- 分包任务、计量、付款均可追溯 `project_id` 和 `contract_id`。
- 分包成本来自计量确认，不来自付款审批。
- 超合同、未计量付款等规则能触发限制或预警。

交付物：

- 分包计量闭环验收记录。

### 6.4 付款与发票闭环

工作内容：

- 基于合同、验收或计量数据发起付款申请。
- 完成付款审批。
- 录入或回写实际付款记录。
- 创建、登记、核验发票。
- 验证付款申请、付款记录、发票、合同金额、成本汇总之间关系。

验收标准：

- 付款申请有明确业务依据。
- 实际付款作为资金流，不被误用为成本发生依据。
- 发票登记可关联付款记录。
- 付款比例、超额、质保金等规则可被验证。

交付物：

- 付款发票闭环验收记录。

### 6.5 结算与归档闭环

工作内容：

- 基于合同创建结算单。
- 自动汇总合同、变更签证、付款记录、成本明细。
- 验证结算单可反查来源。
- 提交结算审批。
- 审批通过后验证结算锁定和合同结算金额回写。

验收标准：

- 结算不重复录入孤立数据。
- 结算金额来源可追溯。
- 结算审批通过后结算单不可随意修改。
- 合同结算金额与结算单最终金额一致。

交付物：

- 结算闭环验收记录。

### 6.6 经营分析与预警闭环

工作内容：

- 查看项目驾驶舱。
- 验证合同金额、动态成本、利润测算、付款比例、结算进度。
- 手动触发预警批量评估。
- 验证合同、成本、付款、结算类预警是否生成。

验收标准：

- 驾驶舱数据能反映项目经营状态。
- 预警规则与业务事实一致。
- 预警能定位到具体项目、合同或业务单据。

交付物：

- 经营分析与预警验收记录。

## 7. 工作包三：权限、安全与多租户验收

优先级：P0  
建议周期：3-5 个工作日  
目标：确认系统在多人、多角色、多项目、多租户场景下不会越权。

### 7.1 角色权限矩阵验收

工作内容：

- 整理角色清单：公司管理层、项目经理、商务经理、成本经理、合同管理员、材料员、设备管理员、分包管理员、财务人员、资料员、审计人员。
- 对照菜单权限、按钮权限、接口权限。
- 按角色执行登录和核心页面访问测试。
- 验证无权限用户不能调用接口。

验收标准：

- 管理员可完整访问管理能力。
- 普通角色仅能访问授权菜单和按钮。
- 后端接口权限与前端菜单按钮一致。
- 未授权接口返回 403 或统一权限错误。

交付物：

- 权限矩阵验收表。

### 7.2 多租户与项目数据隔离验收

工作内容：

- 准备两个租户或两个隔离项目的数据。
- 使用不同用户访问项目、合同、付款、发票、文件、通知。
- 验证列表、详情、下载、删除、审批、通知均不会跨租户或跨项目泄露。

验收标准：

- 列表查询不返回非授权数据。
- 详情查询不能通过 IDOR 访问非授权数据。
- 文件预签名 URL、删除接口受权限控制。
- 通知只发送给正确用户。

交付物：

- 数据隔离验收记录。

### 7.3 安全基线复核

工作内容：

- 复核 JWT、Refresh Token、退出登录、Redis 黑名单。
- 复核上传文件扩展名、大小、路径注入防护。
- 复核敏感字段脱敏。
- 复核生产配置不得硬编码密钥。

验收标准：

- 密码、token、secret 不出现在日志和提交配置中。
- 上传接口不能上传非白名单文件。
- 退出登录后旧 token 不可继续使用。
- 禁用账号不能刷新 token。

交付物：

- 安全基线复核清单。

## 8. 工作包四：端到端测试与质量补强

优先级：P1  
建议周期：5-8 个工作日  
目标：补齐当前测试偏后端服务层的问题，让验收更贴近真实用户操作。

### 8.1 前端端到端测试

建议覆盖场景：

- 登录。
- 新建合同并提交审批。
- 审批待办处理。
- 采购申请或采购订单。
- 材料入库和出库。
- 发票创建、登记、核验。
- 通知列表、未读数、标记已读。
- 结算详情查看和提交审批。

验收标准：

- 核心路径至少有一套可重复执行的浏览器自动化脚本。
- 页面路由、接口调用、权限跳转、提示信息均可验证。
- 自动化脚本可在本地或 CI 中执行。

交付物：

- E2E 测试脚本。
- E2E 测试报告。

### 8.2 并发与一致性测试

建议覆盖场景：

- 审批幂等键重复提交。
- 同一任务并发审批。
- 库存同一物料并发出库。
- 付款申请与合同余额校验。
- 成本生成重复触发。
- 结算审批通过后的二次修改拦截。

验收标准：

- 幂等、唯一约束、乐观锁、悲观锁均有测试覆盖。
- 并发失败路径有明确业务异常。
- 不产生重复成本、重复付款、负库存等数据。

交付物：

- 并发一致性测试报告。

### 8.3 性能基线测试

建议覆盖接口：

- 合同台账分页。
- 成本台账分页。
- 项目驾驶舱。
- 预警批量评估。
- 消息通知分页。
- 结算金额计算。

验收标准：

- 明确测试数据量级。
- 明确接口响应时间基线。
- 超过阈值的接口给出优化建议。

交付物：

- 性能基线报告。

## 9. 工作包五：上线工程保障

优先级：P1  
建议周期：5-10 个工作日  
目标：让系统从“能部署”提升到“可上线、可运维、可回滚”。

### 9.1 应用镜像与发布流程

工作内容：

- 补齐后端 Dockerfile。
- 补齐前端 Dockerfile 或 Nginx 静态资源部署方案。
- 明确镜像 tag 规则。
- 明确 dev、test、prod 环境配置差异。
- 编写发布步骤和回滚步骤。

验收标准：

- 后端和前端可通过镜像部署。
- 环境变量不硬编码。
- 发布失败有可执行回滚方案。

交付物：

- 部署手册。
- 镜像构建说明。

### 9.2 CI/CD 流水线

工作内容：

- 配置后端测试。
- 配置前端类型检查和构建。
- 配置迁移完整性检查。
- 配置产物归档。

验收标准：

- 提交代码后自动执行测试和构建。
- 测试失败阻断合并。
- 构建产物可追溯。

交付物：

- CI 配置文件。
- CI 运行说明。

### 9.3 数据备份与恢复

工作内容：

- 制定 MySQL 备份策略。
- 制定 MinIO 文件备份策略。
- 制定 Redis 可丢失和不可丢失数据边界。
- 做一次恢复演练。

验收标准：

- 有每日备份策略。
- 有恢复步骤。
- 有恢复演练记录。
- 明确 RPO/RTO 目标。

交付物：

- 备份恢复方案。
- 恢复演练记录。

### 9.4 日志、监控与告警

工作内容：

- 明确应用日志路径和格式。
- 明确错误日志采集方式。
- 增加关键业务指标监控建议。
- 增加依赖服务监控建议。

建议监控项：

- 后端应用存活。
- MySQL 连接池。
- Redis 可用性。
- MinIO 可用性。
- 接口 5xx 错误率。
- 登录失败次数。
- 审批失败数。
- 文件上传失败数。
- 预警批处理失败数。

验收标准：

- 生产问题能通过日志定位。
- 关键依赖不可用能被发现。
- 关键业务失败能触发告警。

交付物：

- 监控告警清单。

## 10. 工作包六：二期范围确认与补齐

优先级：P2  
建议周期：3-5 个工作日完成决策，后续按二期计划实施。  
目标：把总体方案中的长期闭环能力拆成清晰二期 Backlog。

### 10.1 移动端正式范围

现状：

- `mobile/` 目录目前为 uni-app 预留。
- 总体方案中提到移动端作为现场主端，企业微信和微信小程序作为轻入口。
- 项目完整度报告未将移动端计入一期核心交付。

建议决策：

- 一期验收：明确移动端不纳入本期上线。
- 二期最小闭环：登录、待办、审批详情、审批操作、通知中心、材料验收拍照上传。
- 三期扩展：现场签证、出入库扫码、进度填报、质量安全记录。

验收标准：

- 验收范围文档明确移动端边界。
- 二期 Backlog 有页面、接口、权限和验收标准。

### 10.2 财务系统接口

现状：

- 当前系统已有付款申请、付款审批、付款记录和发票能力。
- 总体方案要求实际付款数据以财务系统为准。

建议二期工作：

- 定义付款回写接口。
- 定义发票同步接口。
- 定义供应商/合作方银行账户校验接口。
- 定义对账和异常处理机制。

验收标准：

- 实际支付金额、支付日期、付款凭证可从财务系统同步。
- 同一付款申请不会重复回写。
- 回写失败可重试、可追踪。

### 10.3 设备租赁与劳务管理

现状：

- 总体方案将设备租赁、劳务记录列为成本来源。
- 当前主系统已覆盖材料、采购、分包、库存、发票、结算等主链路，设备租赁和劳务可作为二期深化模块。

建议二期工作：

- 设备租赁合同引用。
- 设备进退场记录。
- 机械使用台班。
- 设备维保费用。
- 劳务班组、劳务记录、人工成本归集。

验收标准：

- 设备和劳务成本能进入成本台账。
- 成本来源可通过 `source_type/source_id` 反查。
- 不重复维护合同主数据。

### 10.4 审计资料归档与 BI 分析

建议二期工作：

- 项目资料归档目录。
- 合同、变更、签证、付款、发票、结算资料自动归档索引。
- 审计资料缺失预警。
- BI 分析主题：合同分析、成本分析、付款分析、结算分析、供应商分析。

验收标准：

- 结算时可按项目导出或查看审计资料清单。
- 资料缺失可预警。
- BI 指标口径与成本/付款/结算定义一致。

## 11. 风险与依赖

| 风险 | 影响 | 应对 |
|------|------|------|
| 移动端是否纳入一期范围不明确 | 完整度和排期判断失真 | 在验收前明确移动端为一期外或一期内 |
| 财务系统接口边界不明确 | 付款闭环无法生产化 | 先定义接口契约和回写口径，再开发 |
| 生产运维缺口被低估 | 功能验收后仍无法上线 | 将备份、监控、发布、回滚列为上线门禁 |
| 权限矩阵未业务确认 | 上线后出现越权或误拦截 | 业务角色逐项签字确认 |
| 测试依赖历史环境 | 验收结论不可复现 | 必须执行干净环境复验 |
| 复杂业务链路后续改动引发回归 | 合同、成本、付款、结算数据不一致 | 新增需求必须绑定回归测试和迁移校验 |

## 14. 最终建议

当前项目不建议继续以“补主功能”为主线推进，而应转入“验收收口 + 上线工程化 + 二期范围锁定”的工作模式。

优先顺序建议为：

1. 先完成干净环境复验，证明系统可复现。
2. 再完成业务主闭环验收，证明系统可被业务接受。
3. 同步补齐权限、安全、数据隔离验收，证明系统可受控使用。
4. 上线前补齐发布、备份、监控、回滚，证明系统可运维。
5. 最后将移动端、财务接口、设备租赁、劳务、BI 和审计归档纳入二期计划。

按该顺序推进，项目可以从当前 A- 的“可验收候选”提升为 A 级“可正式交付上线候选”。
