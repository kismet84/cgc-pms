## 3. 测试中发现的 URL 纠正

任务规范中的端点 URL 与实际工作 URL 存在差异：

| 任务规范 URL | 实际工作 URL | 原因 |
|-------------|-------------|------|
| `/api/cost/ledger` | `/api/cost-ledger` | Controller 使用 `@RequestMapping("/cost-ledger")`，非 `/cost/ledger` |
| `/api/notifications` | `/api/api/notifications` | Phase 4 Controller 使用 `@RequestMapping("/api/notifications")`，与 context-path `/api` 叠加 |

## 4. 环境问题与修复

| 问题 | 影响 | 修复 |
|------|------|------|
| ADMIN 角色缺失 | 4/5 端点返回 500 SYSTEM_ERROR | INSERT INTO sys_role (id=4, role_code='ADMIN') + sys_user_role 映射 |
| 多 MySQL 实例混淆 | 数据查询到错误实例 (3306 vs 3307) | 确认 dev profile 连接端口 3307 |
| SUPER_ADMIN vs ADMIN 角色码不匹配 | hasRole('ADMIN') 无法匹配 ROLE_SUPER_ADMIN | 新增 role_code='ADMIN' 角色并分配给 admin 用户 |

## 5. 结论

### 5.1 P95 达标情况

全部 5 个端点 P95 均远低于目标值：

| 端点 | P95 实测 | P95 目标 | 余量 |
|------|---------|---------|------|
| GET /api/contracts | 12.2ms | 500ms | 97.6% |
| GET /api/cost-ledger | 6.1ms | 1000ms | 99.4% |
| GET /api/dashboard/cost-manager | 34.6ms | 2000ms | 98.3% |
| POST /api/alerts/batch-evaluate | 6.1ms | 5000ms | 99.9% |
| GET /api/api/notifications | 5.7ms | 300ms | 98.1% |

### 5.2 关键发现

1. **空载基线已建立**: 在近乎空数据库条件下，所有端点 P95 < 35ms，远低于目标
2. **数据库为极轻量状态**: 多数业务表行数为 0，系统表共约 300 行
3. **本次基线不具备生产代表性**: 无法评估数据量增长后的性能衰减
4. **Dashboard 端点响应波动最大**: 涉及多表 JOIN/聚合，P95=34.6ms（约为简单查询的 3-6 倍）
5. **连接池预热效应**: 各端点首请求普遍比后续请求慢 2-3 倍（HikariCP lazy init）

### 5.3 建议

- **P1**: 使用数据生成器填充 1万~10万行业务数据后重新测试
- **P2**: 在 dashboard 端点加载条件下进行并发测试（当前为串行）
- **P3**: 修复 Phase 4 Controller 双 `/api` 前缀问题
- **P3**: 统一角色码：将 `SUPER_ADMIN` 改为 `ADMIN` 或调整 `@PreAuthorize` 注解
