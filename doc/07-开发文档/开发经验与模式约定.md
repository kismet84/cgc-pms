## 一、前端模式约定

### 1.1 List+Filter 列表页范式

所有列表页遵循统一范式：filter card（筛选区）→ a-table（数据表格）→ 独立的 a-pagination（分页器）。这是从 `pages/project/index.vue` 确立的基础模式，后续所有模块沿用。

**来源**: `learnings.md:4`, `learnings.md:26-28`

**结构示例**:

```
<template>
  <div class="pm-page">
    <!-- 筛选区 -->
    <div class="pm-filter">
      <a-row :gutter="16">
        <a-col :span="6"><!-- 筛选字段 --></a-col>
      </a-row>
      <div class="pm-filter-actions">
        <a-button type="primary" @click="fetchData">查询</a-button>
        <a-button @click="resetFilters">重置</a-button>
      </div>
    </div>
    <!-- 表格 -->
    <a-table :columns="columns" :data-source="dataSource" :loading="loading" />
    <!-- 分页 -->
    <a-pagination v-model:current="pageNo" :total="total" @change="fetchData" />
  </div>
</template>
```

**注意事项**:
- 分页参数统一使用 `pageNo`/`pageSize`（不是 `pageNum`/`pageSize`），因为后端 `PageResult` 使用 `pageNo`。
- 若后端返回不分页的 flat list（如 `/alerts`），由前端通过 `computed` 做客户端分页。

### 1.2 Pinia Store 模式

所有 Pinia store 使用 setup store 风格：`ref` 管理状态、`async` 函数封装 API 调用、`resetState` 方法重置。

**来源**: `learnings.md:5`

**标准模板**:

```typescript
// stores/example.ts
export const useExampleStore = defineStore('example', () => {
  const list = ref<ExampleVO[]>([])
  const loading = ref(false)

  async function fetchList(params: ExampleQueryParams) {
    loading.value = true
    try {
      const res = await getExampleList(params)
      list.value = res.records
    } finally {
      loading.value = false
    }
  }

  function resetState() {
    list.value = []
    loading.value = false
  }

  return { list, loading, fetchList, resetState }
})
```

### 1.3 API 模块模式

所有 API 函数定义在 `src/api/modules/<module>.ts` 下，使用 `request<T>()` 泛型包装器。遵循 `api/modules/workflow.ts` 和 `api/modules/contract.ts` 的风格。

**来源**: `learnings.md:6`, `learnings.md:28`

```typescript
// api/modules/example.ts
import request from '@/utils/request'

export function getExampleList(params: ExampleQueryParams) {
  return request<PageResult<ExampleVO>>({
    url: '/examples',
    method: 'get',
    params
  })
}

export function createExample(data: ExampleCreateDTO) {
  return request<ApiResponse<ExampleVO>>({
    url: '/examples',
    method: 'post',
    data
  })
}
```

**关键约定**:
- `request.ts` 已配置 `baseURL=/api`，所有路径相对 `/api` 即可。
- `request.ts` 自动从 `ApiResponse` 中提取 `.data` 字段，调用方直接拿业务对象。
- 前端类型 ID 使用 `string`（即使后端是 Snowflake Long），JS 精度安全。

每个业务模块使用独立的两字母 CSS 前缀，避免跨模块样式冲突。前缀与模块的对应关系：

## 二、后端模式约定

### 2.1 Controller 路径约定

**来源**: `code-review-fixes/learnings.md:4-14`, `phase-closure-execution/learnings.md:146-147`

`application.yml` 中已设置 `server.servlet.context-path: /api`，**所有 Controller 的 `@RequestMapping` 不得再包含 `/api` 前缀**。

```java
// ✅ 正确
@RestController
@RequestMapping("/contracts")
public class CtContractController { }

// ❌ 错误——导致 /api/api/contracts
@RestController
@RequestMapping("/api/contracts")
public class SomeController { }
```

**已知违规**（Phase 3/4 引入，待修复）:
- `CostLedgerController`、`CostSubjectController`（Phase 3 引入）
- `InvoiceController`、库存 controllers（Phase 4 引入）
- `NotificationController`（Phase 4 引入）

对于这些控制器，实际访问路径为 `/api/api/xxx`。

### 2.2 Service 分层约定

**来源**: `code-review-fixes/learnings.md:8`, `learnings.md:110-114`

| 层级 | 职责 | 禁止行为 |
|------|------|---------|
| Controller | 参数校验 (`@Valid`)、调用 Service、组装响应 | 不可直接操作 Mapper |
| Service | 业务逻辑、事务管理 (`@Transactional`)、租户隔离 | 不可读取 HTTP 请求上下文 |
| Mapper | 数据访问（MyBatis-Plus BaseMapper） | 不可包含业务逻辑 |

Service 内部使用 `LambdaQueryWrapper` 和 `LambdaUpdateWrapper`（类型安全的列引用）。

**来源**: `learnings.md:110`
```java
// 典型 Service 查询模式
LambdaQueryWrapper<Entity> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(Entity::getProjectId, projectId)
       .eq(Entity::getTenantId, UserContext.getCurrentTenantId())
       .orderByDesc(Entity::getCreatedTime);
```

### 2.3 VO/Entity 映射约定

**来源**: `code-review-fixes/learnings.md:17-35`

- VO 用于 API 响应，Entity 用于 DB 映射。两者可同名（前后端各自独立）。
- 前端 VO 的类型名自洽即可，不需要与后端类名精确一致。如前端 `CostSummaryVO` 对应后端 `CostProjectSummaryVO`，结构匹配即可，不强制改名。
- VO 中 ID 字段使用 `String` 类型（Snowflake Long → 字符串，避免 JS 精度丢失）。
- 日期字段在 VO 中格式化为 String（`LocalDateTime` → ISO 字符串），由 Jackson 序列化完成。

### 2.4 多租户模式

**来源**: `code-review-fixes/learnings.md:8`, `phase-closure-execution/learnings.md:394-412`

#### 数据隔离

所有包含业务数据的表必须有 `tenant_id BIGINT NOT NULL DEFAULT 0` 列（共 52 张表）。

**Service 层查询必须追加租户过滤**:

```java
// 列表查询
wrapper.eq(Entity::getTenantId, UserContext.getCurrentTenantId());

// 单条查询——selectById 后校验
Entity entity = mapper.selectById(id);
if (entity == null || !entity.getTenantId().equals(UserContext.getCurrentTenantId())) {
    throw new BusinessException("..._NOT_FOUND");
}
```

#### @Scheduled 线程安全

定时任务方法不在请求线程中执行，没有 `UserContext`。使用方式如下：

**来源**: `learnings.md:138-144`, `code-review-fixes/learnings.md:60-64`

```java
// ❌ 错误：在 @Scheduled 中读取 UserContext
@Scheduled(...)
public void doSomething() {
    Long tenantId = UserContext.getCurrentTenantId();  // 返回 null
}

// ✅ 正确：从业务实体上获取 tenantId
@Scheduled(...)
public void scheduledRefresh() {
    List<PmProject> projects = projectMapper.selectList(...);
    for (PmProject project : projects) {
        service.doWork(project.getTenantId(), project.getId());
    }
}
```

**推荐模式**：提供带 `(Long tenantId, ...)` 参数的重载方法，现有方法内部委托调用（`UserContext.getCurrentTenantId()`）。

#### IDOR 防护

**来源**: `phase-closure-execution/learnings.md:406-407`

所有安全校验返回 HTTP 200 + 统一错误码（如 `PROJECT_NOT_FOUND`），不使用 403/404。这防止了资源存在性枚举攻击（攻击者无法区分"资源不存在"与"资源属于其他租户"）。

### 2.5 审计字段处理（BaseEntity 阴影模式）

**来源**: `phase3-cost-analysis-contract-deepening/learnings.md:142-147`, `phase4-org-inventory-invoice-integration/learnings.md:3-9`

项目在 V22 后变更了审计列命名，导致新旧两种命名共存：

- V1~V21 的旧表（如 `ct_contract`, `wf_instance`）使用 `created_at` / `updated_at`
- V22+ 的新表（如 `cost_target`, `ct_contract_change`, `alert_log`）使用 `created_time` / `updated_time`

`BaseEntity` 默认映射 `createdAt` → `created_at`、`updatedAt` → `updated_at`。对于使用新命名 (`created_time` / `updated_time`) 的实体，使用**阴影覆盖模式**：

```java
@Data
@EqualsAndHashCode(callSuper = true)
@TableName("example_table")
public class ExampleEntity extends BaseEntity {

    // 映射新列名
    @TableField(value = "created_time", fill = FieldFill.INSERT)
    private LocalDateTime createdTime;

    @TableField(value = "updated_time", fill = FieldFill.INSERT_UPDATE)
    private LocalDateTime updatedTime;

    // 屏蔽 BaseEntity 的旧映射
    @TableField(exist = false)
    private LocalDateTime createdAt;

    @TableField(exist = false)
    private LocalDateTime updatedAt;
}
```

**关键点**：
- 使用与原 BaseEntity 相同的 Java 字段名（`createdAt`, `updatedAt`）做阴影并标记 `exist = false`，因为 MyBatis-Plus 的 `MetaObjectHandler` 按字段名自动填充。
- 其余审计字段（`createdBy` → `created_by`, `updatedBy` → `updated_by`, `deletedFlag` → `deleted_flag`, `remark` → `remark`）命名一致，正常继承。

**来源**: `phase4-org-inventory-invoice-integration/learnings.md:106-107`
> ⚠️ **例外**：`V36__init_invoice_table.sql` 创建的 `pay_invoice` 表使用了旧命名 `created_at`/`updated_at`，是 Phase 4 中唯一不一致的表，需修复为 `created_time`/`updated_time`。

### 2.6 权限注解模式

**来源**: `phase3-cost-analysis-contract-deepening/learnings.md:5-8`, `audit-fixes/learnings.md:210-244`

#### 声明式鉴权

```java
@RestController
@RequestMapping("/examples")
public class ExampleController {

    @PreAuthorize("hasRole('ADMIN') or hasAuthority('example:query')")
    @GetMapping
    public PageResult<ExampleVO> list(...) { }

    @PreAuthorize("hasRole('ADMIN') or hasAuthority('example:add')")
    @PostMapping
    public ApiResponse<ExampleVO> create(@Valid @RequestBody ExampleCreateDTO dto) { }
}
```

#### 动态鉴权（审批提交）

审批提交接口需要根据 `businessType` 运行时确定权限码，因此使用编程式检查：

```java
@PostMapping("/submit")
public ApiResponse<WfInstanceVO> submit(@Valid @RequestBody WorkflowSubmitRequest request) {
    checkSubmitPermission(request.getBusinessType());
    // ...
}

private void checkSubmitPermission(String businessType) {
    var authorities = SecurityContextHolder.getContext()
        .getAuthentication().getAuthorities();
    // ROLE_ADMIN 放行
    if (authorities.stream().anyMatch(a -> a.getAuthority().equals("ROLE_ADMIN"))) return;
    // 检查具体的业务权限码
    String requiredPerm = switch (businessType) {
        case "CONTRACT_APPROVAL" -> "contract:submit";
        case "PURCHASE_ORDER" -> "purchase:order:submit";
        // ... 其他 businessType
        default -> throw new IllegalArgumentException("Unknown businessType: " + businessType);
    };
    if (authorities.stream().noneMatch(a -> a.getAuthority().equals(requiredPerm))) {
        throw new AccessDeniedException("无权限提交该类型审批");
    }
}
```

**来源**: `phase-closure-execution/learnings.md:695`
> ⚠️ **已知缺陷**：`checkSubmitPermission()` 的 switch 缺少 `PURCHASE_REQUEST` 分支，会抛出 `IllegalArgumentException` → 500。

### 2.7 Mass Assignment 防护

**来源**: `audit-fixes/learnings.md:110-174`

Controller 直接绑定 Entity 作为 `@RequestBody` 时，攻击者可能通过 JSON 覆写服务端管理的字段（如 `tenantId`, `createdBy`, `approvedStatus`）。使用 `@JsonProperty(access = READ_ONLY)` 阻止反序列化赋值：

```java
// BaseEntity.java 和 PayInvoice 等敏感实体
@JsonProperty(access = JsonProperty.Access.READ_ONLY)
private Long tenantId;

@JsonProperty(access = JsonProperty.Access.READ_ONLY)
private String verifyStatus;

@JsonProperty(access = JsonProperty.Access.READ_ONLY)
private LocalDateTime createdTime;
```

`READ_ONLY` 优于 `@JsonIgnore`：它允许序列化（GET 响应中包含字段），但阻止反序列化（POST/PUT 中忽略字段）。服务层仍可通过 setter 正常赋值。

### 2.8 输入校验模式

**来源**: `audit-fixes/learnings.md:48-80`

Controller 的 `@RequestBody` 参数必须加 `@Valid`，实体类必须加 Jakarta Validation 注解。参考 `CtContract` 的已有模式：

```java
// Controller
@PostMapping
public ApiResponse<InvoiceVO> create(@Valid @RequestBody InvoiceVO vo) { }

// Entity/VO
public class PayInvoice {
    @NotBlank
    private String invoiceNo;

    @NotBlank
    private String invoiceType;

    @NotNull
    private BigDecimal invoiceAmount;
}
```

`GlobalExceptionHandler` 已统一处理 `MethodArgumentNotValidException`，返回 400 + `VALIDATION_ERROR` + 字段名。

### 2.9 编辑/删除守卫模式

**来源**: `code-review-fixes/learnings.md:43-56`, `phase-closure-execution/learnings.md:343-344`

所有可变操作（update/delete/saveItems）必须检查审批状态，阻止在审批中修改。采用统一守卫模式：

```java
// 通用守卫
if (!"DRAFT".equals(entity.getApprovalStatus())) {
    throw new BusinessException("XXX_IN_APPROVAL", "审批中不可编辑");
}

// 加上成本生成标志（如有）
if (entity.getCostGeneratedFlag() != null && entity.getCostGeneratedFlag() == 1) {
    throw new BusinessException("COST_GENERATED", "成本已生成，不可编辑");
}
```

| 实体 | 守卫字段 | 错误码 |
|------|---------|--------|
| MatPurchaseOrder | `approvalStatus != DRAFT` | `ORDER_IN_APPROVAL` |
| MatReceipt | `approvalStatus != DRAFT` | `RECEIPT_IN_APPROVAL` |
| VarOrder | `approvalStatus != DRAFT` + `costGeneratedFlag=1` | `不可编辑/不可删除` |
| StlSettlement | `approvalStatus != DRAFT` | `STL_SETTLEMENT_IN_APPROVAL` |
| CtContractChange | `approvalStatus != DRAFT` + `costGeneratedFlag=1` | `不可编辑/不可删除` |

### 2.10 批量预取模式

**来源**: `phase4-org-inventory-invoice-integration/learnings.md:29`

对于列表数据中需要关联查询的字段，使用批量预取代替逐条查询：

### 3.2 审计列约定

**来源**: `phase3-cost-analysis-contract-deepening/learnings.md:51,60-67`

#### 历史演进

| 版本范围 | 时间列命名 | 状态 |
|---------|-----------|------|
| V1~V21 | `created_at` / `updated_at` | 遗留命名，不再使用 |
| V22+ | `created_time` / `updated_time` | 当前正式命名 |
| V36（例外） | `created_at` / `updated_at` | 唯一违规，待修复 |

#### 新表审计列模板

```sql
`created_time` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
```

### 3.4 索引策略

## 四、审批工作流模式

### 4.1 WorkflowBusinessHandler 实现

**来源**: `phase4-org-inventory-invoice-integration/learnings.md:12-17`, `phase3-cost-analysis-contract-deepening/learnings.md:193-198`

```java
@Component
public class ExampleWorkflowHandler implements WorkflowBusinessHandler {

    @Override
    public String supportBusinessType() {
        return WorkflowBusinessTypes.EXAMPLE; // 常量定义在 WorkflowBusinessTypes 中
    }

    @Override
    public boolean isCritical() {
        return true;  // true=异常回滚事务，false=swallow-and-log
    }

    @Override
    public void onApproved(WorkflowContext context) {
        // 从 context 获取业务 ID
        Long businessId = context.getInstance().getBusinessId();
        // 更新审批状态 → 成本联动 → 汇总刷新
    }

    @Override
    public void onRejected(WorkflowContext context) {
        // 仅更新 approvalStatus=REJECTED，不做金额/成本变更
    }

    @Override
    public void onWithdrawn(WorkflowContext context) {
        // 恢复为 DRAFT 状态
    }

    @Override
    public void beforeSubmit(WorkflowContext context) {
        // 提交前校验（如检查是否有未审批的关联单据）
    }
}
```

**关键规则**:
- **禁止在 Handler 回调中读取 `UserContext`**。Handler 由工作流引擎调用，不在请求线程中。使用 `context.getInstance().getBusinessId()` 获取业务 ID，从业务实体获取 tenantId 等信息。
- `isCritical()` 为 `true` 时，Handler 异常会回滚整个审批事务；`isCritical()` 为 `false` 时异常被吞并记录日志。
- 成本联动型 Handler（Contract、VarOrder、MaterialReceipt、SubMeasure、CtContractChange）的 `isCritical()` 均为 `true`。
- 结算 Handler 的 `isCritical()` 为 `false`，因为它不生成成本。

### 4.3 提交审批模式

**来源**: `phase3-cost-analysis-contract-deepening/learnings.md:207-213`

```java
// Service 层
@Transactional
public void submitForApproval(Long id) {
    Entity entity = mapper.selectById(id);
    // 1. 校验 DRAFT 状态
    if (!"DRAFT".equals(entity.getApprovalStatus())) {
        throw new BusinessException("ALREADY_SUBMITTED");
    }
    // 2. 更新为 APPROVING
    entity.setApprovalStatus("APPROVING");
    mapper.updateById(entity);
    // 3. 调用工作流引擎提交
    workflowEngine.submit(businessType, entity.getId(), title, idempotencyKey);
}
```

### 4.4 审批回调与状态流转

**来源**: `phase-closure-execution/learnings.md:280-325`

| 操作 | 状态变更 | 后续动作 |
|------|---------|---------|
| 提交（Submit） | `DRAFT` → `APPROVING` | 创建工作流实例 + 第一节点任务 |
| 同意（Approve） | 当前任务 → COMPLETED | 创建下一节点任务，最后一节点 → 实例 CLOSED |
| 驳回（Reject） | 实例 → REJECTED | 业务实体 `approvalStatus=REJECTED`，可重新编辑再提交 |
| 撤回（Withdraw） | 实例 → WITHDRAWN | 取消所有待办任务，业务实体恢复 `DRAFT` |
| 重新提交（Resubmit） | 重新激活被驳回节点 | 轮次号 +1，创建新一轮任务 |
| 转办（Transfer） | 原任务 → TRANSFERRED | 为被转办人创建新任务 |
| 加签（Add-sign） | 为当前节点新增待办人 | 顺序模板下按会签执行（所有加签人均需同意） |

### 4.6 结算特定：不可变锁定

**来源**: `phase-closure-execution/learnings.md:332-345`, `phase3-cost-analysis-contract-deepening/learnings.md:256-281`

- 结算是最末端环节，一旦审批通过（`FINALIZED`），所有字段不可修改。
- 结算 **禁止调用 `CostGenerationService`**（循环依赖：结算汇总→成本→结算循环）。
- `autoFillAmounts()` 从 VarOrder (COST 方向, 已确认)、SubMeasure (APPROVED)、PayRecord (SUCCESS) 汇总金额，**不允许手动覆盖**。
- `beforeSubmit()` 校验无未审批的 VarOrder 和 SubMeasure。

### 5.4 结算与成本的边界

**来源**: `phase3-cost-analysis-contract-deepening/learnings.md:256-260`

| | 成本（cost_ledger） | 结算（settlement） |
|---|---|---|
| 何时生成 | 审批通过 | 主动创建并审批 |
| 来源类型 | CT_CONTRACT, MAT_RECEIPT, SUB_MEASURE, VARIATION, CT_CHANGE | 无（结算不生成成本） |
| 可修改 | 否（审计日志） | 仅 DRAFT 状态 |

### 5.5 costGeneratedFlag 标记

**来源**: `phase3-cost-analysis-contract-deepening/learnings.md:81`

所有可能生成成本的实体（MatReceipt, VarOrder, SubMeasure, CtContractChange）都有 `cost_generated_flag TINYINT NOT NULL DEFAULT 0` 字段。它在以下场景中用于判断：

- **编辑/删除守卫**：成本已生成（`flag=1`）时禁止编辑/删除
- **成本幂等性**：重复审批不会重复生成成本

### 7.3 Java Lambda 陷阱

**来源**: `code-review-fixes/learnings.md:81`

```java
// ❌ 错误：lambda 中使用的变量不能被重新赋值
Entity entity = service.getById(id);
entity.setName("changed");
assertThrows(BusinessException.class, () -> service.update(entity));  // entity 是 effectively final 但被修改过

// ✅ 正确：使用快照变量
final Entity snapshot = service.getById(id);
assertThrows(BusinessException.class, () -> service.update(snapshot));
```

```typescript
import { test, expect } from '@playwright/test';

// 登录 helper（直接从 contract.spec.ts 复制）
async function loginAsAdmin(page: Page) {

test.describe('模块名', () => {
  test.beforeEach(async ({ page }) => {

  });

  test('场景描述', async ({ page }) => {
    await page.goto('/module-path');
    await page.waitForSelector('.pm-page');

    // 检查数据可用性，无数据则优雅跳过
    const hasData = await page.locator('.ant-table-row').first()

    }
    // ... 后续操作
  });
});
```

#### 7.4.4 代码规范

- **禁止** `waitForTimeout` > 300ms（仅在 tab 切换后允许 300ms 等待渲染周期）。
- **禁止** 硬编码 `setTimeout`；使用 `waitForSelector`、`waitForURL`。
- 所有 spec 必须包含数据缺失时的优雅跳过逻辑（`console.log` + `return`）。
- `loginAsAdmin()` 从 `contract.spec.ts` 精确复制，保持函数签名一致。
- 截图路径：`e2e/screenshots/{module}-{action}.png`

#### 7.4.5 通知模块特殊性

**来源**: `phase-closure-execution/learnings.md:533-557`

通知模块**没有独立页面**——它以 `NotificationBell.vue` 组件形式嵌入在布局头部。没有 `/notification` 路由。E2E 测试通过 Popover 交互：

```typescript
// 通知组件选择器（overlay-class-name=nb-popover）
await page.click('.nb-trigger');                              // 点击铃铛图标

await page.click('.nb-header button:has-text("全部标为已读")'); // 全部已读
```

#### 7.4.6 驾驶舱特殊性

**来源**: `phase-closure-execution/learnings.md:588-620`

驾驶舱有 5 个角色视图（pm/bm/cost/finance/mgmt），**仅 cost（成本经理）**视图包含 ECharts canvas。Canvas 测试前必须先切换到对应 tab：
