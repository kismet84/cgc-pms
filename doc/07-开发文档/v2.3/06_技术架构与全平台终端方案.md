## 正式版基线说明

本文件已按正式开发交付口径整理，并统一以下架构基线：

1. 一期数据库默认采用 **MySQL 8.0**，审批规则、动态表单和节点规则使用 MySQL JSON 字段。
2. 后端审批引擎内部采用**事务内领域服务直接调用**，不通过 HTTP 调用自身接口。
3. PC Web 是一期完整管理后台；移动端以 uni-app 为现场主端，企业微信和微信小程序作为轻入口。
4. 移动端正式业务开发节奏后移：第 6 周启动基础工程，第 8 周左右进入真实业务功能。
5. 所有业务数据以 `project_id` 为根，涉及合同的业务以 `contract_id` 为纲，成本以业务事实归集，付款与成本口径分离。

## 技术架构正式版补充

### 1. 后端技术栈

| 层级 | 技术 | 说明 |
|---|---|---|
| 后端语言 | Java 21 | 企业级生态成熟，长期维护稳定 |
| 后端框架 | Spring Boot | 快速开发、生态完善 |
| ORM | MyBatis-Plus / MyBatis | 复杂业务 SQL 可控 |
| 权限认证 | Sa-Token / Spring Security | JWT、RBAC、数据权限 |
| 数据库 | MySQL 8.0 | 一期默认数据库 |
| 缓存 | Redis | 会话、权限、字典、待办计数、幂等键 |
| 文件存储 | MinIO / OSS | 合同、照片、发票、结算资料 |
| 消息队列 | RabbitMQ | 审批完成通知、成本生成、付款推送、消息提醒 |
| 定时任务 | XXL-JOB / Quartz | 预警扫描、统计汇总、超时任务 |
| API 文档 | OpenAPI / Swagger | 前后端联调契约 |
| 部署 | Docker + Nginx + CI/CD | 测试、预生产、生产分环境部署 |

### 2. 架构形态

一期采用“模块化单体 + 清晰领域边界 + 可演进微服务”。不建议一期过早拆分大量微服务，以免增加事务、联调、部署和运维复杂度。

### 3. 后端分层

```text
Controller
  ↓
Application Service
  ↓
Domain Service
  ↓
Repository / Mapper
  ↓
MySQL / Redis / MinIO / MQ
```

### 4. 审批与业务集成

审批引擎作为后端领域能力被业务模块调用。同一后端应用内部不得通过 HTTP 调用自身接口。跨系统推送、企业微信消息、财务系统同步等使用消息队列或 Outbox。

## 执行摘要

结合现有方案已确定的统一 `project_id / contract_id`、统一审批引擎、统一附件体系、细粒度权限，以及当前 PC 端以合同台账、复杂筛选、宽表格为主的管理界面，本项目最优解不是“一套前端包打天下”，而是“**PC 浏览器主后台 + uni-app 原生 App 主现场 + 企业微信侧内部轻入口 + 微信小程序外部协作 + PWA 兜底 + 桌面壳延期**”。这更契合建筑工程总包场景中的弱网、拍照/大附件、现场扫码、多人审批、私有化部署与长期维护要求。

## 方案对比

下表按各官方文档能力边界与本项目场景归纳；“周数”为基于**已有统一后端**前提下的相对增量估算。

| 方案 | 支持终端 | 主要优点 | 主要短板 | 适用场景 | 周数 | 优先级 |
|---|---|---|---|---|---:|---|
| PC 浏览器 | PC | 0 安装、最适合复杂台账/报表 | 离线弱、硬件能力弱 | 管理后台主端 | 4-6 | P0 |
| PWA | Web/PWA | 可安装、可离线缓存、可后台同步入口 | 浏览器差异大，持续后台上传/相机链路不稳 | 轻审批、访客、兜底 | 2-4 | P3 |
| 微信小程序 | 微信内 | 触达外部快、免安装、扫码方便 | 沙箱重、长传/大附件受限 | 供应商/分包商轻协作 | 4-8 | P2 |
| 企业微信侧 | 企微内 | 内部身份/消息/审批触达好 | 仍受容器约束 | 内部待办、消息提醒 | 3-6 | P1 |
| uni-app | Android/iOS/H5/多小程序 | Vue/TS 一套多端；API 兼容小程序；支持扫码、定位、拍照、压缩、上传、后台定位等 | 插件质量参差，重交互一致性一般 | 现场 App 首选 | 8-12 | P0 |
| Flutter | Android/iOS/Web/Desktop | 单代码库，渲染一致，性能强，包生态大 | Dart 学习成本高，接微信生态绕 | 重体验独立 App | 12-16 | P2 |
| React Native | Android/iOS/Web | React 团队友好，Expo 生态成熟 | 原生依赖与版本治理成本高 | 既有 React 团队 | 10-14 | P3 |
| Tauri 桌面壳 | Win/macOS/Linux | 官方强调小、快、安全；用系统 WebView；能力按权限/能力集收敛 | 需 Rust，桌面运维需补课 | 私有化内部桌面壳 | +3-5 | P2 |
| Electron 桌面壳 | Win/macOS/Linux | 生态最成熟，桌面能力/更新体系完整 | 包体重，安全面更大，需严格硬化 | 深度桌面集成 | +4-6 | P3 |
| Android/iOS 原生 | 手机 | 性能/后台/相机/文件最强 | 双端成本最高 | 极重现场/强离线 | 16-24 | P4 |

## 关键判断

**Tauri vs Electron**：Electron 官方明确其把 Chromium 和 Node.js 一起带到桌面，JS 可直接触达文件系统与 shell；自动更新成熟，但 macOS 需签名，Windows 仍要处理 Squirrel/MSIX 细节。Tauri 官方强调跨 Linux/macOS/Windows/Android/iOS，使用系统 WebView、能力通过 permissions/capabilities 收敛，并宣称包体可低至 600KB；其更新签名不可关闭。结论：**若未来必须做桌面壳，优先 Tauri；只有在强依赖 Node 生态、桌面插件、团队完全不会 Rust 时才选 Electron。**

**Flutter vs uni-app**：Flutter 官方定位是“单代码库覆盖 mobile/web/desktop”，并依靠硬件加速渲染；官方包管理器 pub.dev 提供大量插件。uni-app 官方则直接支持 iOS、Android、Web 与多种小程序，语法基于 Vue，API 命名兼容小程序，并支持平台特化与原生 SDK 集成。对当前 **Vue + Java 后端** 团队，且需要同时覆盖 App/H5/小程序的国内工程项目，**uni-app 的综合开发成本更低**；若未来目标是“独立品牌 App + 更高 UI 一致性 + 海外分发”，再考虑 Flutter。

**PWA 能否替代原生 App**：PWA 确实可安装、可离线、可在后台做同步，且有 OPFS/File System 等能力；但这些能力受浏览器实现、HTTPS/安全上下文和用户授权约束。对本项目所需的**弱网、长时间后台上传、大图片压缩、持续定位、现场扫码、多角色权限**，PWA 适合作为“轻入口/兜底”，**不宜替代主现场 App**。

**是否需要桌面壳**：当前 PC 页面以宽表格、筛选、审批、台账为主，本质是浏览器后台，而不是本地工具软件；一期没有必要为“壳”承担安装、签名、更新和客户端运维成本。只有当出现**托盘常驻、离线批量上传、USB/打印机/本地文件联动、Windows 单点登录、局域网增强**等明确需求时，再补桌面壳。

**微信小程序与企业微信侧角色分配**：建议把**企业微信侧**定义为“内部员工入口”，承接待办审批、消息提醒、组织身份；把**微信小程序**定义为“外部协作入口”，承接供应商/分包商的扫码、上传、回执。这样能把内部权限边界与外部协作边界拆开，也能降低小程序安全暴露面；尤其要坚持 **AppSecret 只在后端**、前端不落密钥。

## 决策建议

**首选方案**：
PC：纯浏览器后台；
移动：**uni-app 原生 App（Android/iOS）** 作为现场主端；
内部轻端：**企业微信侧**承接审批与消息；
外部轻端：**微信小程序**承接外协扫码/上传；
PWA：只做只读/轻填报/兜底；
桌面壳：一期不上，二期如需再上 **Tauri**。

**分阶段路线**：
阶段一：PC 浏览器 + 企业微信侧待办；
阶段二：uni-app App 上线材料验收、签证、整改、扫码、拍照上传；
阶段三：微信小程序开放给外部合作方；
阶段四：若确有本地增强需求，再补 Tauri 桌面壳。

## 实施清单

任务上，先做 **终端能力矩阵 + POC**：扫码、相机压缩、分片上传、断点续传、离线草稿、弱网重试、定位、消息提醒、版本升级。
接口上，后端至少补齐：`/sync/pull?cursor=`、`/sync/push`、`/upload/init|chunk|complete`、`/device/register`、`/client/version`、`/scan/resolve`、`/location/report`、`/notify/bind`、`/notify/ack`。
同步上，统一采用：**轻离线/弱离线**，本地 SQLite/IndexedDB + **Outbox/Inbox**；服务端按项目与业务表做 **Cursor Sync**；冲突规则以“服务端版本号 + 修改时间 + 审批状态优先”解决。
运维上，App 走商店/企业分发，PC 浏览器走 Nginx/CDN，桌面壳如后续启用再接自动更新；所有端都要做客户端版本兼容和功能开关。
测试用例至少覆盖：弱网断点续传、离线新建后回网同步、同单据双端冲突、扫码失败重试、定位权限拒绝、图片压缩后清晰度、审批重复提交幂等、旧版本客户端兼容。

## 风险与缓解

最大风险不是“技术不能做”，而是**端过多导致能力不一致**。缓解方式是：统一 BFF/API、统一权限与审批状态机、统一文件服务、统一版本策略。
第二个风险是 **uni-app/小程序插件碎片化**；应在立项首月就完成扫码、相机、后台定位、上传 4 个 POC。
第三个风险是 **小程序安全**；必须把密钥、签名、敏感规则放后端，校验域名白名单、文件签名、设备指纹与令牌刷新。
第四个风险是 **未来迁移成本**；前端必须坚持“能力适配层”设计，使 uni-app → Flutter/Tauri 的迁移只替换客户端壳，不重写业务 API。
