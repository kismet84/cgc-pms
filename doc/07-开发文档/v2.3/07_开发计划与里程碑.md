## 正式版基线说明

本文件已按正式开发交付口径整理，并统一以下架构基线：

1. 一期数据库默认采用 **MySQL 8.0**，审批规则、动态表单和节点规则使用 MySQL JSON 字段。
2. 后端审批引擎内部采用**事务内领域服务直接调用**，不通过 HTTP 调用自身接口。
3. PC Web 是一期完整管理后台；移动端以 uni-app 为现场主端，企业微信和微信小程序作为轻入口。
4. 移动端正式业务开发节奏后移：第 6 周启动基础工程，第 8 周左右进入真实业务功能。
5. 所有业务数据以 `project_id` 为根，涉及合同的业务以 `contract_id` 为纲，成本以业务事实归集，付款与成本口径分离。

## 1. 项目概述

### 1.1 项目名称

建筑工程总包项目全过程管理系统

### 1.2 项目定位

本系统面向建筑工程总包单位、项目部、公司职能部门及管理层，建设目标不是单一台账系统，而是覆盖项目、合同、材料设备、分包、成本、付款、审批、资料、结算、经营分析的全过程业务闭环系统。

系统建设主线为：

```text
项目为根
合同为纲
执行为据
成本为果
付款为流
结算为终
```

### 1.3 建设目标

本项目开发目标如下：

1. 打通项目、合同、采购、分包、成本、付款、结算之间的数据链路。
2. 建立统一合同中心，所有采购、分包、付款、结算均通过 `contract_id` 引用合同。
3. 建立统一项目主线，所有核心业务数据必须关联 `project_id`。
4. 建立统一审批引擎，支持顺序审批、会签、或签、转办、加签、撤回、驳回重提。
5. 建立统一成本归集中心，材料验收、分包计量、签证变更等业务自动生成成本。
6. 建立付款申请与实际付款分离机制，付款申请走业务审批，实际付款由财务确认或接口回写。
7. 建立结算收口能力，结算数据从合同、变更、验收、计量、付款、成本自动汇总。
8. 建立角色化驾驶舱，为项目经理、商务经理、成本经理、财务、管理层提供不同视图。
9. 建立前后端统一 JSON 契约，降低前端流程判断复杂度。
10. 支持后续扩展移动审批、财务接口、BI 大屏、OCR 发票识别等增强能力。

## 3. 总体技术架构

## 3.1 架构原则

系统建议采用“模块化单体 + 清晰领域边界 + 可演进微服务”的架构方式。

初期不建议过早拆分为大量微服务，避免增加部署、事务、联调和运维复杂度。应先保证业务主线跑通，后续根据规模逐步拆分。

### 3.1.1 架构分层

```text
前端展示层
  ↓
API 网关 / 统一认证
  ↓
业务应用层
  ↓
领域服务层
  ↓
数据访问层
  ↓
数据库 / 文件存储 / 缓存 / 消息队列
```

### 3.1.2 核心原则

1. 前端不直接判断复杂审批逻辑。
2. 后端返回当前用户可操作动作。
3. 业务模块只关心提交审批和审批结果。
4. 审批引擎统一处理流程流转。
5. 成本由业务单据自动生成。
6. 付款申请和实际付款分离。
7. 结算数据必须可反查来源。
8. 所有关键数据必须有操作日志和审批记录。

## 4. 核心开发原则

## 4.1 数据主线原则

所有核心业务表必须包含：

```text
tenant_id
project_id
status
approval_status

created_at
updated_by

```

涉及合同的业务表必须包含：

```text
contract_id

```

涉及成本来源的表必须通过以下字段追溯：

```text
source_type
source_id
```

---

## 4.2 审批引擎原则

审批相关逻辑必须统一放入审批引擎，不允许每个业务模块单独写一套审批逻辑。

审批引擎必须支持：

```text
顺序审批 SEQUENTIAL
会签 COUNTERSIGN
或签 OR_SIGN
转办 TRANSFER
前加签 ADD_SIGN_BEFORE
后加签 ADD_SIGN_AFTER
并行加签 ADD_SIGN_PARALLEL
撤回 WITHDRAW
驳回 REJECT
重新提交 RESUBMIT
审批轮次 roundNo
审批节点实例 nodeInstanceId
审批任务版本 taskVersion
幂等键 idempotencyKey
```

---

## 4.3 前后端契约原则

前端只负责：

```text
展示数据
渲染按钮
渲染动态表单
提交用户动作
展示审批时间轴
```

后端负责：

```text
权限判断
流程判断
审批人计算
业务规则校验
状态流转
成本生成
付款校验
结算锁定
审批记录留痕
```

前端不得自行判断“是否能审批、是否能转办、是否能加签”，必须以后端返回的 `availableActions` 为准。

# 20. 风险与应对措施

| 风险 | 表现 | 应对措施 |
|---|---|---|
| 需求范围膨胀 | 不断增加功能 | 严格分期，P0 优先 |
| 审批规则复杂 | 流程难以落地 | 审批引擎统一建模 |
| 用户录入压力大 | 现场人员抵触 | 分步表单、默认值、自动带出 |
| 数据口径不一致 | 报表数据被质疑 | 统一口径并可反查来源 |
| 成本自动生成重复 | 成本金额重复 | sourceType + sourceId 唯一约束 |
| 付款与成本混淆 | 付款被当作成本 | 明确付款是资金流，验收计量才是成本 |
| 财务接口不稳定 | 付款状态不同步 | 先手工回写，再接口同步 |
| 审批并发冲突 | 多人同时审批 | taskVersion + 事务锁 + 幂等键 |
| 权限越权 | 非项目人员看到数据 | 项目数据权限强校验 |
| 结算数据不准 | 结算无法审计 | 结算明细必须有来源单据 |
